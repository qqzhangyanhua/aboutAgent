前面咱们搭的智能体，都是"单打独斗"——一个 AI 又查资料、又推理、又写代码，就像一个人同时当项目经理、开发、测试。能干是能干，但效率高不到哪去。

现实中复杂项目怎么搞？团队协作呗。AI 也一样，让不同的 AI 扮演不同角色，术业有专攻，效果往往比一个全能型选手强。

那问题来了：怎么让多个智能体互相配合，比一个智能体做得更好？这篇文章就带你用 Python 实现三种经典模式：主从、辩论、流水线。代码完整可跑，`pip install openai` 就够了。

---

## 一、为什么需要多智能体？

单智能体有个天花板：上下文窗口有限，角色切换容易乱，复杂任务一长就容易跑偏。你让一个 AI 既当调研员又当写手又当审稿人，它的 system prompt 得写多长？一长就容易"精神分裂"——前面说自己是调研员，后面又跑去写结论，角色边界一模糊，输出质量就掉。

多智能体的好处很直接：

- **专业化**：每个 Agent 只干一件事，prompt 更聚焦，效果更稳。就像现实中你不会让同一个人既写需求又写代码又做测试。
- **互相校验**：一个写代码一个 Review，比自写自审靠谱。人都会自我感觉良好，AI 也一样，多一个"挑刺"的视角能少很多低级错误。
- **并行处理**：不相关的子任务可以同时跑，省时间。主从模式里，如果 researcher 和另一个"数据分析"员工任务独立，可以并行调两个 API，总耗时取最大值而不是相加。

下面三种模式，就是把这几个优势用不同方式发挥出来。

---

## 二、模式一：主从模式（Orchestrator-Worker）

主从模式就是：一个"老板"Agent 负责拆任务、分配活，多个"员工"Agent 各自执行。老板统筹全局，员工之间不用互相通信。

先上 Agent 基类，后面三种模式都共用：

```python
from openai import OpenAI
import os

class Agent:
    """统一的 Agent 基类：接受 system_prompt，提供 run(message) 方法"""
    def __init__(self, system_prompt: str, model: str = "gpt-4o-mini"):
        self.client = OpenAI(api_key=os.environ.get("OPENAI_API_KEY"))
        self.system_prompt = system_prompt
        self.model = model

    def run(self, message: str) -> str:
        resp = self.client.chat.completions.create(
            model=self.model,
            messages=[
                {"role": "system", "content": self.system_prompt},
                {"role": "user", "content": message}
            ]
        )
        return resp.choices[0].message.content.strip()
```

主从模式的核心：老板先拆任务，再按计划调员工。

```python
import json

class OrchestratorAgent:
    def __init__(self):
        self.workers = {
            "researcher": Agent("你是一个信息调研专家，负责搜索和收集资料。输出简洁、有条理。"),
            "writer": Agent("你是一个技术写手，负责把资料整理成结构清晰的文章。"),
            "reviewer": Agent("你是一个严格的审稿人，负责检查文章质量并给出修改意见。"),
        }
        self.orchestrator = Agent(
            "你是项目经理。把任务拆解成子任务，指定执行者。"
            "可用执行者：researcher（调研）、writer（写作）、reviewer（审阅）。"
            "只输出 JSON 数组，格式：[{\"worker\":\"xxx\",\"subtask\":\"xxx\"}]"
        )

    def run(self, task: str):
        plan_str = self.orchestrator.run(f"任务：{task}")
        if "```" in plan_str:
            plan_str = plan_str.split("```")[1].replace("json", "").strip()
        plan = json.loads(plan_str)
        results = {}
        for step in plan:
            worker_name = step["worker"]
            subtask = step["subtask"]
            context = subtask
            if results:
                context += "\n\n已有结果：\n" + "\n".join(f"- {k}: {v[:200]}..." if len(str(v)) > 200 else f"- {k}: {v}" for k, v in results.items())
            results[worker_name] = self.workers[worker_name].run(context)
        return results.get("reviewer", list(results.values())[-1])
```

## 三、模式二：辩论模式（Debate）

辩论模式：多个 Agent 对同一问题从不同角度给答案，互相质疑几轮，最后由裁判 Agent 综合。

适合"这个技术方案行不行""要不要上微服务"这类需要多角度分析的决策。

```python
class DebateSystem:
    def __init__(self):
        self.agents = [
            Agent("你是乐观派技术顾问，倾向于看到方案的优势和可能性。回答简洁，100 字以内。"),
            Agent("你是悲观派技术顾问，专注于发现风险和潜在问题。回答简洁，100 字以内。"),
        ]
        self.judge = Agent(
            "你是技术评审裁判。阅读双方辩论，给出客观、平衡的最终结论和建议。200 字以内。"
        )

    def debate(self, question: str, rounds: int = 2):
        history = []
        for r in range(rounds):
            for i, agent in enumerate(self.agents):
                label = "正方" if i == 0 else "反方"
                context = f"议题：{question}\n\n之前的讨论：\n" + "\n".join(history[-4:]) if history else f"议题：{question}\n\n"
                context += f"\n请发表你的第 {r+1} 轮观点："
                resp = agent.run(context)
                history.append(f"【{label}第{r+1}轮】{resp}")
        verdict = self.judge.run(
            f"议题：{question}\n\n完整辩论：\n" + "\n\n".join(history) + "\n\n请给出最终评判："
        )
        return verdict, history
```

**模拟运行效果：**

```
议题：新项目是否应该使用微服务架构？

【正方第1轮】微服务可以让团队独立部署、技术栈灵活，便于快速迭代。
【反方第1轮】团队只有 5 人，微服务会带来巨大运维复杂度和分布式事务问题。
【正方第2轮】可以先从 2-3 个核心服务拆起，用成熟框架如 Spring Cloud 降低复杂度。
【反方第2轮】即使 3 个服务，服务间通信、数据一致性、链路追踪已经够头疼了。

裁判：综合双方观点，建议初期采用模块化单体架构，预留拆分接口，
     等团队规模和业务复杂度增长后再逐步拆分微服务。
```

## 四、模式三：流水线模式（Pipeline）

流水线就是串行处理：每个 Agent 的输出是下一个的输入，像工厂流水线一样。原材料进 drafter，初稿进 polisher，润色稿进 fact_checker，最后 formatter 打包输出。顺序不能乱，因为后一步依赖前一步的成果。

```python
class Pipeline:
    def __init__(self):
        self.stages = [
            {"name": "drafter", "agent": Agent("你是初稿写手。根据主题写技术博客初稿，500 字左右，重点内容完整。")},
            {"name": "polisher", "agent": Agent("你是润色专家。在不改内容的前提下让文章更生动、口语化，去掉 AI 味。")},
            {"name": "fact_checker", "agent": Agent("你是事实核查员。检查技术描述是否准确，标出可疑处，其余保持原样。")},
            {"name": "formatter", "agent": Agent("你是排版专家。格式化为公众号 Markdown，加小标题和分段。")},
        ]

    def run(self, initial_input: str):
        current = initial_input
        for stage in self.stages:
            print(f"📎 阶段：{stage['name']}")
            current = stage["agent"].run(f"请处理以下内容：\n\n{current}")
            print(f"   ✅ 完成，输出 {len(current)} 字")
        return current
```

## 五、三种模式对比

| 维度 | 主从模式 | 辩论模式 | 流水线模式 |
|------|----------|----------|------------|
| 通信方式 | 星型（老板→员工） | 全连接（互相对话） | 线性（前→后） |
| 适合任务 | 可拆解的复杂任务 | 需要多角度的决策/分析 | 有明确步骤的顺序任务 |
| Agent 数量 | 1 + N | 2–3 + 裁判 | N 个串行 |
| 并行能力 | 高（员工可并行） | 低（轮流发言） | 无（严格顺序） |
| 成本 | 中等 | 高（多轮对话） | 中等 |
| 典型应用 | 调研报告、项目执行 | 技术方案评审、风险评估 | 内容生产、数据处理 |

## 六、实战踩坑与优化建议

**控制传递信息长度**：Agent 之间传的内容别太长，否则下游上下文爆炸，又贵又容易丢重点。可以约定"只传摘要"或"只传关键结论"。比如主从模式里，writer 不需要 researcher 的原始 5000 字，给个 300 字摘要就够了。

**Prompt 要聚焦**：每个 Agent 的 system prompt 越专一越好，别让它"啥都干"，否则容易角色混乱。一个 Agent 最好只承担一个清晰职责，比如"只负责润色"比"负责润色和排版"靠谱。

**调试技巧**：给每个 Agent 的输出加前缀，比如 `[researcher]`、`[正方]`，方便看是谁说的，排查问题快很多。辩论模式里尤其有用，不然几轮下来根本分不清哪句是谁说的。

**成本控制**：不是所有任务都需要多 Agent。简单问题单 Agent 够用就别上多智能体，别为了花哨而花哨。多一轮调用就多一份钱，辩论模式跑 3 轮就是 6 次 LLM 调用，成本要心里有数。

## 七、写在最后

多智能体解决了"分工"的问题，但还有一个底层问题没聊——向量数据库该怎么选？前面 RAG 那几篇一直用 ChromaDB 凑合，小项目够用，真要上生产还得看看 Pinecone、Milvus、Qdrant 这些怎么选。下一篇咱们正经对比一下主流向量数据库的适用场景和坑。

核心观点就一句：多智能体不是银弹，本质是"用 token 换质量"。你的任务是否真的复杂到需要多个 AI 协作，这个判断本身就很重要。简单问答、单步翻译，一个 Agent 绰绰有余；复杂调研、多轮决策、长文生产，再考虑上多智能体。
