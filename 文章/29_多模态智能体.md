前面 28 篇文章，我们的智能体一直在"聊天"——输入是文字，输出也是文字。你丢给它一张图，它看不懂；你让它分析一张报表截图，它只能说"请把数据复制成文字给我"。

但你想想实际工作场景：产品经理扔给你一张 Figma 截图说"这个按钮对齐有问题"；运维同事发来一张监控面板说"这个曲线不正常"；客户拍了一张商品照片说"这个跟我买的不一样"——这些场景里，**图片才是核心信息**，纯文字智能体根本接不住。

2025 年之后，主流大模型（GPT-4o、Claude、Gemini、Qwen-VL、DeepSeek-VL）全都支持视觉输入了。给智能体"装眼睛"不再是论文里的概念，而是 `pip install openai` 就能干的事。

这篇文章带你搭三个版本：

- **V1** — 能看图的对话助手：丢一张图过去，它能描述内容、回答问题
- **V2** — 能看图 + 能干活：看到图表后自动提取数据、调用工具做分析
- **V3** — 多模态 RAG：把图片也当"文档"索引，图文混合检索

依赖很少：`pip install openai chromadb pillow`，代码直接能跑。

## 一、多模态智能体到底多了什么？

先跟纯文字智能体做个对比：

| | 纯文字智能体 | 多模态智能体 |
|---|---|---|
| **输入** | 只接受文本 | 文本 + 图片（+ 音频、视频） |
| **理解能力** | 理解自然语言 | 理解自然语言 + 视觉内容 |
| **工具调用** | 基于文字描述做决策 | 基于文字 + 视觉信息做决策 |
| **典型场景** | 问答、写作、代码 | 图表分析、图片审核、视觉RAG |

多模态不是一个新模块，而是在现有的"大脑"模块上**升级了感官**。之前大脑只有"耳朵"（听文字），现在多了"眼睛"（看图片）。其他模块——工具、记忆、规划——不用大改，只是输入变丰富了。

技术上怎么实现？其实就是在 messages 里塞图片：

```python
# 纯文字消息
{"role": "user", "content": "今天天气怎么样？"}

# 多模态消息：文字 + 图片
{"role": "user", "content": [
    {"type": "text", "text": "这张图里有什么？"},
    {"type": "image_url", "image_url": {"url": "data:image/png;base64,..."}}
]}
```

就这么简单。`content` 从字符串变成了列表，列表里可以混排文本和图片。大模型会"看到"图片内容，然后结合文字做回答。

## 二、准备工作

```python
import os
import json
import base64
from openai import OpenAI

api_key = os.getenv("DEEPSEEK_API_KEY")
if not api_key:
    raise ValueError("请设置环境变量 DEEPSEEK_API_KEY")

# DeepSeek VL 或其他支持视觉的模型
# 如果你的模型不支持视觉，可以换成：
#   base_url="https://api.openai.com/v1"  +  MODEL_NAME="gpt-4o"
#   base_url="https://dashscope.aliyuncs.com/compatible-mode/v1"  +  MODEL_NAME="qwen-vl-max"
client = OpenAI(base_url="https://api.deepseek.com", api_key=api_key)
MODEL_NAME = "deepseek-chat"  # 替换为你使用的视觉模型


def encode_image_to_base64(image_path: str) -> str:
    """把本地图片转成 base64 字符串"""
    with open(image_path, "rb") as f:
        return base64.b64encode(f.read()).decode("utf-8")


def make_image_message(text: str, image_source: str) -> dict:
    """
    构造一条带图片的用户消息。
    image_source 可以是本地路径，也可以是 URL。
    """
    content = [{"type": "text", "text": text}]

    if image_source.startswith(("http://", "https://")):
        content.append({
            "type": "image_url",
            "image_url": {"url": image_source}
        })
    else:
        b64 = encode_image_to_base64(image_source)
        ext = image_source.rsplit(".", 1)[-1].lower()
        mime = {"png": "image/png", "jpg": "image/jpeg",
                "jpeg": "image/jpeg", "gif": "image/gif",
                "webp": "image/webp"}.get(ext, "image/png")
        content.append({
            "type": "image_url",
            "image_url": {"url": f"data:{mime};base64,{b64}"}
        })

    return {"role": "user", "content": content}
```

`make_image_message` 这个函数把"构造图片消息"这件事封装好了——传个路径或 URL 进去，自动处理 base64 编码。后面三个版本都会用到它。

## 三、V1 —— 能看图的对话助手

最简单的多模态智能体：你丢一张图 + 一个问题，它看图回答。

```python
def multimodal_chat(system_prompt: str, image_source: str, question: str) -> str:
    """最基础的多模态对话：一张图 + 一个问题 → 一个回答"""
    messages = [
        {"role": "system", "content": system_prompt},
        make_image_message(question, image_source)
    ]

    response = client.chat.completions.create(
        model=MODEL_NAME,
        messages=messages,
        max_tokens=1024
    )
    return response.choices[0].message.content
```

就这么几行。跟之前纯文字的对话几乎一样，唯一的区别是用户消息里多了一张图。

来测试几个典型场景：

```python
# ---- 场景 1：描述图片内容 ----
answer = multimodal_chat(
    system_prompt="你是一个视觉助手，用中文简洁地描述图片内容。",
    image_source="https://upload.wikimedia.org/wikipedia/commons/thumb/3/3a/Cat03.jpg/1200px-Cat03.jpg",
    question="这张图里有什么？"
)
print("场景1:", answer)
# → "图片中是一只橘色的猫，正趴在地上，眼睛直视镜头……"

# ---- 场景 2：读取图片中的文字（OCR） ----
answer = multimodal_chat(
    system_prompt="你是一个 OCR 助手。提取图片中的所有文字，保持原始格式。",
    image_source="receipt.png",  # 本地的一张收据照片
    question="请提取这张收据上的所有信息。"
)
print("场景2:", answer)
# → "商家：星巴克（三里屯店）\n日期：2025-03-10\n拿铁（大杯）×1  ¥38\n……"

# ---- 场景 3：分析图表 ----
answer = multimodal_chat(
    system_prompt="你是一个数据分析师。分析图表内容，指出关键趋势和异常。",
    image_source="sales_chart.png",  # 一张折线图
    question="这张销售趋势图有什么值得关注的地方？"
)
print("场景3:", answer)
# → "从图表看，Q3 销量有明显下滑，从 850 降到 620，降幅约 27%……"
```

V1 能干的事已经不少了——图片描述、OCR 提取、图表解读。但它有个明显的局限：**看完就完了，不会采取行动**。它能告诉你"Q3 销量下滑了"，但不会自动去查数据库找原因、不会生成报告、不会发告警邮件。

这就是 V2 要解决的问题。

## 四、V2 —— 能看图 + 能干活：视觉驱动的 Tool Use

V1 只是"看了说说"，V2 要让智能体**看到图片后自己决定调什么工具**。

比如：你扔给它一张数据库监控截图，它看到 CPU 飙到 95% 了，自动调工具去查进程列表、然后帮你定位问题。这才是真正的多模态"智能体"——不只是看，还要做。

### 4.2 模拟工具执行

```python
def execute_tool(name: str, arguments: dict) -> str:
    """模拟执行工具（生产环境替换为真实实现）"""
    if name == "extract_chart_data":
        return json.dumps({
            "status": "success",
            "chart_type": arguments["chart_type"],
            "data_points": arguments.get("data_points", []),
            "message": f"成功提取 {len(arguments.get('data_points', []))} 个数据点"
        }, ensure_ascii=False)

    elif name == "generate_analysis_report":
        report = f"""
📊 {arguments['title']}

【数据摘要】{arguments['data_summary']}

【整体趋势】{arguments['trend']}

【异常发现】
{chr(10).join('• ' + a for a in arguments.get('anomalies', ['无']))}

【建议措施】
{chr(10).join('• ' + r for r in arguments.get('recommendations', ['暂无']))}
"""
        return json.dumps({"status": "success", "report": report.strip()},
                          ensure_ascii=False)

    elif name == "send_alert":
        return json.dumps({
            "status": "sent",
            "severity": arguments["severity"],
            "recipients": arguments.get("recipients", ["default_admin"]),
            "message": f"告警已发送：{arguments['message']}"
        }, ensure_ascii=False)

    return json.dumps({"error": f"未知工具: {name}"})
```

### 4.3 多模态智能体循环

核心来了——把图片理解和工具调用串成一个完整的 Agent Loop：

```python
def multimodal_agent(image_source: str, question: str, max_rounds: int = 5) -> str:
    """
    多模态智能体：看图 → 理解 → 调用工具 → 给出最终回答。
    """
    system_prompt = """你是一个多模态数据分析智能体。你能够：
1. 观察和理解用户提供的图片（图表、截图、照片等）
2. 从图片中提取关键信息和数据
3. 调用工具进行深度分析、生成报告或发送告警
4. 基于视觉信息和分析结果给出最终建议

工作流程：
- 先仔细观察图片，理解其内容
- 如果是图表，先用 extract_chart_data 提取数据
- 根据数据情况决定是否需要生成报告或发送告警
- 最后给用户一个清晰的总结"""

    messages = [
        {"role": "system", "content": system_prompt},
        make_image_message(question, image_source)
    ]

    for round_num in range(max_rounds):
        response = client.chat.completions.create(
            model=MODEL_NAME,
            messages=messages,
            tools=tools,
            max_tokens=2048
        )
        msg = response.choices[0].message
        messages.append(msg)

        if not msg.tool_calls:
            return msg.content

        for tool_call in msg.tool_calls:
            func_name = tool_call.function.name
            func_args = json.loads(tool_call.function.arguments)
            print(f"  [轮次{round_num + 1}] 调用工具: {func_name}")

            result = execute_tool(func_name, func_args)
            messages.append({
                "role": "tool",
                "tool_call_id": tool_call.id,
                "content": result
            })

    return "达到最大轮次限制，请简化问题后重试。"
```

跟第 1 篇的智能体循环一模一样的结构——`while` 循环 + 判断 `tool_calls`。唯一的区别是第一条消息带了图片。这就是多模态智能体的本质：**输入多了一个维度，核心架构不变**。

## 五、V3 —— 多模态 RAG：图文混合检索

前两个版本处理的都是"用户主动发一张图"的场景。但很多时候，**图片是知识库的一部分**——产品说明书里有实物图、技术文档里有架构图、合同里有公章照片。你需要把这些图片也索引进 RAG，让智能体在检索时能同时找到文字和图片。

### 5.1 思路：用大模型给图片生成文字描述，然后索引描述

最直接的方案不需要什么花哨的多模态 Embedding——让大模型看一遍图片，生成一段文字描述，然后把描述当成普通文本索引进向量库。检索的时候搜文字描述，命中了就把原图一起返回。

```
图片 → 大模型看图生成描述 → 描述文本做 Embedding → 存入向量库
                                                           ↑
用户提问 → 问题做 Embedding → 向量检索 → 命中描述 → 返回描述 + 原图
```

简单粗暴，但非常实用。

### 5.2 构建图文混合知识库

```python
import chromadb

chroma_client = chromadb.Client()
collection = chroma_client.create_collection(
    name="multimodal_docs",
    metadata={"hnsw:space": "cosine"}
)


def describe_image(image_source: str, context: str = "") -> str:
    """用大模型为图片生成文字描述，用于索引"""
    prompt = "请详细描述这张图片的内容，包括关键数据、文字、布局等所有信息。"
    if context:
        prompt += f"\n背景信息：{context}"
    return multimodal_chat(
        system_prompt="你是一个精确的图片描述器。描述要尽可能详细和结构化，便于后续检索。",
        image_source=image_source,
        question=prompt
    )


def get_embedding(text: str) -> list[float]:
    """获取文本的 Embedding 向量"""
    response = client.embeddings.create(
        model="text-embedding-3-small",  # 或其他 Embedding 模型
        input=text
    )
    return response.data[0].embedding


# 模拟一个图文混合的知识库
KNOWLEDGE_BASE = [
    {

    },
    {
        "id": "doc_2",
        "type": "text",
        "title": "产品规格说明",

    },
    {
        "id": "img_1",
        "type": "image",
        "title": "Q3各区域销售对比图",
        "image_path": "regional_sales_q3.png",
        "context": "2024年Q3各销售区域的柱状对比图"
    },
    {
        "id": "img_2",
        "type": "image",
        "title": "Model-X 产品外观图",
        "image_path": "model_x_photo.png",

    },
    {
        "id": "doc_3",
        "type": "text",
        "title": "退货政策",
        "content": "7天无理由退货，15天质量问题换货，1年保修。退货需保持原包装完好，配件齐全。大家电需预约上门取件。"
    }
]


def build_multimodal_index(knowledge_base: list[dict]):
    """构建图文混合索引"""
    ids, documents, metadatas, embeddings = [], [], [], []

    for item in knowledge_base:
        item_id = item["id"]

        if item["type"] == "text":
            text = f"{item['title']}\n{item['content']}"
        else:
            # 图片类型：先让大模型生成描述，再索引描述文本
            print(f"  正在为图片 {item_id} 生成描述...")
            description = describe_image(item["image_path"], item.get("context", ""))
            text = f"{item['title']}\n[图片描述] {description}"

        ids.append(item_id)
        documents.append(text)
        metadatas.append({
            "type": item["type"],
            "title": item["title"],
            "has_image": item["type"] == "image",
            "image_path": item.get("image_path", "")
        })
        embeddings.append(get_embedding(text))

    collection.add(ids=ids, documents=documents,
                   metadatas=metadatas, embeddings=embeddings)
    print(f"索引完成，共 {len(ids)} 条文档（文本 + 图片）")


build_multimodal_index(KNOWLEDGE_BASE)
```

关键设计：图片类型的文档先调 `describe_image` 生成描述，然后把描述当成普通文本走 Embedding。metadata 里记着 `has_image` 和 `image_path`，检索命中后可以把原图带出来。

### 5.3 多模态 RAG 检索 + 生成

```python
def multimodal_rag_query(question: str, top_k: int = 3) -> str:
    """多模态 RAG：检索图文混合知识库，生成回答"""

    # 1. 检索
    query_embedding = get_embedding(question)
    results = collection.query(
        query_embeddings=[query_embedding],
        n_results=top_k,
        include=["documents", "metadatas", "distances"]
    )

    # 2. 拼接上下文（区分文本和图片）
    context_parts = []
    image_paths = []

    for doc, meta, dist in zip(results["documents"][0],
                               results["metadatas"][0],
                               results["distances"][0]):
        relevance = 1 - dist
        if relevance < 0.3:
            continue

        if meta["has_image"]:
            context_parts.append(f"[图片文档] {doc}\n（相关度：{relevance:.2f}）")
            if meta["image_path"]:
                image_paths.append(meta["image_path"])
        else:
            context_parts.append(f"[文本文档] {doc}\n（相关度：{relevance:.2f}）")

    context = "\n\n---\n\n".join(context_parts)

    # 3. 构造消息（如果检索到图片，把原图也传给大模型）
    system_prompt = """你是一个知识库助手。根据检索到的文档内容回答用户问题。
规则：
- 只基于提供的文档内容回答，不要编造
- 如果提供了图片，结合图片内容一起分析
- 如果文档中没有相关信息，明确告知用户"""

    user_content = [
        {"type": "text", "text": f"检索到的相关文档：\n\n{context}\n\n用户问题：{question}"}
    ]

    # 把检索到的原图也塞进消息
    for img_path in image_paths:
        if os.path.exists(img_path):
            b64 = encode_image_to_base64(img_path)
            ext = img_path.rsplit(".", 1)[-1].lower()
            mime = {"png": "image/png", "jpg": "image/jpeg",
                    "jpeg": "image/jpeg"}.get(ext, "image/png")
            user_content.append({
                "type": "image_url",
                "image_url": {"url": f"data:{mime};base64,{b64}"}
            })

    messages = [
        {"role": "system", "content": system_prompt},
        {"role": "user", "content": user_content}
    ]

    response = client.chat.completions.create(
        model=MODEL_NAME,
        messages=messages,
        max_tokens=1024
    )
    return response.choices[0].message.content
```

检索到图片类文档后，不只是返回描述文本，还把**原始图片**塞进消息传给大模型——让它既看到检索的文字描述，又能直接看到原图。这样回答的准确度比纯靠描述文本高得多。

## 六、进阶：多图对比分析

实际场景中经常需要同时看多张图。比如"对比这两张截图，UI 改了什么？"或者"这三张报表有什么共同趋势？"

```python
def multi_image_analysis(images: list[dict], question: str) -> str:
    """
    多图对比分析。
    images: [{"path": "img1.png", "label": "修改前"}, ...]
    """
    system_prompt = """你是一个视觉分析专家。用户会提供多张图片进行对比分析。
请仔细观察每张图片的细节差异，给出结构化的对比结论。"""

    content = [{"type": "text", "text": question + "\n\n以下是需要对比的图片："}]

    for img in images:
        content.append({"type": "text", "text": f"\n【{img['label']}】"})
        if img["path"].startswith(("http://", "https://")):
            content.append({
                "type": "image_url",
                "image_url": {"url": img["path"]}
            })
        else:
            b64 = encode_image_to_base64(img["path"])
            content.append({
                "type": "image_url",
                "image_url": {"url": f"data:image/png;base64,{b64}"}
            })

    messages = [
        {"role": "system", "content": system_prompt},
        {"role": "user", "content": content}
    ]

    response = client.chat.completions.create(
        model=MODEL_NAME,
        messages=messages,
        max_tokens=2048
    )
    return response.choices[0].message.content


# 使用示例
result = multi_image_analysis(
    images=[
        {"path": "ui_before.png", "label": "改版前"},
        {"path": "ui_after.png", "label": "改版后"}
    ],
    question="对比这两个版本的 UI 设计，列出所有变化点。"
)
print(result)
```

一条消息里塞多张图，用 label 标注每张图的身份。大模型会逐张看过去，然后做交叉对比。这个能力在 UI Review、竞品分析、质检场景里非常好用。

## 七、踩坑备忘

搭多模态智能体过程中最常踩的几个坑：

### 7.1 图片尺寸与 token 消耗

大模型处理图片是要烧 token 的。一张 1024×1024 的图大约消耗 765 个 token（以 GPT-4o 为例），分辨率越高越贵。

```python
def resize_image_if_needed(image_path: str, max_size: int = 1024) -> str:
    """大图自动缩放，控制 token 消耗"""
    from PIL import Image
    import io

    img = Image.open(image_path)
    if max(img.size) > max_size:
        ratio = max_size / max(img.size)
        new_size = (int(img.size[0] * ratio), int(img.size[1] * ratio))
        img = img.resize(new_size, Image.LANCZOS)

    buffer = io.BytesIO()
    img.save(buffer, format="PNG")
    return base64.b64encode(buffer.getvalue()).decode("utf-8")
```

经验法则：信息密度不高的图片（产品照片、UI 截图）缩到 512px 就够了；信息密度高的（数据表格、密集文字）保持 1024px。

### 7.2 图片描述的质量决定 RAG 效果

V3 的多模态 RAG 里，图片描述是关键环节。描述写得烂，后面检索就找不到。

```python
# ❌ 糟糕的描述 prompt
"描述这张图片。"
# → "这是一张柱状图。" —— 太笼统，检索几乎没用

# ✅ 好的描述 prompt
"请详细描述这张图片的内容。如果是图表，请列出所有数据标签、数值和趋势。如果有文字，请逐字提取。"
# → "这是一张2024年Q3各区域销售额柱状图。华东区680万元，华南区590万元，
#     华北区520万元……整体呈现华东>华南>华北的格局。" —— 信息丰富，检索准确
```

### 7.3 不是所有模型都支持 tool_calls + 图片同时传

有些模型支持视觉但不支持在视觉消息里同时做工具调用。遇到这种情况，用"两阶段"策略：第一次调用让模型看图并输出结构化文字描述，第二次调用把描述当文字传入并启用工具。

```python
def two_stage_vision_agent(image_source: str, question: str) -> str:
    """两阶段策略：先看图提取信息，再用工具分析"""
    # 阶段 1：纯视觉理解，不启用工具
    visual_analysis = multimodal_chat(
        system_prompt="仔细观察图片，输出所有你看到的信息，尽可能结构化。",
        image_source=image_source,
        question=question
    )

    # 阶段 2：把视觉分析结果当文字输入，启用工具
    messages = [
        {"role": "system", "content": "你是一个数据分析智能体。根据以下视觉分析结果，调用合适的工具完成任务。"},
        {"role": "user", "content": f"图片分析结果：\n{visual_analysis}\n\n用户需求：{question}"}
    ]

    response = client.chat.completions.create(
        model=MODEL_NAME, messages=messages,
        tools=tools, max_tokens=2048
    )
    # ... 后续工具调用循环同 V2 ...
    return response.choices[0].message.content
```

## 八、三个版本对比

| | V1 | V2 | V3 |
|---|---|---|---|
| **能力** | 看图 + 聊天 | 看图 + 工具调用 | 图文混合检索 |
| **核心升级** | 多模态输入 | 视觉驱动决策 | 图片也能被搜到 |
| **场景** | 图片描述、OCR | 图表分析、监控告警 | 产品图检索、文档图问答 |
| **复杂度** | 低 | 中 | 中高 |
| **关键代码** | messages 里塞图片 | 图片 + tool_calls | 图片描述 → Embedding → 检索 |

## 写在最后

多模态智能体的核心其实就一句话：**messages 里除了文字，还能塞图片了**。

架构上没有根本性的变化——还是那个"大模型 + 工具 + 循环"的老配方。V1 加了视觉输入，V2 让视觉驱动工具调用，V3 让图片也变成可检索的知识。每一步都是在现有能力上叠加，而不是推倒重来。

但这个"加了眼睛"的升级，在实际业务里的价值非常大。以前用户发一张图过来，智能体只能说"请描述一下图片内容"；现在它自己就能看、能分析、能行动。客服场景里看商品照片判断售后问题，运维场景里看监控截图自动告警，办公场景里看 Excel 截图提取数据——这些都是多模态智能体的杀手级应用。
