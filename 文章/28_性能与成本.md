用大模型做智能体，最终绕不开两个字：**烧钱**。

GPT-4o 一百万 token 15 美元，一个客服 Agent 每天处理 1000 个对话，每个对话平均 5 轮，每轮 2000 token——一天就是 1000 万 token，光 API 费用就是 150 美元。一个月 4500 美元。这还只是一个 Agent。

而且还有性能问题：大模型每次调用需要 1-10 秒，用户可等不了那么久。

这篇文章讲怎么在**不牺牲智能体质量**的前提下，把成本砍下来、把速度提上去。核心思路就一句话：**把"聪明"做在架构里，而不是堆大模型**。

## 一、成本分析：钱花在哪了？

### 1.1 成本构成

```python
def calculate_monthly_cost(
    daily_conversations: int = 1000,
    turns_per_conversation: int = 5,
    tokens_per_turn: int = 2000,
    price_per_million_tokens: float = 1.0,  # DeepSeek: ¥1/百万 token
    rag_calls_ratio: float = 0.6,
    avg_rag_tokens: int = 3000,
    tool_calls_ratio: float = 0.3,
    avg_tool_overhead_tokens: int = 500,
) -> dict:
    """计算月度成本"""
    # 基础 LLM 调用
    daily_tokens = daily_conversations * turns_per_conversation * tokens_per_turn
    monthly_base = daily_tokens * 30 * price_per_million_tokens / 1_000_000

    # RAG 检索带来的额外 token
    rag_conversations = daily_conversations * rag_calls_ratio
    daily_rag_tokens = rag_conversations * avg_rag_tokens * turns_per_conversation
    monthly_rag = daily_rag_tokens * 30 * price_per_million_tokens / 1_000_000

    # 工具调用的额外开销
    tool_conversations = daily_conversations * tool_calls_ratio
    daily_tool_tokens = tool_conversations * avg_tool_overhead_tokens * turns_per_conversation
    monthly_tool = daily_tool_tokens * 30 * price_per_million_tokens / 1_000_000

    total = monthly_base + monthly_rag + monthly_tool
    return {
        "daily_conversations": daily_conversations,
        "monthly_base_cost": round(monthly_base, 2),
        "monthly_rag_cost": round(monthly_rag, 2),
        "monthly_tool_cost": round(monthly_tool, 2),
        "monthly_total": round(total, 2),
        "daily_tokens": daily_tokens,
        "monthly_tokens": daily_tokens * 30
    }


# DeepSeek 成本
cost = calculate_monthly_cost(price_per_million_tokens=1.0)
print(f"=== DeepSeek 月度成本估算 ===")
for key, value in cost.items():
    print(f"  {key}: {value}")

# GPT-4o 成本对比
cost_gpt = calculate_monthly_cost(price_per_million_tokens=15.0)
print(f"\n=== GPT-4o 月度成本估算 ===")
print(f"  月总费用: ${cost_gpt['monthly_total']}")
```

### 1.2 成本的大头在哪？

通常排序是：**LLM 调用 > RAG 上下文膨胀 > 工具调用开销**。

最贵的不是模型单价，而是**不必要的 token 消耗**——塞了太多无关的检索结果、保留了太多历史对话、System Prompt 写太长。

## 二、优化策略一：模型路由

不是所有问题都需要最贵的模型。

```python
import os
import time
from openai import OpenAI

client = OpenAI(
    base_url="https://api.deepseek.com",
    api_key=os.getenv("DEEPSEEK_API_KEY")
)


class CostAwareRouter:
    """成本感知的模型路由器"""

    MODELS = {
        "cheap": {"name": "deepseek-chat", "cost_per_1k": 0.001},
        "strong": {"name": "deepseek-chat", "cost_per_1k": 0.002},
    }

    SIMPLE_PATTERNS = [
        "你好", "谢谢", "再见", "是的", "不是",
        "查订单", "查物流", "什么时候到",
    ]

    def route(self, query: str) -> str:
        """快速路由：简单问题用便宜模型"""
        # 规则路由（零成本判断）
        if len(query) < 20 or any(p in query for p in self.SIMPLE_PATTERNS):
            return "cheap"

        # 复杂问题用强模型
        if any(kw in query for kw in ["对比", "分析", "为什么", "怎么设计", "架构"]):
            return "strong"

        return "cheap"

    def call(self, query: str, system_prompt: str = "你是一个助手") -> dict:
        model_key = self.route(query)
        model = self.MODELS[model_key]

        start = time.time()
        response = client.chat.completions.create(
            model=model["name"],
            messages=[
                {"role": "system", "content": system_prompt},
                {"role": "user", "content": query}
            ],
            temperature=0.7
        )
        latency = time.time() - start

        tokens = 0
        if response.usage:
            tokens = response.usage.total_tokens

        return {
            "model": model_key,
            "content": response.choices[0].message.content,
            "tokens": tokens,
            "cost": tokens * model["cost_per_1k"] / 1000,
            "latency_ms": int(latency * 1000)
        }


router = CostAwareRouter()

queries = [
    "你好",
    "订单 12345 到哪了？",
    "帮我对比一下 FastAPI 和 Django 的架构设计理念、性能差异和适用场景",
]

total_cost = 0
for q in queries:
    result = router.call(q)
    total_cost += result["cost"]
    print(f"  [{result['model']}] {q[:25]}... → {result['tokens']} tokens, ¥{result['cost']:.4f}")

print(f"\n总费用: ¥{total_cost:.4f}")
```

## 三、优化策略二：缓存

相同或相似的问题，不需要每次都调模型。

```python
import hashlib
import json


class ResponseCache:
    """响应缓存：相同问题直接返回缓存结果"""

    def __init__(self, max_size: int = 1000):
        self.cache: dict[str, dict] = {}
        self.max_size = max_size
        self.hits = 0
        self.misses = 0

    def _make_key(self, query: str, system_prompt: str) -> str:
        """生成缓存 key"""
        content = f"{system_prompt}::{query}"
        return hashlib.md5(content.encode()).hexdigest()

    def get(self, query: str, system_prompt: str = "") -> dict | None:
        """查缓存"""
        key = self._make_key(query, system_prompt)
        if key in self.cache:
            entry = self.cache[key]
            if time.time() - entry["cached_at"] < entry["ttl"]:
                self.hits += 1
                return entry["response"]
            else:
                del self.cache[key]
        self.misses += 1
        return None

    def set(self, query: str, response: dict, system_prompt: str = "", ttl: int = 3600) -> None:
        """写缓存"""
        if len(self.cache) >= self.max_size:
            oldest_key = min(self.cache, key=lambda k: self.cache[k]["cached_at"])
            del self.cache[oldest_key]

        key = self._make_key(query, system_prompt)
        self.cache[key] = {"response": response, "cached_at": time.time(), "ttl": ttl}

    def stats(self) -> dict:
        total = self.hits + self.misses
        return {
            "hits": self.hits,
            "misses": self.misses,
            "hit_rate": f"{self.hits / total:.0%}" if total > 0 else "N/A",
            "cache_size": len(self.cache)
        }


# 带缓存的模型调用
cache = ResponseCache()


def cached_call(query: str, system_prompt: str = "你是一个助手") -> dict:
    """先查缓存，miss 了再调模型"""
    cached = cache.get(query, system_prompt)
    if cached:
        return {**cached, "from_cache": True}

    result = router.call(query, system_prompt)
    cache.set(query, result, system_prompt, ttl=1800)
    return {**result, "from_cache": False}


# 同一个问题问两次
for _ in range(2):
    r = cached_call("Python 的 GIL 是什么？")
    source = "缓存" if r.get("from_cache") else "模型"
    print(f"  [{source}] tokens={r.get('tokens', 0)}, cost=¥{r.get('cost', 0):.4f}")

print(f"缓存统计：{cache.stats()}")
```

缓存的适用场景：**高频的事实性问题**（"你们的退货政策是什么？"）。不适合个性化问题（"我的订单到哪了？"）。

## 四、优化策略三：上下文压缩

第 16 篇讲了上下文工程，这里从成本角度再强调几个具体手段。

### 4.1 检索结果精简

```python
def slim_retrieval(docs: list[dict], query: str, max_docs: int = 2, max_chars: int = 500) -> str:
    """精简检索结果：减少塞入上下文的 token 数"""
    # 只取最相关的 N 篇
    top_docs = sorted(docs, key=lambda d: d.get("score", 0), reverse=True)[:max_docs]
    # 每篇只取前 max_chars 个字符
    parts = []
    for i, doc in enumerate(top_docs):
        content = doc["content"][:max_chars]
        parts.append(f"[{i+1}] {content}")
    return "\n".join(parts)
```

### 4.2 历史对话压缩

```python
def compress_history_for_cost(history: list[dict], max_turns: int = 3) -> list[dict]:
    """成本优化的历史压缩：只保留最近 N 轮"""
    system = [m for m in history if m["role"] == "system"]
    conv = [m for m in history if m["role"] != "system"]
    return system + conv[-(max_turns * 2):]
```

## 五、优化策略四：流式输出

流式输出不省钱，但**省体验**。用户不用干等 5 秒，第一个字 0.5 秒就能看到。

```python
def stream_response(query: str, system_prompt: str = "你是一个助手") -> None:
    """流式输出：用户体验优化"""
    stream = client.chat.completions.create(
        model="deepseek-chat",
        messages=[
            {"role": "system", "content": system_prompt},
            {"role": "user", "content": query}
        ],
        stream=True,
        temperature=0.7
    )

    print("助手：", end="", flush=True)
    for chunk in stream:
        if chunk.choices[0].delta.content:
            print(chunk.choices[0].delta.content, end="", flush=True)
    print()
```

## 六、优化策略五：异步并行

多个独立的 LLM 调用可以并行执行。

```python
import asyncio

async def parallel_calls(queries: list[str]) -> list[str]:
    """并行调用多个查询"""

    async def single_call(query: str) -> str:
        response = client.chat.completions.create(
            model="deepseek-chat",
            messages=[{"role": "user", "content": query}],
            temperature=0
        )
        return response.choices[0].message.content

    # 并行执行所有查询
    tasks = [single_call(q) for q in queries]
    results = await asyncio.gather(*tasks)
```

## 八、省钱的优先级

如果你只能做一件事，做**模型路由**。

这是我的亲身经历：我们有一个客服 Agent，上线时全用 GPT-4o，月费 $4,200。加了一个最简单的路由——判断用户消息长度，短于 20 字的用 DeepSeek，其余用 GPT-4o——月费降到了 $1,800。没有任何用户体验下降，因为"你好""查订单 12345""谢谢"这类消息用 DeepSeek 和 GPT-4o 回答质量没有区别。

后来又加了缓存，降到 $800。最后压缩了 System Prompt 里的冗余内容（有 500 token 的内容几乎从不生效），降到 $600。

优化顺序很重要：

1. **模型路由**（投入小，收益最大）
2. **缓存**（需要一定的请求量才有效果，但一旦有效就非常显著）
3. **上下文压缩**（细活，要逐个排查哪些 token 是浪费的）
4. **并行调用**（不省钱但省时间，用户体验好了间接省运营成本）
5. **流式输出**（不省一分钱，但用户"感觉"快了）

别一上来就做流式——那是最不省钱的优化。先看账单，找最大的开销，一刀切下去。
