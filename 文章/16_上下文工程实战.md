你可能听过"Prompt Engineering"——提示词工程。2023 年最火的词之一，满世界都是"一句话让 ChatGPT 效率翻倍"的教程。

但到了 2025 年，搞智能体的人已经不太提这个了。取而代之的是一个新词：**Context Engineering（上下文工程）**。

区别在哪？Prompt Engineering 关心的是"怎么把一句话写好"；Context Engineering 关心的是"**喂给大模型的所有信息怎么组装**"——系统提示、用户消息、检索结果、工具输出、历史对话、Few-shot 示例……这些东西怎么选、怎么排、怎么压缩、怎么动态拼装，才能让大模型在有限的上下文窗口里发挥最大战力。

打个比方：Prompt Engineering 是写菜谱，Context Engineering 是当大厨——你不仅要知道菜谱怎么写，还得知道食材怎么选、怎么备、怎么上桌。这篇文章就带你从"写菜谱"升级到"当大厨"。

依赖很简单：`pip install openai chromadb tiktoken`，代码直接能跑。

## 一、为什么 Prompt 不够了？

### 1.1 智能体的输入远不止一句话

你跟 ChatGPT 聊天，输入就是一句 prompt。但智能体的输入是一个复杂的上下文包：

```
┌─────────────────────────────────────────────┐
│              Context Window                  │
│                                              │
│  ┌─────────────┐  ┌──────────────────────┐  │
│  │ System Prompt│  │ 用户当前问题          │  │
│  │ （角色设定）  │  │                      │  │
│  └─────────────┘  └──────────────────────┘  │
│                                              │
│  ┌─────────────┐  ┌──────────────────────┐  │
│  │ 历史对话     │  │ RAG 检索结果          │  │
│  │ （记忆）     │  │ （外部知识）          │  │
│  └─────────────┘  └──────────────────────┘  │
│                                              │
│  ┌─────────────┐  ┌──────────────────────┐  │
│  │ 工具调用结果  │  │ Few-shot 示例        │  │
│  │ （中间产物）  │  │ （行为示范）         │  │
│  └─────────────┘  └──────────────────────┘  │
│                                              │
└─────────────────────────────────────────────┘
```

每一块都在抢上下文窗口的空间。GPT-4o 有 128K token，听起来很大，但你算算：3 轮历史对话 4K、RAG 检索 5 段文档 8K、工具调用结果 3K、系统提示 2K……一下子就 17K 了。更别提复杂场景下这些数字会翻好几倍。

### 1.2 上下文的质量决定输出的质量

这是最关键的一点：**你塞进去什么，大模型就基于什么来回答**。

塞了一堆无关文档？模型会在噪音里迷路。漏掉关键信息？模型会编造答案。顺序排得不对？模型可能忽略中间部分（Lost in the Middle 问题）。

> 一个残酷的事实：80% 的智能体质量问题，根源不在模型能力，而在喂给模型的上下文质量。

### 1.3 上下文工程的核心问题

说白了就四个字：**选、排、压、拼**。

| 操作 | 问题 | 举例 |
|------|------|------|
| **选** | 哪些信息该放进去？ | 检索了 10 段文档，只有 3 段相关 |
| **排** | 信息按什么顺序放？ | 重要信息放开头还是结尾？ |
| **压** | 信息太多怎么压缩？ | 20 轮对话怎么压成 3K token？ |
| **拼** | 各部分怎么组装？ | System Prompt + 历史 + 检索 + 工具结果的拼装模板 |

下面一个一个讲，每步都有代码。

## 二、选：上下文过滤与相关性控制

第一步是"选"——不是所有信息都值得塞进上下文。

### 2.1 检索结果的相关性过滤

RAG 检索回来的文档，不是每段都有用。很多人直接把 top-5 全塞进去，其实 top-5 里可能只有 2 段真正相关。

```python
import os
from openai import OpenAI

os.environ["OPENAI_API_KEY"] = "你的API Key"
client = OpenAI(
    base_url="https://api.deepseek.com",
    api_key=os.getenv("DEEPSEEK_API_KEY")
)
MODEL_NAME = "deepseek-chat"


def filter_by_relevance(query: str, documents: list[dict], threshold: float = 0.6) -> list[dict]:
    """用 LLM 评估每段文档与问题的相关性，过滤掉不相关的"""
    filtered = []
    for doc in documents:
        response = client.chat.completions.create(
            model=MODEL_NAME,
            messages=[{
                "role": "user",
                "content": (
                    f"判断以下文档与问题的相关性。\n\n"
                    f"问题：{query}\n"
                    f"文档：{doc['content'][:500]}\n\n"
                    f"只返回一个 0 到 1 之间的数字，表示相关程度。"
                    f"1 = 完全相关，0 = 完全无关。只返回数字，不要其他内容。"
                )
            }],
            temperature=0
        )
        try:
            score = float(response.choices[0].message.content.strip())
            if score >= threshold:
                doc["relevance_score"] = score
                filtered.append(doc)
        except ValueError:
            pass
    return sorted(filtered, key=lambda x: x["relevance_score"], reverse=True)


# 模拟 RAG 检索结果
retrieved_docs = [
    {"content": "Python 的 GIL（全局解释器锁）限制了多线程的并行执行效率，对于 CPU 密集型任务建议使用多进程。"},
    {"content": "FastAPI 是一个高性能的 Python Web 框架，支持异步处理，性能接近 Node.js 和 Go。"},
    {"content": "昨天北京天气晴朗，最高气温 28 度，适合户外活动。"},
    {"content": "Django REST Framework 提供了完善的序列化、认证和权限管理功能，适合构建企业级 API。"},
    {"content": "Python 3.12 引入了更好的错误提示信息和性能优化。"},
]

query = "Python Web 框架怎么选？"
relevant = filter_by_relevance(query, retrieved_docs, threshold=0.5)
print(f"原始文档 {len(retrieved_docs)} 段，过滤后 {len(relevant)} 段：")
for doc in relevant:
    print(f"  [{doc['relevance_score']:.1f}] {doc['content'][:60]}...")
```

这种做法的代价是每段文档都要调一次 LLM，慢且贵。生产中更常见的做法是**用 Embedding 相似度做粗筛 + LLM 做精筛**，或者直接用 Rerank 模型（第 2 篇讲过）。

### 2.2 历史对话的选择性保留

不是所有历史对话都值得保留。用户前 5 轮在闲聊，第 6 轮才进入正题——你把闲聊也塞进去，白白浪费 token。

```python
def select_relevant_history(
    history: list[dict],
    current_query: str,
    max_turns: int = 5
) -> list[dict]:
    """从历史对话中挑选与当前问题相关的轮次"""
    if len(history) <= max_turns * 2:
        return history

    system_msgs = [m for m in history if m["role"] == "system"]
    conversations = [m for m in history if m["role"] != "system"]

    # 按轮次配对
    turns = []
    for i in range(0, len(conversations) - 1, 2):
        if i + 1 < len(conversations):
            turns.append((conversations[i], conversations[i + 1]))

    # 最近 2 轮一定保留（上下文连贯性）
    must_keep = turns[-2:] if len(turns) >= 2 else turns[:]
    candidates = turns[:-2] if len(turns) > 2 else []

    # 对更早的轮次，用关键词重叠度做简单相关性判断
    scored = []
    query_words = set(current_query)
    for turn in candidates:
        turn_text = turn[0]["content"] + turn[1]["content"]
        turn_words = set(turn_text)
        overlap = len(query_words & turn_words) / max(len(query_words), 1)
        scored.append((overlap, turn))

    scored.sort(key=lambda x: x[0], reverse=True)
    selected = [t for _, t in scored[:max_turns - len(must_keep)]]

    # 按原始顺序排列
    all_selected = selected + must_keep
    result_msgs = system_msgs[:]
    for turn in turns:
        if turn in all_selected:
            result_msgs.extend(turn)

    return result_msgs
```

核心思路：**最近的一定保留（保证连贯），更早的按相关性挑选**。

## 三、排：信息的摆放顺序

选好了信息，接下来是"排"——放在哪里很重要。

### 3.1 Lost in the Middle 问题

2023 年斯坦福的一篇论文发现：大模型对上下文窗口的注意力分布不均——**开头和结尾的信息更容易被关注，中间的容易被忽略**。

这意味着：
- 最重要的检索结果应该放在**开头或结尾**
- 不太重要的放中间
- 指令性内容（"请基于以上信息回答"）放在**最后**

### 3.2 上下文排列策略

```python
def arrange_context(
    system_prompt: str,
    retrieved_docs: list[dict],
    history: list[dict],
    user_query: str,
    tool_results: list[str] | None = None
) -> list[dict]:
    """按最优顺序组装上下文"""
    messages: list[dict] = []

    # 1. System Prompt 永远在最前面
    messages.append({"role": "system", "content": system_prompt})

    # 2. 历史对话紧跟其后（提供对话连贯性）
    messages.extend(history)

    # 3. 构建当前轮的用户消息：检索结果 + 工具结果 + 用户问题
    #    关键：把最相关的文档放在开头，用户问题放在最后（结尾位置）
    context_parts = []

    if retrieved_docs:
        # 按相关性降序，最相关的排最前
        docs_text = "\n\n".join(
            f"【参考资料 {i+1}】\n{doc['content']}"
            for i, doc in enumerate(retrieved_docs)
        )
        context_parts.append(f"以下是与问题相关的参考资料：\n\n{docs_text}")

    if tool_results:
        tools_text = "\n\n".join(
            f"【工具输出 {i+1}】\n{result}"
            for i, result in enumerate(tool_results)
        )
        context_parts.append(f"以下是工具执行的结果：\n\n{tools_text}")

    # 用户问题放最后（结尾位置，注意力最强）
    context_parts.append(f"基于以上信息，请回答：{user_query}")

    messages.append({"role": "user", "content": "\n\n---\n\n".join(context_parts)})
    return messages


# 示例
messages = arrange_context(
    system_prompt="你是一个技术顾问，基于提供的资料回答问题。如果资料不足，明确说明。",
    retrieved_docs=[
        {"content": "FastAPI 支持异步，性能极高"},
        {"content": "Django 生态完善，适合大型项目"},
    ],
    history=[
        {"role": "user", "content": "我想做一个 API 服务"},
        {"role": "assistant", "content": "好的，你对性能有什么要求？"},
    ],
    user_query="FastAPI 和 Django 哪个更适合？"
)

for msg in messages:
    role = msg["role"].upper()
    content = msg["content"][:100]
    print(f"[{role}] {content}...")
```

排列的核心原则：**System Prompt → 历史 → 参考资料（重要在前）→ 工具结果 → 用户问题（放最后）**。

## 四、压：上下文压缩

信息太多，窗口装不下？那就压缩。

### 4.1 对话历史压缩

第 5 篇讲了滑动窗口和摘要压缩。这里给一个更实用的方案——**渐进式压缩**：最近的保留原文，稍早的压缩成摘要，更早的只保留关键事实。

```python
import tiktoken


def count_tokens(text: str, model: str = "gpt-4o") -> int:
    """计算文本的 token 数"""
    try:
        enc = tiktoken.encoding_for_model(model)
    except KeyError:
        enc = tiktoken.get_encoding("cl100k_base")
    return len(enc.encode(text))


def progressive_compress(
    history: list[dict],
    max_tokens: int = 4000
) -> list[dict]:
    """渐进式压缩：近的保原文，远的压摘要，更远的提关键事实"""
    system_msgs = [m for m in history if m["role"] == "system"]
    conversations = [m for m in history if m["role"] != "system"]

    # 按轮次分组
    turns = []
    for i in range(0, len(conversations) - 1, 2):
        if i + 1 < len(conversations):
            turns.append([conversations[i], conversations[i + 1]])

    if not turns:
        return history

    total_text = " ".join(m["content"] for m in conversations)
    total_tokens = count_tokens(total_text)

    if total_tokens <= max_tokens:
        return history

    # 三层策略：最近 3 轮保原文，3-8 轮压摘要，8 轮以前提取关键事实
    recent = turns[-3:]
    middle = turns[-8:-3] if len(turns) > 3 else []
    old = turns[:-8] if len(turns) > 8 else []

    compressed: list[dict] = system_msgs[:]

    # 老对话 → 提取关键事实
    if old:
        old_text = "\n".join(
            f"用户：{t[0]['content']}\n助手：{t[1]['content']}"
            for t in old
        )
        facts_response = client.chat.completions.create(
            model=MODEL_NAME,
            messages=[{
                "role": "user",
                "content": (
                    f"从以下对话中提取关键事实，用简短的要点列表表示，"
                    f"只保留对后续对话有用的信息：\n\n{old_text}"
                )
            }],
            temperature=0
        )
        facts = facts_response.choices[0].message.content
        compressed.append({
            "role": "system",
            "content": f"[早期对话关键事实]\n{facts}"
        })

    # 中期对话 → 摘要
    if middle:
        middle_text = "\n".join(
            f"用户：{t[0]['content']}\n助手：{t[1]['content']}"
            for t in middle
        )
        summary_response = client.chat.completions.create(
            model=MODEL_NAME,
            messages=[{
                "role": "user",
                "content": f"用 2-3 句话概括以下对话的主要内容：\n\n{middle_text}"
            }],
            temperature=0
        )
        summary = summary_response.choices[0].message.content
        compressed.append({
            "role": "system",
            "content": f"[近期对话摘要]\n{summary}"
        })

    # 近期对话 → 保留原文
    for turn in recent:
        compressed.extend(turn)

    return compressed


# 模拟：12 轮对话
history = [{"role": "system", "content": "你是一个技术顾问"}]
for i in range(12):
    history.append({"role": "user", "content": f"第{i+1}轮问题：关于{['Python', 'Java', 'Go', 'Rust'][i % 4]}的性能问题"})
    history.append({"role": "assistant", "content": f"第{i+1}轮回答：这是关于该语言性能的详细分析，包含基准测试数据和优化建议..." * 3})

original_tokens = count_tokens(" ".join(m["content"] for m in history))
compressed = progressive_compress(history, max_tokens=2000)
compressed_tokens = count_tokens(" ".join(m["content"] for m in compressed))

print(f"压缩前：{original_tokens} tokens")
print(f"压缩后：{compressed_tokens} tokens")
print(f"压缩率：{(1 - compressed_tokens / original_tokens) * 100:.0f}%")
```

### 4.2 检索文档压缩

检索到的文档往往很长，但真正有用的可能只有几句话。用 LLM 做文档压缩：

```python
def compress_document(doc: str, query: str, max_sentences: int = 3) -> str:

    response = client.chat.completions.create(
        model=MODEL_NAME,
        messages=[{
            "role": "user",
            "content": (
                f"从以下文档中提取与问题最相关的内容，"
                f"最多 {max_sentences} 句话，保留关键数据和结论。\n\n"
                f"问题：{query}\n\n"
                f"文档：{doc}"
            )
        }],
        temperature=0
    )
    return response.choices[0].message.content
```

## 五、拼：动态上下文组装

最后是"拼"——把选好、排好、压好的各部分动态拼装成最终的 messages。

### 5.1 上下文预算分配

上下文窗口就这么大，各部分怎么分配额度？我用的策略是**先保证指令，再保证当前问题，剩下的按优先级分配**：

```python
class ContextBudget:
    """上下文预算管理器"""

    def __init__(self, max_tokens: int = 8000, reserved_for_output: int = 2000):
        self.max_tokens = max_tokens
        self.reserved_for_output = reserved_for_output
        self.available = max_tokens - reserved_for_output
        self.allocations: dict[str, int] = {}

    def allocate(self) -> dict[str, int]:
        """按优先级分配 token 预算"""
        budget = self.available
        alloc: dict[str, int] = {}

        # 优先级 1：系统提示（固定，通常 500-1000）
        alloc["system_prompt"] = min(1000, budget)
        budget -= alloc["system_prompt"]

        # 优先级 2：用户当前问题（固定，通常 200-500）
        alloc["current_query"] = min(500, budget)
        budget -= alloc["current_query"]

        # 优先级 3：检索结果（占剩余的 50%）
        alloc["retrieved_docs"] = int(budget * 0.5)
        budget -= alloc["retrieved_docs"]

        # 优先级 4：工具输出（占剩余的 40%）
        alloc["tool_results"] = int(budget * 0.4)
        budget -= alloc["tool_results"]

        # 优先级 5：历史对话（占剩余的全部）
        alloc["history"] = budget

        self.allocations = alloc
        return alloc

    def report(self) -> str:
        if not self.allocations:
            self.allocate()
        lines = [f"总预算：{self.max_tokens} tokens（输出预留 {self.reserved_for_output}）"]
        for name, tokens in self.allocations.items():
            pct = tokens / self.available * 100
            lines.append(f"  {name}: {tokens} tokens ({pct:.0f}%)")
        return "\n".join(lines)


budget = ContextBudget(max_tokens=8000, reserved_for_output=2000)
print(budget.report())
```

### 5.2 完整的上下文组装引擎

把前面的"选、排、压"串起来，做成一个完整的上下文组装流程：

```python
class ContextEngine:
    """上下文工程引擎：负责选、排、压、拼"""

    def __init__(self, max_context_tokens: int = 8000):
        self.budget = ContextBudget(max_tokens=max_context_tokens)
        self.budget.allocate()

    def build_context(
        self,
        system_prompt: str,
        user_query: str,
        history: list[dict] | None = None,
        retrieved_docs: list[dict] | None = None,
        tool_results: list[str] | None = None
    ) -> list[dict]:
        """一站式上下文组装"""
        messages: list[dict] = []

        # 1. 系统提示（裁剪到预算内）
        sys_tokens = self.budget.allocations["system_prompt"]
        trimmed_system = self._trim_to_budget(system_prompt, sys_tokens)
        messages.append({"role": "system", "content": trimmed_system})

        # 2. 历史对话（压缩到预算内）
        if history:
            hist_tokens = self.budget.allocations["history"]
            compressed_history = self._compress_history(history, hist_tokens)
            messages.extend(compressed_history)

        # 3. 构建用户消息
        user_parts: list[str] = []

        # 3a. 检索结果（过滤 + 压缩到预算内）
        if retrieved_docs:
            doc_tokens = self.budget.allocations["retrieved_docs"]
            filtered_docs = self._filter_docs(retrieved_docs, user_query)
            docs_text = self._format_docs(filtered_docs, doc_tokens)
            if docs_text:
                user_parts.append(f"参考资料：\n{docs_text}")

        # 3b. 工具输出
        if tool_results:
            tool_tokens = self.budget.allocations["tool_results"]
            tools_text = self._trim_to_budget(
                "\n\n".join(f"[工具输出 {i+1}] {r}" for i, r in enumerate(tool_results)),
                tool_tokens
            )
            user_parts.append(f"工具执行结果：\n{tools_text}")

        # 3c. 用户问题（永远在最后）
        user_parts.append(f"请回答：{user_query}")

        messages.append({"role": "user", "content": "\n\n---\n\n".join(user_parts)})
        return messages

    def _trim_to_budget(self, text: str, max_tokens: int) -> str:
        tokens = count_tokens(text)
        if tokens <= max_tokens:
            return text
        ratio = max_tokens / tokens
        return text[:int(len(text) * ratio * 0.9)]

    def _compress_history(self, history: list[dict], max_tokens: int) -> list[dict]:
        non_system = [m for m in history if m["role"] != "system"]
        total = count_tokens(" ".join(m["content"] for m in non_system))
        if total <= max_tokens:
            return non_system
        # 只保留最近几轮
        result: list[dict] = []
        current_tokens = 0
        for msg in reversed(non_system):
            msg_tokens = count_tokens(msg["content"])
            if current_tokens + msg_tokens > max_tokens:
                break
            result.insert(0, msg)
            current_tokens += msg_tokens
        return result

    def _filter_docs(self, docs: list[dict], query: str) -> list[dict]:
        # 简化版：如果有相关性分数就用，否则全部保留
        if any("relevance_score" in d for d in docs):
            return [d for d in docs if d.get("relevance_score", 0) >= 0.5]
        return docs

    def _format_docs(self, docs: list[dict], max_tokens: int) -> str:
        parts: list[str] = []
        current_tokens = 0
        for i, doc in enumerate(docs):
            text = f"【{i+1}】{doc['content']}"
            t = count_tokens(text)
            if current_tokens + t > max_tokens:
                break
            parts.append(text)
            current_tokens += t
        return "\n\n".join(parts)


# 使用示例
```

## 六、System Prompt 设计模式

System Prompt 是上下文的"灵魂"——它定义了智能体的行为边界。分享几个在生产中验证过的模式。

### 6.1 结构化 System Prompt 模板

```python
def build_system_prompt(
    role: str,
    capabilities: list[str],
    constraints: list[str],
    output_format: str,
    context_instructions: str = ""
) -> str:
    """构建结构化的 System Prompt"""
    sections = [
        f"# 角色\n你是{role}。",
        f"# 能力\n" + "\n".join(f"- {c}" for c in capabilities),
        f"# 约束\n" + "\n".join(f"- {c}" for c in constraints),
        f"# 输出格式\n{output_format}",
    ]
    if context_instructions:
        sections.append(f"# 上下文使用说明\n{context_instructions}")
    return "\n\n".join(sections)

    role="一个专业的 Python 技术顾问",

        "根据用户需求推荐合适的技术方案",
```

### 6.2 动态 System Prompt

```python
PROMPT_TEMPLATES: dict[str, dict] = {

    }

def classify_intent(query: str) -> str:
    """简单的意图分类"""
    response = client.chat.completions.create(
        model=MODEL_NAME,
        messages=[{
            "role": "user",
            "content": (
                f"判断以下用户输入属于哪个类别，只返回类别名：\n"
                f"- technical（技术问题）\n"
                f"- casual（闲聊）\n"
                f"- code_review（代码审查）\n\n"
                f"用户输入：{query}\n\n"
                f"只返回一个类别名，不要其他内容。"
            )
        }],
        temperature=0
    )
    intent = response.choices[0].message.content.strip().lower()
    return intent if intent in PROMPT_TEMPLATES else "technical"


def get_dynamic_system_prompt(query: str) -> str:
    """根据用户意图动态选择 System Prompt"""
    intent = classify_intent(query)
    template = PROMPT_TEMPLATES[intent]
    return (
        f"你是{template['role']}。\n"
        f"回答风格：{template['style']}。\n"
        f"约束：{template['constraints']}。"
    )
```

## 七、实战：上下文感知的智能体

把前面所有技巧串起来，做一个完整的"上下文感知"智能体：

```python
import json
import chromadb


class ContextAwareAgent:
    """上下文感知智能体：自动管理上下文质量"""

    def __init__(self, max_tokens: int = 8000):
        self.engine = ContextEngine(max_context_tokens=max_tokens)
        self.history: list[dict] = []
        self.knowledge_base = self._init_knowledge_base()

    def _init_knowledge_base(self) -> chromadb.Collection:
        """初始化知识库"""
        chroma = chromadb.Client()
        collection = chroma.get_or_create_collection("tech_docs")
        # 预存一些技术文档
        docs = [
            "FastAPI 是高性能 Python Web 框架，基于 Starlette，支持异步，基准测试 9000+ req/s。",
            "Django 是全功能 Python Web 框架，内置 ORM、Admin、认证系统，适合大型项目。",
            "Flask 是轻量级 Python Web 框架，核心精简，通过扩展添加功能，适合小型项目和微服务。",
            "Uvicorn 是 ASGI 服务器，常配合 FastAPI 使用，支持 HTTP/2 和 WebSocket。",
            "SQLAlchemy 是 Python 最流行的 ORM，支持多种数据库，Django 内置自己的 ORM。",
            "pytest 是 Python 最流行的测试框架，支持参数化、fixture、插件等高级功能。",
        ]
        ids = [f"doc_{i}" for i in range(len(docs))]
        existing = collection.get()["ids"]
        new_docs = [(id_, doc) for id_, doc in zip(ids, docs) if id_ not in existing]
        if new_docs:
            collection.add(
                ids=[x[0] for x in new_docs],
                documents=[x[1] for x in new_docs]
            )
        return collection

    def _retrieve(self, query: str, top_k: int = 3) -> list[dict]:
        """从知识库检索相关文档"""
        results = self.knowledge_base.query(query_texts=[query], n_results=top_k)
        docs = []
        if results["documents"]:
            for doc, dist in zip(results["documents"][0], results["distances"][0]):
                relevance = max(0, 1 - dist)
                docs.append({"content": doc, "relevance_score": relevance})
        return docs

    def chat(self, user_input: str) -> str:
        """对话入口"""
        # 1. 动态选择 System Prompt
        system_prompt = get_dynamic_system_prompt(user_input)

        # 2. 检索相关文档
        docs = self._retrieve(user_input)

        # 3. 用上下文引擎组装
        messages = self.engine.build_context(
            system_prompt=system_prompt,
            user_query=user_input,
            history=self.history,
            retrieved_docs=docs,
        )

        # 4. 调用 LLM
        response = client.chat.completions.create(
            model=MODEL_NAME,
            messages=messages,
            temperature=0.7
        )
        answer = response.choices[0].message.content

        # 5. 更新历史
        self.history.append({"role": "user", "content": user_input})
        self.history.append({"role": "assistant", "content": answer})

        return answer

    def show_stats(self) -> None:
        """显示上下文使用情况"""
        total_history_tokens = count_tokens(
            " ".join(m["content"] for m in self.history)
        )
        print(f"对话轮数：{len(self.history) // 2}")
        print(f"历史 token 数：{total_history_tokens}")
        print(f"预算分配：")
        print(self.engine.budget.report())


# 运行
agent = ContextAwareAgent(max_tokens=6000)

questions = [
    "FastAPI 和 Django 哪个性能更好？",
    "如果我要做一个内部管理系统呢？",
    "那测试怎么写？用什么框架？",
]

for q in questions:
    print(f"\n用户：{q}")
    answer = agent.chat(q)
    print(f"助手：{answer}")

print("\n--- 统计 ---")
agent.show_stats()
```

## 八、上下文工程的常见坑

写到最后，列一下我见过的高频翻车场景：

### 坑 1：检索结果全塞进去

"反正窗口大，多塞点总没坏处"——错。无关信息会稀释有效信息，模型的注意力被分散。宁可少塞，也别塞垃圾。

### 坑 2：System Prompt 写成小说

见过有人写 2000 字的 System Prompt，恨不得把产品需求文档贴进去。System Prompt 应该**精炼**——角色、能力、约束、输出格式，四部分就够了。

### 坑 3：忽略 token 预算

不算 token 就往里塞，结果要么超限报错，要么留给模型输出的空间太小，回答被截断。**永远预留足够的输出空间**。

### 坑 4：历史全量保留

20 轮对话一个字不丢地塞进去，token 烧钱不说，模型还容易在历史信息里打转，忽略当前问题。

### 坑 5：不做 A/B 测试

上下文的组装方式没有"标准答案"，不同的排列、压缩策略对不同任务效果不一样。生产环境一定要做实验。

## 九、总结

| 技术 | 解决什么问题 | 核心思路 |
|------|-------------|----------|
| **相关性过滤** | 检索噪音 | 不是 top-k 全要，只要真正相关的 |
| **信息排列** | Lost in the Middle | 重要的放开头和结尾 |
| **渐进式压缩** | 历史太长 | 近的保原文，远的压摘要，更远提事实 |
| **预算分配** | 窗口有限 | 先保指令和问题，剩余按优先级分 |
| **动态 System Prompt** | 场景多变 | 根据意图自动切换角色和约束 |
| **上下文引擎** | 串联所有环节 | 选→排→压→拼的自动化流水线 |

Prompt Engineering 教你怎么说话，Context Engineering 教你怎么**准备上场前的一切**。在智能体开发中，后者往往比前者更重要——因为模型能力再强，垃圾进、垃圾出。
