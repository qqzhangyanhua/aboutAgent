

# 能不能用 Skill 直接替代 RAG？

# 这是很多 AI Agent 团队都会踩的坑

上周在一个 AI Agent 技术群里，看到两个工程师吵了起来。

起因是一个很常见的问题：他们的客服智能体回答不准，项目负责人说要上 RAG，其中一个开发直接反驳：

> "RAG 一整套链路搭下来，起码两三周。把知识直接写进 Skill 不行吗？一天就能上线。"

另一个开发说："你今天写 20 条规则进去行，明天变成 200 条呢？到时候全是屎山。"

两个人谁也没说服谁。

但这件事本身就很有代表性——**几乎所有做 AI Agent 的团队，在早期都会面临同样的选择困惑。**

先说结论：

> **RAG 解决"AI 知道什么"，Skill 解决"AI 怎么做"。**
> **两者不是竞争关系，而是分工关系。**

但这句话人人都能说出来，真正有价值的问题是：什么场景下可以只用 Skill 先顶上？什么时候必须上 RAG？中间的灰色地带怎么判断？

这篇文章就把这件事讲透。

---

# 一、先搞清楚本质区别：知识 vs 方法

很多争论之所以反复纠缠，是因为一开始就没区分清楚两者的"能力边界"。

### RAG 解决的是"知识供给"问题

模型的训练数据有截止日期，不知道你公司昨天发布的新产品，也不知道你们内部的审批流程。RAG 的本质就是**给模型外挂一个可实时检索的知识库**。

它的核心能力链路是这样的：

```
用户提问 → 理解意图 → 从知识库检索相关文档片段 → 基于检索结果生成回答
```

**适合解决的问题：**

- 模型不知道最新信息（产品更新、政策变动）
- 模型不知道企业内部资料（内部文档、制度、FAQ）
- 知识量太大，不可能全部塞进提示词
- 回答需要可追溯、可引用原始出处

### Skill 解决的是"行为规范"问题

模型虽然聪明，但做事不稳定。同一个任务执行三次可能给出三种完全不同的输出格式。Skill 的本质是**给模型写一本操作手册**，告诉它遇到什么场景，按什么步骤，用什么格式，调用什么工具。

它的核心能力是这样的：

```
触发条件 → 执行步骤 → 输出规范 → 质量检查 → 异常处理
```

**适合解决的问题：**

- 模型做事不稳定，每次步骤不一样
- 不会按团队规范输出（格式、语气、结构）
- 不会正确调用工具和 API
- 复杂任务缺乏合理的执行顺序

**一句话总结本质区别：**

> **RAG 提供"材料"，Skill 规定"做法"。**
> 就像厨师做菜——RAG 是食材仓库，Skill 是菜谱。缺食材做不出菜，缺菜谱做出来的菜不稳定。

---

# 二、为什么很多团队想用 Skill 替代 RAG？

因为从工程角度看，RAG 的建设成本并不低。

一个能投入生产的 RAG 系统，你至少要搞定这些：

| 环节       | 要做的事                                       |
| -------- | ------------------------------------------ |
| 数据采集     | 对接多种数据源（文档、数据库、API、网页）                     |
| 文档切片     | 选择切片策略（按段落、按语义、按固定长度），直接影响检索质量             |
| 向量化      | 选 Embedding 模型，处理多语言、长文本等边界情况              |
| 检索与排序    | 向量检索 + 关键词检索混合，加上 Rerank 模型                |
| 知识更新     | 文档变更后的增量更新、版本管理                            |
| 引用溯源     | 回答要标注来源，能链接回原始文档                           |
| 质量控制     | 召回率、准确率监控，幻觉检测                             |

这一套做下来，对于刚起步的团队来说，两三周算快的。

反过来，Skill 看起来轻快得多：写一套规则、封装一个流程、补几个模板。一个下午就能出效果。

所以很多团队的想法很自然：

> 我能不能把常用知识、规则、模板都写进 Skill，让智能体直接照着做，省掉 RAG？

**这个想法不离谱。在某些场景下甚至是正确的。**

关键在于——什么场景下可以，什么场景下不行。

---

# 三、四种场景下，Skill 可以先顶上

以下场景里，直接用 Skill 承载知识，问题不大，甚至比 RAG 更合适。

## 场景一：知识量很小（< 50 条规则级别）

比如：

- 只有十几条审核规则
- 只有几个固定输出模板
- 只有一套明确的 SOP
- 只有少量稳定的术语解释

**判断标准**：这些内容全部写出来，不超过一两页 A4 纸。这种量级做 RAG 反而是杀鸡用牛刀。

## 场景二：知识几乎不变（更新频率 > 3 个月）

比如：

- 稳定的品牌规范（语气、措辞、禁忌词）
- 固定的交付格式（报告结构、文档模板）
- 长期不变的审查标准
- 通用的写作方法论

**判断标准**：这些内容最近三个月没有改过，未来三个月大概率也不会改。

## 场景三：核心不在"查"，而在"做"

比如：

- 把访谈记录整理成用户洞察
- 把会议纪要改写成周报
- 把客服对话转成工单摘要
- 把一份材料按固定结构重新组织

这些任务表面上也有"知识"成分，但核心价值不在知识检索，而在**流程执行和格式规范**。这正是 Skill 最擅长的领域。

## 场景四：还在 0→1 验证阶段

产品还没定型，业务规则还在变，RAG 的数据源和知识结构都没稳定下来。

这时候先用 Skill 快速验证需求，是非常合理的路线。**先证明 Agent 有价值，再决定怎么做得更好。**

> **总结：知识少、变化慢、重执行、先验证——这四种情况，Skill 先顶上完全没问题。**

# 四、四个危险信号：该上 RAG 了

如果你在实际使用中出现了以下任何一种情况，说明 Skill 已经扛不住了，必须认真考虑上 RAG。

## 信号一：Skill 越写越长，开始像"伪知识库"

最开始写 10 条规则，后来变成 30 条、50 条、100 条。

到某个阶段你会发现：这个 Skill 已经不是 Skill 了——它变成了一个**没有检索能力、没有版本管理、没有引用能力的知识库**。

维护成本会急剧上升：每次改一点规则都要重新理解整个文件、处理冲突、担心遗漏。而且模型的上下文窗口是有限的，塞太多内容进去，**注意力会被稀释，反而导致关键规则被忽略。**

> **经验值参考**：当你的 Skill 文件超过 3000 字，或者规则条目超过 50 条，就该警惕了。

## 信号二：知识更新后，Skill 没跟着改

比如：

- 产品上了新功能，Skill 里还是老功能的说明
- 价格政策调整了，Skill 里还是上个月的价格
- 内部制度变了，Skill 里的审批流程还是旧的

这会导致一个非常隐蔽的问题：

> **AI 的回答流程很标准，格式很漂亮，但依据已经过期了。**

这比"答不出来"更危险——因为它看起来特别可信，用户不会怀疑，错误会悄悄扩散。

RAG 天然支持"更新后即可检索"，文档一改，检索结果就会跟着变。而 Skill 需要人工同步更新，很容易遗漏。

## 信号三：任务依赖"从大量资料中找到最相关那条"

有些任务的关键不只是"知道规则"，而是要**从几百甚至几千份文档里精准定位最相关的那一段**。

典型场景：

- 合同条款匹配——"这个场景适用哪条免责条款？"
- 法规依据查找——"这个操作违反了哪条规定？"
- 帮助中心问答——"我的 XX 功能怎么用？"（产品有几百个功能）
- 内部制度查询——"出差报销的标准是多少？"（制度手册几百页）

这里的核心能力是**语义检索**。不用 RAG，模型只能靠你提前塞进去的有限内容去猜，知识覆盖率和准确率会迅速下降。

## 信号四：回答需要"出处"和"可追溯性"

在很多企业场景中，回答对不对只是一半，**另一半是能不能说清楚依据从哪来**。

典型场景：法务、合规、客服、医疗、金融、企业知识问答。

RAG 可以直接给出文档片段、来源链接、页码引用。而 Skill 本身不擅长做这件事——它更像"工作说明书"，不是"证据系统"。

> **总结：Skill 膨胀失控、知识悄悄过期、需要语义检索、需要引用出处——出现任何一个信号，就该上 RAG 了。**

# 五、成熟架构不是二选一，而是三层设计

真正有经验的团队，不会纠结"用 Skill 还是 RAG"。他们的做法是**分三层来设计**，让两者各司其职。

## 第一层：能写死的规则 → 写进 Skill

```markdown
# 客服回复规范（Skill）
- 输出格式：先问候 → 再回答 → 最后引导
- 语气：专业但亲切，禁止使用"不知道""没办法"
- 敏感问题：涉及法律、安全、隐私时，直接转人工
- 工具调用：查询订单用 order_api，查询物流用 logistics_api
```

这些是**稳定的、流程性的、不依赖外部数据的规则**。

## 第二层：会变化的知识 → 放进 RAG

```markdown
# 知识库内容（RAG）
- 产品功能文档（随版本更新）
- 退换货政策（随活动调整）
- 常见问题 FAQ（持续扩充）
- 活动规则说明（每周可能变化）
```

这些是**量大、会变、需要检索的内容**。

## 第三层：让 Skill 调度 RAG——这才是最关键的一层

```markdown
# 智能客服流程（Skill 调度 RAG）

步骤 1：判断问题类型
  - 如果是"订单/物流查询" → 调用 order_api
  - 如果是"产品功能问题" → 进入步骤 2
  - 如果是"投诉/敏感问题" → 直接转人工

步骤 2：检索知识库
  - 从 RAG 检索与用户问题最相关的 3 个文档片段
  - 如果相关度评分 < 0.7，告知用户"这个问题我不太确定，帮你转接专员"

步骤 3：生成回答
  - 基于检索结果，按客服模板输出
  - 必须在回答末尾标注信息来源
  - 如果检索结果之间有矛盾，以最新版本为准

步骤 4：质量自检
  - 回答是否直接回应了用户问题？
  - 是否包含了来源引用？
  - 语气是否符合品牌规范？
```

这才是 Agent 的完整设计：

> **RAG 负责"找"，Skill 负责"用"。Skill 决定什么时候检索、检索后怎么判断、怎么引用、怎么输出。**

# 六、实战推演：一个客服智能体的演进过程

用一个具体案例，把从"只有 Skill"到"Skill + RAG"的演进过程走一遍。

### 阶段一：只有 RAG，没有 Skill

```
用户问题 → 检索知识库 → 返回相关文档 → 模型自由发挥生成答案
```

**实际表现：**

- 能找到相关资料，但回答风格不统一
- 不会先安抚情绪，直接丢答案
- 不区分"可直接回答"和"需要转人工"
- 格式混乱，有时长篇大论，有时一句话带过

**问题本质**：有知识，没方法。

### 阶段二：只有 Skill，没有 RAG

```
用户问题 → Skill 判断类型 → 按模板生成回答
```

**实际表现：**

- 回答格式统一、流程标准，用户体验好
- 但遇到超出 Skill 覆盖范围的问题就无法回答
- 产品更新了新功能，Skill 里没有，AI 不知道
- 活动政策变了，Skill 还是旧的，答错了用户没察觉

**问题本质**：有方法，没知识。

### 阶段三：Skill + RAG 协同

```
用户问题
→ Skill 判断问题类型和意图
→ 根据意图选择是否检索 RAG
→ RAG 返回最相关的文档片段
→ Skill 判断检索结果是否足够支撑回答
→ Skill 按客服模板生成标准化回复
→ 自动标注信息来源
→ 必要时触发升级/转人工流程
```

**实际表现：**

- 回答准确且有据可查
- 知识实时更新，不怕政策变动
- 格式统一、流程标准
- 能识别自己"不确定"的问题并主动转人工

**这才是一个合格的 AI 客服应该有的样子。**

> **真正决定智能体"像不像一个靠谱员工"的，是 Skill。**
> **真正决定它"知不知道最新政策"的，是 RAG。**

# 七、一张表帮你快速决策

做智能体时，面对任何一块功能，都可以用这张表快速判断。

| 判断维度         | 偏 Skill                            | 偏 RAG                               |
| ------------ | ---------------------------------- | ----------------------------------- |
| **核心需求**     | "按规范做事"（跑流程、格式化输出）                | "找到正确信息"（知识检索、问答）                   |
| **知识规模**     | < 50 条规则，1-2 页 A4 纸                | 几十到几千份文档                            |
| **更新频率**     | 季度级别以上才变                           | 周级别甚至天级别变化                          |
| **是否需要引用出处** | 不需要                                | 需要（法务、合规、客服等场景）                     |
| **当前阶段**     | 0→1 验证期                            | 已验证，准备长期运营                          |
| **团队资源**     | 有限，先快速验证                           | 有余力搭建检索链路                           |

**最实用的一个判断公式：**

> 如果你能把要用到的全部知识在一个 Markdown 文件里写完，而且三个月内不需要大改——用 Skill。
> 否则，上 RAG。

**从 Skill 向 RAG 的迁移路径建议：**

1. **第一步**：先用 Skill 快速验证需求，跑通核心流程
2. **第二步**：当 Skill 文件膨胀或知识开始频繁变动时，把"变化的知识"抽离出来
3. **第三步**：将抽离出的知识灌入 RAG 系统，保留 Skill 作为流程控制层
4. **第四步**：让 Skill 调度 RAG——实现"三层架构"

这个过程是渐进式的，不需要一步到位。

# 八、结语

很多团队做 AI Agent 时，最容易犯的错，不是模型选错了，而是**把"知识问题"和"流程问题"混在了一起**。

于是出现两种常见误区：

- 以为上了 RAG，智能体就会做事（缺 Skill）
- 以为做了 Skill，就不需要 RAG（缺知识）

真正合理的理解是：

> **RAG 让 AI 知道更多，Skill 让 AI 做得更稳。**

想用 Skill 替代 RAG？在小场景、稳定知识、早期验证阶段——完全可以。

但别把临时方案当长期架构。当知识开始膨胀、开始过期、开始需要检索时，该上 RAG 就上。

最终，成熟的 Agent 不是二选一，而是：

> **用 RAG 管知识，用 Skill 管流程，用 Skill 调度 RAG。**

这才是接近真实生产环境的答案。
