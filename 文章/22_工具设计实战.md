```python
def do_everything(action: str, params: dict) -> str:
    if action == "query_order":
        ...
    elif action == "cancel_order":
```

一个超级函数搞定一切。大模型调用它时需要传一个 `action` 参数来区分要干啥。听起来省事？实际上是噩梦——大模型得从 30 个 action 里选一个，选错的概率极高；schema 描述写不清楚；出了 bug 不知道是哪个分支的问题。

这篇文章讲**工具设计**的方法论：怎么拆、拆多细、拆完怎么组合。

## 一、为什么"一个大工具"是错的

### 1.1 大模型选择工具的认知负担

大模型选工具靠的是什么？**工具名 + 描述**。你给它一个工具叫 `do_everything`，描述是"执行各种操作"——它怎么知道什么时候该调？

对比一下：

```
❌ 一个大工具：
   名称：do_everything
   描述：执行订单相关的所有操作，通过 action 参数区分

✅ 拆成小工具：
   名称：query_order        描述：根据订单号查询订单详情
   名称：cancel_order       描述：取消指定订单
   名称：apply_refund       描述：提交退款申请
   名称：track_logistics    描述：查询包裹物流状态
```

后者每个工具名和描述都明确告诉大模型"这个工具干什么"。大模型选对的概率大幅提升。

## 二、工具拆分的五个原则

### 原则 1：单一职责

一个工具只做一件事。判断标准：**能否用一句话描述它的功能**。

```
❌ "管理用户和订单的相关操作" → 做了两件事
✅ "根据订单号查询订单详情" → 一件事
```

### 原则 2：输入最小化

只要完成任务必需的参数，不要多余的。

```python
# ❌ 参数太多，大模型容易填错
{
    "name": "query_order",
    "parameters": {
        "order_id": "string",
        "user_id": "string",      # 不需要，系统能根据 session 获取
        "include_history": "bool", # 不需要，默认就好
        "format": "string",       # 不需要，工具内部决定
    }
}

# ✅ 只要必需的
{
    "name": "query_order",
    "parameters": {
        "order_id": "string"
    }
}
```

### 原则 3：描述要精准

描述是大模型选工具的唯一依据。写得含糊就等着选错。

```python
# ❌ 含糊
"description": "处理订单"

# ✅ 精准
"description": "根据订单号查询订单的当前状态、商品信息和物流进度"
```

### 原则 4：输出要结构化

工具的返回值要结构化，让大模型容易理解和引用。

```python
# ❌ 非结构化
def query_order(order_id: str) -> str:
    return "订单12345，状态已发货，2件商品，快递圆通"

# ✅ 结构化
def query_order(order_id: str) -> str:
    result = {
        "order_id": order_id,
        "status": "已发货",
        "items": [
            {"name": "无线鼠标", "quantity": 1, "price": 79},
            {"name": "鼠标垫", "quantity": 1, "price": 29}
        ],
        "logistics": {"carrier": "圆通", "tracking_no": "YT12345"}
    }
    return json.dumps(result, ensure_ascii=False)
```

### 原则 5：幂等优先

同一个调用执行多次，结果应该一样。因为大模型可能重复调用同一个工具。

```python
# ❌ 非幂等：每次调用都创建新记录
def create_ticket(title: str) -> str:
    ticket_id = db.insert({"title": title})  # 调两次就创建两条
    return f"工单 {ticket_id} 已创建"

# ✅ 幂等：先检查是否已存在
def create_ticket(title: str, idempotency_key: str) -> str:
    existing = db.find({"idempotency_key": idempotency_key})
    if existing:
        return f"工单 {existing.id} 已存在"
    ticket_id = db.insert({"title": title, "idempotency_key": idempotency_key})
    return f"工单 {ticket_id} 已创建"
```

## 三、工具粒度：拆到什么程度？

拆太粗不好，拆太细也不好。有个简单的判断标准：

### 3.1 粒度判断矩阵

| | 工具太粗 | 粒度刚好 | 工具太细 |
|---|---|---|---|
| **特征** | 一个工具做 5 件事 | 一个工具做 1 件事 | 一个操作拆成 3 个工具 |
| **问题** | 大模型选不准 | - | 大模型要调 3 次才能完成 |
| **例子** | `manage_order` | `cancel_order` | `find_order` → `check_cancellable` → `do_cancel` |

### 3.2 拆还是不拆的决策树

```python
def should_split_tool(tool_description: str) -> str:
    """判断一个工具要不要拆分"""
    checks = {
        "描述里有"和""并""或者"等连接词？": "功能不单一，建议拆",
        "参数超过 5 个？": "可能混合了多个功能，建议拆",
        "有些参数只在特定场景用？": "不同场景该是不同工具",
        "名字很长或很泛？": "说明职责不清晰，建议拆",
    }
    results = []
    for question, suggestion in checks.items():
        results.append(f"  检查：{question} → {suggestion}")
    return "\n".join(results)


# 举例
print("=== 工具：manage_user_and_order ===")
print(should_split_tool("管理用户信息和订单，支持查询、修改、删除"))
print("\n=== 工具：query_order ===")
print(should_split_tool("根据订单号查询订单详情"))
```

### 3.3 正确的粒度示例

以电商客服场景为例：

```
太粗（1 个工具）：
  order_system(action, order_id, ...)  → 大模型老选错 action

太细（15 个工具）：
  get_order_id → get_order_status → get_order_items →
  get_order_price → get_order_address → ...  → 调太多次

刚好（6 个工具）：
  query_order       → 查订单
  cancel_order      → 取消订单
  apply_refund      → 退款
  track_logistics   → 查物流
  update_address    → 改地址
  create_ticket     → 提工单
```

## 四、Skill = 工具的编排

单个工具是原子操作，但用户的需求往往需要多个工具配合。这时候需要 **Skill** —— 工具的编排层。

```python
from typing import Callable, Any


# 原子工具
def query_order(order_id: str) -> dict:
    return {"order_id": order_id, "status": "已发货", "amount": 299, "cancelable": False}

def check_refund_eligible(order: dict) -> dict:
    eligible = order["status"] in ["已发货", "已签收"]
    return {"eligible": eligible, "reason": "订单状态允许退款" if eligible else "当前状态不可退款"}

def calculate_refund(order: dict) -> dict:
    return {"refund_amount": order["amount"], "refund_method": "原路退回"}

def submit_refund(order_id: str, amount: float, reason: str) -> dict:
    return {"ticket_id": "RF20250310001", "status": "审核中", "estimated_days": 3}


class SkillEngine:
    """技能引擎：管理和执行 Skill"""

    def __init__(self):
        self.skills: dict[str, dict] = {}

    def register(
        self,
        name: str,
        description: str,
        steps: list[Callable],
        required_params: list[str]
    ) -> None:
        self.skills[name] = {
            "description": description,
            "steps": steps,
            "required_params": required_params
        }

    def execute(self, skill_name: str, params: dict) -> dict:
        """执行一个 Skill"""
        if skill_name not in self.skills:
            return {"error": f"未知技能：{skill_name}"}

        skill = self.skills[skill_name]
        context = dict(params)
        results: list[dict] = []

        for i, step in enumerate(skill["steps"]):
            try:
                result = step(**{k: v for k, v in context.items() if k in step.__code__.co_varnames})
                context.update(result)
                results.append({"step": i + 1, "function": step.__name__, "result": result})
            except Exception as e:
                return {"error": f"步骤 {i + 1}（{step.__name__}）执行失败：{e}", "completed_steps": results}

        return {"success": True, "steps": results, "final_context": context}

    def list_skills(self) -> list[dict]:
        return [
            {"name": name, "description": info["description"], "params": info["required_params"]}
            for name, info in self.skills.items()
        ]


# 注册 Skill
engine = SkillEngine()

engine.register(
    name="退款申请",
    description="完整的退款流程：查订单 → 检查资格 → 计算金额 → 提交申请",
    steps=[
        lambda order_id: query_order(order_id),
        lambda **ctx: check_refund_eligible({"status": ctx.get("status", ""), "amount": ctx.get("amount", 0)}),
        lambda **ctx: calculate_refund({"amount": ctx.get("amount", 0)}),
        lambda order_id, **ctx: submit_refund(order_id, ctx.get("refund_amount", 0), "用户申请退款"),
    ],
    required_params=["order_id"]
)

# 执行
print("可用技能：")
for s in engine.list_skills():
    print(f"  {s['name']}：{s['description']}（参数：{s['params']}）")
```

### Tool vs Skill 的关系

```
用户说："我要退款，订单号 12345"

大模型选择 Skill：退款申请
                    │
          ┌────────┼────────┐────────┐
          ↓        ↓        ↓        ↓
     查订单    检查资格   算金额   提交申请
    (Tool 1)  (Tool 2)  (Tool 3)  (Tool 4)
```

大模型选的是 **Skill**（一个高层意图），Skill 内部自动串联多个 **Tool**（原子操作）。大模型不需要知道退款流程有几步，它只需要知道"退款申请"这个 Skill 需要一个 `order_id`。

## 五、工具描述的写法

工具描述是大模型选工具的唯一依据。写好描述比写好代码更重要。

### 5.1 好描述的四要素

```python
GOOD_DESCRIPTION_TEMPLATE = """
一个好的工具描述包含四要素：

1. 做什么（What）：这个工具的功能
2. 什么时候用（When）：在什么场景下应该调用它
3. 需要什么（Input）：参数说明（在 parameters 里）
4. 返回什么（Output）：返回值的说明

示例：
❌ "查询数据库"
✅ "根据订单号查询订单的当前状态、商品列表和物流信息。
    当用户询问订单状态、发货情况或商品详情时使用。
    返回 JSON 格式的订单详情。"
"""
```

### 5.2 参数描述的写法

```python
# ❌ 参数描述太简单
{
    "order_id": {"type": "string", "description": "订单号"}
}

# ✅ 参数描述带格式提示
{
    "order_id": {
        "type": "string",
        "description": "订单号，格式为纯数字，如 1234567890。用户可能说"订单12345"或"单号12345"，提取其中的数字部分即可。"
    }
}
```

大模型从用户自然语言中提取参数时，描述里的格式提示能大幅提升准确率。

## 六、踩坑回顾

回过头看，工具设计最容易犯的错误就是**按开发者的思维而不是大模型的思维来设计**。

开发者觉得一个 `manage_order(action, order_id, ...)` 函数很优雅——参数灵活，一个函数搞定所有事。但大模型面对这种工具时，它得先判断 action 应该传什么值、不同 action 下哪些参数是必填的。判断链越长，出错概率越高。

反过来，`query_order(order_id)` + `cancel_order(order_id, reason)` + `refund_order(order_id, amount)` 这三个工具虽然"不优雅"，但大模型几乎不可能选错——用户说"查一下我的订单"，它 100% 会选 `query_order`。

所以我现在的经验法则是：**给大模型设计工具时，把自己想象成一个实习生。如果实习生拿到这个工具的说明书，5 秒内能知道什么时候用、怎么用，那就是好的设计。** 如果还需要看半天文档、理解一堆枚举值，那就该拆了。
