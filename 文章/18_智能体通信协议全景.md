第 7 篇咱们聊了 MCP，把它比作 USB——给智能体的工具搞了一套标准接口。但 MCP 只解决了"Agent 怎么调工具"的问题。如果你想让**两个不同的 Agent 互相合作**呢？比如你的客服 Agent 需要调用供应商的库存 Agent，两个 Agent 可能在不同的云上、用不同的框架写的、甚至用不同的大模型。

这就需要 Agent 之间的通信协议了。

2023-2025 年，智能体的通信方式经历了三代演进：

1. **Function Calling**（2023）—— AI 学会"打电话"：告诉模型有哪些函数可以调，模型返回调用参数
2. **MCP**（2024）—— 工具变成"USB 设备"：标准化工具接口，即插即用
3. **A2A**（2025）—— Agent 之间"握手"：不同平台的 Agent 互相发现、互相委托任务

这篇文章把三代协议串起来，讲清楚每代解决了什么问题、长什么样、怎么用。重点放在 MCP 进阶和 A2A，因为 Function Calling 大家已经很熟了。

---

## 一、全景图：三代协议各解决什么问题？

先用一张表对齐认知：

| | Function Calling | MCP | A2A |
|---|---|---|---|
| **解决的问题** | AI 怎么调用外部函数 | 工具怎么标准化接入 | Agent 之间怎么协作 |
| **通信方向** | LLM → 函数 | Client → Server | Agent ↔ Agent |
| **核心比喻** | 打电话（号码写死） | USB 插口（即插即用） | 外交协议（互相发现、谈判、委托） |
| **发起者** | OpenAI（2023.6） | Anthropic（2024.11） | Google（2025.4） |
| **适用场景** | 单 Agent + 几个工具 | Agent + 多种工具/数据源 | 多 Agent 跨平台协作 |
| **标准化程度** | 各厂商各搞各的 | 开放标准，社区推动 | 开放标准，多厂商支持 |

三者不是替代关系，而是**层层递进**：Function Calling 是基础，MCP 在其上做了工具层的标准化，A2A 又在更上层做了 Agent 间的标准化。

## 二、Function Calling：起点

### 2.1 回顾基本流程

Function Calling 的本质很简单：你告诉 LLM"你有哪些函数可以用"，LLM 决定要不要调、传什么参数，你拿到参数后自己执行函数，再把结果喂回去。

```python
import os
import json
from openai import OpenAI

client = OpenAI(
    base_url="https://api.deepseek.com",
    api_key=os.getenv("DEEPSEEK_API_KEY")
)
MODEL_NAME = "deepseek-chat"

# 定义工具 schema
tools = [
    {
        "type": "function",
        "function": {
            "name": "get_stock_price",
            "description": "查询股票实时价格",
            "parameters": {
                "type": "object",
                "properties": {
                    "symbol": {"type": "string", "description": "股票代码，如 AAPL"}
                },
                "required": ["symbol"]
            }
        }
    }
]

def get_stock_price(symbol: str) -> str:
    """模拟股价查询"""
    prices = {"AAPL": "189.52", "GOOGL": "141.80", "MSFT": "417.23"}
    return prices.get(symbol, "未找到")

# 调用
messages = [{"role": "user", "content": "苹果公司股价多少？"}]
response = client.chat.completions.create(model=MODEL_NAME, messages=messages, tools=tools)

msg = response.choices[0].message
if msg.tool_calls:
    call = msg.tool_calls[0]
    args = json.loads(call.function.arguments)
    result = get_stock_price(**args)
    print(f"调用 {call.function.name}({args}) → {result}")
```

### 2.2 Function Calling 的痛点

用了两年 Function Calling，痛点越来越明显：

**痛点 1：Schema 格式不统一**。OpenAI 用一种格式，Claude 用另一种，Gemini 又不一样。换个模型就得改 schema。

**痛点 2：工具和 Agent 强耦合**。工具定义写死在代码里，想共享给别的 Agent？复制粘贴。

**痛点 3：只管"调用"，不管"发现"**。你得提前告诉 Agent 有哪些工具，它不能自己去找。

这三个痛点催生了 MCP。

## 三、MCP 进阶：超越第 7 篇的入门

第 7 篇讲了 MCP 的基础——Server、Client、stdio 通信。这里进阶讲三个生产中更重要的话题。

### 3.1 MCP 的 Resource 和 Prompt 能力

很多人只用了 MCP 的 Tools 能力，忽略了另外两个同样重要的原语：

```python
from mcp.server import Server
from mcp.server.stdio import stdio_server
import mcp.types as types

app = Server("advanced-server")

# ---- 1. Tools（第 7 篇讲过）----
@app.list_tools()
async def list_tools() -> list[types.Tool]:
    return [
        types.Tool(
            name="query_database",
            description="查询数据库",
            inputSchema={
                "type": "object",
                "properties": {
                    "sql": {"type": "string", "description": "SQL 语句"}
                },
                "required": ["sql"]
            }
        )
    ]

@app.call_tool()
async def call_tool(name: str, arguments: dict) -> list[types.TextContent]:
    if name == "query_database":
        # 模拟查询
        return [types.TextContent(type="text", text=f"查询结果：{arguments['sql']} → 3 条记录")]
    return [types.TextContent(type="text", text=f"未知工具：{name}")]


# ---- 2. Resources（只读数据源）----
@app.list_resources()
async def list_resources() -> list[types.Resource]:
    return [
        types.Resource(
            uri="db://schema/users",
            name="用户表结构",
            description="users 表的字段定义",
            mimeType="application/json"
        ),
        types.Resource(
            uri="db://schema/orders",
            name="订单表结构",
            description="orders 表的字段定义",
            mimeType="application/json"
        )
    ]

@app.read_resource()
async def read_resource(uri: str) -> str:
    schemas = {
        "db://schema/users": '{"id": "int", "name": "varchar(100)", "email": "varchar(200)"}',
        "db://schema/orders": '{"id": "int", "user_id": "int", "amount": "decimal", "status": "varchar(20)"}',
    }
    return schemas.get(str(uri), "未找到")


# ---- 3. Prompts（提示词模板）----
@app.list_prompts()
async def list_prompts() -> list[types.Prompt]:
    return [
        types.Prompt(
            name="sql_expert",
            description="SQL 查询专家模式",
            arguments=[
                types.PromptArgument(
                    name="task",
                    description="要完成的查询任务",
                    required=True
                )
            ]
        )
    ]

@app.get_prompt()
async def get_prompt(name: str, arguments: dict | None) -> types.GetPromptResult:
    if name == "sql_expert":
        task = (arguments or {}).get("task", "")
        return types.GetPromptResult(
            messages=[
                types.PromptMessage(
                    role="user",
                    content=types.TextContent(
                        type="text",
                        text=f"你是 SQL 专家。先读取表结构（用 resources），再根据任务写查询：\n任务：{task}"
                    )
                )
            ]
        )
    raise ValueError(f"未知 Prompt: {name}")


async def main():
    async with stdio_server() as (read_stream, write_stream):
        await app.run(read_stream, write_stream)

if __name__ == "__main__":
    import asyncio
    asyncio.run(main())
```

三个原语的分工：

| 原语 | 用途 | 类比 |
|------|------|------|
| **Tools** | 可执行的操作 | USB 设备的"写"功能 |
| **Resources** | 只读的上下文数据 | USB 设备的"读"功能 |
| **Prompts** | 预置的交互模板 | USB 设备的"使用说明" |

### 3.2 SSE 远程通信

第 7 篇用的是 stdio（本地进程通信），生产环境更常见的是 SSE（远程 HTTP 通信）：

```python
# 远程 MCP Server（SSE 模式）
from mcp.server import Server
from mcp.server.sse import SseServerTransport
from starlette.applications import Starlette
from starlette.routing import Route, Mount

app = Server("remote-server")

# ... 注册 tools/resources/prompts（同上）...

sse = SseServerTransport("/messages/")

async def handle_sse(request):
    async with sse.connect_sse(request.scope, request.receive, request._send) as streams:
        await app.run(streams[0], streams[1], app.create_initialization_options())

starlette_app = Starlette(
    routes=[
        Route("/sse", endpoint=handle_sse),
        Mount("/messages/", app=sse.handle_post_message),
    ]
)

# 启动：uvicorn this_file:starlette_app --host 0.0.0.0 --port 8080
```

```python
# 远程 MCP Client
from mcp import ClientSession
from mcp.client.sse import sse_client

async def use_remote_server():
    async with sse_client("http://server-host:8080/sse") as (read, write):
        async with ClientSession(read, write) as session:
            await session.initialize()

            # 和本地用法完全一样
            tools = await session.list_tools()
            print(f"远程工具：{[t.name for t in tools.tools]}")

            result = await session.call_tool("query_database", {"sql": "SELECT * FROM users"})
            print(f"结果：{result.content[0].text}")
```

关键点：**Client 侧的代码完全不用改**——无论 Server 在本地还是远程，Client 的 API 都一样。这就是协议标准化的好处。

### 3.3 MCP 的局限

MCP 解决了"Agent 怎么用工具"的问题，但没解决"Agent 怎么找到其他 Agent"和"Agent 之间怎么合作"的问题。你的 Agent 能通过 MCP 调一个数据库工具，但它不能通过 MCP 找到另一个 Agent 并委托任务给它。

这就是 A2A 要解决的问题。

## 四、A2A：Agent 之间的"外交协议"

### 4.1 A2A 解决什么问题？

想象这个场景：

你的公司有一个"客服 Agent"，负责回答用户问题。用户问："我上周下的单到哪了？"客服 Agent 需要查物流信息，但物流系统是合作方维护的，他们也有一个"物流 Agent"。

问题来了：两个 Agent 怎么合作？

- 它们可能跑在不同的服务器上
- 用不同的框架（你用 LangGraph，对方用 CrewAI）
- 背后用不同的模型（你用 GPT，对方用 Claude）
- 它们需要互相发现、互相信任、互相传递任务

MCP 搞不定这件事——它只管 Agent 和工具之间的关系，不管 Agent 和 Agent 之间的关系。

**A2A（Agent-to-Agent Protocol）** 就是来填这个坑的。Google 在 2025 年 4 月发布了 A2A 协议，目标是让不同平台、不同框架、不同模型的 Agent 能互相协作。

### 4.2 A2A 的核心概念

A2A 定义了五个关键概念：

| 概念 | 说明 | 类比 |
|------|------|------|
| **Agent Card** | Agent 的"名片"，描述自己能干啥 | LinkedIn 个人主页 |
| **Task** | 一次委托任务，有生命周期 | 外包合同 |
| **Message** | Agent 之间的消息 | 邮件 |
| **Artifact** | 任务产出物（文件、数据等） | 交付物 |
| **Streaming** | 实时更新进度 | 项目进度报告 |

### 4.3 Agent Card：Agent 的名片

每个 A2A Agent 需要在 `/.well-known/agent.json` 发布自己的 Agent Card：

```python
# 模拟一个 A2A Agent Card
agent_card = {
    "name": "物流查询 Agent",
    "description": "查询包裹物流状态、预计送达时间",
    "url": "https://logistics-agent.example.com",
    "version": "1.0.0",
    "capabilities": {
        "streaming": True,
        "pushNotifications": False
    },
    "skills": [
        {
            "id": "track_package",
            "name": "包裹追踪",
            "description": "根据运单号查询包裹当前位置和预计送达时间",
            "inputModes": ["text"],
            "outputModes": ["text"]
        },
        {
            "id": "estimate_delivery",
            "name": "送达预估",
            "description": "预估包裹送达时间",
            "inputModes": ["text"],
            "outputModes": ["text"]
        }
    ],
    "authentication": {
        "schemes": ["Bearer"]
    }
}

import json
print(json.dumps(agent_card, indent=2, ensure_ascii=False))
```

Agent Card 的作用：**让别的 Agent 能发现你、了解你能干啥、知道怎么和你通信**。就像你的 LinkedIn 主页告诉别人你会什么技能、怎么联系你。

### 4.4 A2A 的交互流程

一次完整的 A2A 交互分四步：

```
1. 发现（Discovery）
   客服 Agent 访问 https://logistics.example.com/.well-known/agent.json
   获取物流 Agent 的 Agent Card，了解它的能力

2. 发起任务（Send Task）
   客服 Agent 发送：
   POST https://logistics.example.com/a2a
   {
     "method": "tasks/send",
     "params": {
       "id": "task-001",
       "message": {
         "role": "user",
         "parts": [{"type": "text", "text": "查询运单 SF1234567890 的物流状态"}]
       }
     }
   }

3. 执行 & 流式更新（Streaming）
   物流 Agent 开始处理，通过 SSE 推送进度：
   → "正在查询运单..."
   → "已定位包裹，当前在深圳转运中心"
   → 最终结果

4. 返回结果（Task Complete）
   {
     "id": "task-001",
     "status": {"state": "completed"},
     "artifacts": [
       {
         "parts": [{"type": "text", "text": "运单 SF1234567890：深圳转运中心，预计明天送达"}]
       }
     ]
   }
```

### 4.5 用 Python 模拟 A2A 交互

下面用纯 Python 模拟一个简化的 A2A 交互流程：

```python
import uuid
from enum import Enum


class TaskState(str, Enum):
    SUBMITTED = "submitted"
    WORKING = "working"
    COMPLETED = "completed"
    FAILED = "failed"


class A2AMessage:
    """A2A 消息"""
    def __init__(self, role: str, text: str):
        self.role = role
        self.text = text

    def to_dict(self) -> dict:
        return {"role": self.role, "parts": [{"type": "text", "text": self.text}]}


class A2ATask:
    """A2A 任务"""
    def __init__(self, task_id: str | None = None):
        self.id = task_id or str(uuid.uuid4())
        self.state = TaskState.SUBMITTED
        self.messages: list[A2AMessage] = []
        self.artifacts: list[dict] = []

    def to_dict(self) -> dict:
        return {
            "id": self.id,
            "status": {"state": self.state.value},
            "messages": [m.to_dict() for m in self.messages],
            "artifacts": self.artifacts
        }


class A2AServer:
    """模拟 A2A Server（被调用方）"""

    def __init__(self, name: str, skills: list[dict]):
        self.name = name
        self.skills = skills
        self.tasks: dict[str, A2ATask] = {}

    def get_agent_card(self) -> dict:
        return {
            "name": self.name,
            "skills": self.skills,
            "capabilities": {"streaming": False}
        }

    def handle_task(self, task_id: str, message: A2AMessage) -> A2ATask:
        """处理一个任务"""
        task = A2ATask(task_id)
        task.messages.append(message)
        task.state = TaskState.WORKING

        # 模拟处理逻辑
        query = message.text
        if "运单" in query or "物流" in query:
            task.artifacts.append({
                "parts": [{"type": "text", "text": f"查询结果：包裹在深圳转运中心，预计明天 18:00 前送达"}]
            })
            task.state = TaskState.COMPLETED
        else:
            task.artifacts.append({
                "parts": [{"type": "text", "text": "抱歉，我只能处理物流查询相关任务"}]
            })
            task.state = TaskState.FAILED

        self.tasks[task.id] = task
        return task


class A2AClient:
    """模拟 A2A Client（调用方）"""

    def __init__(self, name: str):
        self.name = name
        self.known_agents: dict[str, dict] = {}

    def discover(self, server: A2AServer) -> dict:
        """发现远程 Agent"""
        card = server.get_agent_card()
        self.known_agents[card["name"]] = card
        return card

    def send_task(self, server: A2AServer, text: str) -> A2ATask:
        """发送任务"""
        task_id = str(uuid.uuid4())
        message = A2AMessage(role="user", text=text)
        return server.handle_task(task_id, message)


# —— 模拟完整流程 ——

# 1. 物流 Agent（Server）
logistics_agent = A2AServer(
    name="物流查询 Agent",
    skills=[
        {"id": "track", "name": "包裹追踪", "description": "根据运单号查询物流状态"}
    ]
)

# 2. 客服 Agent（Client）
customer_service = A2AClient(name="客服 Agent")

# 3. 发现
card = customer_service.discover(logistics_agent)
print(f"发现 Agent：{card['name']}")
print(f"  能力：{[s['name'] for s in card['skills']]}")

# 4. 发送任务
task = customer_service.send_task(logistics_agent, "帮我查一下运单 SF1234567890 的物流状态")
print(f"\n任务状态：{task.state.value}")
print(f"结果：{task.artifacts[0]['parts'][0]['text']}")
```

### 4.6 A2A 与 MCP 的关系

很多人问：有了 MCP 还需要 A2A 吗？它俩是什么关系？

```
┌──────────────────────────────────────────────┐
│                 A2A 协议层                     │
│    Agent A  ←──── 任务委托 ────→  Agent B     │
│       │                             │         │
│       ↓                             ↓         │
│  ┌─────────┐                  ┌─────────┐    │
│  │MCP Client│                 │MCP Client│    │
│  └────┬────┘                  └────┬────┘    │
│       ↓                            ↓         │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐     │
│  │工具 A    │  │工具 B    │  │工具 C    │     │
│  │(MCP Srv) │  │(MCP Srv) │  │(MCP Srv) │     │
│  └─────────┘  └─────────┘  └─────────┘     │
└──────────────────────────────────────────────┘
```

一句话：**MCP 管"Agent 到工具"，A2A 管"Agent 到 Agent"**。它们在不同层次，完全可以配合使用。

一个 Agent 内部通过 MCP 调工具（查数据库、读文件），Agent 之间通过 A2A 协作（委托任务、交换结果）。

## 五、Function Calling 的演进方向

Function Calling 虽然是最早的方案，但它一直在进化。

### 5.1 从私有格式到行业标准

2023 年各家格式不统一。现在的趋势是向 JSON Schema + OpenAI 格式看齐：

```python
# 2023 年：各家各有各的格式
# OpenAI 格式
openai_tool = {
    "type": "function",
    "function": {"name": "...", "parameters": {"type": "object", ...}}
}

# Claude 格式（早期）
claude_tool = {
    "name": "...",
    "input_schema": {"type": "object", ...}
}

# 2025 年：趋于统一
# Claude 现在也支持 OpenAI 格式的 tools 参数
# Gemini 也基本对齐
# DeepSeek、Qwen 等国产模型更是直接兼容 OpenAI 格式
```

### 5.2 Structured Outputs：让格式更可靠

Function Calling 的一个痛点是模型可能返回格式不对的参数。OpenAI 在 2024 年推出了 Structured Outputs，保证输出严格遵循 JSON Schema：

```python
from pydantic import BaseModel


class WeatherQuery(BaseModel):
    city: str
    unit: str = "celsius"


class WeatherResponse(BaseModel):
    city: str
    temperature: float
    unit: str
    description: str


# 用 Structured Outputs 保证格式
response = client.chat.completions.create(
    model=MODEL_NAME,
    messages=[{"role": "user", "content": "北京天气"}],
    tools=[{
        "type": "function",
        "function": {
            "name": "get_weather",
            "description": "查询天气",
            "parameters": WeatherQuery.model_json_schema(),
            "strict": True  # 开启严格模式
        }
    }]
)

# strict=True 保证返回的参数一定符合 schema
# 不会出现缺字段、类型错误等问题
```

### 5.3 Parallel Function Calling

现在的模型支持一次返回多个工具调用，不用串行等待：

```python
# 一个问题触发多个并行调用
messages = [{"role": "user", "content": "帮我查北京和上海的天气，再算一下 1024+2048"}]

response = client.chat.completions.create(
    model=MODEL_NAME,
    messages=messages,
    tools=tools  # 包含 get_weather 和 calculate
)

msg = response.choices[0].message
if msg.tool_calls:
    print(f"并行调用 {len(msg.tool_calls)} 个工具：")
    for call in msg.tool_calls:
        print(f"  - {call.function.name}({call.function.arguments})")
    # 可以并行执行这些调用，再一次性把结果喂回去
```

## 六、选型指南：什么场景用什么协议？

| 场景 | 推荐方案 | 理由 |
|------|---------|------|
| 单 Agent + 3-5 个工具 | **Function Calling** | 最简单，直接在代码里定义 schema 就行 |
| 单 Agent + 很多工具/数据源 | **MCP** | 工具标准化，即插即用，不用每个都写一遍 |
| 工具需要被多个 Agent 共享 | **MCP** | 一个 Server 多个 Client 复用 |
| 同一团队的多 Agent 协作 | **框架内置机制**（如 LangGraph） | 同一框架内用框架自己的编排 |
| 跨团队/跨平台的 Agent 协作 | **A2A** | 不同框架、不同模型的 Agent 需要标准化通信 |
| Agent 需要被外部发现和调用 | **A2A** | Agent Card 提供了标准化的能力声明 |

实际项目中，三者经常**混合使用**：

```
用户
  ↓
客服 Agent（Function Calling 调用内部函数）
  ├── MCP → 查订单数据库
  ├── MCP → 读产品文档
  └── A2A → 委托给物流 Agent（外部合作方）
                ├── MCP → 查物流系统
                └── Function Calling → 调用内部计算函数
```

## 七、协议互操作：MCP ↔ A2A 桥接

一个常见需求：你的 Agent 通过 MCP 暴露了一堆工具能力，现在想让它也能被其他 Agent 通过 A2A 调用。需要一个"桥接层"：

```python
class MCPtoA2ABridge:
    """
    桥接层：把 MCP Server 包装成 A2A Agent
    让通过 MCP 暴露的工具，也能通过 A2A 被其他 Agent 调用
    """

    def __init__(self, mcp_tools: list[dict], agent_name: str):
        self.mcp_tools = mcp_tools
        self.agent_name = agent_name

    def generate_agent_card(self) -> dict:
        """从 MCP 工具列表自动生成 A2A Agent Card"""
        skills = []
        for tool in self.mcp_tools:
            func = tool.get("function", tool)
            skills.append({
                "id": func["name"],
                "name": func["name"],
                "description": func.get("description", ""),
                "inputModes": ["text"],
                "outputModes": ["text"]
            })
        return {
            "name": self.agent_name,
            "description": f"通过 MCP 桥接的 Agent，提供 {len(skills)} 个能力",
            "skills": skills,
            "capabilities": {"streaming": False}
        }

    def route_a2a_to_mcp(self, skill_id: str, text_input: str) -> str:
        """把 A2A 任务路由到对应的 MCP 工具"""
        for tool in self.mcp_tools:
            func = tool.get("function", tool)
            if func["name"] == skill_id:
                # 实际生产中这里会调用 MCP Client
                return f"[MCP 桥接] 调用工具 {skill_id}，输入：{text_input}"
        return f"未找到对应的 MCP 工具：{skill_id}"


# 使用
bridge = MCPtoA2ABridge(
    mcp_tools=[
        {"function": {"name": "query_database", "description": "查询数据库"}},
        {"function": {"name": "read_document", "description": "读取文档"}},
    ],
    agent_name="数据服务 Agent"
)

card = bridge.generate_agent_card()
print(f"桥接后的 Agent Card：")
print(json.dumps(card, indent=2, ensure_ascii=False))

result = bridge.route_a2a_to_mcp("query_database", "SELECT * FROM users LIMIT 10")
print(f"\n路由结果：{result}")
```

## 八、总结

| 协议 | 解决什么 | 核心设计 | 适用场景 |
|------|---------|---------|---------|
| **Function Calling** | AI 调函数 | Schema 声明 + LLM 生成参数 | 单 Agent + 少量工具 |
| **MCP** | 工具标准化 | Client-Server + 三种原语 | Agent + 多工具/数据源 |
| **A2A** | Agent 互操作 | Agent Card + Task 生命周期 | 跨平台多 Agent 协作 |

三代协议的演进方向很清楚：**从"AI 能调函数"到"工具即插即用"到"Agent 互联互通"**。

如果你刚开始做智能体，Function Calling 就够了。工具多了上 MCP。多 Agent 跨平台协作用 A2A。三者不冲突，按需选用，混合使用也完全没问题。
