# 量化是什么？让大模型在消费级显卡上跑起来

70B 的模型要几百 GB 显存？别慌，用量化把它压到能在你的游戏显卡上跑。这事儿就像把 4K 电影压成 720P——画质稍微糊一点，但手机能播了，而且大多数人根本看不出差别。

---

## 一、为什么大模型这么吃显存

先说个数字：一个 **Llama 3.1 70B** 模型，用 FP16 精度存，光权重就要 **140GB**。两张 A100 80G 才勉强装得下，普通玩家一张 4090 24G 直接劝退。

为啥这么能吃？大模型的本质是一堆**参数**——几十亿、几百亿个浮点数，每个数都要占内存。70B 就是 700 亿个参数，FP16 每个数占 2 字节，算下来 140GB 没跑。再算上推理时的激活值、KV Cache，显存需求还得往上加。

打个比方：模型就像一本超厚的字典，每个词条是一个参数。FP16 相当于每个字都用高清印刷，清晰是清晰，但书厚得搬不动。**量化**干的事，就是换一种"印刷方式"——字小一点、纸薄一点，书能塞进背包了，内容还是那本字典。

---

## 二、量化的核心思想

量化的核心就一句话：**牺牲一点精度，换巨大的空间节省**。

这跟 JPEG 压缩图片是一个道理。一张 RAW 照片几十 MB，转成 JPEG 质量 90%，体积砍到几 MB，肉眼几乎看不出区别。量化对大模型做的事一模一样——把高精度的浮点数（FP16、FP32）映射到低精度的整数（INT8、INT4），存储和计算都省了。

具体来说：**FP16** 用 16 位表示一个数，能表达 65536 种不同的值，精度很高。**INT8** 只有 8 位，256 种值，精度砍半，但体积也砍半。**INT4** 更狠，16 种值，体积再砍一半，精度再降一档。

大模型有个特点：**对精度没那么敏感**。训练好的权重，大部分数值分布比较集中，用少几位去近似，模型输出差不了多少。就像你记电话号码，记个 138 开头的 11 位就行，不用精确到小数点后 10 位。所以量化在 NLP 领域特别管用，换到图像生成可能就得谨慎点。

---

## 三、量化级别解释

市面上常见的精度档位，可以类比成**图片分辨率**：

**FP32**：原图，32 位浮点，精度最高，体积最大。训练用这个，推理一般用不上。

**FP16**：高清版，16 位浮点，体积减半，精度几乎无损。很多推理框架的默认格式。

**INT8**：标清版，8 位整数，体积再减半。**Q8** 量化通常能做到"几乎无损"，肉眼和 FP16 分不出来。

**INT4**：流畅版，4 位整数，体积再砍一半。**Q4** 系列是消费级显卡的常客，7B 模型 4GB 显存就能跑。

**Q4_K_M、Q5_K_S 这些是啥？** 这是 **GGUF** 格式里的细分档位。K 代表"分块量化"（block-wise quantization），同一块内的参数共享一个缩放因子，比全局量化更精细。M 是 Medium，S 是 Small，数字越大精度越高、体积越大。**Q4_K_M** 是 INT4 里的"均衡款"，**Q5_K_S** 是 INT5 里的"轻量款"，**Q8_0** 接近无损。

选哪个？显存紧张就 Q4，想要质量就 Q8，折中就 Q5 或 Q6。

## 四、常见量化方法

不同量化方法，本质都是"把高精度映射到低精度"，但实现思路不一样。

**GPTQ**（GPT Quantization）：**逐层量化**，针对 Transformer 的每一层单独做校准，用真实数据跑一遍，找到最优的量化参数。优点是精度高，缺点是量化过程慢，而且得自己跑一遍。适合对质量要求高、愿意花时间做离线量化的场景。**AutoGPTQ**、**GPTQ-for-LLaMA** 是常见实现。

**AWQ**（Activation-aware Weight Quantization）：**激活感知量化**。它发现一个规律：有些权重对输出影响大，有些影响小。AWQ 给重要的权重少压一点，不重要的多压一点，在同样体积下比 GPTQ 更保真。量化速度也比 GPTQ 快。**vLLM**、**SGLang** 都支持 AWQ 模型。

**GGUF**（GPT-Generated Unified Format）：**Ollama 和 llama.cpp 的御用格式**。它不是某一种量化算法，而是一个**容器格式**，里面可以装 Q4、Q5、Q8 等各种量化版本。GGUF 的设计目标就是"易用"——一条命令拉下来就能跑，不用自己量化。社区里 Qwen、Llama、Mistral 的 GGUF 版本一抓一大把，**Ollama** 默认用的就是它。

总结：**要省事用 GGUF**，要极致精度用 GPTQ 或 AWQ 自己量化。普通人玩本地模型，GGUF 够用了。

## 五、实操：用 Ollama 跑量化模型

Ollama 已经把量化封装好了，你根本不用关心底层是 Q4 还是 Q8，选个模型名就行。

拉一个 7B 的 Q4 量化版：

```bash
ollama pull qwen2.5:7b-instruct-q4_K_M
```

跑起来：

```bash
ollama run qwen2.5:7b-instruct-q4_K_M
```

完事。一行拉模型，一行跑，没有配置文件，没有"请先安装 CUDA"。

想试试更大的？70B 的 Q4 量化版，显存需求从 140GB 压到 40GB 左右，两张 4090 能跑：

```bash
ollama pull qwen2.5:72b-instruct-q4_K_M
ollama run qwen2.5:72b-instruct-q4_K_M
```

Ollama 的模型名里带 `q4`、`q8` 的就是量化版。`ollama list` 看已安装的，`ollama pull <模型名>` 拉新的。不用自己转格式，不用写脚本，**开箱即用**。

## 六、量化对性能的影响

量化会损失精度，但损失有多大？得看任务。

**通用对话、写作、翻译**：Q4 和 FP16 的差距，普通人很难察觉。你问"写一封辞职信"，两种精度生成的文案质量差不多。**代码生成**：Q4 偶尔会出点小毛病，比如少个括号、变量名拼错，但整体逻辑没问题。**数学推理**：这里差距会明显一点，Q4 在复杂推理题上更容易翻车，Q8 更稳。

有团队做过实测：7B 模型上，**Q4_K_M 相比 FP16，在 MMLU 等基准上掉分大约 1–3%**，在 HumanEval 代码任务上掉 2–5%。对日常使用来说，这点损失可以接受。**Q8 基本可以视为无损**，和 FP16 的差距在误差范围内。

另一个好处：**量化后推理更快**。INT4 的矩阵运算比 FP16 省算力，同样显存下，Q4 的吞吐量能比 FP16 高一大截。所以很多生产环境反而主动用 Q4——又快又省，效果够用。

## 七、怎么选量化级别

本质是**显存和精度的平衡**。

**显存 8GB**：老老实实 Q4，7B 模型能跑，再大就别想了。

**显存 12–16GB**：Q4 跑 7B 绰绰有余，可以试试 13B 的 Q4。想要更好效果就 7B 的 Q8。

**显存 24GB**：7B 的 Q8、13B 的 Q4、甚至 32B 的 Q4 都能玩。24G 是消费级的天花板，选择空间很大。

**显存 48GB 及以上**：70B 的 Q4 能跑，想要更高质量就 Q5 或 Q6。FP16 就别想了，除非你有多张 A100。

**实用建议**：先拉个 Q4 试试，跑得动、效果能接受就别折腾。觉得糊就换 Q8，显存不够就换更小的模型。量化级别不是越高越好，**够用就行**。

---

## 结尾

量化不是什么黑科技，就是"用更少的比特存更多的信息"——和 JPEG 压缩、MP3 编码一个思路。大模型天生适合这种压缩，所以才能在消费级显卡上跑起来 70B 的模型。

下次有人跟你说"没 A100 别想跑大模型"，你可以回他：**用量化，游戏显卡也能玩**。一张 4090，Q4 的 70B 跑得动；一张 3060，Q4 的 7B 绰绰有余。门槛没那么高，动手试试就知道了。
