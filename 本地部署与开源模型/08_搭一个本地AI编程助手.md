# 搭一个本地的 AI 编程助手（替代 Copilot）

GitHub Copilot 一年一百多美刀，Cursor Pro 也不便宜。如果告诉你，在自己电脑上就能跑一个差不多的，一分钱不用花，你会不会心动？

别急着说"本地跑不动"。2024 年的开源代码模型，7B 级别在普通笔记本上就能流畅跑，补全速度不比云端慢多少。关键是：**数据不出你的电脑**，想怎么用就怎么用，没有订阅费，没有 token 计费。今天就把这事儿说清楚——怎么搭、用什么模型、几种方案怎么选，以及和 Copilot 比到底差在哪。

---

## 一、本地编程助手的架构

本质上，本地 AI 编程助手就是两件事：**本地跑的代码模型** + **能调用它的编辑器插件**。

这就像你家里装了个净水器，水从自家水管出来，不依赖外面的桶装水配送。模型在本地推理，插件负责把当前代码、光标位置、项目上下文打包发给模型，再把返回的补全或建议展示给你。数据流全程在你机器上，不经过任何第三方服务器。

**核心组件**：

- **推理引擎**：Ollama、LM Studio、TabbyML 等，负责加载模型、跑推理、提供 API
- **模型**：DeepSeek Coder、Qwen2.5 Coder、CodeLlama 等，专门为代码训练
- **编辑器插件**：VSCode 的 Continue、Cursor 的本地模型配置、Tabby 的官方插件等，负责和推理引擎通信

三者搭配好了，体验上就和 Copilot 很像：边写边补全、选中代码问问题、让 AI 帮你改一段逻辑。区别只是模型在你本地跑，不花钱。

---

## 二、模型选择

代码模型和通用聊天模型不一样，训练数据里代码占比高，对语法、库、框架更敏感。选错了模型，补全质量会明显拉胯。

**DeepSeek Coder**：国内团队出品，专攻代码。6.7B 版本在 HumanEval 上表现很好，中英文代码都吃得开，**性价比极高**。Ollama 里叫 `deepseek-coder`，量化后 4GB 显存就能跑。如果你主要写 Python、JavaScript、Go，闭眼选这个基本不会错。

**Qwen2.5 Coder**：阿里通义千问的代码专项模型，7B 规格，训练了 5.5 万亿 token 代码数据，覆盖 92 种语言。**中文注释、中文变量名**处理得比 DeepSeek 更自然，适合国内项目。Ollama 里叫 `qwen2.5-coder`。

**CodeLlama**：Meta 出品，基于 Llama 微调，有 7B、13B、34B 多个规格。英文代码表现稳定，生态成熟，但中文支持一般。如果你主要写英文代码、用英文文档，CodeLlama 是稳妥之选。

**简单对比**：

| 模型 | 显存需求（量化后） | 中文代码 | 英文代码 | 推荐场景 |
|------|-------------------|----------|----------|----------|
| DeepSeek Coder 6.7B | 4–6GB | 良好 | 优秀 | 综合首选，性价比高 |
| Qwen2.5 Coder 7B | 4–6GB | 优秀 | 良好 | 中文项目、国内技术栈 |
| CodeLlama 7B | 4–6GB | 一般 | 优秀 | 英文为主、要生态 |

**实用建议**：16GB 内存的笔记本，优先试 **DeepSeek Coder** 或 **Qwen2.5 Coder**，两个都拉下来对比一下，看哪个更顺手。显存紧张就选 `:7b-q4_0` 这类量化版，体积砍半，效果差不了多少。

## 三、方案一：Ollama + Continue（最简方案）

这是上手最快的组合：Ollama 负责跑模型，Continue 是 VSCode/Cursor 的插件，专门对接各种本地和云端模型。

**安装步骤**：

1. 装 Ollama：去 https://ollama.com 下载，Mac 拖进应用程序，Windows 双击安装包，Linux 用官方脚本。装完在终端跑一次 `ollama run deepseek-coder`，确认能正常对话。
2. 拉代码模型：`ollama pull deepseek-coder`（或 `qwen2.5-coder`），等下载完成。
3. 装 Continue 插件：在 VSCode 或 Cursor 的扩展市场搜 "Continue"，安装。
4. 配置模型：打开 Continue 面板，点设置，在 "Models" 里添加 Ollama，模型名填 `deepseek-coder`，保存。

**使用体验**：

Continue 支持**行内补全**（类似 Copilot 的 ghost text）和**聊天面板**（选中代码提问、让 AI 改逻辑）。补全触发方式和 Copilot 差不多，边写边出建议，按 Tab 采纳。聊天模式可以 @ 文件、@ 文件夹，把项目上下文塞给模型，让它基于整个目录回答问题。

优点是**零配置**：Ollama 默认开 API，Continue 自动发现，填个模型名就能用。缺点是 Continue 的补全有时不如专门做补全的插件灵敏，但对日常写代码、问问题已经够用。

---

## 四、方案二：Ollama + Cursor（配置本地模型为后端）

Cursor 本身支持把本地模型当后端，不用买 Pro 也能用 AI 能力，只是模型换成你自己跑的。

**配置方法**：

1. 确保 Ollama 在跑，且已拉好代码模型。
2. 打开 Cursor 设置（Cmd/Ctrl + ,），搜 "OpenAI Base URL" 或 "Custom API"。
3. 把 Base URL 改成 `http://localhost:11434/v1`，API Key 随便填（如 `ollama`），模型名填 `deepseek-coder`。
4. 保存后，Cursor 的补全和聊天会走本地 Ollama，不再请求云端。

**体验**：

Cursor 的补全 UI 和 Copilot 很像，行内灰字、Tab 采纳。聊天面板可以选中代码、问问题、让 AI 改多行。本地模型响应速度取决于你的 CPU/GPU，一般 7B 量化版在 M1/M2 Mac 或 3060 显卡上，延迟在可接受范围内。

适合已经习惯 Cursor、不想换编辑器、又希望数据本地的用户。缺点是 Cursor 的 Agent 能力（多文件编辑、复杂任务拆解）主要针对云端大模型优化，本地 7B 模型做复杂任务会吃力，但补全和简单问答没问题。

## 五、方案三：TabbyML（专为代码补全设计的开源方案）

TabbyML 是专门为**代码补全**设计的开源项目，不是通用聊天模型，而是针对"下一个 token 预测"优化过的。就像 Copilot 的核心是补全，Tabby 只做这一件事，但做得更专注。

**特点**：

- 支持 **StarCoder2**、**Qwen2.5-Coder**、**DeepSeek-Coder** 等，自带量化、缓存优化
- 提供 **VSCode 插件**，补全延迟低，对击键响应快
- 支持 **CUDA / Metal / CPU**，Mac 和 Linux 都能跑
- 完全本地，无云端依赖

**安装**：

1. 去 https://tabby.tabbyml.com 下载对应系统的 Tabby 服务端，或 Docker 部署。
2. 在 VSCode 装 "Tabby" 插件，配置里填 Tabby 服务地址（默认 `http://localhost:8080`）。
3. 在 Tabby 里选择模型、下载，启动服务。

**使用体验**：

补全延迟比 Ollama + Continue 更低，因为 Tabby 针对补全场景做了优化，模型加载和推理都更轻量。但**没有聊天功能**，只能做行内补全，不能选中代码问问题。适合"我只要补全、不要聊天"的用户，或者和 Continue 搭配用：Tabby 负责补全，Continue 负责聊天。

## 六、体验对比：本地方案 vs Copilot / Cursor Pro

实话实说，本地 7B 代码模型和 Copilot、Cursor Pro 用的云端大模型，**有差距**，但没想象中那么大。

**补全速度**：本地模型推理在你机器上，网络延迟为零，首 token 出来往往比云端还快。持续输出时，7B 量化版在中等配置上也能做到基本实时，不会明显卡顿。

**代码质量**：简单补全——变量名、常见 API、重复结构——本地模型和 Copilot 差距不大，甚至因为延迟低，体感更流畅。复杂逻辑、跨文件理解、需要"想几步"的任务，云端大模型明显更强。本地 7B 容易在长依赖、复杂上下文上翻车。

**上下文理解**：Copilot 和 Cursor 会利用整个项目、打开的文件、光标位置，做更精准的补全。本地方案依赖插件传多少上下文，传得少就弱一些。Continue 支持 @ 文件，能缓解，但和云端"全项目索引"比，还是差一档。

**诚实评价**：

- **日常写业务代码、补全变量和函数**：本地方案够用，甚至更顺手（无延迟、无计费）
- **复杂重构、多文件联动、需要深度推理**：还是云端大模型靠谱
- **企业合规、离线开发、预算敏感**：本地方案是刚需，值得投入

## 七、适合什么场景

**企业合规**：代码不能出内网，不能走云端 API。本地部署是唯一选择，模型、数据全在你自己控制的机器上。

**离线开发**：出差、通勤、网络不稳，云端一断就歇菜。本地模型插上电就能跑，不依赖网络。

**省钱**：Copilot 一年 100 刀，Cursor Pro 也要订阅。本地跑起来之后，电费之外没有边际成本，适合学生、个人开发者、小团队。

**学习**：想搞懂代码模型怎么工作、补全怎么实现的，本地部署是最好的实验环境。换模型、调参数、看日志，随时折腾。

**隐私**：公司内部项目、未公开的创意、敏感逻辑，不想让任何第三方看到。本地推理，数据不出机器。

## 八、优化技巧

**选对模型大小**：7B 是性价比甜点，16GB 内存的机器跑量化版没问题。14B、32B 质量更好，但显存需求翻倍，除非有 24GB+ 显卡，否则先别上。

**调 GPU 层数**：Ollama 支持 `OLLAMA_NUM_GPU` 控制多少层放 GPU，剩下放 CPU。显存紧张时，减少 GPU 层数，用 CPU 兜底，能跑起来比跑不动强。

**配置缓存**：Ollama 会把模型加载到内存，第一次推理慢，后面就快。确保 `ollama serve` 常驻，别每次用都重启。Tabby 也有类似缓存，保持服务开着即可。

**量化换速度**：`q4_0`、`q5_0` 这类量化版，体积小、推理快，质量损失有限。先试量化版，不够再上完整版。

**System Prompt**：如果用的是支持 System Prompt 的方案（如 Continue 聊天），可以加一句"你是一个专业的编程助手，优先给出简洁、可运行的代码"，能稍微提升输出风格的一致性。

---

## 结尾

本地 AI 编程助手不是要取代 Copilot，而是多一个选择。当你需要隐私、离线、省钱，或者单纯想折腾一下开源模型，本地部署是一条实实在在的路。Ollama + Continue 十分钟能跑起来，Tabby 专注补全，Cursor 配本地模型也能用。选好模型、调好配置，日常写代码的体验不会差到哪去。

至于和顶配云端模型的差距，心里有数就行：简单场景够用，复杂场景再交给云端。两者不冲突，可以并存。
