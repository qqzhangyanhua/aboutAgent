# Token 是什么？为什么 AI 按"字"收费

我朋友上周用 ChatGPT 写了一篇 2000 字的行业报告。他输入的问题只有两行，不到 50 个字。一看账单：烧了 47 块。

他懵了：我就问了句话，怎么这么贵？

我帮他查了查：那 2000 字里，输入加输出一共烧了将近 5 万个 token。按 API 的计价方式，输出比输入贵三倍——模型"写"的每一个字，都在烧钱。他这才明白：原来 AI 不是按"字"收费，而是按一种叫 token 的东西收费。而 token 和字，根本不是一回事。

答案藏在一个叫 **token** 的东西里。今天把它说透。

---

## 一、同样一段话，中文比英文贵 50%——这不是歧视，是数学

先说一个很多人不知道的事实，也是整篇文章最值得记住的一点：**同样意思的一段话，用中文和用英文，AI 的"计价单位"数量差很多**。

"人工智能正在改变世界"——9 个字，可能被算成 9 个 token。换成英文 "AI is changing the world"——5 个词，可能只占 5 到 6 个 token。信息量差不多，中文的"计价单位"几乎是英文的两倍。

所以当你用中文跟 ChatGPT 聊天，同样的对话轮数，账单往往比用英文高一大截。这不是平台故意坑你，而是**分词算法的"母语"是英文**。GPT 的词表是在英文为主的互联网语料上训练出来的，对英文的"切法"更优化。

知道这一点，你就明白：想省 token，用英文 prompt 有时候真能省 30% 到 50%。做多语言产品时，也要把"中文用户更耗 token"算进成本里。

这不是谁故意设计的，而是历史和技术路径决定的。互联网上的英文文本远多于中文，模型"学"出来的切法，自然更照顾英文。未来会不会变？有可能——随着中文语料和中文用户的重要性提升，会有更多模型针对中文优化。但在那之前，知道这个差异，就能少踩坑。

---

## 二、Token 不是"字"也不是"词"，是模型"看得懂"的最小单位

那 token 到底是什么？

**Token 既不是字，也不是词，而是模型"看得懂"的最小单位**。你可以把它想象成一块蛋糕被切成的每一小块。整段文字是一块完整的蛋糕，**tokenization**（分词）就是按某种规则把它切成小块的过程。

有的切法细——一块很小，模型看得更精细；有的切法粗——一块很大，算得快但可能漏细节。GPT 用的是一种叫 **BPE** 的切法，在细和粗之间找平衡。为什么叫"最小单位"？因为模型只能识别这些小块，不能识别更细的东西。你输入的文字，必须先被切成 token，模型才能处理。

对中文来说，一个 token 可能是一个字，也可能是半个词、一个词、甚至一个标点。对英文来说，"apple" 可能是一个 token，"unbelievable" 可能被拆成 "un"、"believ"、"able" 三个。所以当你问"为什么我输入 100 个字，token 数不是 100？"——因为 token 不是按字算的，是按模型的分词规则算的。

**意外发现**：标点符号也占 token。一个句号、一个逗号，都可能单独成块。所以"少用废话"不只是文风问题，更是成本问题——每多一个"嗯"、"啊"、"那个"，都在给账单加码。写 prompt 的时候，能一句话说清的，别拆成三句。

---

## 三、BPE：AI 的"切词"算法，像在玩一种特殊的拼图

那模型到底怎么切的？用的是 **BPE**（Byte Pair Encoding，字节对编码）。

想象你在玩一种特殊的拼图游戏。一开始，你手里只有最基础的碎片——可能是单个字符，比如 "a"、"b"、"你"、"好"。然后你从海量文本里统计：哪两块碎片最经常挨在一起？统计出来之后，就把这俩粘成一块新的"大碎片"，加入你的拼图盒。下次再统计，再粘……如此反复，拼图盒越来越大，但每块都是"高频组合"。

比如英文里 "th" 和 "e" 经常连在一起，BPE 就会把 "the" 粘成一块；"un" 和 "believable" 经常一起出现，就粘成 "unbelievable"。中文同理——"中国"、"人民"、"互联网" 这种高频词，往往会被粘成一块 token；而"魑魅魍魉"这种生僻字，可能一个字就是一个 token。

所以 BPE 的本质是：**让模型用最少的"块"来表达最多的信息**。高频组合粘一块，生僻的保持原样。这样既控制了词表大小，又让常见文本的 token 数尽可能少。这是一场在"表达力"和"效率"之间的精妙平衡。理解了这一点，你就理解了为什么不同语言、不同文本，token 数会有那么大差异。

**意外发现**：为啥不直接按"词"切？因为词太多了。中文的词理论上无穷无尽，新词、网络用语、专业术语层出不穷，全塞进词表不现实。BPE 的聪明之处在于：**用数据驱动的方式，把文本切成"对模型来说最省事"的小块**。既不是纯按字切（那样太碎，序列太长），也不是纯按词切（词表会爆炸），而是取一个折中。

你可以自己试试：打开一个在线的 token 计数器，输入同样的中文和英文句子，对比一下数字。那种"明明我写得少，怎么反而贵"的憋屈感，会瞬间找到答案。数据不会骗人。

## 四、中文为什么"吃亏"？具体数据告诉你

我们做个更具体的对比。

一段 500 字的中文文章，可能轻松破 1000 token；同样信息量的英文，可能只要 600 到 700。也就是说，**中文的 token 效率大约是英文的 60% 到 70%**。你发 1000 个中文字，可能相当于发了 1500 个英文单词的"计价单位"。

有没有针对中文优化的方案？有。有些模型会专门在中文语料上做 BPE 训练，把"人工智能"、"机器学习"、"深度学习"这类词粘成一块，能明显提升中文的 token 效率。国产大模型在这方面往往做得更好。但整体上，目前主流模型的词表还是英文主导，中文用户确实要"多付一点"。

**金句**：Token 不是按字收费，而是按"模型看得懂的小块"收费。小块切得越碎，数量越多，账单越厚。

再举个更狠的例子：一篇 3000 字的深度报道，中文可能破 4000 token；同样信息量的英文，可能 2500 就够。做产品时，如果你的用户主要是中文用户，成本预算得比别人多留 30% 到 50% 的余量。

## 五、上下文窗口：AI 的"工作记忆"有多大？

说到 token，就绕不开 **上下文窗口**（Context Window）。

你可以把它想象成 AI 的"书桌"。书桌有多大，它一次能"摊开"多少内容来看？8K token 的书桌，大概能放几千个英文单词；128K token 的书桌，能放一整本中篇小说。你输入的问题、系统提示词、历史对话、检索到的文档……全都要摆在这张书桌上。超了怎么办？要么截断（把最早的内容扔掉），要么报错。

**为什么聊久了 AI 会"忘事"？** 因为窗口有限。你跟它聊了 20 轮，每轮都带历史，上下文可能已经堆到几万 token。新问题进来的时候，最早几轮的内容可能已经被"挤出去"了——不是模型不努力记，是书桌放不下了。所以长对话里，AI 容易忘记你一开始说的"请用第三人称"、"请控制在 500 字以内"这类早期指令。

**为什么 RAG 要把文档切块？** 也是因为窗口。一篇 10 万字的 PDF，不可能整篇塞进 prompt。RAG 的做法是：先把文档切成小块（比如每块 500 字），存起来；用户提问时，只把"最相关"的几块检索出来塞进上下文。这样既能利用长文档，又不会把窗口撑爆。切块的大小，本质上就是在平衡"信息完整性"和"token 预算"。

块切得太小，可能把一句完整的话拆散，模型看不懂；切得太大，一块就占几千 token，塞不了几块就满了。所以做 RAG 的团队，往往要反复调"块大小"和"重叠长度"——这是 token 预算和检索效果之间的博弈。

**意外发现**：窗口大不等于"记得牢"。就像你的书桌能摊开很多书，但你不一定每本都认真看。模型也一样，上下文太长的时候，中间部分容易被"忽略"，效果反而可能变差。所以别盲目追求 128K，够用就行。

实战建议：如果你发现 AI 在长对话里开始"答非所问"或"忘掉你之前的设定"，别怪它笨——先看看是不是该开新对话了。适时用一句"总结一下我们刚才讨论的要点"把前面的内容压缩，再开新局，往往比硬撑着聊下去更省 token、效果更好。

## 六、一个真实的 API 调用，算算一天要花多少钱

API 收费，基本都是按 token 算的。但有个细节很多人忽略：**输出 token 比输入 token 贵**。为啥？因为生成比读取难。每生成一个 token，模型都要做一次完整的推理，所以输出更耗算力。

我们算一笔具体的账。以某主流 API 为例：输入 1 元/百万 token，输出 3 元/百万 token。假设你做一个"文档问答"产品，用户上传 PDF 后提问：

- 每次请求：用户问题 200 token + 检索到的 3 段文档 3000 token + 系统提示词 500 token = 输入 3700 token
- 模型回复：平均 800 token
- 单次成本：3700×1 + 800×3 = 6100/百万 ≈ 0.006 元

看起来每次才半分钱，但架不住量大。一天 1 万次请求，就是 60 块。一个月 1800 块。如果用户问题更长、文档塞得更多、回复更长，成本轻松翻倍。**省 token 就是省钱**——少写废话、少塞无关的上下文、控制输出长度，都能直接反映在账单上。

再算一个更简单的场景：日常聊天。你问一个问题 200 token，模型回 500 token，单次成本 0.00017 元。一天聊 100 轮，也就 1.7 块。但如果你每次都带着 10 轮历史（每轮 500 token），输入就变成 5000+ 了，单次成本直接翻 10 倍。所以**带历史上下文是成本的大头**，能省就省。

## 七、Token 经济学：为什么 OpenAI 不停降价？

过去两年，GPT 的 API 价格降了好几轮。2022 年 GPT-3.5 刚出来时，百万 token 要几美金；现在同样规格，可能只要几毛钱。**Token 成本下降的速度，比大多数人想象得快**。

为什么？算力在变便宜，模型在变高效，竞争在加剧。OpenAI、Google、Anthropic、国内大厂……都在抢开发者。每一轮降价，都是在用"更少的钱处理更多的 token"来换市场份额。对开发者来说，这是好事——同样的预算，能调更多次 API、处理更多用户。但也要警惕：**成本降了，用量可能涨得更快**。以前不敢做的"每次请求塞 10 篇文档"的功能，现在敢做了，总账单可能不降反升。

**金句**：Token 经济学告诉我们——便宜不是省钱的理由，控制用量才是。

有业内人士预测：未来两三年，token 单价可能再降一个数量级。到那时，真正制约你的可能不是"能不能调得起 API"，而是"模型能不能在合理时间内算完"。成本在降，但**用量和效率的博弈，会一直存在**。提前理解 token，就是提前理解这场博弈的规则。

## 八、聊得越久，AI 回得越慢——不是 bug，是数学

你可能有这种感觉：聊得越久，AI 回得越慢。

因为模型是**自回归**的——每生成一个 token，都要把"当前已有的全部 token"再算一遍。你发了 10 轮对话，上下文可能有几千 token；模型生成第一个字的时候，要处理这几千 token；生成第一百个字的时候，要处理几千+100……**序列越长，单步计算量越大**，所以越往后越慢。

这就像你写作文。写第一句的时候，脑子里只有题目；写第十句的时候，你得不断回头看前面九句。句子越多，每次"回头看"的负担越重。大模型也一样，只不过它"回头看"的方式是数学运算——每个 token 都要参与注意力计算，序列长度一上去，计算量呈平方级增长。所以同样的模型，处理 1K token 和 10K token，速度可能差一个数量级。

想提速？要么清空历史重新开聊，要么用摘要把长对话压缩成短上下文。做产品的时候，如果发现"聊久了就卡"，别急着怪服务器——先看看是不是上下文堆太长了。

**金句**：AI 的"慢"不是服务器的问题，是数学的问题。你喂给它的上下文越长，它每一步要算的东西就越多。这是架构决定的，不是优化能解决的。

## 九、省 Token 的实用技巧

说了这么多，最后给点实在的。

**第一，压缩 prompt。** 别把整篇文档原封不动塞进去，先做摘要、抽关键句，或者只塞相关段落。系统提示词能短就短，每多一个 token，每次请求都要多传一遍。一个 500 字的 system prompt，每次请求都要付 500 个输入 token 的钱——如果一天调 1 万次，就是 500 万 token 花在重复的内容上。

**第二，控制输出长度。** 在 API 里设置 `max_tokens`，别让模型自由发挥写小作文。需要表格就让它输出表格，需要列表就让它输出列表，格式越紧凑，token 越少。

**第三，善用缓存。** 有些 API 支持 prompt caching——如果系统提示词或前缀不变，可以只算一次，后续请求复用。长文档、长指令，能缓存就缓存。这对 RAG 类应用尤其重要：同一批文档被反复检索时，缓存能省下一大笔输入 token 的钱。

**第四，考虑用英文。** 如果场景允许，用英文写 prompt 和期望输出，往往能省 30% 到 50% 的 token。对成本敏感的项目，这点很关键。

**第五，定期清空上下文。** 对话太长不仅慢，还容易"遗忘"早期的重点。适时开新对话，或者用一句总结把前面的内容"压缩"进新问题里，既能省 token，也能保持模型"脑子清醒"。

**第六，选对模型。** 不同模型的 token 定价差很多。如果任务简单，用便宜的小模型可能够用；如果必须用大模型，就尽量把 prompt 设计得紧凑，别浪费。

**第七，审计你的 prompt。** 很多团队做成本优化，第一件事就是审计：哪些 prompt 写得太啰嗦？哪些地方塞了没必要的历史？系统提示词里有没有可以删掉的"示例"？一压缩，成本能降一大截。这是最容易被忽视、也最容易见效的一步。

**金句**：Token 优化的本质，是让每一块"小蛋糕"都物有所值。不浪费，就是省钱。

---

## 结尾

Token 不是什么玄乎的东西，就是模型处理文本的最小单位。理解它，你就能看懂账单、理解速度、知道怎么省钱。

下次再看到"消耗了 50 万 token"，你心里就有数了——那不是 50 万个字，而是模型按自己的规则"切"出来的 50 万块小蛋糕。切法不同，数量就不同；数量不同，成本和速度就不同。

**知道规则的人，才能在不公平的棋盘上，找到自己的活法。**
