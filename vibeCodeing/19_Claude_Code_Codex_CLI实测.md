# Claude Code / Codex CLI 实测：终端里的 AI 编程

有个做运维的朋友，上个月跟我吐槽：他平时在服务器上改配置、写脚本，根本用不上 Cursor——那玩意儿得开个 IDE、连上项目，他 SSH 进机器就一个黑乎乎的终端，能咋办？我说那你试试终端里的 AI 呗。他愣了：终端里也能用 AI 写代码？

能。而且不止一家在做。Anthropic 出了 Claude Code，OpenAI 出了 Codex CLI，都是**不依赖 IDE、直接在终端里干活**的 AI 编程工具。装好之后，在项目目录敲一条命令，就能跟 AI 一起读代码、改文件、跑测试，全程不用开网页、不用复制粘贴。

这篇是咱们 Vibe Coding 系列的第 19 篇，专门聊聊这两个"终端派"选手：它们是什么、怎么用、跟 Cursor 那种 IDE 方案有啥区别，以及——你到底该选谁。

---

## 一、为什么有人不用 Cursor，偏选终端？

这事儿得用吃饭来打比方。

**去餐厅吃饭**：环境好、服务全、菜单丰富，但你得出门、得等位、得按人家的节奏来。对应 IDE 里的 AI 编程——Cursor、Windsurf、Copilot，功能齐全，但你得在编辑器里干活，项目得在本地、得能打开。

**叫外卖**：在家就能吃，但送来的东西你没法现场改。对应那种"把代码贴给 ChatGPT、再复制回来"的方式——灵活是灵活，但来回粘贴、上下文容易断。

**自己做**：厨房就是你的地盘，想加啥料加啥料，但前提是你得在厨房。对应**终端里的 AI**——你 SSH 进服务器、在 CI 流水线里、在 Docker 容器里、在只有命令行没有图形界面的环境里，照样能让 AI 帮你写代码、改配置、跑命令。

所以选终端的，往往不是"不喜欢 Cursor"，而是**场景不允许**。服务器运维、CI/CD 管道里的自动化、大型项目的批量重构、远程开发环境——这些地方，IDE 要么进不去，要么太重。终端轻、快、哪儿都能跑，才是刚需。

## 二、Claude Code 是什么？

Claude Code 是 Anthropic 官方的 AI 编程工具，**核心卖点**：不绑 IDE，直接在终端里读代码、改文件、跑命令，一条龙。

装完以后，在项目目录下敲 `claude`，就进交互模式了。你跟它说"给用户模块加个分页"，它会自己打开相关文件、理解结构、改代码、跑测试。你不需要复制粘贴，不需要告诉它"去第三个文件看看"——它直接读写你的文件系统，直接执行你允许的命令。

**几个核心特点**：

直接操作文件系统。不像那种"生成代码让你复制"的聊天机器人，Claude Code 是真的在改你的文件。你看着它 diff，确认了就应用，不确认就回滚。

能跑命令。你说"启动一下开发服务器看看有没有报错"，它真的会跑 `npm run dev`，然后把输出给你看。修 bug 的时候，它能自己复现问题、自己验证修复。

无 IDE 依赖。你可以在 VS Code 里用它的插件，也可以在纯终端里用。SSH 到服务器、在 CI 里、在 WSL 里，只要有个终端，就能用。

支持 MCP（Model Context Protocol）。可以接第三方工具、自定义工作流，扩展性不错。

**适合谁**：已经在用 Claude、喜欢 Anthropic 那套"安全可控"风格的人；或者需要在非 IDE 环境（服务器、CI、远程开发）里用 AI 编程的人。

安装也简单：`curl -fsSL https://claude.ai/install.sh | bash`（macOS/Linux），或者 `brew install --cask claude-code`。装完进项目目录敲 `claude` 就行，不用配 IDE、不用开网页。

## 三、Codex CLI 是什么？

Codex CLI 是 OpenAI 的终端方案，用 Rust 写的，主打**沙箱执行**和**安全模式**。

跟 Claude Code 一样，你进项目目录敲 `codex`，就进了一个全屏的终端 UI。可以对话、可以看 diff、可以批准或拒绝每一步操作。但 Codex 多了一层**权限控制**：默认是 Auto 模式——在工作目录内读文件、改文件、跑命令可以自动做，但动到工作目录外的东西、或者要联网，会先问你。

还有 Read-only 模式：只读不写，纯当顾问，适合"先让它分析，我再决定怎么改"。

以及 Full Access 模式：放开了干，适合你完全信任当前任务的时候。这种分级，对生产环境、对敏感项目，挺实用。

**几个核心特点**：

沙箱与审批。每一步操作都可以配置成"先展示计划，你批准再执行"。不会出现"AI 一激动把你整个 home 目录删了"这种惨剧。

非交互式 `exec`。除了交互模式，还支持 `codex exec "fix the CI failure"` 这种一条命令搞定。可以写进脚本、接 CI/CD，自动化程度高。

内置代码审查。敲 `/review` 可以针对当前 diff、某个 commit、或者未提交的改动做审查，不直接改代码，只给建议。适合提交前自查。

支持 MCP、多 agent 协作（实验性）、图片输入（截图、设计稿）。功能矩阵很全，想扩展的话空间不小。

**适合谁**：已经在用 ChatGPT Plus/Pro、想在同一生态里搞定终端编程的人；或者对"AI 乱改文件"有顾虑、想要审批流程的人。

安装方式：`npm install -g @openai/codex` 或 `brew install codex`。随 ChatGPT Plus、Pro、Business 等订阅一起用，不单独收费。

## 四、实测对比：同一个任务，两个工具的表现

我拿同一个任务测了一轮：**给一个现有的小型 Web 项目加一个"导出为 CSV"的功能**。项目有前端列表页、后端 API、数据库，需要新增一个导出接口、一个导出按钮，并处理权限。这个任务不算复杂，但能考验工具对项目结构的理解、多文件联动的能力、以及能不能听懂"别写本地文件"这种约束。

**Claude Code**：进项目目录，`claude` 启动，直接说"给用户列表加个导出 CSV 功能"。它先扫了一圈代码结构，然后列了个计划：1）后端加导出接口 2）前端加导出按钮 3）鉴权沿用现有逻辑。我点头，它开始改。改的过程中，有一处它想用 `fs` 写临时文件，我提醒"我们这是 API，别写本地文件"，它立刻改成流式返回。整体节奏顺畅，理解项目结构很快，改完还自己跑了 `npm test` 验证。

**Codex CLI**：同样进目录，`codex` 启动，同样的需求。它也是先分析、再出计划，但多了"每步都要我点批准"的环节。第一次跑的时候我选了 Auto，它自动改；后来我故意换成需要审批的模式，它就每一步都停一下，等我确认。`exec` 模式我也试了：`codex exec "add CSV export to user list"`，非交互跑完，把结果打到 stdout，适合写进脚本。

**对比小结**：两者都能完成这个任务，质量差不多。Claude Code 的交互更"一气呵成"，少点确认、多点信任；Codex 的审批机制更细，适合不想放权的场景。Codex 的 `exec` 在自动化、CI 集成上更顺手；Claude Code 对项目结构的理解速度略快一点，可能跟模型有关。整体来说，**日常开发两者都能打，差异主要在风格和生态**。

有个小细节值得提：Codex 支持 `/review` 做提交前审查，不直接改代码，只给建议。我试了试，对"有没有漏掉边界条件"、"这段逻辑会不会有性能问题"这类问题，反馈挺实用。Claude Code 目前没有内置的 review 模式，但你可以口头让它"审查一下这段改动"，效果类似，只是没那么结构化。

## 五、和 IDE 方案的定位差异：什么场景用终端更合适？

Cursor、Copilot 这类 IDE 方案，优势是**和写代码的流程深度绑定**：你边写边补全、边改边问，上下文天然在编辑器里。适合从零开发、日常迭代、需要频繁看多文件、做重构的场景。

终端方案的优势是**轻、可脚本化、不挑环境**。下面这些场景，用终端更合适：

**服务器运维**：SSH 上去改 nginx 配置、写 cron 脚本、调 systemd 服务。没有 IDE，只有终端，Claude Code 或 Codex 就能直接上。

**CI/CD 管道**：在流水线里自动修失败用例、更新 changelog、做简单的代码检查。`codex exec` 这类非交互模式，接脚本很方便。

**大型项目批量重构**：改几百个文件的命名、统一格式、升级依赖。在 IDE 里容易卡，终端里跑，资源占用更可控。

**远程开发 / 容器内开发**：只有命令行、没有图形界面的时候，终端 AI 是唯一选择。

所以不是"终端更好"或"IDE 更好"，而是**场景决定工具**。你 80% 的时间在本地 IDE 里写业务代码，Cursor 就够；你经常要进服务器、搞 CI、做批量操作，终端方案就该备一个。

还有个现实因素：很多公司的开发机权限管得严，装不了完整的 IDE，或者只能用 Web IDE、远程桌面。这种时候，终端 + AI 往往是唯一能用的组合。轻量、不挑环境、能脚本化，这三个特点凑在一起，终端方案的价值就出来了。

## 六、优缺点总结

| 维度 | Claude Code | Codex CLI |
|------|-------------|-----------|
| **上下文理解** | 强，扫项目快，多文件联动顺畅 | 强，计划清晰，步骤可追溯 |
| **执行安全性** | 直接操作，依赖你本机权限 | 沙箱 + 审批模式，可控性更高 |
| **上手门槛** | 低，会敲命令就行 | 低，交互模式友好，exec 需一点脚本基础 |
| **自动化 / CI** | 支持，但 exec 不如 Codex 成熟 | 强，`exec` 设计完善，适合管道 |
| **价格** | 需 Claude 订阅或 Anthropic 账号 | 随 ChatGPT Plus/Pro 等，有订阅就有 |
| **生态** | Anthropic 系，MCP 支持好 | OpenAI 系，和 ChatGPT 打通 |

**Claude Code 的短板**：审批粒度不如 Codex 细，想要"每步都问我"得自己兜着；非交互的脚本化能力还在完善。

**Codex CLI 的短板**：依赖 OpenAI 生态，如果你主力用 Claude，会有割裂感；Rust 写的虽然快，但某些环境下的安装偶尔会踩坑。

两者都支持 MCP，可以接数据库、API、自定义工具，扩展空间都不小。如果你已经在用 Cursor 的 MCP 配置，迁移到终端方案时，很多配置可以复用。

## 七、怎么选：一张决策树

- **你 90% 的时间在 IDE 里写代码**：继续用 Cursor/Copilot，终端方案当备选即可。
- **你经常 SSH 进服务器、搞运维**：两个都试试，看哪个跟你的工作流更合拍。倾向审批流程选 Codex，倾向一气呵成选 Claude Code。
- **你要把 AI 接进 CI/CD**：优先 Codex，`exec` 模式成熟，文档全。
- **你已经在 ChatGPT 订阅里**：Codex 几乎零额外成本，直接装。
- **你已经在 Claude 订阅里**：Claude Code 更顺手，生态统一。
- **你对"AI 直接改文件"不放心**：Codex 的 Read-only、审批模式更适合你。

一句话：**IDE 方案解决"写代码时的效率"，终端方案解决"写代码之外、但还是要动代码"的效率**。一个管"写"，一个管"改"和"跑"，分工明确。两者不互斥，很多人的最佳组合是：日常 Cursor，关键时刻掏终端。

如果你两个都没用过，建议先装一个试试。选哪个？看你手头已有的订阅：有 Claude 就试 Claude Code，有 ChatGPT Plus 就试 Codex。用一周，感受一下"在终端里跟 AI 一起改代码"的节奏，再决定要不要长期用。很多人的反馈是：一旦习惯，就回不去了——尤其是那些经常跟服务器、跟 CI 打交道的人。

---

## 八、结尾

我那个做运维的朋友，后来装了 Codex，在服务器上改配置、写脚本的时候，终于不用再把代码复制到网页里再贴回来了。他说，终端里有个能读能改能跑的 AI，感觉像多了个随叫随到的同事，而且不占工位。

AI 编程的战场，早就从"哪个 IDE 最强"扩展到了"在什么环境里、用什么方式跟 AI 协作"。终端里的 Claude Code 和 Codex CLI，把这条线补上了。下一篇，咱们聊另一个很多人忽视的话题：**AI 写的代码，怎么测才放心**——测试策略、回归范围、以及怎么不让 AI 帮你写出一堆"能跑但不敢上线"的玩意儿。
