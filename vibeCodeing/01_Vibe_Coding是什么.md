# 不写一行代码，30 分钟做出一个 App？聊聊最近爆火的 Vibe Coding

上个月一个做前端的朋友跟我吐槽，说他花了整整两天，写了一个简单的待办事项小工具——从搭项目、写组件、调样式到处理各种小 bug，两天时间搭进去了。

结果呢？隔壁工位的同事，打开一个叫 Cursor 的编辑器，对着 AI 说了半小时需求，一个功能差不多的成品就出来了。

他凑过去看了一眼，发现对方几乎没怎么敲代码。整个过程就是：**说一句需求 → 点一下"接受" → 跑一下看效果 → 再说一句**。循环往复，程序就长出来了。

他当时脑子里就一个念头：**我苦学两年的编程技术，是不是白学了？**

这种焦虑你可能也有。但更魔幻的是——连 OpenAI 的联合创始人、前特斯拉 AI 总监 Andrej Karpathy，也在用这种方式写代码。他还给这种写法起了个名字：**Vibe Coding**，直译过来叫"氛围编程"。

这名字听着就挺玄乎。今天咱们就掰开了聊聊：这到底是个什么东西？为什么突然火了？跟普通人有什么关系？

---

## 一、这个词是谁发明的？

2025 年 2 月，Karpathy 在 X（原推特）上发了一条帖子，把这种开发方式正式"命名"了。

他说了什么呢？我给你翻译一下核心意思：

> 我发现了一种新的编程方式，叫"Vibe Coding"。你完全沉浸在氛围里，甚至忘了代码的存在。我现在连键盘都懒得碰，直接用语音跟 AI 说话。永远点"全部接受"，不再一行行看代码改了什么。报错了？直接把错误信息复制粘贴给 AI，通常它自己就能修好。修不好就绕过去。代码量早就超出我能理解的范围了。严格来说这不算在"写代码"——就是看看、说说、跑跑、复制粘贴，大部分时候居然能跑。

你品品这段话。一个 AI 领域的顶级大佬，公开说自己"不看代码、不碰键盘、报错就粘贴"。这要是放在五年前，程序员圈子得炸锅。

但他说了一句更狠的总结：**"看、说、跑、粘贴，基本就能跑。"**

其实早在 2023 年，Karpathy 就放过一句话："最热门的编程语言是英语。" 当时很多人觉得是玩笑。两年后回头看——这句话成了预言。

---

## 二、所以 Vibe Coding 到底是什么？

一句话解释：**你用人话告诉 AI 你想要什么，AI 帮你把代码写出来，你只管看效果、提意见。**

听着是不是太简单了？我们用两个类比来感受一下。

**类比一：打车 vs 自己开车。**

传统编程就像自己开车。你得会踩油门、打方向盘、看导航、应对各种路况，每一步都是你在操作。

Vibe Coding 就像打车。你上车跟司机说"去三里屯那家海底捞"，然后你就坐着了。路线对不对你瞄一眼，到了没你看一下，但方向盘你是不碰的。

**类比二：装修。**

传统编程是你自己当设计师+工人——画图纸、选材料、搬砖砌墙，全程亲力亲为。

Vibe Coding 是你当业主——你说"我要一个采光好的客厅、开放式厨房、北欧风"，设计师出方案，你看效果图，满意就过，不满意就改。你不需要知道承重墙怎么算，你只需要知道自己想住什么样的房子。

**核心区别就一个：**

- 传统编程的核心能力是"**理解代码**"——你得知道每一行在干什么
- Vibe Coding 的核心能力是"**描述意图**"——你得能说清楚你要什么

可能有人会问：那如果我让 AI 写代码，但每一行我都仔细看了、都理解了，算不算 Vibe Coding？

还真不算。那只是把 AI 当了一个打字更快的助手。**真正的 Vibe Coding 是：你不逐行看代码，靠运行结果和反复对话来推进。** 代码在后台"生长"，你更像是在跟一个会写代码的搭档聊天。

## 三、为什么 2025 年才火？

"用人话指挥 AI 写代码"这个想法，其实 2023 年就有人在尝试了。但为什么到 2025 年初才真正出圈？

因为三个条件同时成熟了。

**第一，AI 变聪明了。**

2023 年的 AI 写代码，就像一个刚学会做菜的新手——简单的炒蛋没问题，但你让他做一桌年夜饭，大概率翻车。到了 2024 年底、2025 年初，新一代 AI 模型在写代码这件事上明显上了一个大台阶。不光能写单个功能，还能理解你整个项目的上下文，记住你之前说过的需求，多轮对话也不会"忘事儿"。

用 Karpathy 的话说：AI 好到你可以放心点"全部接受"，项目也不会崩。

**第二，工具变顺手了。**

以前用 AI 写代码，你得这样操作：打开 ChatGPT → 把需求打进去 → 等它生成代码 → 复制 → 粘贴到你的编辑器里 → 跑一下 → 报错了再复制错误信息回去……来回折腾，体验非常割裂。

现在呢？Cursor、Windsurf 这些新一代编程工具，把 AI 直接"装"进了编辑器里。你在编辑器里说一句"把这个按钮改成蓝色"，AI 改完直接显示在代码里，你点一下"接受"就生效了。整个过程丝滑得像跟同事对话。

**第三，价格降下来了。**

早期用 AI 写代码，按使用量收费，聊一下午可能要花几十上百块。现在各种模型竞争激烈，价格大幅下降，普通人也能负担得起"跟 AI 聊一下午"的成本。

**所以 Vibe Coding 不是突然发明的，是时机到了。** AI 够聪明、工具够顺手、价格够便宜——三者叠加，这种开发方式才从极客的玩具变成了普通人的工具。

## 四、来，感受一下 Vibe Coding 的过程

光说概念太抽象，我们来走一遍真实场景。

假设你想做一个"每日一句"的小网页：打开就显示一句励志语录，点一下按钮就换一句。很简单的需求对吧？

**如果用传统方式**，你得：打开编辑器 → 新建好几个文件 → 写页面结构 → 写样式 → 写逻辑 → 查资料 → 调格式 → 修 bug……折腾一两个小时是常事。

**如果用 Vibe Coding**，整个过程是这样的：

你打开 Cursor，直接跟 AI 说：

> "帮我做一个简单的网页，打开就显示一句励志语录，下面有个按钮，点一下就换一句。风格要简洁，白底黑字，有点留白的感觉。"

十几秒后，AI 生成了一版代码。你点"接受"，运行一下——页面出来了，但按钮太小了。你接着说：

> "按钮太小了，做大一点，颜色改成深蓝色。"

AI 改完，你再看——嗯，按钮好了，但字体太普通。你又说：

> "换个更有设计感的字体，语录文字再大一号。"

就这样，**说一句、看一下、再说一句**，三五轮下来，一个像模像样的页面就出来了。全程你没写一行代码，做的事情就是"提需求"和"看效果"。

如果中间报错了怎么办？Karpathy 的做法简单粗暴：**直接把报错信息复制粘贴给 AI，一个字都不解释。** AI 通常能根据错误信息自己修好。修不好就换个说法再试，或者暂时绕过去——反正对一个小项目来说，能跑就行。

这就是 Vibe Coding 的日常：**你负责"我要什么"，AI 负责"怎么实现"。**

## 五、Vibe Coding 不是非黑即白，它有三个"段位"

很多人以为 Vibe Coding 就是"完全不写代码"。其实不是，它更像一个光谱，从浅到深可以分成三个段位：

**青铜段位：AI 帮你补全。** 你还是在写代码，但 AI 会猜你下一行想写什么，自动补全。就像手机打字时的联想输入——你打"今天天"，它自动建议"气不错"。目前最常见的就是 GitHub Copilot 的代码补全功能。

**白银段位：你说需求，AI 写代码。** 你不再一行行写了，而是用对话的方式描述"我要什么"，AI 生成整块代码，你看看对不对，不对就让它改。Cursor 和 Windsurf 就是这个段位的代表工具。

**王者段位：你只管验收。** 你给一个大目标，比如"帮我做一个记账 App"，AI 自己拆解任务、创建文件、写逻辑。你主要就是运行、看效果、提反馈。Karpathy 说的"我不再看代码改了什么"，就是这个段位。

大多数人目前在青铜到白银之间。但随着 AI 越来越强，王者段位正在变得越来越可行。

## 六、谁在用？不只是程序员

你可能觉得 Vibe Coding 是程序员的事。还真不是。

**程序员在用它提效。** 那些重复性的、模板化的代码交给 AI 写，自己把精力省下来做更有价值的事——比如想清楚产品逻辑、设计系统架构。就像有了洗衣机你不会再手洗衣服，但你还是得决定穿什么。

**不会编程的人在用它做产品。** 《纽约时报》的科技记者 Kevin Roose 用 Vibe Coding 做了好几个小应用，他管这叫"一个人的软件"——以前你得雇一个开发团队才能实现的想法，现在你一个人跟 AI 聊聊就能做出来。

**创业者在用它快速试错。** 硅谷最知名的创业孵化器 Y Combinator，2025 年冬季批次里，**25% 的创业公司代码有 95% 是 AI 写的**。先用 AI 快速搭一个能跑的原型，验证想法靠不靠谱，再决定要不要投入人力做"正经"开发。

所以 Vibe Coding 不只是"程序员偷懒的新姿势"，它真正在做的事情是——**降低"把想法变成软件"的门槛**。以前这道门槛叫"会写代码"，现在这道门槛变成了"会说清楚你要什么"。

## 七、先别太兴奋——Vibe Coding 不是万能的

聊了这么多好处，得泼点冷水了。Karpathy 自己也说了，这种方式适合"周末随手做的小项目"。对于真正重要的、长期维护的系统，Vibe Coding 有明显的边界。

**第一个坑：代码质量没保障。** AI 写的代码，表面上能跑，但里面可能藏着各种问题——多余的代码、不合理的结构、隐藏的 bug。你不逐行检查就不知道。就像你请了一个装修队，房子看着漂亮，但水管接得对不对、电线走得安不安全，你不拆开看就不知道。

**第二个坑：项目大了就失控。** 小项目没问题，但如果代码越来越多，你又不了解里面的结构，将来要改功能、修问题，就会陷入"改一个地方，另一个地方崩了"的窘境。

**第三个坑：安全问题。** 有研究显示，AI 写的代码出现安全漏洞的比例更高。如果你做的是涉及用户隐私、支付、敏感数据的产品，这个风险不能忽视。

**所以一句话总结：Vibe Coding 适合做原型、个人项目、一次性小工具。真要做正经产品上线，还是需要有人懂代码、做把关。**

具体有哪些坑、怎么避，我们后续文章会专门展开聊。

## 写在最后

Vibe Coding 本质上是在回答一个问题：**当 AI 能听懂人话、能写出能跑的代码时，"编程"这件事会变成什么样？**

答案正在浮现——编程正在从"写代码"变成"描述目标 + 验收结果"。你不再需要精通编程语法，但你需要一种新能力：**会说清楚你要什么、会判断结果好不好、会一步步迭代到满意。**

这其实是一件挺激动人心的事。以前"把想法变成软件"是少数人的特权，现在这个能力正在扩散到更多人手里。

下次你看到有人对着屏幕说话、很少敲键盘却在"写程序"，别惊讶——他可能正在 Vibe Coding。
