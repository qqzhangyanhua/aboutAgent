# 同一个工具，效率差 10 倍？关键在你怎么跟 AI 说话

上周有个读者私信我，说他跟同事都用 Cursor 做同一个需求——给后台管理系统加个导出 Excel 的功能。

同事半小时搞定了。他折腾了一下午，AI 要么生成的是错的库、要么格式不对、要么直接报错。他纳闷：同一个工具，差距咋这么大？

我让他把对话记录发来看看。看完就明白了——问题不在 Cursor，在**他怎么跟 AI 说话的**。同事的 prompt 是："项目用 React + Ant Design，已有 Table 组件在 src/pages/OrderList，数据源是 orderList 这个 state。帮我加个导出按钮，点击后把当前表格数据导出成 xlsx，用 sheetjs 库。" 他的 prompt 是："帮我做个导出 Excel 的功能。"

同一件事，不同的说法，结果天差地别。今天咱们就聊聊：**怎么说话，AI 才能写出你想要的代码。**

---

## 一、为什么"怎么说"比"用什么工具"更重要

很多人有个误解：AI 这么聪明，我说个大概它应该能懂吧？错了。**AI 不是读心术**，它只能根据你给的信息来推理。你给得模糊，它就只能猜；你给得具体，它才能精准。

打个比方。你跟实习生说"做个登录页"，他可能给你做个纯 HTML 的、可能用 Vue、可能用 React、可能丑得没法看。你跟他说"用 React + Tailwind，要有邮箱密码输入框、记住我复选框、忘记密码链接，风格参考我们官网"，他至少知道往哪个方向使劲。AI 也一样——**你说得越清楚，它产出越靠谱。**

AI 编程的核心是"你说，AI 写"。如果你的"说"本身就不及格，那 AI 写得再卖力，也是在错误的方向上狂奔。所以，**你怎么跟 AI 说话，直接决定了效率上限**。工具再强，也救不了糟糕的需求描述。

---

## 二、Prompt 五要素：SCOPE 框架

我总结了一个 **SCOPE** 框架，五个字母对应五个要素。把这五个都照顾到，你的 prompt 质量能上一个台阶。

**S：Specific（具体）——别让 AI 猜**

你要什么，就说什么。

坏 prompt："做个登录页。"

好 prompt："用 React + Tailwind 做登录页，要有邮箱输入框、密码输入框、记住我复选框、忘记密码链接，按钮用深蓝色，整体风格简洁。"

坏 prompt："加个分页。"

好 prompt："给用户列表加前端分页，每页 10 条，用 Ant Design 的 Pagination 组件，放在表格下方。"

**C：Context（上下文）——告诉 AI 你的项目长什么样**

AI 不知道你项目长什么样。你得告诉它：用什么技术栈、已有什么代码、文件在哪。

坏 prompt："帮我写个获取用户列表的接口。"

好 prompt："项目用 Next.js 14 + App Router，API 在 app/api 下。帮我写 app/api/users/route.ts，从数据库查用户列表，支持分页参数 page 和 limit，返回 JSON。"

坏 prompt："修一下这个 bug。"

好 prompt："@OrderList.tsx 这个组件在点击导出时报错 'Cannot read property map of undefined'，数据源 orderList 可能是空数组，需要加空值判断。"

**O：Output（输出格式）——说清楚你要什么样的结果**

你要完整文件还是代码片段？要注释还是不要？要符合现有风格吗？

坏 prompt："写个工具函数。"

好 prompt："在 src/utils/formatDate.ts 里写一个 formatDate 函数，接收 Date 对象，返回 'YYYY-MM-DD' 格式字符串。用 JSDoc 注释，风格跟同目录下 formatMoney 保持一致。"

坏 prompt："加个校验。"

好 prompt："在表单提交前加邮箱格式校验，校验失败时在输入框下方显示红色错误提示，用我们现有的 FormItem 组件。"

**P：Priority（优先级）——别让 AI 摊大饼**

需求一多，AI 容易摊大饼。你得告诉它先做啥、后做啥。

坏 prompt："做个完整的用户管理模块，要有列表、搜索、新增、编辑、删除、权限控制、导出、批量操作。"

好 prompt："先做用户列表和搜索，列表用 Table，搜索支持姓名和手机号。新增编辑删除后面再说。"

坏 prompt："把这个页面优化一下。"

好 prompt："先修加载状态和空数据展示，样式美化放最后。"

**E：Example（示例）——给 AI 看参考，比说一百句都管用**

给 AI 看一个类似的参考，比你说一百句都管用。

坏 prompt："写个类似的组件。"

好 prompt："参考 @UserCard.tsx 的结构和样式，写一个 ProductCard 组件，把用户信息换成商品信息（名称、价格、图片）。"

坏 prompt："API 返回格式跟之前一样。"

好 prompt："返回格式跟 @getOrderList 这个接口一样，{ code, data, message }，data 里是数组。"

## 三、迭代式 Prompt：别指望一句话搞定

很多人幻想：我说一句，AI 给我一个完美的成品。现实是：**复杂需求几乎不可能一句话搞定**。迭代才是正道。

**第一轮：搭骨架**

先让 AI 把主流程跑通，别纠结细节。

"用 React 做一个待办事项应用，有输入框、添加按钮、列表展示、删除功能。先不管样式，能跑就行。"

**第二轮：补细节**

骨架有了，再补交互和边界情况。

"加个完成/未完成的切换，点一下可以标记完成。空列表时显示'暂无待办'。输入框为空时点击添加要提示用户。"

**第三轮：修 bug**

跑一跑，把报错和异常情况丢给 AI。

"@App.tsx 删除最后一项时报错，可能是数组越界，修一下。"

**第四轮：优化体验**

功能没问题了，再搞样式和体验。

"给完成的任务加删除线，按钮加点 hover 效果，整体风格简洁一点。"

这就是典型的**多轮迭代**。每一轮只解决一类问题，不贪多。AI 不会一次给你完美答案，但你可以通过多轮对话，一步步把它"磨"成你要的样子。

## 四、八个实战 Prompt 模板

直接套用，改改就能用。

**新建项目**

"用 [技术栈] 创建一个 [项目类型]，包含 [核心功能 1]、[功能 2]。目录结构清晰，先实现主流程，样式后面加。"

例："用 Next.js 14 + TypeScript 创建一个博客模板，包含文章列表页、文章详情页、简单的导航栏。用 MDX 解析 markdown，先不管样式。"

**修 Bug**

"@[相关文件] 在 [操作/场景] 时报错：[报错信息]。可能的原因是 [你的猜测]。请修复并加必要的空值判断。"

例："@OrderList.tsx 在点击导出按钮时报错 'xlsx is not defined'，可能是 sheetjs 没正确导入。请检查并修复。"

**重构代码**

"@[目标文件] 这段代码 [问题描述]。请重构为 [目标形态]，保持功能不变，符合 [规范/风格]。"

例："@useAuth.ts 这个认证相关的工具超过 100 行，逻辑混在一起。请拆成 useLogin、useLogout、useUser 三个小模块，保持对外接口不变。"

**写测试**

"为 @[目标文件] 写单元测试，覆盖 [核心逻辑/边界情况]。用 [测试框架]，风格跟 @[已有测试文件] 一致。"

例："为 formatDate 函数写 Jest 测试，覆盖正常日期、非法输入、边界日期。参考 @formatMoney.test.ts 的写法。"

**加功能**

"在 @[相关文件/模块] 中加 [功能描述]。使用现有的 [组件/工具]，保持 [风格/约定]。"

例："在 OrderList 页面加导出 Excel 按钮，用现有的 Button 组件，点击后导出当前表格数据，用 sheetjs，风格跟其他操作按钮一致。"

**做 UI**

"参考 @[参考组件/设计] 做 [目标组件/页面]。要求 [具体样式/交互]。用 [技术栈]。"

例："参考 @UserCard 的布局和配色，做一个 ProductCard 组件，展示商品图、名称、价格、库存。用 Tailwind，hover 时卡片轻微上浮。"

**写 API**

"在 [路径] 写 [接口名称] 接口。请求 [方法] [路径]，参数 [描述]，返回 [格式]。用 [数据库/ORM]。"

例："在 app/api/products/route.ts 写商品列表接口。GET 请求，支持 query 参数 category、page、limit。返回 { code, data: { list, total } }。用 Prisma 查数据库。"

**做优化**

"@[目标文件] 当前 [性能问题/可读性问题]。请优化 [具体方向]，不改变对外行为。"

例："@ProductList.tsx 当前一次渲染 1000 条数据会卡顿。请加虚拟滚动，用 react-window，保持筛选和排序功能正常。"

## 五、常见踩坑：这些错误别再犯了

有些坑，踩过一次就记住了。

**一口气塞太多需求**

坏："帮我做用户管理，要有列表、搜索、新增、编辑、删除、导入、导出、权限、日志、统计报表。"

好：拆成多次。"先做列表和搜索。""列表好了，加新增和编辑。""再加删除和导出。"

**描述太模糊**

坏："把这个弄好一点。""优化一下。""修一下。"

好："把加载状态从转圈改成骨架屏。""把这段重复逻辑抽成函数。""在 orderList 为空时加判断，避免 map 报错。"

**不给上下文**

坏："写个登录接口。"

好："项目用 Next.js，API 在 app/api 下，数据库用 Prisma，已有 User 模型。写 app/api/auth/login/route.ts，接收 email 和 password，校验后返回 JWT。"

**不验证就继续**

坏：AI 生成代码，看都不看直接 Accept，接着提下一个需求。结果前面的代码有 bug，后面的全建在错误基础上。

好：每轮改完，跑一下、点一下、看一眼。确认没问题再继续。有问题就先把问题修掉。

## 六、进阶技巧

**让 AI 先分析再动手**

复杂需求，先让它拆解，再一步步做。

"我要给这个表格加可编辑功能，用户可以直接在单元格里改数据。请先分析现有代码结构，列出需要改动的文件和步骤，我确认后再动手。"

**让 AI 先列计划再写代码**

大功能先要个执行计划，避免方向跑偏。

"我们要做一个小程序的后台管理系统，包含用户、订单、商品三个模块。请先给出整体架构建议和开发顺序，包括技术选型、目录结构、每个模块的核心接口。"

**分步验证**

每完成一步，就验证一步。

"先只做数据获取和展示，不做编辑。我跑一下看看数据结构对不对，对了再加编辑功能。"

## 写在最后

AI 编程的本质是**人机协作**。你负责想清楚、说清楚，AI 负责写出来。Prompt 就是你跟 AI 的接口——接口设计得好，协作顺畅；设计得烂，互相折磨。

下次跟 AI 提需求前，不妨先问自己：我有没有说清楚要什么？有没有给它足够的上下文？有没有拆成它能一步步完成的小任务？

记住：**好的 Prompt 不是越长越好，而是该有的都有，不该有的别废话。** 具体、有上下文、有优先级、能迭代——做到这几点，你的 AI 编程体验会好很多。
