# 别让 AI 一口吃个胖子——大项目要这么拆才不翻车

「帮我用 React 做一个完整的电商系统，要有商品列表、购物车、支付、用户中心、后台管理……」—— 去年有个朋友这么跟 Cursor 说了一嘴，然后盯着屏幕等奇迹。两小时后，AI 吐出了一万多行代码，他兴冲冲跑起来，发现登录页点不进去、购物车加不了商品、支付接口压根没接。更惨的是，那一万多行混在一起，他想改个按钮颜色都不知道从哪儿下手。最后他删了重来，老老实实从「先能显示一个商品」开始。

这个故事我听过太多次了。不是 AI 不行，是你让它干的事儿太大了。就像让一个刚入职的实习生一天之内搞定整个双十一大促——不是他笨，是你分配任务的方式有问题。

今天咱们聊聊：**怎么把大需求拆成 AI 能啃得动的小块**，以及一套可复用的渐进式开发工作流。

---

## 一、为什么大需求一把梭必翻车

先说清楚：不是 AI 偷懒，是它的「生理结构」决定了它干不了这事儿。

**第一，AI 的上下文窗口有限。** 哪怕 128K、200K，它一次能「认真看」的也就那么多。你让它做一个完整电商系统，它得同时考虑：前端路由、状态管理、接口设计、数据库表结构、支付流程、权限控制……这些信息塞进一个 prompt 里，它要么顾此失彼，要么干脆给你一个「大而空」的架子，细节全是错的。就像让人背一整本百科全书去考试，他只能给你背个目录。

**第二，需求越大，歧义越多。** 「商品列表」——要分页吗？每页多少条？要不要搜索？要不要筛选？要不要排序？「用户中心」——包含哪些功能？订单列表要不要？地址管理要不要？你觉得自己说清楚了，AI 可不知道你脑子里想的是哪个版本。它只能猜，猜错了就写错。大需求 = 无数个这种「你觉得不用说、它觉得得猜」的岔路口。

**第三，无法分步验证。** 一把梭出来的代码，跑不起来的时候，你根本不知道问题出在哪儿。是路由配错了？是接口没对上？是数据库连接有问题？一万行代码堆在那儿，debug 就像在迷宫里找出口。而如果你每次只做一个小功能，跑不通立刻就知道是这一步的问题，改起来也快。

所以，**大任务拆解** 不是「建议」，是「必须」。不拆，AI 再强也救不了你。

---

## 二、任务拆解的三种策略

拆法有很多，我总结三种最实用的，你可以按项目类型选，也可以混着用。

**策略一：功能维度拆分 —— 先做核心，再做辅助**

就像盖房子先打地基再装修。把需求按「没了它系统就不成立」 vs 「有了更好」来分。比如博客系统：能发文章、能看文章，这是核心；评论、标签、搜索，这是辅助。先让核心跑通，再一个个往上加。这样每一步都有「可用的东西」，不会出现「写了一堆功能但整体跑不起来」的尴尬。

**策略二：垂直切片拆分——每次做完整的一块**

这是我最推荐的方式。不是「先把所有接口写完，再写所有前端」，而是「这次把文章列表做完整」——前端列表页、后端接口、数据库查询，一口气全搞定。这样每做完一块，你都有一个**从头到尾能验证**的功能。就像吃披萨，一口咬下去从饼底到芝士到 toppings 都有，而不是先吃一盘子饼底再吃一盘子芝士。

**策略三：层级拆分 —— 先搭架子再填肉**

适合你对整体架构心里有数的情况。先建好项目结构、路由、基础组件、数据库表设计，但都是「空壳」——能跑，但没业务逻辑。然后再一层层往里填。好处是 AI 一开始就有「全局视角」，不会后面发现架构不对推倒重来。坏处是前期看不到具体效果，得耐得住寂寞。

三种策略可以组合：比如先用层级拆分搭个骨架，再用垂直切片一块块做。关键是你心里要有数：**这一步做完，我能验证什么**。

## 三、渐进式开发的工作流

光知道「要拆」还不够，得有一套可执行的工作流。我自己的流程是这样的：

**Step 1：用 AI 生成项目骨架**

第一个 prompt 不要贪多。就说：「用 React + Vite 建一个博客项目，要有路由、基础布局、一个首页。技术栈用 TypeScript，样式用 Tailwind。」跑起来，能打开页面，就算成功。这一步的目的是：**有一个能跑的起点**，后面的工作都在这个基础上长出来。

**Step 2：一块一块迭代**

每个小功能一个对话，或者一个明确的任务单元。比如：「在现有项目基础上，加一个文章列表页。从接口拉数据，展示标题、摘要、发布时间。要有加载中和空数据的提示。」做完，跑通，再下一个：「加文章详情页，点列表项跳转，根据 id 请求详情。」

**Step 3：每做完一块就验证 + 提交**

这是最容易偷懒的一步，但也是最关键的。每完成一个小功能，**立刻**跑一遍、点一遍、确认没问题，然后提交一次（用 git 的话就是 `git commit`）。不要攒着一起提交。这样有两个好处：一是出问题能立刻定位到「是上一块搞坏的」；二是你有了一串清晰的检查点，随时可以回滚。

**Step 4：在已有代码基础上继续**

每次给 AI 新任务时，**把相关文件 @ 进来**（Cursor 里可以 @ 文件名），或者明确说「在现有的 xxx 基础上改」。不要让 AI 凭空想象项目结构，它猜不准。你打开文章列表那个文件，说「在这个页面里加一个分页」，比你说「给文章列表加分页」靠谱一百倍。

---

## 四、检查点管理——给每一步买个保险

AI 会改坏代码。不是偶尔，是经常。所以你得有一套「改坏了也不怕」的机制。

**用 git 管理每个阶段。** 每完成一个可验证的里程碑，就提交一次。提交说明写清楚：「文章列表页」「文章详情页」「Markdown 编辑器」。这样你随时能一眼看出项目进展到哪了。

**AI 改坏了随时回滚。** 如果某次 AI 的修改把东西搞崩了，别跟它较劲，直接回滚到上一个版本。回滚完了再重新给一个更精确的指令，往往比让它「修好」更快。记住：你的时间比 AI 的 token 值钱。

**分支策略。** 如果项目稍微大一点，可以开个开发分支，每个小功能在单独分支上做，做完 merge 回主分支。这样主分支永远是「能跑」的状态，你随时可以切回去。用 Cursor 的话，它改的往往是当前打开的文件，所以养成习惯：改之前先看一眼当前状态，改坏了能快速恢复。

Checkpoint 的本质是：**把「可恢复」当成默认设置**。不指望 AI 一次做对，而是假设它可能做错，然后让回滚成本趋近于零。

## 五、一个实战案例：用 AI 做一个博客系统的正确姿势

拿博客系统举个完整的例子，你可以照着这个节奏来。

**阶段 1：项目初始化 + 最简页面**

Prompt：「用 Vite + React + TypeScript 创建一个博客项目，Tailwind 样式。有一个首页，显示 'Hello Blog'。配置好路由。」  
验证：`npm run dev` 能跑，浏览器能看到字。  
Commit：`init: 项目骨架`

**阶段 2：文章列表和详情**

Prompt：「加一个文章列表页，路由 /posts。用 JSON 占位数据（写死在组件里就行），展示 3 篇假文章，每篇有 title、excerpt、date。点击标题可以跳转到 /posts/:id 详情页，详情页显示完整内容。」  
验证：能点、能跳、能看。  
Commit：`feat: 文章列表和详情（假数据）`

**阶段 3：接真实数据**

Prompt：「把假数据换成从 API 获取。新建一个 api/posts.ts，用 fetch 请求 https://jsonplaceholder.typicode.com/posts，把前 10 条转成我们需要的格式。列表页和详情页都从这儿取数据。」  
验证：刷新能看到真实数据。  
Commit：`feat: 接入 API 数据`

**阶段 4：Markdown 编辑器**

Prompt：「加一个写文章页面，路由 /posts/new。用 textarea 输入，支持 Markdown。预览区域用 react-markdown 渲染。暂时不保存，刷新就没了也行。」  
验证：能输入、能预览。  
Commit：`feat: Markdown 编辑和预览`

**阶段 5：评论系统（可选）**

Prompt：「在文章详情页底部加评论列表。用公开的评论接口，根据文章 id 过滤。展示评论者名字和内容。」  
验证：详情页能看到评论。  
Commit：`feat: 评论展示`

**阶段 6：用户登录（可选，更复杂）**

这里可以再拆：先做登录页界面，再做登录接口对接，再做登录状态存储和权限校验。每一步都单独验证、单独提交。

你看，每个阶段都是「一个小目标 + 可验证 + 可回滚」。没有「一口气做完整博客」那种一步登天的幻想。

## 六、任务拆解模板（可直接使用的 prompt 模板）

最后给几个可以直接复制用的 prompt 模板，你改改就能用。

**模板 1：开新功能**

```
在现有项目基础上，实现 [具体功能描述]。

要求：
- 技术栈与项目一致（[列出关键约束]）
- 保持现有代码风格
- 不要改动无关文件

完成后需要能：[可验证的验收标准]
```

**模板 2：在已有页面上扩展**

```
打开 @[文件名]，在现有逻辑基础上 [具体改动]。

注意：
- 不要删除现有功能
- 不要大改，只做增量修改
- 改完后 [具体验证方式]
```

**模板 3：阶段性总结（给 AI 看上下文用）**

```
当前项目状态：
- 已完成：[列出已实现的功能]
- 技术栈：[React/Vue/...]
- 下一步要做：[下一个功能的简要描述]

请基于以上信息，实现 [具体任务]。
```

这些模板的核心思路都一样：**把「做什么」「做到什么程度」「怎么验证」说清楚**，别让 AI 猜。

---

## 写在最后

AI 编程最大的坑，不是 AI 写不出代码，而是我们总想让它一步到位。就像健身，你不可能一天练成八块腹肌，得一天天来。**任务拆解** 是尊重 AI 的能力边界，**渐进式开发** 是尊重你自己的认知负荷，**检查点管理** 是尊重「事情可能搞砸」这个现实。

下次再想「让 AI 帮我做个 xxx 系统」的时候，先停一秒，问自己：**我能把这个系统拆成多少个「今天就能验证」的小块？** 拆好了，再开工。别让 AI 一口吃个胖子，也别让自己一口吃个胖子。

觉得有用的话，点赞、转发、评论区聊聊你拆项目的经验。下一篇咱们聊：**两天时间，用 AI 从零做一个能上线的团队工具**——全栈实战，手把手带你走一遍。
