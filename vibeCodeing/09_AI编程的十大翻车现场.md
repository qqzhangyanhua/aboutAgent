# AI 帮你写代码的十大翻车现场（附急救手册）

你有没有过这种体验：让 AI 帮你写代码，看着它噼里啪啦输出一大段，跑起来也没报错，你心想"稳了"。

结果三天后，你发现某个边界情况直接崩了，或者逻辑压根就不对——这时候你才意识到，前面那三天等于白干。

这不是你一个人的遭遇。AI 编程就像找了个特别能说的实习生：干活快、态度好，但你不盯着，他真敢给你挖坑。今天咱们就来聊聊 AI 编程最常见的十大翻车现场，以及——怎么救。

---

## 一、「完美运行」的幻觉——看起来对，逻辑有隐蔽 bug

代码跑通了，测试用例也过了，你以为万事大吉。直到用户反馈"这个按钮点了没反应"或者"数据怎么对不上"，你才发现 AI 写的逻辑在某个分支上根本走不通。

为什么会这样？AI 是基于概率生成文本的，它擅长"看起来像代码"的代码，但不保证"逻辑正确"。就像一个人背熟了语法，不代表能写出没漏洞的程序。它容易忽略边界条件、空值处理、并发场景，这些恰恰是 bug 的高发区。

**怎么救**：把 AI 当草稿机，不当终稿机。生成代码后，**至少自己走一遍关键路径**，重点检查：空值、边界、异常分支。有条件的话，补上单元测试，尤其是边界用例。记住：能跑 ≠ 对了。

---

## 二、上下文丢失——聊着聊着 AI 忘了前面的约定

你明明说过"用 TypeScript""别用 any""接口返回格式是 xxx"，结果 AI 下一段代码又给你整出 JavaScript、满屏 any、接口结构完全对不上。你问它"我不是说了吗"，它一脸无辜。

为什么会这样？大模型的"记忆"容量再大，也有衰减。对话越长，它越容易"忘记"前面的约束。而且它没有真正的项目记忆，每次回复都是基于当前可见的内容做推理，前面的约定一旦被挤出窗口，就跟没说一样。

**怎么救**：**把关键约定写进 Prompt 开头**，每次新对话都带上。用 Cursor 的话，把项目规范、技术栈、命名约定写进 `.cursorrules` 或项目文档，让 AI 每次都能"看到"。重要决策不要只在聊天里说，要落到文件里。

---

## 三、过度工程——你要个轮子，它给你造了辆车

你只想写个简单的数据导出功能，AI 给你搭了一套抽象层、工厂模式、依赖注入，还顺带建了七八个文件。你看着那一坨架构，心想：我就想导个 CSV 啊。

为什么会这样？AI 的训练数据里，大量是"教科书式"的、追求可扩展性的代码。它没见过太多"够用就行"的小脚本，所以容易往"大而全"的方向偏。再加上它没有成本意识，不知道多写代码 = 多维护负担。

**怎么救**：**明确说"保持简单""不要过度设计"**。在 Prompt 里加一句"只实现当前需求，不要预测未来扩展"。如果它已经开始造车了，直接打断："删掉多余的抽象，只保留核心逻辑。"

---

## 四、依赖幻觉——引用不存在的库或 API

AI 写了个 import 引用某个包，你 `npm install` 一跑，发现根本没这个包。或者它调用了某个库的某个方法，你查文档发现这方法三年前就废弃了。

为什么会这样？AI 会"幻想"出合理的 API 名字和用法，因为它学的是模式，不是实时的包库。它可能把多个库的 API 混在一起，或者编造一个"听起来很对"的函数名。

**怎么救**：**生成代码后先跑一遍**，看有没有 import 报错。对不熟悉的库，顺手查一下官方文档，确认 API 是否存在、用法是否匹配。用 `package.json` 锁版本，避免"在我这儿能跑在你那儿崩"的玄学问题。

---

## 五、风格混乱——同一个项目里三种代码风格

这个文件用双引号，那个文件用单引号；这里用 function 声明，那里用箭头函数；命名一会儿驼峰一会儿下划线。整个项目读起来像三个不同的人写的。

为什么会这样？AI 没有"项目一致性"的概念，每次生成都是独立采样。不同对话、不同模型版本，可能采样出不同风格。它也不知道你项目里已有的约定。

**怎么救**：**用代码检查工具和格式化工具强制统一**（比如 ESLint、Prettier）。把格式化规则写进项目，提交前自动跑一遍。在 `.cursorrules` 里写明：缩进用 2 空格还是 4 空格、引号用单还是双、命名规范是什么。让 AI 有章可循。

---

## 六、越改越烂——AI 修 bug 引入更多 bug

你发现一个 bug，让 AI 修。它改了一处，结果别的地方崩了。你再让它修，它又改出新的问题。几轮下来，代码比原来还难搞。

为什么会这样？AI 是局部优化的高手，全局理解的高手它算不上。修一个点的时候，它可能没考虑到这个改动对上下游的影响。而且它没有"回归测试"的概念，改完就交差，不负责验证别的地方有没有被牵连。

**怎么救**：**修 bug 时尽量缩小改动范围**，能改一行就别改十行。改完之后，自己跑一遍相关功能，或者写个简单的回归用例。如果 AI 连续改坏两次，就别让它继续了，自己上手更稳。

---

## 七、安全裸奔——密钥硬编码、SQL 注入、XSS 漏洞

AI 生成的代码里，API Key 直接写在源码里；用户输入拼进数据库查询语句（容易引发 SQL 注入）；渲染网页时没做转义（容易引发 XSS 跨站脚本攻击）。你部署上线，安全扫描一跑，红了一片。

为什么会这样？训练数据里有大量"能跑就行"的示例代码，很多本身就不安全。AI 学的是功能实现，不是安全最佳实践。它也不知道你的密钥应该放环境变量，不知道用户输入永远不可信。

**怎么救**：**安全相关的代码必须人工审查**。明确告诉 AI："不要硬编码密钥，用环境变量""用户输入要做参数化查询/转义"。上线前用静态代码扫描工具扫一遍，至少把明显的漏洞拦住。

---

## 八、性能黑洞——看似能跑，数据量大时崩掉

本地测试一百条数据，一切正常。上了生产，十万条数据一查，接口超时、内存爆掉、数据库连接池耗尽。你才发现 AI 写的是全表扫描、N+1 查询（循环里一次次查库，查 N 次而不是一次批量查）、或者一次性加载全部数据到内存。

为什么会这样？AI 默认假设"数据量不大"。它生成的代码往往追求正确性，不追求性能。循环里查库、没有分页、没有索引优化——这些在它看来都是"能跑"的。

**怎么救**：**提前说明数据规模**。在 Prompt 里写"这个接口可能被高频调用""数据量可能到百万级"。生成代码后，问自己：有没有在循环里反复查库？有没有全量加载？有没有该加索引的地方？必要时做一次压测。

---

## 九、重复造轮子——项目里已有的功能 AI 又写了一遍

你们项目里明明有封装好的 request 工具、统一的错误处理、现成的权限校验，AI 又给你写了一套新的。结果一个项目里两套类似的实现，以后改起来都不知道改哪边。

为什么会这样？AI 看不到你整个项目的结构，它只能根据你当前打开的文件和对话内容来推断。它不知道你们有公共模块，不知道哪些东西已经实现过了。所以它倾向于"你要啥我写啥"，从头造轮子。

**怎么救**：**在 Prompt 里主动告诉它项目里有什么**。"我们已经有 `@/utils/request`，用这个发请求""错误处理用 `handleError`，不要自己写 try-catch"。或者把关键文件用 `@` 引用进来，让 AI 能看到现有实现。

---

## 十、「我来帮你重构」——没事把好好的代码改坏了

你让 AI 加个小功能，它说"我顺便帮你优化了一下结构"。你一看改动记录，好家伙，把原来能跑的代码重构了一遍，变量名全改了，逻辑拆得七零八落，还引入了新 bug。

为什么会这样？AI 有"优化冲动"。它看到代码就想"改进"，觉得自己的写法更优雅、更符合最佳实践。但它不理解你原来的设计意图，不知道哪些"冗余"其实是有意为之，重构完可能破坏原有逻辑。

**怎么救**：**明确限制它的权限**。"只加功能，不要改现有代码""不要重构，不要重命名"。如果它已经开始乱改了，直接说"回滚，只做我要求的那一处修改"。记住：能跑的代码，别乱动。

---

## 一个系统性的防翻车策略

说了这么多坑，最后给一个可执行的**小抄**，每次用 AI 写代码前后都可以过一遍：

**写之前**：把技术栈、风格规范、已有模块写进 Prompt 或规则文件；明确说"保持简单""不要过度设计"；如果有安全/性能要求，提前说明。

**写之后**：至少跑一遍，确认能编译、能运行；检查引用和依赖，确认没有幻觉 API；走一遍关键逻辑，重点看边界和异常；安全相关代码人工审查，避免硬编码和注入。

**改的时候**：修 bug 尽量小范围改动；明确禁止"顺便重构"；改完跑一遍相关功能，做简单回归。

## 写在最后

AI 编程不是银弹，它是个加速器——用得好，效率翻倍；用不好，翻车翻到怀疑人生。关键是：**你才是那个对代码负责的人**。AI 可以写草稿，可以提建议，但最终的质量关，还得你自己把。

下次再让 AI 写代码的时候，不妨把这篇当个小抄，写前写后扫一眼。少翻一次车，就多省一天时间。
