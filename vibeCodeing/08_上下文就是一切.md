# 为什么同样用 AI 写代码，有人效率翻倍，有人还不如自己写？

两个人用同一款 AI 编程工具，一个说"这玩意儿太神了，效率翻倍"，另一个说"生成的全是废代码，还不如自己写"。

你猜差别在哪？不是谁更会 prompt，也不是谁订阅了更贵的套餐——**差别在上下文**。你给 AI 的信息决定了它的上限。

就像给一个新员工入职：你丢给他一张工位图就让他干活，他肯定懵；你给他一份完整的项目文档、代码规范、目录说明，他上手就快。AI 也一样。今天咱们就聊聊：怎么把"上下文"喂对，让 AI 真正成为你项目里的熟手。

---

## 一、AI 编程的本质：你喂什么，它产出什么

很多人把 AI 编程想得太简单：我有个需求，AI 帮我写代码，完事。现实是：**AI 的能力 × 你给的信息质量 = 输出质量**。模型再强，你给的信息稀烂，它也只能瞎猜。

打个比方。你让 AI "加个用户列表页"。它不知道你用 React 还是 Vue，不知道你有现成的表格组件，不知道你们接口返回格式是 `{ code, data }` 还是 `{ success, result }`。它只能按"最常见"的方式猜——可能猜对了，也可能跟你项目格格不入。但如果你告诉它：项目用 React + Ant Design，已有 `UserTable` 在 `src/pages/users`，接口封装在 `src/api/user.ts`，返回格式是 `{ code, data, message }`——它立刻知道往哪使劲，生成出来的代码直接能用。

**你给 AI 的信息决定了它的上限**。所以，AI 编程的本质不是"我会说话"，而是**把项目的关键信息喂给 AI**，让它从"陌生人"变成"熟读你项目的老员工"。

## 二、.cursorrules / .windsurfrules —— 项目级"员工手册"

`.cursorrules`（Cursor 用）和 `.windsurfrules`（Windsurf 用）是同一类东西：放在项目根目录的规则文件，AI 在生成代码时会自动读取。你可以把它理解成**给 AI 写的员工手册**——新员工入职第一天，发他一本，他就知道你们团队怎么干活。

**文件放哪**：项目根目录，跟 `package.json`（项目配置文件）同级。文件名固定，不能乱改。

**写什么内容**：技术栈声明、编码规范、命名约定、架构规则。关键是写**可执行的约束**，别写"代码要好看"这种废话，要写"组件用函数式写法"、"接口调用统一走 `src/api`"、"禁止 any，不确定用 unknown"。

下面给两个完整示例，你可以直接抄了改。

**React + TypeScript 项目示例**：

```
# 技术栈
- React 18 + TypeScript 严格模式
- 构建：Vite
- 样式：Tailwind CSS
- 状态：Zustand（全局）/ useState（局部）
- 请求：axios，封装在 src/api

# 组件规范
- 一律用函数式组件 + hooks，禁止 class
- 一个文件一个组件，文件名与组件名一致（PascalCase）
- 组件放在 src/components，页面放在 src/pages
- Props 必须定义 TypeScript 接口，禁止 any

# 样式规范
- 用 Tailwind 类名，禁止内联 style
- 复杂样式用 cn() 组合，不写超过 5 个类的长串

# API 规范
- 所有请求走 src/api 下的封装，不要直接 fetch/axios
- 错误处理统一在拦截器，组件里用 try-catch 处理业务逻辑
- 接口返回格式：{ code: number, data: T, message: string }

# 禁止项
- 禁止 any，不确定用 unknown
- 禁止直接修改 props
- 禁止超过 80 行的函数
- 禁止在组件里写业务 API 调用，抽到 hooks 或 service
```

**Python 后端项目示例**：

```
# 技术栈：Python 3.10+ / FastAPI / SQLAlchemy 2.0 async / Pydantic v2
# 结构：app/routers（路由）、app/services（服务）、app/models（模型）
# 规范：type hints 必写、异步用 async/await、router 不写 SQL、错误用 HTTPException
# API：响应 { code, data, message }，分页 page/page_size
# 禁止：硬编码配置、print 调试、同步 requests
```

有了这套规则，AI 生成新代码时会自动遵循你的约定。省掉大量"生成-发现不对-再改"的来回。

## 三、CLAUDE.md / AGENTS.md —— 给 AI 的"项目百科"

Rules 解决的是"怎么写"，CLAUDE.md 和 AGENTS.md 解决的是"项目长什么样"。前者是规范，后者是**项目百科**——背景、结构、核心模块、常用命令，让 AI 一眼看懂你的项目。

**CLAUDE.md** 和 **AGENTS.md** 都是项目说明文件，不同 AI 工具会识别不同的文件名，但作用类似。你可以只写一个，也可以两个都写、内容互补。

**写什么**：

1. **项目背景**：这是什么项目、解决什么问题、目标用户是谁
2. **目录结构说明**：每个文件夹干什么，别让 AI 猜
3. **核心模块介绍**：关键文件、核心逻辑、数据流
4. **常用命令**：怎么跑、怎么测、怎么部署
5. **设计原则**：你们为什么这么设计，有什么约定

**示例**（下面是一份项目说明的写法，AI 读到就能快速理解你的项目）：

```markdown
# 项目概述
B 端后台管理系统，多租户 + 权限控制。前端 React，后端 NestJS。

# 目录结构
- frontend/ - React + Vite（前端）
- backend/src/modules/ - 业务模块（user、order、product）
- backend/src/common/ - 鉴权、异常、日志
- docs/ - 设计文档、接口文档、需求文档
- scripts/ - 部署、迁移脚本

# 核心模块
- 鉴权：JWT 令牌，中间件在 common/auth
- 权限：RBAC 权限模型在 modules/rbac
- 多租户：请求头 X-Tenant-Id，中间件解析注入

# 常用命令
cd frontend && npm run dev   # 启动前端
cd backend && npm run start:dev   # 启动后端
./scripts/deploy.sh staging   # 部署到测试环境
```

AI 读到这些，就知道"用户模块在哪"、"鉴权怎么做的"、"要改个接口该动哪些文件"。不用你每次对话都重复一遍。

## 四、项目结构本身就是上下文

目录命名、文件组织，本身就是最强的上下文。**清晰的目录让 AI 更容易理解**，乱七八糟的命名会让它迷路。

举个例子。你的接口层是叫 `api`、`services`、`request` 还是 `http`？如果统一叫 `api`，AI 一看就知道"请求相关在这"。如果你有的叫 `fetchUtils`、有的叫 `apiClient`、有的叫 `request`，AI 会懵：到底用哪个？

**项目组织方式**：有的团队把前端、后端、文档放一个仓库，AI 可以一次看到全貌，适合小团队。但目录层级深了，AI 容易"看不过来"。有的团队分多个仓库，结构简单，但 AI 看不到另一个仓库的代码。折中做法：在根目录放一个清晰的 README，说明各子项目的关系和入口；每个子项目有自己的 .cursorrules 和 README。

**README 的重要性**：很多人觉得 README 是给人看的，AI 会自己读代码。错。README 是 AI 的**第一眼**。一个写清楚的 README（项目是什么、怎么跑、目录结构、关键概念）能让 AI 少走很多弯路。反例：README 只有一句"这是一个项目"，AI 只能靠猜。

## 五、文档即上下文：让 AI 读你的文档

很多团队有设计文档、API 文档、PRD，但都放在 Confluence、飞书、Swagger 上，AI 读不到。**把文档放进项目，或者接入 @docs**，AI 才能按你的规范来。

**@docs 引用**：Cursor 支持接入自定义文档（URL 或本地路径），索引后可以在对话里 @docs 引用。比如你们有内部 API 文档，接入后你说"@docs 用户中心 API，帮我写一个获取用户详情的接口封装"，AI 会按文档里的路径、参数、响应格式来写，而不是瞎猜。

**把设计文档、API 文档放在项目里**：在项目根目录建一个 `docs/` 文件夹，放架构图、接口说明、设计规范。AI 读得到，你 @ 引用也方便。比如 `docs/api/user.md` 里写了用户相关接口，你 @ 这个文件，AI 就知道该调什么、传什么。

**PRD 到代码的工作流**：需求文档（PRD）写清楚功能、交互、边界情况，放在 `docs/prd/`。开发时 @ 对应的 PRD，让 AI 按需求实现。这样 AI 不是在"猜你要什么"，而是在"按文档实现"。需求变更时，先更新 PRD，再让 AI 根据更新后的 PRD 改代码，形成闭环。

## 六、上下文管理的最佳实践（十条建议）

十条具体建议，可以直接当 checklist 用：

1. **项目根目录必须有 .cursorrules**：写技术栈、编码规范、禁止项，越具体越好。
2. **必须有 CLAUDE.md 或 AGENTS.md**：写项目概述、目录结构、核心模块、常用命令。
3. **README 要写清楚**：不只"怎么跑"，还要写"项目是什么、目录干什么、关键概念有哪些"。
4. **目录命名要统一**：api 就 api，别混用 services、request、http。
5. **把设计文档放进项目**：docs/ 下放架构、接口文档、需求文档，别只放外部链接。
6. **接入 @docs**：有接口文档、设计系统、内部规范，接入后 @docs 引用。
7. **Rules 里写"不要做什么"**：除了要做什么，也写禁止项，减少 AI 踩坑。
8. **定期更新 Rules 和文档**：技术栈变了、新约定出来了，及时同步，别让 AI 按过时信息生成。
9. **大项目拆小任务**：别一次 @ 整个大仓库，@ 相关文件夹或文件，减少噪音。
10. **关键逻辑加注释**：复杂业务、历史包袱、特殊约定，写清楚，AI 读代码时能理解。

## 七、一个真实案例：上下文优化前后的效果对比

有个朋友做 React 后台，之前没有 .cursorrules，也没有项目文档。他让 AI "加个订单导出功能"，AI 生成了用 xlsx 库、直接 fetch、内联样式的代码——跟他们项目用 sheetjs、走 axios 封装、Tailwind 的规范完全对不上。他改了半天，还不如自己写。

后来他加了 .cursorrules（技术栈、API 规范、样式规范）和 CLAUDE.md（目录结构、核心模块说明），再让 AI 做类似需求。这次 AI 生成的代码：用 sheetjs（他们已有依赖）、走 `src/api/order.ts` 的封装、按钮用 Tailwind 类名，直接符合项目风格。他只需要微调一下边界情况，就上线了。

**差别就在上下文**。同样的模型、同样的需求，有上下文和没上下文，产出质量差一个数量级。花半小时把 Rules 和文档写好，后续每次对话都能省掉大量返工。这才是性价比最高的投入。

## 写在最后

AI 编程工具不是魔法，它不会读心术。你给什么，它产出什么。**上下文就是一切**——Rules、文档、项目结构、@ 引用，都是把"你的项目"翻译成"AI 能理解"的方式。把这些做好了，AI 才能真正成为你项目里的熟手，而不是一个每次都猜错的陌生人。

花点时间把 .cursorrules 写好、把 CLAUDE.md 写全、把 docs 建起来、把 @ 引用用熟。你会发现：AI 编程工具不是替代你写代码，而是把你从重复劳动里解放出来，让你把精力放在架构、设计、判断上。这才是"真正用好"的样子。
