# 90% 的人只用了 Cursor 10% 的能力——一篇讲透它的正确用法

我用了半年 Cursor。直到有一天同事问我："你怎么还在手动复制 AI 生成的代码？"

我才反应过来：自己一直在用 Tab 补全和 Chat，压根没碰过 Composer。后来花了两周把规则、@ 引用、Agent 模式摸透，效率直接翻倍。回头一看，之前那半年，我大概只用了 Cursor 10% 的能力。

如果你也觉得自己"在用 Cursor，但好像没用到点子上"，这篇就是为你写的。从交互模式到实战技巧，一次性讲清楚。

---

## 一、Cursor 的三种交互模式：实习生、电话指导、还是同事坐你工位？

Cursor 本质上是个"三合一"的 AI 编程环境，但很多人只用了其中一种。搞清楚这三种模式的区别，是高效使用的前提。

**Tab 补全**：最基础的一层。你敲代码，它猜下一行。就像有个实习生坐在旁边，你写一句，他帮你续一句。适合写重复性逻辑、常见模式。但它的局限也很明显：**只看当前文件、只看光标附近**，不知道你项目里有什么现成的工具函数，不知道团队约定。所以补出来的代码经常"语法对、风格不对"。

**Chat（Cmd+L）**：对话式交互。你问问题，它答；你描述需求，它生成代码。就像资深同事在电话里指导你——能解释代码、修 bug、写注释、给建议。但有个痛点：**你得自己复制粘贴**。AI 的回复在聊天框里，你要手动选中、复制、切回编辑器、粘贴。而且默认情况下，Chat 对你项目的了解有限——除非你主动 @ 引用文件，否则它是在"盲猜"。

**Composer / Agent（Cmd+I）**：这是 Cursor 的杀手锏。你给一个需求，比如"给用户列表加个分页"，它会自己打开相关文件、理解结构、改多个文件、甚至跑终端验证。你不需要复制粘贴，不需要告诉它"去第三个文件改"。**就像同事直接坐你工位上，键盘鼠标自己动。**

三种模式各有适用场景：写小片段用 Tab，问问题用 Chat，做功能开发、重构、修复杂 bug 用 Composer。

## 二、@ 引用系统——精准投喂上下文，让 AI 不再盲猜

AI 再聪明，看不到你的代码也是白搭。**@ 引用** 就是告诉 AI"这些是你要看的"，相当于给同事递材料。

**@file**：引用单个文件。比如你问"这段代码为什么报错"，@ 一下报错信息和相关源文件，AI 能立刻定位问题。适合针对性强、涉及文件少的场景。

**@folder**：引用整个文件夹。比如你 @src/components，AI 会索引这个目录下的所有文件，理解你的组件结构。适合"帮我看看这个模块有没有重复逻辑"、"给这个目录下的组件统一加个属性"这类需求。

**@codebase**：引用整个代码库。AI 会做语义搜索，找到跟你的问题相关的代码。适合"项目里有没有现成的 XXX 实现"、"这个功能跟哪个模块有关"这类探索性问题。注意：大项目用 @codebase 会消耗较多资源（token），响应可能慢一点。

**@web**：让 AI 去网上查资料。比如查某个库的最新用法、查最佳实践。适合查文档。

**@docs**：引用你接入的自定义文档（后面会讲）。比如你公司有内部 API 文档、设计规范，接入后 @docs 就能让 AI 按你的规范来。

**@git**：引用 Git 相关的上下文，比如代码变更记录、提交信息。适合做代码审查、理解某次改动的意图。

**什么时候用什么 @**：问具体报错 → @file 报错信息 + 相关源文件；做功能开发 → @folder 或 @codebase 让 AI 理解项目结构；查外部资料 → @web；有内部文档 → @docs。原则就一个：**给够上下文，但别给太多**。@ 一整个超大型项目，不如 @ 相关的几个文件夹。

## 三、.cursorrules——给 AI 写"员工手册"，让它秒懂你的项目

AI 不知道你团队用什么风格写组件，不知道你变量名喜欢驼峰还是下划线，不知道你 API 层要统一用什么方式发请求。**Rules（规则）** 就是把这些约定写下来，让 AI 从"第一次来你项目"变成"熟读你项目规范的老员工"。

**项目级规则 vs 全局规则**：`.cursorrules` 是个配置文件，放在项目根目录，只对当前项目生效；全局规则在 Cursor 设置里配置，对所有项目生效。一般建议：项目特有的约定（框架、目录结构、命名规范）放 `.cursorrules`；个人偏好（比如"用中文注释"、"解释要简洁"）放全局规则。

**怎么写好一个 .cursorrules 文件**：不要写废话，要写**可执行的约束**。比如"用 TypeScript 严格模式"、"组件用函数式写法"、"API 调用统一走 `src/api` 下的封装"、"错误处理用 try-catch"。越具体，AI 生成的结果越符合预期。

**实际例子——React 项目的规则模板**：

```
- 使用 React 18 + TypeScript 严格模式
- 组件一律用函数式写法，禁止老式 class 组件
- 状态管理：简单状态用 useState，跨组件用 Context 或 Zustand
- 样式：Tailwind CSS，禁止内联 style
- API 调用：统一用 src/api 下的 request 封装，不要直接 fetch
- 组件文件：一个文件一个组件，文件名与组件名一致（首字母大写）
- 禁止 any，不确定的类型用 unknown
```

有了这套规则，AI 生成新组件时会自动遵循：函数式、TypeScript、Tailwind、走你的 API 封装。省掉大量"生成-发现不对-再改"的来回。

## 四、Docs 功能——让 AI 读你的文档，按你的规范来写

很多团队有内部文档：API 规范、设计系统、架构说明。这些文档 AI 默认读不到，所以生成的代码经常跟你们的标准对不上。**Docs 功能**就是把这些文档"喂"给 Cursor，让 AI 在生成代码时参考。

**接入方式**：在 Cursor 设置里找到 Docs，可以添加 URL（比如你们的设计系统文档、API 文档链接）或者本地文件路径。Cursor 会索引这些内容，之后你在对话里 @docs，AI 就能引用。

**结合 @docs 使用**：比如你要写一个调用用户中心 API 的接口，你们有内部 API 文档。接入后，你 @docs 选用户中心文档，然后说"帮我写一个获取用户详情的接口封装"，AI 会按你们文档里的路径、参数、响应格式来写，而不是瞎猜。

适合的场景：有接口文档、有设计系统规范、有内部技术文档的团队。文档越规范，AI 用得越准。

## 五、Agent 模式深度拆解——让 AI 自己干活，你只负责验收

Composer（Cmd+I）是 Cursor 最强大的模式，理解它的工作方式，才能"指挥"得好。

**自动读文件、改文件、跑终端**：你给需求后，Agent 会自己决定要看哪些文件、改哪些地方。它会自动读取文件、修改代码、运行命令，你不需要一步步指导。比如你说"给登录页加个忘记密码的链接"，它会打开登录组件、找到合适的位置、插入链接、可能还会检查路由配置。如果改完你想验证，它还能自己跑开发服务器看效果。

**Yolo 模式**：默认情况下，Agent 执行终端命令前会问你确认。但有些操作（比如安装依赖、打包构建）你希望它直接跑，不用每次都点确认。开启 Yolo 模式后，Agent 可以自动执行命令，无需确认。注意：Yolo 适合信任的场景，别在敏感目录开，否则它可能执行你不想执行的命令。

**MCP 工具接入**：MCP 是一种协议，让 Cursor 能接外部工具——查数据库、调 API、读设计稿、操作文件等。比如你们有内部接口文档服务，接入后 Agent 可以直接查"用户列表接口的请求参数是什么"，而不需要你手动复制文档内容。MCP 把 Cursor 从"只能改代码"扩展成"能跟你的工作流深度集成"。

**用好 Agent 的诀窍**：需求描述要清晰，最好带上下文。不说"加个分页"，而说"给 UserList 组件加前端分页，每页 10 条，用现有的 Pagination 组件"。@ 引用关键文件，减少它"找错文件"的概率。复杂任务可以拆成几步，先让它做第一步，确认方向对了再继续。

## 六、10 个让效率翻倍的实战技巧

**技巧 1：用 @ 引用替代口头描述**。与其说"我们项目里有个认证相关的工具函数"，不如直接 @ 那个文件。AI 看到真实代码，比听你描述准一百倍。

**技巧 2：Chat 和 Composer 分工**。问概念、查语法、解释代码用 Chat；做功能、改多文件、跑命令用 Composer。别在 Chat 里让它"帮我把这三个文件都改一下"，它做不到，切 Composer。

**技巧 3：Rules 里写"反面教材"**。除了写要做什么，也可以写"不要做什么"。比如"不要用老式 var 声明"、"不要直接修改传入的参数"、"不要写超过 50 行的函数"。约束越清晰，生成质量越高。

**技巧 4：选中代码再提问**。在 Chat 里选中一段代码，然后问"这段有什么问题"或"怎么优化"，AI 会自动把选中内容当上下文，不用你复制粘贴。

**技巧 5：用 Composer 做"批量操作"**。比如"把 src/components 下所有组件的类型定义统一改成 TypeScript"、"给所有 API 调用加上错误处理"。这类重复性工作，Agent 比人手快得多。

**技巧 6：报错时 @ 完整错误信息**。不要只贴一行错误提示，把完整的报错堆栈、相关文件都 @ 进去。错误信息越完整，AI 定位越准。

**技巧 7：分步骤验证复杂需求**。一个需求涉及 10 个文件，别一次性丢给 Agent。先让它做核心部分，你跑一遍、确认没问题，再让它做剩余部分。避免它改了一堆，最后发现方向错了要全部回滚。

**技巧 8：善用 Cmd+K 行内编辑**。选中一段代码，Cmd+K，直接说"加个加载状态"或"改成异步写法"。适合小范围、单文件的修改，比开 Chat 快。

**技巧 9：Docs 接入你们最常用的文档**。API 文档、设计规范、部署流程——接进去之后，@docs 就能用。一次配置，长期受益。

**技巧 10：定期更新 Rules**。项目规范会变，新加的技术栈、新的约定，及时写进 .cursorrules。Rules 不是一劳永逸的，要跟着项目演进。

## 七、Cursor 的局限和注意事项

Cursor 很强，但不是银弹。有些事要心里有数。

**上下文窗口有限**：再大的模型，能"看到"的代码量也有限。超大型项目、超长文件，AI 可能漏掉关键信息。这时候 @ 引用要更精准，或者把任务拆小。

**生成质量依赖你的输入**：需求描述模糊，生成结果就模糊。Rules 写得好，输出才稳定。把 AI 当队友，就要像跟队友沟通一样，把需求讲清楚。

**敏感信息**：不要在有密钥、密码、真实用户数据的项目里开 Yolo 模式。Agent 可能把敏感信息发到云端（取决于你的配置），注意隐私和合规。

**复杂业务逻辑**：AI 对"常规模式"很熟，对你们公司特有的业务规则、历史包袱可能理解不到位。关键逻辑、核心算法，人的把关不可少。

**网络和订阅**：Cursor 依赖云端模型，需要网络。Pro 订阅有额度限制，重度使用可能触发限制，可以关注用量。

从"只会 Tab 补全"到"Rules + @ 引用 + Agent 一条龙"，Cursor 的能力是分层递进的。很多人卡在第一层，觉得"也就那样"，其实是没往上走。
