# AI 编程 + CI/CD：自动化工作流闭环

上周五下午，小张提了个 PR，等 Code Review 等到下班——老李在开会，没人点 approve。周一早上 CI 跑挂了，红彤彤一片，他对着日志翻了半天，发现是依赖版本冲突，修完再 push，又等一轮 CI。周二终于 merge 了，要发版，Release Notes 还得自己写，照着 commit 一条条抄。小张跟我吐槽：从写到上线，每一步都要人盯着，AI 不是能写代码吗，能不能把后面这摊子也包了？

能。而且已经有人在这么干了。这篇是 Vibe Coding 系列的第 23 篇，也是本系列补充的最后一篇。上一篇聊了 Copilot Workspace 和 Cursor Composer，这篇专门掰扯：**AI 怎么嵌进从开发到部署的整条链路，让「写代码」和「跑起来」真正闭环**。

---

## 一、从写到上线：AI 能插手的每个环节

先把整条链路捋清楚。你写完代码，接下来通常是：本地跑测试 → 提 PR → 等人 Review → CI 跑一遍（lint、测试、构建）→ 通过后 merge → CD 部署到测试/生产环境 → 发版时写 Release Notes。每一步都是「人动手」或「人等结果」。小张的遭遇就是典型：等 Review 要等、CI 挂了要自己修、发版要自己抄说明，一环扣一环，哪环卡住都烦。

AI 能嵌进去的地方，说白了就是：**凡是「重复劳动」和「等别人」的环节，都能试着让 AI 顶上**。不是要 AI 取代人，而是让人少干点「机器能干」的活。

开发阶段，AI 帮你写代码，这个系列前面聊得够多了。测试阶段，AI 可以自动给新代码补测试用例——你加了个函数，它给你写对应的单元测试，甚至直接塞进 CI 里，每次 push 都自动生成、自动跑。Review 阶段，AI 可以当「第一道把关」——PR 一上来，AI 先扫一遍，有没有明显 bug、风格对不对、安全有没有坑，给你一份自动 Review 报告，人再重点看 AI 标出来的地方。CI 跑挂了，AI 可以读日志、分析原因、自动提交修复——不用你对着报错自己猜。发版时，AI 可以根据 commit 和 PR 自动生成 Release Notes、Changelog，省得你一条条抄。

整条链路串起来，就是「AI 编程 + CI/CD 的闭环」：你（或 AI）写代码 → AI 补测试 → AI 做 Review → CI 跑 → 挂了 AI 修 → 过了自动部署 → 发版 AI 写说明。人主要干两件事：**定方向**和**兜底**。剩下的，交给管道和 AI 去跑。

## 二、AI 自动生成测试：CI 管道里的「随写随测」

传统流程是：你写代码，自己记得补测试，忘了就漏了。AI 的做法是：**代码一进 CI，AI 先看看改了什么，自动给新改动的部分写测试，写完了在同一个 pipeline 里跑**。

打个比方：你往锅里加了新菜，AI 不是等你吃完再说「该加盐了」，而是菜一下锅它就尝一口，告诉你「这块可能淡了」，顺便把盐递过来。测试就是那勺盐——AI 在 CI 里自动生成、自动跑，发现问题立刻反馈，不用等你 merge 到主分支、等 QA 测完才发现。

具体实现方式有好几种。有的工具在 CI 的「测试阶段」前加一步：解析本次 commit 的 diff，把新增或修改的函数/模块喂给 AI，让它生成对应的测试用例，写入临时文件，然后正常跑测试。有的集成在 GitHub Actions、GitLab CI 里，用 step 调用 AI API，把 diff 和项目结构传过去，拿回测试代码再执行。关键是：**测试的生成和运行都在管道里，不依赖你本地「记得写」**。有的团队更进一步：本地 commit 前也跑一轮 pre-commit hook，AI 先给改动补测试，你一起 commit 进去，PR 一上来就带着测试，Review 的人看着也放心。本地预生成 + CI 再补一轮，双重保险。

当然，AI 生成的测试不是百分百靠谱。可能漏边界情况、可能断言写得太松。所以更稳妥的做法是：AI 生成 + 人抽查，或者 AI 生成后先跑一遍，失败的用例再让人修。但至少，**「忘了写测试」这个坑，能被填上一大半**。

## 三、AI 代码审查：PR 的「第一道门卫」

Code Review 最烦人的地方，不是「看不懂代码」，而是「等人」。PR 提上去，卡在 Review 队列里，谁有空谁看，没空就晾着。AI 可以当「门卫」：PR 一创建，AI 先扫一遍，几分钟内给你一份报告——哪里可能有 bug、哪里风格不一致、哪里可能有安全风险。人来了，重点看 AI 标出来的，省时间；AI 没标的地方，快速过一眼就行。

市面上已经有成熟产品。**CodeRabbit** 是专门做 AI Code Review 的，接 GitHub、GitLab，PR 一开它就自动评论，逐文件、逐行给建议，还能解释「为啥这么改更好」。**GitHub Copilot 的 PR Review** 也类似，在 PR 页面点一下，AI 帮你过一遍，生成总结和改进建议。还有 **Bolt.new**、**Cursor** 等，都能在 PR 场景里做「AI 先看、人再定」的流程。

AI Review 的价值，不是取代人，而是**把人的精力用在刀刃上**。那些「缩进不对」「变量命名太长」之类的，AI 能搞定；真正需要「这段逻辑在业务上对不对」「和别的模块会不会打架」的判断，还是得人来。所以 AI Review 更像是「预筛」——粗活它干，细活你干。

## 四、CI 跑挂了？让 AI 修

这是最「爽」的一环。你 push 完代码，去接杯水，回来一看 CI 红了。小张那种「对着日志翻半天发现是依赖版本冲突」的经历，换谁都不爽。以前你得：打开日志、找报错、猜原因、改代码、再 push、再等 CI。现在可以：**CI 失败时触发一个 step，把错误日志和 diff 喂给 AI，让它分析原因、生成修复、自动提交一个新 commit**。

流程大概是：CI 的某个 job 失败 → 触发「AI 修复」job → 把失败日志、相关代码、项目结构传给 AI → AI 返回修复方案（或直接生成 patch）→ 自动 commit、push 到原分支或新分支 → 再跑一轮 CI。如果修好了，你回来看到的是绿勾；如果没修好，至少 AI 给了个方向，你接着调。像依赖冲突、类型错误、lint 报错这类「模式比较固定」的问题，AI 修起来成功率挺高；真遇到「业务逻辑错了」那种，AI 可能修偏，但至少能帮你缩小排查范围。

工具方面，有 **Warp**、**Mend** 等在做「CI 失败自动修复」；也有团队自己用 GitHub Actions + OpenAI API 搭一套，把 `on: failure` 的 job 接到 AI，逻辑不复杂，核心就是「失败时调 API、把上下文传过去、解析返回的 diff、用 git 应用并 push」。难点在于：**AI 得能正确理解日志**——有些错误信息含糊、有些是环境问题不是代码问题、有些要结合多行日志才能判断，AI 可能误判。所以更稳妥的是：AI 先给建议，人确认后再提交；或者 AI 自动提交，但限制在「非主分支」「非生产相关」的场景，避免乱改。

还有个细节：AI 修完的 commit，最好打上明显标签，比如 `[bot] fix: ...` 或 `[AI] ...`，方便以后回溯。万一修错了，你也知道是 AI 干的，排查起来有方向。

## 五、AI 生成 Release Notes 和 Changelog

发版前最后一哆嗦：写 Release Notes。照着 commit 一条条抄，格式不统一、重点不突出，还容易漏。小张周二 merge 完要发版，对着几十条 commit 一条条整理，半小时过去了。AI 可以干这事：**把本次 release 包含的 commit、PR 描述、甚至 diff 摘要喂给 AI，让它生成结构化的 Release Notes 和 Changelog**。

比如：输入「从 v1.2 到 v1.3 的所有 merge 的 PR」，AI 输出「新功能：xxx；Bug 修复：xxx；Breaking Changes：xxx」，按类别分好，语言统一，直接贴到 GitHub Release 或发版公告里。有的工具已经集成在 CI 里，比如 **release-drafter** 可以配合 AI 增强，**Changeset** 也能和 AI 结合，自动从 commit 归纳变更说明。你还可以定制格式——有的团队要中英双语、有的要按模块分、有的要突出安全更新，AI 都能按模板来。

这块 AI 的准确度相对高——主要是「归纳」和「润色」，不涉及改代码逻辑。唯一要注意的是：**别漏掉重要的 breaking change**，AI 有时会把它归到「小改动」里，发版前让人扫一眼比较保险。另外，commit message 写得好，AI 归纳起来更准；乱七八糟的「fix」「update」堆一块，AI 也难办。养成写清楚 commit 的习惯，等于给 AI 和未来的自己省事。

## 六、完整链路示例：从 commit 到 deploy

把上面几块串起来，一个典型的 AI + CI/CD 闭环长这样：

1. **开发**：你（或 AI）写代码，commit 并 push。
2. **PR 创建**：AI 自动 Review，在 PR 里留下评论和建议。你打开 PR 页面，先看 AI 的总结，再点进标红的地方细看。
3. **CI 触发**：push 后 CI 启动。第一步，AI 根据 diff 自动生成/补充测试用例；第二步，跑 lint、跑测试、跑构建。
4. **CI 失败**：若某步失败，触发 AI 修复 job，把日志和代码喂给 AI，AI 分析并提交修复，CI 再跑一轮。你收到通知时，可能已经绿了。
5. **Merge**：人 Review 通过（重点看 AI 标的地方），merge 到主分支。
6. **CD**：主分支更新触发部署，自动发布到测试/生产环境。
7. **发版**：打 tag 或点发布时，AI 根据 commit 和 PR 生成 Release Notes，人确认后发布。

这套流程不是「全有或全无」，可以按需裁剪。小团队可能只上 AI Review + Release Notes，测试生成和 CI 修复先不做；大团队可能全上，再配上更细的权限和审批。关键是：**让 AI 和管道替你扛住重复劳动**，你专注在「写什么」和「对不对」上。

全程下来，人主要做三件事：**写代码（或给 AI 提需求）、做关键决策（merge 不 merge、发不发版）、兜底（AI 搞不定时接手）**。剩下的「补测试、预 Review、修 CI、写说明」，都能交给 AI 和管道自动跑。小张那种「等 Review 等到下班、CI 挂了手动修、发版自己抄 Release Notes」的流程，可以压缩成「提 PR、看 AI 报告、点 merge、扫一眼发版说明」——省下来的时间，够多写两行代码了。

## 七、注意事项：权限、兜底、成本

闭环很香，但不能无脑上。有几件事得想清楚。

**权限控制**：AI 在 CI 里能自动 commit、能改代码、能访问仓库，权限要给够，但不能给过。建议：AI 自动提交只允许在 feature 分支、或需要人 approve 才能 merge 的分支；生产环境的部署，别让 AI 直接动；敏感信息（密钥、token）别喂给 AI，日志里也要脱敏——CI 日志经常包含环境变量，AI 分析时万一泄露到外部就麻烦了。有的团队会单独建一个「AI 修复」用的 bot 账号，权限最小化，只允许 push 到特定分支。

**人工兜底**：AI 会错。生成的测试可能漏 case，Review 可能漏掉严重问题，修 CI 可能改错地方。所以关键环节要留人：merge 前至少一人看过；发版前 Release Notes 扫一眼；AI 自动修的代码，能 review 就 review，不能就至少跑一遍完整测试再 merge。别指望「全自动」——AI 是加速器，不是替代品。出过一次 AI 修错导致 regression 的，你就知道人工兜底有多重要。

**成本**：AI 在管道里每跑一次都要调 API，按 token 计费。PR 多、commit 频繁的团队，成本会上去。可以做的优化：只对「有实质改动」的 PR 跑 AI Review，改 README 的那种就跳过；CI 修复只在失败时触发，不每次跑；测试生成可以按「改动文件数」设阈值，小改动不生成。算清楚再上，别月底一看账单懵了。有的团队会设「每日 AI 调用上限」，超了就不跑，第二天再说，成本可控。

**落地顺序**：别一上来就全上。建议先上 AI Review——门槛低、见效快、不涉及自动改代码，团队接受度高。再上 Release Notes 生成——纯归纳，风险小。测试生成和 CI 修复可以放后面，等团队习惯「AI 在管道里」这件事再说。一步步来，比一口气全自动稳得多。

## 八、结尾：Vibe Coding 补充系列的收尾

这篇是 Vibe Coding 系列补充的最后一篇。从第 14 篇「AI 写的代码能上生产吗」开始，我们聊了安全红线、成本账、团队落地、测试策略、代码迁移、Copilot Workspace 和 Cursor Composer，到这篇「AI + CI/CD 闭环」，把 **AI 编程从「写代码」延伸到「跑起来」** 的整条链路捋了一遍。

整个补充系列的主题，其实就一句话：**AI 能帮你写，但写完之后的事，一样重要**。代码能不能上生产、安不安全、成本划不划算、团队怎么落地、测试怎么搞、迁移怎么弄、工具怎么选、管道怎么闭环——这些都不是「写」本身，却是「写好」不可或缺的部分。Vibe Coding 不是「躺平让 AI 写」，而是「和 AI 一起，把整条链路跑顺」。

AI 编程不是「让 AI 替你写就完事了」。写只是第一步，后面还有测、审、修、部署、发版。这些环节里，AI 能嵌入的地方越来越多，但**人的角色不会消失**——定方向、做决策、兜底，这些 AI 干不了。把 AI 当「流水线上的熟练工」，把自个儿当「产线主管」，活能轻松不少，出事了也能接得住。

回到小张那个场景。等他下次提 PR，AI 先 Review 一版，CI 里自动补测试，挂了 AI 先试着修，过了自动部署，发版时 AI 写好 Release Notes，他只需要：看看 AI 的 Review、点个 merge、发版前扫一眼说明。等 Code Review 等到下班？那种事，可以少一点了。

Vibe Coding 系列到这儿告一段落。前面 13 篇讲「怎么用 AI 写」——从 Copilot 到 Cursor、从 Prompt 心法到实战、从翻车现场到安全红线；后面这 10 篇补「写了之后怎么办」——安全、成本、团队、测试、迁移、工具、闭环，该聊的都聊了。AI 编程这件事，写只是入口，测、审、修、部署、发版，一环都不能少。把 AI 嵌进整条链路，才是真正的「Vibe Coding」——不慌不忙，该 AI 干的 AI 干，该人定的人定，流水线跑顺了，码字才有 vibe。
