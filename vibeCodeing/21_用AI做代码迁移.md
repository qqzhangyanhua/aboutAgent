# 用 AI 做代码迁移（Python 2→3、JS→TS）

上个月朋友老张接了个活儿：公司有个跑了七八年的 Python 2 项目，老板说「咱们迁到 Python 3 吧，Python 2 都停更多少年了」。老张打开代码库一看，三千多行，print 满天飞，还有一堆不知道从哪抄来的 `__future__` 和 `six`。他试着用 2to3 跑了一遍，报错一百多个，修完还有隐性的 unicode 问题，测都测不过来。他问我：能不能让 AI 干这事儿？

另一边，有个做前端的哥们儿，老板说「咱们全面上 TypeScript」。他手头一堆 JS 文件，没类型、没接口定义，全是 `any` 满天飞的那种。他问：AI 能帮我迁吗？

这事儿太典型了。**代码迁移**听起来像「换个语法」，其实是一场「搬家」——不只是把家具挪个位置，还得考虑新房子有没有对应的插座、水管接不接得上、邻居的规矩跟以前一不一样。工具能帮你改语法，改不了生态兼容；能帮你换关键字，换不了你脑子里的习惯。这篇是 Vibe Coding 系列的第 21 篇，上一篇聊了 AI 编程的测试策略，这篇专门说说：用 AI 做代码迁移，能帮多少忙、会在哪儿挖坑、怎么用才不翻车。

---

## 一、代码迁移为什么痛苦

老板一句「咱们迁吧」，底下人可能要忙一个月。先说一个扎心的事实：代码迁移的痛苦，八成不在「语法替换」上。语法替换，工具都能干，2to3、ts-migrate、codemod 一大堆。真正要命的是三件事：**生态变了、习惯变了、隐含逻辑藏不住了**。

生态变了，意思是「以前能用的库，现在可能没了或者 API 全改了」。Python 2 时代一堆库只支持 2，迁到 3 得找替代品；JS 迁 TS，很多第三方库没类型定义，你得自己写 `.d.ts` 或者忍受满屏的 `any`。这就像搬家：老房子的家具，新房子不一定摆得下，有的得扔、有的得改、有的得重新买。更烦的是，有时候「替代品」不完全等价——API 长得像，但细节不一样，迁完跑起来才发现某个参数含义变了、某个返回值结构改了，又得回头修。

习惯变了，意思是「写代码的方式不一样了」。Python 2 的 `print` 是语句，Python 3 是函数；Python 2 的除法 `/` 默认整除，Python 3 是浮点除；JS 的 `var` 和 TS 的 `const/let` 作用域不一样。这些不是「改个关键字」就完事，而是整个思维模式要换。你让一个习惯了 Python 2 的人写 Python 3，他可能下意识还会写 `print x`，工具能帮你改语法，改不了你的肌肉记忆。团队协作的时候更明显：有人已经按新习惯写了，有人还在用老写法，代码风格分裂，review 的时候得两套标准一起看，心累。

隐含逻辑藏不住了，最狠。很多老代码，能跑是因为「碰巧对了」——没类型检查、没严格模式，一些边界情况从来没被触发过。一迁到 TS、一上 Python 3 的严格检查，原来「藏着」的问题全暴露了：这个变量可能是 undefined、那个函数可能返回 null、除零、编码错误……以前「能跑」是因为没人踩到雷，一迁移，雷全炸了。

还有个隐形成本：**测试和回归**。迁完你得确认「没把功能搞坏」，但老项目往往测试不全，甚至没测试。你改了一千行，怎么保证行为没变？只能靠人肉点、靠上线后用户反馈。这种不确定性，让很多人在「迁」和「不迁」之间反复横跳，最后拖成技术债，越拖越难迁。

所以，代码迁移不是「改改语法」，而是**理解老代码在干什么、在新环境里怎么等价地干、还要把以前没暴露的问题一并修掉**。这三件事，传统工具只能帮你做第一件，后两件得靠人——或者，靠 AI。

理解了这些，你再看「用 AI 做迁移」就有底气了：AI 不是来替代你的，是来帮你扛那「八成」机械活的。剩下的「两成」——生态、习惯、隐含逻辑、测试验证——还得你盯着。但至少，你不用一个人扛全部了。

## 二、AI 做代码迁移的原理

那 AI 能干啥？跟传统工具有啥不一样？AI 做迁移，跟 2to3 那种「规则替换」不一样。规则替换是：看到 `print x` 就改成 `print(x)`，看到 `1/2` 就改成 `1//2`。机械、快、但容易漏。AI 的方式是：**理解源码的语义，再转换成目标语言的等价实现，还要适配目标语言的生态**。

打个比方：规则替换像翻译软件，逐词对应；AI 像懂两种语言的人，先听懂你在说什么，再用另一种语言把意思说出来。前者可能翻出「你好吗？我很好，谢谢，你呢？」这种教科书句子；后者能根据语境翻出「还行吧」「凑合」这种更贴切的表达。当然，AI 也不是真「懂」，它只是从海量代码里学到了很多迁移模式，能根据上下文做概率推断。所以它比规则强，但比真正懂业务的人弱——这就是为什么「AI 打头阵，人兜底」。

具体到代码迁移，AI 会做几件事：第一，**读懂上下文**——这个变量是干啥的、这个函数在什么场景下被调用、这个库在新环境里有没有替代品。第二，**做语义等价的转换**——不是简单换关键字，而是保证「行为一致」。比如 Python 2 的 `unicode` 和 `str`，迁到 Python 3 得变成 `str` 和 `bytes`，AI 得根据实际用法判断该用哪个。第三，**适配目标生态**——该加类型注解的加、该写 `.d.ts` 的写、该换库的换。第四，**处理边界**——除零、空值、编码、异步回调里的 this，这些容易出问题的地方，AI 会尝试给出合理实现，但未必每次都对，所以需要人复核。

当然，AI 不是万能的。它也会漏、也会错、也会产生「看起来对了但运行时崩」的代码。它没见过你的业务、不知道你们的历史包袱、更不会替你背锅。所以，用 AI 做迁移，本质是「AI 打头阵，人兜底」——AI 负责把大头的机械活干了，人负责查漏补缺、跑测试、修边界。后面会具体说怎么分工。

## 三、实测一：Python 2 → Python 3

说再多原理，不如看实战。老张那个项目，我让他试了试用 Cursor 的 Composer 迁。挑了几个经典坑点，看看 AI 表现如何。

**print 和 unicode**。这两样是 Python 2→3 的「保留节目」。AI 处理得很好：`print x` 改成 `print(x)`，`print "hello"` 改成 `print("hello")`，基本不会漏。unicode 那块，`u"中文"` 在 Python 3 里可以直接写成 `"中文"`，AI 也能识别；但有个地方，老代码里混用了 `str` 和 `unicode`，做字符串拼接的时候，AI 一开始漏了一个 `encode('utf-8')`，导致某个接口返回的 JSON 里中文乱码。后来加了更具体的 prompt——「这段代码要输出到 HTTP 响应，注意编码」——AI 才修对。所以，**涉及 I/O 和编码的地方，得额外提醒 AI**。

**除法**。Python 2 的 `3/2` 是 1，Python 3 是 1.5。AI 一般能识别「这里是整除意图」还是「这里要浮点除」。但有一处，老代码里 `count / total * 100` 算百分比，在 Python 2 里因为整除会得到 0，本来就是个 bug；AI 迁的时候改成了浮点除，结果「顺便」把 bug 修了。这种算好事，但说明：**迁移过程中可能改变行为**，你得跑测试确认。

**包导入**。`urllib`、`urllib2` 在 Python 3 里合并成 `urllib`，`ConfigParser` 改成 `configparser`。AI 对这些标准库的迁移很熟，基本不会错。但第三方库就悬了：有个老项目用了 `MySQLdb`，Python 3 推荐用 `mysqlclient` 或 `pymysql`，AI 会建议换，但不会自动改调用方式——因为 API 有差异，得人工确认。所以，**第三方库迁移，AI 能给方向，不能全自动**。

**迭代器**。Python 2 的 `dict.keys()` 返回列表，Python 3 返回视图；`range` 在 2 里是函数返回列表，在 3 里是惰性对象。AI 一般能识别「这里需要 list」的场景，自动加 `list()` 包装。但有些隐式迭代，比如 `for k in d.keys()`，在 2 和 3 里都能跑，AI 可能不会动——这种其实不用动，但说明 AI 对「哪些必须改、哪些可以不动」的判断还算靠谱。

整体看，Python 2→3 这种「语法+标准库」层面的迁移，AI 能搞定七八成；剩下两成是编码、第三方库、边界行为，需要人盯着。老张的反馈是：比纯手迁快一倍，但最后那几天修 bug 的酸爽，一点没少。所以，别指望「丢给 AI 就完事」，做好「迁完再修一轮」的心理准备，反而更踏实。

## 四、实测二：JavaScript → TypeScript

Python 迁完，再看看前端的。JS 迁 TS，难点不在语法，而在**类型**。TS 的核心价值是类型安全，但老 JS 代码根本没类型，AI 得「猜」——猜对了省事，猜错了就留下一堆 `any` 或者类型错误。

**类型推断**。AI 会根据变量用法推断类型：`let count = 0` 推成 `number`，`let name = "foo"` 推成 `string`，数组、对象也能推。但遇到「先声明后赋值」「多分支赋值不同类型」的情况，AI 容易推成 `any`。比如 `let x; if (flag) x = 1; else x = "a";`，AI 可能直接给 `x` 标 `any`，因为 union 类型 `number | string` 需要更细的分析。**简单场景 AI 能推，复杂场景容易降级成 any**。

**any 的坑**。一旦有了 `any`，类型检查就形同虚设。AI 为了「先跑起来」，经常大面积用 `any`，尤其是函数参数、返回值、第三方库。你迁完一看，类型覆盖率上去了，但 `any` 占了一半，等于白迁。所以，**迁 TS 的时候要明确要求「尽量少用 any，能推就推、推不出再 any」**，否则 AI 会偷懒。

**第三方库类型**。很多 npm 包没有 `@types/xxx`，AI 会给你写 `declare module "xxx" { ... }`，但往往写不全，只 declare 你用到的部分。这能解决编译报错，但类型信息不完整，以后用这个库的其他 API 还是没提示。**第三方库类型，能找现成的 @types 就用，没有的话 AI 写的 declare 要人工补全**。

还有个细节：JS 的 `this` 在 TS 里要显式标注，尤其是回调函数。AI 有时候会漏，导致 `this` 为 undefined 的运行时错误。**涉及 this 的地方，迁完要重点检查**。

**strict 模式**。TS 的 `strict` 一开，`null` 和 `undefined` 就得显式处理，很多老 JS 代码根本没考虑这些。AI 可能给你加一堆 `!` 或者 `??`，能过编译，但不一定符合你的业务逻辑。比如 `user.name` 可能真是 undefined，你加个 `!` 说「肯定有」，运行时照样崩。**strict 迁移可以分阶段：先关 strict 迁过去，再逐步打开，一块一块修**。

## 五、实测三：框架迁移

语言迁完了，框架呢？Vue、React 这种大版本升级，也是迁移。框架迁移比语言迁移更「玄学」——Vue 2→3、React Class→Hooks，不只是语法变，连**心智模型**都变了。

**Vue 2→3**。Options API 变 Composition API，`this` 没了，`data`、`methods`、`computed` 全要拆成 `ref`、`reactive`、`computed`。AI 能帮你做机械转换，比如把 `data()` 里的属性改成 `ref()`，但**逻辑拆分**不一定合理。有的组件逻辑复杂，拆成多个 `setup` 里的函数，AI 可能按「原来 methods 的顺序」拆，而不是按「逻辑内聚」拆，结果代码能跑，但结构乱七八糟。所以，**框架迁移建议分两步：先让 AI 做语法级转换，再人工重构结构**。另外，Vue 3 的 `$emit`、`$attrs`、`$slots` 用法有变化，AI 一般能改对，但 `provide/inject`、自定义指令这些用得少的 API，可能要人工检查。

**React Class→Hooks**。`componentDidMount` 变 `useEffect`，`this.state` 变 `useState`，AI 对这些映射很熟。但 Class 的 `this` 绑定、生命周期顺序、`getDerivedStateFromProps` 这种复杂场景，AI 容易搞错。尤其是「在 useEffect 里要模拟 componentDidUpdate 的 prevProps 比较」，AI 可能写出一版能跑的，但依赖数组没设对，导致多余渲染或者该更新没更新。**React 迁移，生命周期相关的逻辑要重点 review**。`useCallback`、`useMemo` 这类性能优化，AI 会加，但加得对不对、有没有过度优化，也得人判断。

框架迁移还有个共性：**生态里的组件库、工具库可能还没跟上**。Vue 3 刚出的时候，很多 Vue 2 的库不兼容；React 18 的 concurrent 模式，老库可能有坑。AI 不知道你项目里用了哪些库、哪些版本，迁移时可能只改了框架代码，没改依赖。**迁移前先查清楚：目标框架的生态兼容性，再决定迁不迁、怎么迁**。

还有个框架迁移的「坑中坑」：**混用**。有的项目是 Vue 2 + Vue 3 混用、或者 Class 组件和 Hooks 混用，因为迁移是渐进式的。这种混用状态，AI 容易搞混——它可能把新代码写成旧风格，或者把旧代码改成新风格但没跟旁边的代码对齐。所以，**混用期要明确告诉 AI：这个文件是 Vue 2 的、那个是 Vue 3 的**，别让它自由发挥。

## 六、AI 迁移的效果评估

说了这么多，到底值不值？老张最后迁完了，我帮他粗略统计了一下：三千多行 Python 2，AI 一次性迁了大概 85%，剩下 15% 是人工修的。人工修的主要是：三处编码问题、两个第三方库的 API 替换、五六个除法和类型相关的边界 case。整体算下来，**AI 省了大概 60% 的时间**——如果纯手迁，估计得两周；用 AI，一周搞定，但最后两天基本在修 AI 漏掉的坑。

JS 迁 TS 那个哥们儿，项目小一点，一千多行。AI 迁完，类型覆盖率从 0 到 70%，但 `any` 占了 30%。他后来花了两天，把关键的 `any` 一个个换成具体类型，才敢上生产。**迁移成功率**可以理解为「迁完能跑」——这个 AI 基本能做到；**需要人工修复的比例**，根据项目复杂度，大概在 10%～30%。项目越老、越乱、越多「历史包袱」，人工修的比例越高。

不同迁移类型，AI 的表现也不一样。**纯语法迁移**（比如 Python 2→3 的 print、除法）AI 最稳，人工修的比例能压到 5% 以内。**类型迁移**（JS→TS）次之，因为「推断类型」本身有不确定性，any 和类型错误会多一些。**框架迁移**最悬，心智模型变了，AI 容易在结构、生命周期、依赖关系上出错，人工修的比例可能冲到 30% 甚至更高。所以，评估「AI 能帮多少」的时候，得先看你是哪种迁移。

还有个变量：**项目年龄**。越老的项目，历史包袱越多，AI 越容易在「没见过」的写法上翻车。新项目、规范清晰的项目，AI 迁起来更顺。所以，别拿「别人用 AI 迁得很快」来要求自己的老破小项目，每个项目的情况不一样，预期要合理。

还有个指标：**行为一致性**。迁完跑同样的测试，有没有结果不一样的？老张的项目里，有一处因为除法语义变了，测试断言得改；还有一处因为 unicode 处理方式变了，一个边缘 case 的测试从 pass 变 fail。所以，**迁移完一定要跑全量测试**，不能指望 AI 100% 保持行为一致。

最后说个心态问题：**别追求一次迁完美**。第一次迁完，能跑、测试过、没明显 bug，就可以先上线。剩下的类型优化、结构重构、性能调优，可以慢慢来。迁移是个持续过程，不是一锤子买卖。先把「迁过去」这件事搞定，再追求「迁得好」。

## 七、最佳实践：怎么用 AI 做迁移才不翻车

总结几条实操经验，照着做能少踩坑。这些不是银弹，但能显著降低「迁完发现一堆问题」的概率。

**分模块迁，别一口吃**。整个项目一次性丢给 AI，容易乱。按模块、按文件、按功能拆，一块一块迁。迁完一块，跑测试，过了再迁下一块。这样出问题也好定位，不会「改了一千行不知道哪错了」。而且，AI 的上下文窗口有限，一次塞太多文件，它可能漏掉跨文件的依赖关系。小块迁，每次上下文清晰，质量更稳。

**先有测试，再迁**。没测试的项目，迁完你根本不知道对不对。能补测试的先补，哪怕只补核心路径；补不了的，至少手动跑一遍关键流程。**迁移的本质是「行为等价」**，没测试就没法验证等价，迁了也心虚。

**渐进式，别一步到位**。比如 JS 迁 TS，可以先把 `.js` 改成 `.ts`，类型全标 `any`，先让项目能编译；再一块一块把 `any` 换成具体类型。Python 2→3 也可以先迁到 2.7+ 的兼容写法，再逐步去掉 `six`、去掉 `__future__`。一步到位容易崩，渐进式能随时停、随时回滚。而且，渐进式迁移能跟业务迭代并行：这周迁一个模块、下周迁另一个，不用专门腾出两周「闭关迁移」，对团队节奏更友好。

**给 AI 足够的上下文**。迁移的时候，把「这段代码在干什么」「用了哪些第三方库」「目标环境是什么」说清楚。比如「这是 Django 1.11 的项目，要迁到 Django 4，Python 3.10」——AI 知道了，才能做对决策。上下文越少，AI 越容易瞎猜。还可以把项目里的 `requirements.txt`、`package.json` 一起丢给 AI，让它知道依赖范围；把迁移文档、官方迁移指南的链接也附上，AI 会参考这些做更准确的转换。

**关键路径人工 review**。编码、除法、第三方库、生命周期、this 绑定——这些「高雷区」迁完一定要人看。AI 能兜住大部分，但漏掉的那几处，往往是运行时才爆的坑。宁可多花半小时 review，也别上线后半夜被叫起来修。

**保留迁移记录**。迁的时候，顺便记一下：哪些文件是 AI 迁的、哪些是人工修的、哪些地方有「已知风险」。这样以后出问题，查起来有线索。也方便团队其他人接手——「这段当时为啥这么写」有个文档总比没有强。

**选对工具**。Cursor 的 Composer、GitHub Copilot、Claude、GPT——都能做迁移，但体验不一样。Composer 适合整块文件迁移，能一次改多个文件；Copilot 适合边写边迁，补全的时候顺带改；Claude/GPT 适合「把代码贴进去、让它输出迁移结果」。根据你的项目规模和迁移方式，选一个主战场，别换来换去，否则上下文容易丢。

最后一条：**别在迁移的同时加新功能**。迁就迁，别顺手「优化一下」「顺便加个 feature」。迁移和加功能混在一起，出问题了你都不知道是迁错了还是新代码写错了。先把迁完成，稳定了，再想别的。

---

## 八、结尾

回到老张：他迁完 Python 3，请我喝了顿酒，说「下次再迁，我还用 AI，但我会先补测试」。这话说到点子上了——AI 做迁移，能省时间、能兜住大部分机械活，但**测试和人工兜底**不能省。迁移不是「改完就完」，而是「改完要确认行为没变」。没测试，迁了等于蒙眼过河；有测试，AI 才是真正的助力。

代码迁移这事儿，以后会越来越多——老项目要续命、要换技术栈、要跟上时代，迁是迟早的。用 AI 迁，能少掉不少头发，但别指望全自动。**AI 打头阵，人兜底**，才是正道。工具在进步，AI 的迁移能力也会越来越强，但「理解业务、验证行为、承担后果」这件事，短期内还是得人来做。把 AI 当助手，别当替身，迁起来才踏实。

老张后来跟我说，他打算把这次迁移的经验写成文档，以后团队再迁别的项目，就有章可循了。这事儿值得做——迁移不是一次性的，而是会反复出现。沉淀下来的经验，比单次迁完就忘，值钱多了。

下一篇，咱们聊聊另一个热门话题：**Copilot Workspace 和 Cursor Composer**——两个「AI 帮你写整块功能」的工具，到底有啥区别、该怎么选。到时候见。
