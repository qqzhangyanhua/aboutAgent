# 两天时间，我用 AI 从零做了一个能上线的团队工具

上周末我突发奇想：团队每天写日报，但没人愿意花时间看别人的。能不能做个工具，让大家把日报丢进去，AI 自动汇总成一份"今日团队动态"？说干就干，我打开 Cursor，从零开始，两天时间把一个能用的团队小工具搞上线了。全程没写几行代码，主要是在跟 AI 聊天。

这篇文章就是这次实战的完整复盘——从需求定义到设计、编码、部署，每一步怎么跟 AI 配合，哪些地方 AI 干得漂亮，哪些地方你得自己兜底。如果你也想用 AI 搞点小项目，这篇可以直接当操作手册用。

---

## 一、项目定义：做什么

先想清楚你要解决什么问题。我的场景很简单：**团队日报收集 + AI 总结**。

具体需求是：每个人每天提交一份日报（今天做了什么、遇到什么问题、明天计划），系统存起来；管理员可以一键触发 AI 汇总，把所有人的日报合并成一份"今日团队简报"，方便 leader 快速浏览。

技术选型我直接让 AI 推荐，它给了这套组合：**Next.js + Tailwind + Supabase + Vercel**。简单说：Next.js 是前后端一体的网页框架，Tailwind 写样式快，Supabase 是免费的云数据库（还自带登录功能），Vercel 部署点几下就能上线。全是主流方案，文档多、坑少，AI 训练数据里也吃得饱。

---

## 二、Phase 1：用 AI 生成项目骨架

**这一步要解决什么问题？** 从零开始，得先有个能跑起来的空壳子，后面的功能才能往上加。

项目从零开始，第一步是搭架子。我给 AI 的 prompt 是这样的：

> 用 Next.js 14 App Router + TypeScript + Tailwind 创建一个新项目。目录结构要清晰：app 下放页面和 API 路由，components 放可复用组件，lib 放工具函数和 Supabase 客户端。先不要装 Supabase，只搭好基础结构，确保能跑起来。

AI 生成的目录大概长这样：

```
app/
  layout.tsx
  page.tsx
  api/
components/
lib/
public/
```

跑 `npm run dev`，本地能打开。这一步 AI 基本不会翻车，因为**项目初始化是高度模板化的**，模型见过无数遍。你只需要确认：项目能跑起来、样式能生效就行。

---

## 三、Phase 2：核心功能——日报提交表单

**这一步要解决什么问题？** 用户得有个地方填日报、点提交，数据才能进系统。

接下来是核心：一个表单，让用户填日报内容并提交。

我跟 AI 说的原话是：

> 在 app/page.tsx 做一个日报提交表单。字段包括：日期（默认今天）、今日完成、遇到的问题、明日计划。用 Tailwind 做样式，表单要简洁，提交后显示成功提示。数据先不存数据库，用 console.log 打印出来就行。

AI 很快生成了一个表单，样式还行，但有几个小问题：日期选择器用的是原生 input type="date"，在 Safari 上丑得一批；提交后没有清空表单。我补了两句 prompt："日期用个好看点的选择器，提交成功后清空表单"，AI 改完就 OK 了。

**这里的经验是**：第一次描述可以粗一点，先跑通流程；发现问题再迭代。不要一上来就写超长 prompt，AI 容易摊大饼，你也容易漏需求。

---

## 四、Phase 3：后端——Supabase 数据库和 API

**这一步要解决什么问题？** 表单提交的数据得存下来，不然刷新就没了。Supabase 是免费的云数据库服务，不用自己搭服务器。

表单有了，下一步是存数据。我让 AI 设计数据库结构：

> 我们要存日报。每条日报有：id、用户 id、日期、今日完成、遇到的问题、明日计划、创建时间。用 Supabase，表名 daily_reports。帮我写建表 SQL，以及 Next.js API 路由：POST 存日报，GET 按日期查日报列表。

AI 给出的结构中规中矩，但漏了一个点：**用户 id 当时还没做登录**，我临时改成了可选的 user_id，先让数据能存进去。等后面接登录功能再补。

接口这块，AI 生成得不错。Supabase 的用法、环境变量怎么配，它都写对了。我唯一改的是：加了一层简单的校验，防止空数据入库。**AI 擅长写"能跑"的代码，但边界条件、安全校验，往往需要你提醒。**

---

## 五、Phase 4：AI 总结功能——接入大模型

**这一步要解决什么问题？** 一堆日报堆在那儿，leader 没时间一条条看。让 AI 自动汇总成一份简报，几秒钟就能了解全貌。

这是整个项目最有趣的部分：把一堆日报喂给大模型，让它输出一份汇总。

我的 prompt 是：

> 写一个 API 路由，接收日期参数，从数据库查出当天所有日报。把日报内容拼接成一段文本，调用 OpenAI API，让 GPT 总结成一份"今日团队简报"。简报要包含：整体进展、主要问题、明日重点。返回格式是 Markdown。

AI 生成的代码逻辑没问题，但 **prompt 设计需要自己调**。第一版 AI 给的 system prompt 太泛，汇总出来的东西像流水账。我改成了更具体的指令：

> 你是一个团队日报汇总助手。输入是多个成员的日报原文，请输出一份简洁的"今日团队简报"，包含：1）今日整体进展（2-3 句）；2）需要关注的问题（如有）；3）明日重点事项。语气专业但简洁，不要重复原文，用概括性语言。

改完效果明显好很多。所以：**调大模型的 prompt，是典型的"AI 做骨架、人工做精修"**。模型不知道你的业务语境，你得告诉它你要什么风格、什么结构。

---

## 六、Phase 5：用户登录

**这一步要解决什么问题？** 日报得知道是谁写的，不然混在一起分不清。Supabase 自带登录功能，支持邮箱密码、魔法链接、第三方登录等，我选了最简单的邮箱密码。

跟 AI 说的：

> 集成 Supabase Auth。需要登录页、注册页、登出逻辑。登录成功后跳转到首页，未登录用户访问首页时重定向到登录页。用 Supabase 的 signIn、signUp、signOut 方法。

AI 把登录注册页、服务端校验、路由保护都写出来了。但有一个坑：**登录状态在不同环境下的配置**不一样，本地跑没问题，上 Vercel 后登录状态拿不到。我查了文档，补了相关配置，才修好。

这类"环境相关"的问题，AI 经常照顾不到，因为训练数据里各种环境混在一起。**部署前在目标环境跑一遍，能省很多事后排查。**

---

## 七、Phase 6：部署上线

**这一步要解决什么问题？** 本地跑得再欢，关掉电脑就没了。部署就是把项目放到云端，别人通过链接就能访问。

最后一步：扔到 Vercel 上。

> 这个 Next.js 项目要部署到 Vercel。需要配置的环境变量有：NEXT_PUBLIC_SUPABASE_URL、NEXT_PUBLIC_SUPABASE_ANON_KEY、OPENAI_API_KEY。帮我写一个 .env.example 说明每个变量的用途，以及部署时要注意什么。

AI 给出了 .env.example 和简要说明。Vercel 部署本身没啥难度：连 GitHub、选仓库、填环境变量、点 Deploy，几分钟就搞定。

唯一要提醒的是：**数据库的访问权限要配好**。Supabase 有两种密钥，公开的那种可以放前端，另一种千万别泄露。行级权限（RLS，就是"谁只能看自己的数据"这种规则）你得自己在 Supabase 后台配，这部分 AI 没法替你点鼠标。

---

## 八、复盘：哪些 AI 做得好，哪些需要人工介入

整个流程跑下来，可以总结成一张表：

**AI 擅长的**：项目初始化、增删改查接口、表单界面、样式布局、Supabase 基础用法、调用大模型 API。这些要么是模板代码，要么是文档里写得很清楚的标准操作，模型练得多，出错率低。

**需要人工的**：业务逻辑边界（比如"同一天同一人只能提交一份日报"要不要做）、安全配置（权限、环境变量、密钥管理）、提示词调优、环境差异（本地 vs 线上）、性能优化（比如日报多了要不要分页）。这些要么依赖业务理解，要么依赖对运行环境的熟悉，AI 容易漏或者给通用解。

打个比方：**AI 像是一个很能干的实习生，你描述清楚需求，它能快速出活；但验收标准、边界情况、上线前的最后一道关，还得你自己把。**

## 九、全流程 Prompt 清单（可复用）

如果你要照着做一版，可以直接用下面这套 prompt，按顺序喂给 AI：

1. **项目初始化**：用 Next.js 14 App Router + TypeScript + Tailwind 创建项目，目录结构包含 app、components、lib。

2. **日报表单**：做日报提交表单，字段：日期、今日完成、遇到的问题、明日计划，Tailwind 样式，提交后清空。

3. **数据库设计**：设计 daily_reports 表，字段包含 id、user_id、date、content 等，给出 Supabase 建表 SQL。

4. **API 路由**：写 POST 和 GET 接口，对接 Supabase，支持按日期查询。

5. **AI 汇总**：写 API 接收日期，查日报，调 OpenAI 总结成团队简报，返回 Markdown。

6. **用户认证**：集成 Supabase Auth，登录、注册、登出，保护需要登录的页面。

7. **部署说明**：列出环境变量，说明 Vercel 部署步骤和注意事项。

每一轮都可以基于上一轮的产物继续迭代，比如"在刚才的表单基础上，加上 XX 校验"。

## 写在最后

两天时间，从零到上线一个能用的团队小工具，放在以前得至少一两周。AI 编程的价值不在于"取代程序员"，而在于**把重复劳动压缩掉**，让你把精力放在需求澄清、产品判断、边界把控这些真正需要人脑的地方。

下次你想搞个周末项目，不妨试试：先想清楚要解决什么问题，选一套主流技术栈，然后跟 AI 一轮一轮聊。记得，说清楚比说得多重要；跑起来比一次做完美重要。
