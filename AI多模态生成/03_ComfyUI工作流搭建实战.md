# ComfyUI 工作流搭建实战

上个月有个做电商的朋友跟我吐槽，说每天要出几十张产品图，用 WebUI 一张一张点：改 Prompt、调参数、点生成、等出图、再下一张。一晚上下来手都酸了，效率低得离谱。他想批量跑，但 WebUI 的批量功能要么藏得深，要么不够灵活，换个模型还得重新设一遍。后来他听说有个叫 ComfyUI 的东西，能像搭积木一样把流程串起来，一次配置、反复用，还能分享给别人。他问我：这玩意儿靠谱吗？上手难不难？

说实话，ComfyUI 刚出来的时候我也觉得"节点式"听着就头大，但用习惯之后发现，它解决的就是 WebUI 那种"表单式"操作的根本痛点——不可复用、不可分享、批量操作笨重。今天这篇，不讲高深原理，就用几个接地气的比喻，把 ComfyUI 从安装到第一个工作流、再到进阶玩法给你捋清楚。看完你至少能自己搭一条"文生图流水线"，以后想批量出图，点一下就行。

---

## 一、ComfyUI 是什么：乐高积木 vs 填表

先搞清楚 ComfyUI 和 WebUI 的本质区别。

WebUI（比如 Automatic1111）的交互方式，说白了就是**填表**。你打开一个页面，上面一堆输入框：模型选哪个、Prompt 写什么、采样步数多少、尺寸多大……填完点"生成"，图就出来了。下次想换个模型？重新选一遍。想加个 ControlNet？再找对应的 tab 勾上。每次操作都是"一次性"的，流程没法固化，也没法分享给别人。就像你去银行办业务，每次都要重新填一遍表格，柜员不会记得你上次填了什么。

ComfyUI 走的是另一条路：**节点式工作流**。你把"加载模型""编码 Prompt""采样""解码出图"这些步骤，拆成一个个小方块（节点），用线把它们连起来。连好之后，整条链路就是一个"流水线"——数据从左边流到右边，中间经过哪些处理，一目了然。想改？动某个节点就行。想复用？把整个工作流保存成文件，下次直接加载。想分享？把文件发给别人，他打开就能用，不用自己从头搭。

打个比方：WebUI 像**点菜**，每次都要跟服务员说一遍你要什么；ComfyUI 像**自己下厨**，菜谱（工作流）写好了，照着做就行，而且菜谱可以复印、可以传给朋友。再往深一点说，ComfyUI 的节点就像**乐高积木**，每个积木块负责一件事，你按自己的逻辑把它们拼起来，拼成什么形状自己说了算。也有人把它比作**电路板**——每个节点是元器件，连线是导线，电流（数据）从输入端流到输出端，中间经过各种"加工"。

所以 ComfyUI 的核心优势就四个字：**可复用、可分享**。一次搭好，终身受益。

## 二、为什么越来越多人转 ComfyUI

除了可复用和可分享，还有几个实实在在的理由。

**第一，适合批量生产**。做电商的、做自媒体的、做游戏素材的，经常要一次性出几十上百张图。WebUI 的批量功能要么藏在犄角旮旯，要么只能简单循环 Prompt，想"每张图换一个 LoRA"或者"先文生图再图生图再超分"这种复杂流程，基本没戏。ComfyUI 可以把整条流水线搭好，然后写个脚本或者用自带的队列功能，一口气跑完。就像工厂的流水线，机器开起来，产品一个接一个出来，你不用每个都亲手拧螺丝。

**第二，更灵活**。WebUI 的界面是固定的，开发者给你什么你就用什么。想加个新功能？等插件更新。ComfyUI 的节点是模块化的，社区里有人做了新节点，你装上去就能用，不用等官方。ControlNet、换脸、超分辨率、多模型串联……想怎么组合就怎么组合，没有"官方没做所以不行"这回事。

**第三，工作流即文档**。你搭好的工作流，打开看一眼就知道"这张图是怎么出来的"——用了什么模型、加了什么 LoRA、ControlNet 控制了什么、采样步数多少。新人接手、自己隔了半年再看，都能秒懂。WebUI 那种"上次怎么调的忘了"的尴尬，在 ComfyUI 里不存在。

**第四，社区生态成熟了**。早两年 ComfyUI 的教程少、节点乱，上手确实劝退。现在不一样了，GitHub 上现成的工作流一大堆，Civitai、OpenArt 这些站也能下载，很多大佬分享的"一键出图"工作流，直接导入就能用。你先用别人的，再慢慢学着改，学习曲线平滑了很多。

## 三、安装和基础配置

ComfyUI 的安装其实不复杂，坑主要在"模型放哪"和"常见报错"。

**安装方式**：推荐用 GitHub 直接克隆。打开终端，找个目录执行 `git clone https://github.com/comfyanonymous/ComfyUI`，然后进目录，按官方文档装依赖。Windows 用户可以直接下 release 的压缩包解压，省事。如果你已经装过 WebUI，ComfyUI 可以复用它的模型——在启动脚本里指定 `--clip-path` 和模型路径就行，不用重复下载几十个 G。

**模型文件放哪**：ComfyUI 的目录下有个 `models` 文件夹，里面再分子目录：`checkpoints` 放主模型（.safetensors），`loras` 放 LoRA，`vae` 放 VAE，`controlnet` 放 ControlNet 模型，`upscale_models` 放超分模型。记住这个结构，以后下载模型直接对号入座，放错地方 ComfyUI 找不到。

**常见报错**：一是"CUDA out of memory"——显存爆了，解决方法是换小一点的模型、降低分辨率、或者开 `--lowvram` 模式。二是"No module named xxx"——缺依赖，照着报错信息 pip install 一下。三是"could not find model"——模型路径错了或者文件名不对，检查 `models` 下有没有放对。四是节点显示"红色"——说明连线有问题，数据类型不匹配，比如把图像输出接到了需要文本输入的端口，把线重新连对就行。五是插件冲突——装了一堆自定义节点之后启动报错，可以试试禁用最近装的插件，逐个排查。

还有个实用技巧：第一次启动会慢，因为要加载模型。之后如果只是改 Prompt、不改工作流结构，可以用"队列"模式连续跑，不用每次重启。工作流记得随时 Ctrl+S 保存，不然崩了或者关错了，一上午白搭。

## 四、核心概念：节点、连线、工作流

把这三个概念捋清楚，后面搭什么都顺。

**节点（Node）**：就是工作流里的"积木块"。每个节点干一件事：加载模型、编码文本、采样、解码、保存图片……节点有输入端口和输出端口，输入在左边，输出在右边。有的节点只有输出（比如"加载模型"），有的只有输入（比如"保存图像"），大部分是两头都有。

**连线**：把输出端口拖到输入端口，就形成一条"数据流"。比如"加载模型"的输出连到"采样"的 model 输入，"编码 Prompt"的输出连到"采样"的 positive 输入。线连对了，数据才能从上游流到下游。连错了，节点会报红，运行时会报错。

**工作流**：一堆节点加连线，组成一个完整的"流水线"。从最左边的"原料"（模型、Prompt、图片）到最右边的"成品"（生成的图），中间经过哪些加工步骤，全在画布上。工作流可以保存成 .json 文件，也可以导出成 .png（ComfyUI 支持把工作流嵌进图片，扫一眼就能导入）。

用流水线比喻：节点是工位，每个工位负责一道工序；连线是传送带，把半成品从一个工位送到下一个；工作流就是整条生产线。原料从左边进去，成品从右边出来，中间怎么加工的，一目了然。

## 五、第一个工作流：文生图

从零搭一条最简单的文生图流水线，走通整条链路。

**第一步，加载模型**。在画布上右键，选择 "Load Checkpoint"，会出来一个节点，让你选主模型。如果你在 `models/checkpoints` 里放过 .safetensors 文件，下拉框里会显示。选一个，比如 SDXL 的 base 模型。

**第二步，编码 Prompt**。加两个 "CLIP Text Encode" 节点，一个填正向 Prompt（你想画什么），一个填负向 Prompt（你不想画什么）。这两个节点都需要"连接"到模型——把 Load Checkpoint 的 CLIP 输出连到 CLIP Text Encode 的 clip 输入。这样模型才知道用哪个编码器来理解你的文字。

**第三步，准备 latent**。加一个 "Empty Latent Image" 节点，设置宽高（比如 1024x1024）和 batch size（一次生成几张）。这个节点输出一个"空的 latent"，相当于给采样器一块画布。

**第四步，采样**。加一个 "KSampler" 或 "KSampler Advanced" 节点。把 Load Checkpoint 的 model 输出连到它的 model 输入，把两个 CLIP Text Encode 的 output 分别连到 positive 和 negative，把 Empty Latent Image 的 output 连到 latent_image。然后设置采样步数（20-30）、采样器（如 Euler）、调度器（如 normal）。这个节点会输出采样后的 latent。采样步数别设太高，30 步和 50 步肉眼差别不大，但时间差不少，新手 20 步起步就够。

**第五步，解码出图**。加一个 "VAE Decode" 节点，把 Load Checkpoint 的 VAE 输出连到它的 vae 输入，把 KSampler 的 output 连到 samples 输入。VAE 负责把 latent 转成像素图。有的模型自带 VAE，有的需要单独加载，看模型说明。

**第六步，保存**。加一个 "Save Image" 节点，把 VAE Decode 的 output 连到它的 image 输入。设置保存路径（可选），然后点 "Queue Prompt"，图就会生成并保存。

整条链路是：模型 + Prompt → 编码 → 空 latent → 采样 → 解码 → 保存。走通这一遍，你就理解了 ComfyUI 的基本逻辑。后面加 LoRA、加 ControlNet，都是在这条链路上"插入"新节点。

## 六、进阶：图生图、ControlNet、换脸、超分

基础工作流跑通之后，可以往链路上加"增强模块"。

**图生图**：在文生图的基础上，把 "Empty Latent Image" 换成 "VAE Encode" + 一张输入图。先把图片用 VAE 编码成 latent，再喂给 KSampler，这样采样就是在"原图的基础上"改，而不是从噪声开始。适合做风格迁移、局部重绘。

**ControlNet 控制**：加一个 "Load ControlNet" 节点，选好 ControlNet 模型（如 depth、canny、openpose），再加一个 "Apply ControlNet" 节点，把 ControlNet 和正反向 Prompt 一起接到采样器。这样生成的图会"服从"你提供的控制图——比如你给一张线稿，它就按线稿的轮廓画；你给一张深度图，它就按深度关系排布物体。ControlNet 是精确控制构图、姿势的利器。

**换脸**：用 InstantID、IP-Adapter Face 这类节点，把一张人脸图作为参考，让生成的结果"长着这张脸"。适合做角色一致性、证件照风格统一。

**超分辨率**：在 VAE Decode 之后、Save Image 之前，加一个 "Upscale Model" 节点，选 Real-ESRGAN 或类似模型，把 1024 的图放大到 2048 甚至更高。适合出大图、打印、高清壁纸。

这些都可以在同一个工作流里组合。比如：文生图 → 图生图微调 → ControlNet 精修 → 超分放大 → 保存。一条流水线干完所有事，点一下，等结果就行。刚开始可能觉得节点太多、线太乱，别慌——先把每个"增强模块"单独搭一遍，跑通了再往主流程上接。就像拼乐高，先拼好各个小部件，最后再组装成大城堡。

## 七、实用插件推荐

ComfyUI 的扩展生态很丰富，有几个必装。

**ComfyUI Manager**：相当于"应用商店"。在里面可以浏览、安装、更新自定义节点和插件，不用自己手动 git clone。装好之后，很多功能都能一键搞定，强烈建议第一个装。

**ComfyUI-Impact-Pack / ComfyUI-Impact-Subpack**：提供大量实用节点，比如批量处理、条件分支、图像处理工具。做复杂工作流的时候经常用到。

**ComfyUI-Advanced-ControlNet**：ControlNet 的增强版，支持更多预处理器、更细的控制选项。玩 ControlNet 的人基本都会装。

**ComfyUI-InstantID**：换脸、人脸一致性。做角色设计、证件照风格统一的时候用得上。

**ComfyUI-essentials**：一些常用的小工具节点，比如图像混合、颜色调整、批量保存。属于"装了不亏"的类型。

装插件的通用流程：ComfyUI Manager 里搜索名字，点 Install，重启 ComfyUI。如果 Manager 里没有，就去 GitHub 搜 "ComfyUI-xxx"，clone 到 `custom_nodes` 文件夹，然后 pip install 它的 requirements.txt。装完插件如果报错，多半是依赖没装全，看插件目录里的 requirements.txt，一条条 pip 进去就行。

## 八、结尾

回到开头那个做电商的朋友。他后来花了一个下午，照着教程搭了一条"产品图流水线"：固定模型和 LoRA，批量读 Prompt 列表，自动出图、自动保存。以前一晚上手点几十张，现在挂机跑，他去泡杯咖啡回来就收图。他说唯一后悔的是没早点学 ComfyUI——早学会，少受多少罪。

ComfyUI 的门槛确实比 WebUI 高一点，但换来的是一次配置、长期复用，以及"想怎么组合就怎么组合"的自由。如果你也受够了表单式操作、想批量出图、或者想精确控制生成流程，不妨花点时间把第一个工作流搭起来。搭完你会发现，节点式思维一旦习惯，就回不去了。
