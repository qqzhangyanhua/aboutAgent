# GPU 服务器选型与成本计算

老板推门进来：「上个 GPU 服务器要多少钱？」你查了查云厂商报价，报了个数。老板沉默了三秒：「……你再说一遍？」没错，GPU 贵。A100 按小时计费，跑满一个月能买一辆车；H100 更狠，直接上六位数。但 AI 应用离了 GPU 又跑不动，这笔账怎么算、怎么省、怎么选，是每个做 AI 落地的团队迟早要面对的问题。这篇文章把 GPU 选型、云上租赁、自建决策、成本估算、省钱技巧和常见坑，一次性讲清楚。

---

## 一、为什么 AI 应用离不开 GPU：搬砖的比喻

先打个比方。CPU 像是一个博士生——啥都会，逻辑推理、复杂计算、串行任务样样精通，但一次只能干一件事。GPU 呢？像是一百个搬砖工人——每个人只会「搬一块砖」这种简单动作，但一百个人同时搬，总量就上去了。

大模型推理在干什么？本质上就是海量的矩阵乘法。每个计算单元（神经元）做的是「乘加」这种最基础的运算，但架不住数量多——70B 模型一次前向传播，要算几百亿次乘加。CPU 一个核心吭哧吭哧算，算到天亮；GPU 几千个 CUDA 核心一起上，几秒搞定。这就是**并行计算**的威力：单次操作简单，但架不住人多力量大。

所以别纠结「能不能用 CPU 跑」了。小模型、低频调用、对延迟不敏感，CPU 凑合能跑；但生产环境、实时对话、大批量推理，CPU 就是蜗牛爬。GPU 不是奢侈品，是刚需。选型的问题不是「要不要 GPU」，而是「要哪种 GPU、要多少、搁哪儿跑」。

还有个类比：CPU 是瑞士军刀，GPU 是流水线。前者万能但慢，后者专精但快。大模型推理这种「同一种操作重复亿万次」的活儿，天生适合流水线。

## 二、主流 GPU 型号一览：谁适合干什么

市面上的 GPU 五花八门，按定位大致分几档。

**A100 / A800**：数据中心级旗舰。80G 显存，算力强，支持 FP16/BF16，训练和推理都顶。A800 是国产合规版，算力略降但能进机房。适合：大模型训练、70B 级别推理、高并发生产。缺点：贵，按小时计费肉疼，包月也不便宜。

**H100 / H800**：新一代旗舰，比 A100 快一截，显存带宽更高，推理吞吐量翻倍。适合：对延迟和吞吐都极其敏感的场景，钱不是问题的团队。缺点：更贵，供应紧张，很多云厂商还没铺开。

**L40S**：消费级架构的数据中心卡。24G 显存，推理性能不错，价格比 A100 友好不少。适合：7B～13B 模型推理、中等并发、预算有限但又要稳定性的团队。很多云厂商的「性价比」机型就是 L40S。

**RTX 4090 / 4090D**：消费级天花板。24G 显存，单卡推理 7B 模型绰绰有余，13B 也能跑。适合：小团队、原型验证、自建单机、对成本极度敏感。缺点：不是为 7×24 高负载设计的，长时间满负载可能出问题；云上很少提供，多半得自己买卡。

**V100 / T4**：老一代，但还在用。V100 32G 显存，算力不如 A100；T4 16G，适合小模型或低并发。价格相对便宜，适合预算紧、模型不大的场景。新项目一般不推荐，除非是存量迁移。

选型口诀：**大模型、高并发选 A100/H100；中等规模、要性价比选 L40S；小团队、自建、省钱选 4090；老项目、老黄历选 V100/T4**。

还有个细节：显存和模型大小的对应关系。7B 模型 FP16 大约 14G，INT4 量化后 4G 左右；13B 翻倍；70B 要 140G+，单卡 A100 80G 装不下，得多卡或量化。选卡前先算清楚：你的模型加载要多少显存，留 20% 余量给 KV cache 和激活，别卡着边买。

## 三、云 GPU 方案对比：谁家香谁家坑

云上租 GPU，主流就那几家，各有各的脾气。

**AWS**：产品线全，p4d、p5 实例，A100、H100 都有。按需、预留、Spot 三种计费，灵活。稳定性好，适合企业级、合规要求高的场景。缺点：贵，按小时计费比其他家高一大截；配置复杂，新手容易晕。

**阿里云 / 腾讯云 / 华为云**：国内首选。有 A100、A800、L40S 等，按量付费和包月都有。优势是网络好、合规、支持国内支付和发票。缺点：GPU 机型经常缺货，要抢；价格也不便宜，但比 AWS 略友好。

**AutoDL**：国内创业团队最爱。按分钟计费，关机不计费，用多少算多少。机型多，4090、A100、H100 都有，价格透明。适合：个人开发者、小团队、实验阶段、用量不稳定。缺点：稳定性看运气，高峰期可能排队；不适合对 SLA 要求极高的生产环境。

**Lambda Labs**：海外性价比之选。A100、H100 价格比 AWS 低不少，按小时计，适合海外团队。缺点：国内访问可能慢，支付要信用卡。

**Vast.ai / RunPod**：共享 GPU 市场，个人矿机主出租算力。价格极低，但稳定性、安全、网络都看运气。适合：纯实验、不差那点稳定性、预算紧到极致。生产环境别碰。有人拿来跑训练，跑一半机器被回收了， checkpoint 都没来得及存，血泪教训。

**Google Cloud / Azure**：也有 GPU 实例，GCP 的 A100 价格和 AWS 差不多，Azure 的 NC 系列老一点。选这两家多半是因为已经在用他们的其他服务，统一在一个云上管理方便。单独为 GPU 选他们，性价比一般。

总结：**企业级、合规、稳定选 AWS 或国内大云；创业团队、灵活用量选 AutoDL；海外、要性价比选 Lambda；纯玩选 Vast.ai**。

价格这块没法给具体数字，波动太大——缺货时涨价、促销时打折，各家计费方式也不一样。建议自己跑一遍：同一机型、同一时长，AWS、阿里云、AutoDL 各算一遍，心里就有数了。别光看单价，还要看最低计费单位（按分钟还是按小时）、关机是否计费、包月有没有折扣。

## 四、自建 vs 租赁：什么时候该自己买卡

自建和租赁，本质是**固定成本 vs 可变成本**的权衡。

**自建**：一次性投入大——买卡、买服务器、机房、电费、运维。但跑起来之后，边际成本低，用多用少一个价。适合：用量大、长期跑、对数据合规有要求、有运维能力。粗略算：一张 4090 一万多，A100 十万级，跑满 2～3 年能回本的话，自建就值。

**租赁**：按量付费，用多少花多少，不用操心硬件。适合：用量小、用量波动大、起步阶段、不想养运维。缺点：长期跑下来，总成本可能比自建高；而且云上 GPU 经常缺货，关键时刻抢不到很抓狂。

决策框架：**月用量 < 200 小时，租；> 500 小时且稳定，考虑自建；中间地带，算一算回本周期再定**。还有一类：数据不能出内网、合规要求高，那基本只能自建或私有云。

自建还有个隐性成本：人的时间。装驱动、配环境、搞监控、处理故障，都得有人兜着。小团队没专职运维，自建容易变成「开发兼职网管」，半夜被叫起来重启机器。租赁省心，但钱花在明处；自建省钱，但操心在暗处。算总账的时候，把人力成本也算进去。

## 五、成本估算模型：一个月要花多少钱

核心公式：**月成本 ≈ 推理次数 × 单次推理成本**。

单次推理成本怎么算？取决于：模型大小、序列长度、GPU 型号、利用率。粗略经验值（仅供参考）：7B 模型、512 token 输入输出，A100 单次约 0.001～0.003 元；13B 翻倍；70B 再翻几倍。具体要压测，不同框架、不同 batch 大小，差异很大。vLLM、TGI、SGLang 这些推理框架，吞吐能差一倍，选对框架也能省不少。

推理次数：日活 × 每人每天调用次数 × 每次请求的 token 数。比如 1 万日活，每人每天 10 次，每次 500 token，一天就是 5000 万 token。按 7B 模型、A100 粗算，一天光推理成本可能就几百到上千，一个月上万很正常。

再加上：GPU 闲置时的费用（按量付费不用不花钱，包月则要摊）、存储、网络、网关、缓存等其他组件。所以做预算时，**推理成本往往是大头**，先把这块算清楚，再往上加 20%～30% 的 buffer，差不多就是真实账单。

有个实用技巧：先小规模压测，记录「单次推理耗时」和「GPU 利用率」，反推单次成本。比如 A100 按小时 20 块，单次推理 100ms，理论上每小时能跑 3.6 万次，单次成本约 0.0005 元。实际会有排队、调度开销，再乘个 1.5～2 的系数，就接近真实值了。

## 六、省钱技巧：能省一点是一点

GPU 贵，能抠一点是一点。

**竞价实例 / Spot**：云厂商的闲置算力，价格能打到按需的 1/3 甚至更低。缺点：可能随时被回收，适合能容忍中断的任务——比如离线推理、批量任务、训练。在线服务慎用，除非做了容错和迁移。

**模型量化**：FP16 变 INT8、INT4，显存减半甚至更多，同样的卡能跑更大的模型，或者小模型可以多开几个实例提高吞吐。量化会损失一点精度，但对很多场景够用。7B 模型 INT4 量化后，4090 能跑，省一张 A100 的钱。

**批处理（Batching）**：单条请求单次推理，GPU 利用率低；攒一批一起推理，吞吐上去，单次成本下来。前提是能接受一点延迟——批要等凑齐，不是实时响应。适合后台任务、异步场景。

**模型路由**：简单问题用小模型，复杂问题用大模型。根据 query 难度或用户等级路由到不同规格的 GPU，避免「杀鸡用牛刀」。成本能省 30%～50%，架构上要多一层路由逻辑。比如：意图识别用 1B 小模型，真正生成用 7B；或者免费用户走 7B，付费用户走 70B。分层设计，钱花在刀刃上。

**缓存**：相同或相似请求直接走缓存，不调模型。RAG 场景里，重复问题很多，缓存命中率上去，推理次数下来，钱就省了。语义缓存更狠——问题表述不同但意思一样，也能命中，命中率能再提一截。本系列前面有专门讲缓存的文章，可以翻回去看。

**包月 vs 按量**：用量稳定、每月跑满几百小时，包月通常比按量便宜。用量波动大，按量更灵活。算清楚自己的用量曲线，再选计费方式。

**冷启动优化**：模型加载要时间，实例从零到就绪可能几分钟。频繁扩缩容的话，冷启动会拖累体验。可以预留少量「热」实例常驻，或者用模型预加载、池化，减少冷启动次数。省的不是 GPU 钱，是用户等的那几分钟。

## 七、常见踩坑：前人栽树后人乘凉

**显存不够（OOM）**：模型太大、batch 太大、序列太长，显存爆了直接崩。解决：换大显存卡、量化、减 batch、截断序列。部署前用 `nvidia-smi` 和压测工具摸清显存峰值，别凭感觉。还有个坑：PyTorch 默认会预分配一大块显存，`PYTORCH_CUDA_ALLOC_CONF` 可以调，但治标不治本，根本还是模型和 batch 别超。

**带宽瓶颈**：模型计算没问题，但数据从 CPU 搬到 GPU、从存储读到内存，成了瓶颈。多卡时，卡间通信也可能拖后腿。解决：用 NVMe、高带宽存储；多卡时注意拓扑，尽量用 NVLink 直连。PCIe 4.0 x16 和 NVLink 的带宽差一个数量级，多卡训练尤其要注意。

**驱动版本**：CUDA、驱动、PyTorch 版本要对上，对不上就报错。上一篇文章讲了 Docker 化，把环境锁死在镜像里，能少很多「在我这儿能跑」的悲剧。新机器部署，先查兼容性矩阵，别硬上。

**排队等待**：云上 GPU 紧俏，高峰期创建实例要排队，有时排几小时。有紧急需求时很抓狂。应对：提前预留、包月锁资源；或者多选几家云，这家没货换那家。

**闲置浪费**：包了月，晚上没人用，GPU 空转，钱照扣。应对：做弹性伸缩，低峰期缩容；或者用 Serverless GPU，按请求计费，不用不花钱（目前支持的厂商和模型还有限）。

**计费陷阱**：有些云厂商「关机不收费」指的是关机状态，但磁盘、公网 IP 可能照扣。删实例前看清楚，别以为关机就没事了。还有按秒计费、不足一分钟按一分钟算的，高频启停会多花钱，注意计费规则。

**多租户隔离**：一台机器跑多个模型或多个租户，显存、算力要隔离好。否则一个租户把 GPU 吃满，其他全卡死。Docker 的 `--gpus` 可以指定卡，K8s 有 GPU 分片，按需配置，别一锅粥乱炖。

## 八、小结与下一篇预告

老板问「上个 GPU 要多少钱」，你报完数把人吓一跳——但吓完之后，得把账算清楚。GPU 是 AI 应用的刚需，选型看模型规模和并发，云上租还是自己买看用量和合规，成本估算抓住「推理次数 × 单次成本」这根主线，省钱从量化、批处理、缓存、竞价实例入手，坑提前躲——显存、带宽、驱动、排队，都是前人踩过的。

从 Docker 化部署到 GPU 选型，你的 AI 应用已经能跑起来了。接下来要面对的是：上线之后怎么平稳放量、怎么验证新模型新功能、怎么少踩生产环境的雷。下一篇聊 **灰度发布与 A/B 测试**——新模型不敢全量推，先放 5% 流量试试水；两个版本同时跑，看哪个效果更好。把发布和实验流程搞顺，AI 应用才算真正上了生产。
