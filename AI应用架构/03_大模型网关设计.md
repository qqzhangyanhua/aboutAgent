# 大模型网关设计：负载均衡、限流、降级、模型路由

直接调 OpenAI API 能跑，这谁都知道。写几行代码，发个请求，拿个回复，Demo 阶段完美。但等你真把东西上线，用户一多，问题就来了：限流 429 满天飞，某天 OpenAI 宕机你的服务跟着挂，月底一看账单——怎么多了两个零？

这时候你就需要一个**大模型网关**。它就像你家门口的总闸，水龙头（API 调用）多了它能分流，水管爆了它能关闸，还能帮你算清楚每个月用了多少水。

---

## 一、为什么需要模型网关

先说几个真实场景。

**单点故障**：你的应用直接连 OpenAI，某天凌晨三点 API 挂了，你的用户全在等转圈。没有备用通道，没有自动切换，你只能爬起来发公告。网关能帮你做**故障转移**，主模型挂了自动切备用，用户无感知。

**限流**：OpenAI 的 API 有 RPM（每分钟请求数）和 TPM（每分钟 token 数）限制。免费 tier 更狠，企业级也有上限。流量一上来，429 错误成片出现，你的应用直接卡死。网关可以**多 Key 轮转**，把压力分散到多个 API Key，相当于把一条窄路拓宽成多条。

**成本控制**：大模型按 token 计费，没有预算控制的话，一个异常用户或者一个 bug 就能让你的账单爆炸。某公司因为循环调用导致一夜烧掉几万美元的新闻不是段子。网关可以**限流、配额、告警**，在失控前掐住。

**多模型管理**：GPT-4、Claude、Gemini、国产模型……你不可能在每个业务代码里写一堆 if-else 去选模型。网关统一管理，业务方只关心"我要调大模型"，至于调哪个、怎么调，网关说了算。

说白了，**网关就是把"调大模型"这件事从业务逻辑里抽出来，做成一个可观测、可控制、可扩展的基础设施层**。

## 二、网关的核心功能

### 负载均衡

想象你有 5 张 OpenAI 的 API Key，每张都有独立的限流额度。直接用一个 Key，流量大了就撞墙；用网关做**轮转**，请求依次打到不同 Key 上，相当于把 5 条小水管的流量合并成一条大河。

负载均衡不只是多 Key，还可以是**多模型分发**。比如你有 3 个推理节点跑同一个模型，网关可以把请求均匀分配到不同节点，避免某个节点过载、其他节点闲置。

### 限流

限流分两种：**保护上游**（别把 API 打爆）和**保护预算**（别把钱包打爆）。

保护上游：每个 Key 的 RPM/TPM 是固定的，网关需要记录每个 Key 的调用量，超了就排队或者拒绝。可以用**令牌桶**或**滑动窗口**算法实现。

保护预算：按用户、按项目、按天设置 token 或金额上限。比如某个内部工具每人每天最多 10 万 token，超了直接拒绝，并触发告警。这对防止"忘记关掉的测试脚本"特别有用。

### 降级

主模型挂了怎么办？**降级策略**就是答案。

比如你默认用 GPT-4o，当连续 3 次请求失败或超时，自动切换到 Claude 或国产模型。用户可能感觉回复风格略有不同，但总比一直转圈强。降级可以是**模型级**（GPT-4 降 GPT-3.5），也可以是**供应商级**（OpenAI 降 Anthropic）。

关键是要有**健康检查**：定期探测各上游是否可用，故障时快速剔除，恢复时自动加回。

### 模型路由

这是网关最"智能"的部分。不是简单地把请求转发到某个模型，而是**根据请求特征决定用哪个模型**。

简单任务用便宜模型：比如情感分析、关键词提取，用 GPT-3.5 或小模型就够了，没必要上 GPT-4。**按任务类型路由**，能省一大笔钱。

复杂任务用贵模型：需要推理、多步规划、代码生成，再上 GPT-4 或 Claude。**按复杂度路由**，既保证效果又控制成本。

还可以**按延迟路由**：用户等不及的实时场景，优先选响应快的模型；后台批处理可以选慢但便宜的。

## 三、开源网关方案

不想从零造轮子，可以看看这几个开源方案。

**LiteLLM**：Python 写的，支持 100+ 模型供应商，统一成 OpenAI 兼容的接口。你写一套调用逻辑，换模型只需改配置。自带代理模式，可以当网关用，支持多 Key 轮转、重试、简单的路由。适合快速接入多模型，代码量小。

**One API**：国产开源，界面友好，支持多种大模型 API 的统一管理和转发。可以配置 Key、模型映射、额度限制，适合中小团队自建。部署简单，Docker 一把梭。

**OpenRouter**：严格说是托管服务而不是自建方案，但它本质上是一个"模型聚合网关"。你调它的 API，它帮你路由到背后的几十个模型，按使用量付费。适合不想维护网关、又想用多模型的场景。缺点是依赖第三方，数据经过他们的服务器。

选型建议：**快速验证用 LiteLLM 或 OpenRouter**；**要完全掌控、数据不出内网用 One API 或自建**。

## 四、自建网关的核心代码

自建网关不复杂，核心就是一个**转发层 + 中间件**。下面用 Python 写一个最小可用版本，展示负载均衡、限流、降级的思路。

```python
import hashlib
import time
from collections import defaultdict
from typing import List, Optional

class LLMGateway:
    def __init__(self, api_keys: List[str], models: List[str]):
        self.api_keys = api_keys
        self.models = models
        self.key_index = 0
        self.rate_limits = defaultdict(lambda: {"count": 0, "window_start": time.time()})
        self.max_rpm = 60  # 每分钟最多 60 次

    def _get_next_key(self) -> str:
        """轮转选择 API Key"""
        key = self.api_keys[self.key_index % len(self.api_keys)]
        self.key_index += 1
        return key

    def _check_rate_limit(self, user_id: str) -> bool:
        """简单限流：滑动窗口"""
        now = time.time()
        window = self.rate_limits[user_id]
        if now - window["window_start"] > 60:
            window["count"] = 0
            window["window_start"] = now
        if window["count"] >= self.max_rpm:
            return False
        window["count"] += 1
        return True

    def call_with_fallback(self, prompt: str, user_id: str = "default"):
        """带降级的调用：主模型失败则切备用"""
        if not self._check_rate_limit(user_id):
            raise Exception("Rate limit exceeded")

        errors = []
        for model in self.models:
            try:
                key = self._get_next_key()
                # 实际调用时替换为 openai.Client(api_key=key).chat.completions.create(...)
                return {"model": model, "key_used": key[:8] + "..."}
            except Exception as e:
                errors.append((model, str(e)))
                continue

        raise Exception(f"All models failed: {errors}")
```

这段代码体现了三个核心逻辑：**轮转选 Key**、**按用户限流**、**模型失败自动切换**。生产环境需要加上超时、重试、熔断、指标上报，但骨架就是这样。

## 五、模型路由策略

路由策略决定了"这个请求该用哪个模型"。常见几种：

**按任务类型**：在请求里带一个 `task_type` 字段，比如 `classification`、`generation`、`code`。网关维护一个映射表：分类任务走 GPT-3.5，代码生成走 GPT-4，创意写作走 Claude。业务方在调用时声明意图，网关负责匹配。

**按成本**：设置预算优先级。比如"本月预算紧张，所有请求默认走便宜模型；VIP 用户或特定接口才走贵模型"。可以按用户等级、按接口、按时间（如非高峰用便宜模型）来区分。

**按延迟**：实时对话场景，优先选首 token 延迟低的模型；批处理任务可以选慢但便宜的。有的网关会维护各模型的延迟百分位（P99），动态选择。

**按内容长度**：输入输出特别长的请求，token 消耗大，可能优先路由到按量计费更划算的模型，或者直接拒绝超长请求。

实际落地时，往往是**多策略组合**：先按任务类型粗分，再按当前负载、预算余量做细调。可以用规则引擎，也可以上简单的 ML 模型做预测。

## 六、监控和告警

网关是流量中枢，不监控等于裸奔。至少要盯这几类指标：

**调用量**：QPS、每日/每月请求数、token 消耗量。按模型、按用户、按接口维度拆。发现某个用户或接口异常飙高，及时介入。

**延迟**：P50、P95、P99 延迟，首 token 延迟（TTFT）。延迟突然升高可能是上游限流或网络问题，需要告警。

**错误率**：4xx、5xx 比例，超时比例。错误率 spike 通常是上游故障或配置问题，要能快速发现。

**成本**：按模型、按用户的 token 消耗和费用。设置预算阈值，超 80% 预警，超 100% 自动熔断或降级。

告警不要贪多，否则容易麻木。优先保证：**服务不可用**、**错误率突增**、**成本超预算** 这三类能及时触达。用 Prometheus + Grafana 或云厂商的监控服务都能实现。

## 七、最佳实践

**网关和业务解耦**：业务代码只调网关的统一接口，不关心底层是 OpenAI 还是 Claude。这样换模型、加降级、调路由，都不需要改业务逻辑。

**配置外置**：Key、模型列表、限流阈值、路由规则全部放配置文件或配置中心，改配置不动代码，支持热更新。

**幂等与重试**：大模型调用可能超时，网关要有重试机制。注意区分可重试错误（5xx、超时）和不可重试错误（4xx、内容违规）。重试时最好用指数退避，避免雪崩。

**审计日志**：谁在什么时候调了什么模型、消耗多少 token，都要落库。出问题时能追溯，做成本分析也方便。

**灰度与回滚**：新模型、新路由策略上线，先灰度一部分流量，观察指标再全量。有问题能快速回滚到旧配置。

---

## 结尾

大模型网关不是什么高深的东西，本质就是一个**带策略的转发层**。但把它做好，能让你从"Demo 能跑"进化到"生产敢用"。

直接调 API 是骑自行车，加个网关是开汽车。自行车能到目的地，但汽车有安全带、有仪表盘、有备胎。生产级的大模型调用，值得多花这点功夫。

如果你已经在用大模型做业务，不妨回头看看：你的调用链路里，有没有那层"总闸"？
