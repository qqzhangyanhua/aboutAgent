# AI 应用的可观测性：怎么追踪一次完整的 LLM 调用链

用户发来一条消息：「你们家 AI 回答错了。」

你问：哪里错了？用户说：反正不对。你再问：当时你问了什么？用户说：忘了，大概就是问了个产品问题。

然后你打开后台，发现——你不知道那次请求的完整链路是什么、模型收到了什么 prompt、检索到了哪些文档、为什么最终输出了那个答案。你只能对着代码猜，或者让用户再试一次，祈祷能复现。

这就是**可观测性**要解决的问题：当 AI 出问题时，你能从哪查、怎么查、查什么。

---

## 一、AI 应用可观测性的特殊挑战

传统后端服务出了 bug，你打个断点、看个堆栈、复现一下就能定位。AI 应用不一样，有三座大山。

**非确定性输出**：同样的输入跑两次，模型可能给出不同的回答。你没法「复现」一次错误的调用，只能靠事后回溯。就像你去医院看病，医生每次把脉结果可能略有不同，你不能指望「再摸一次脉」来验证上次的诊断——你得有当时的检查记录。

**长链路**：一次用户请求，背后可能是 API 入口 → 输入校验 → RAG 检索 → Embedding 计算 → 向量搜索 → Prompt 组装 → LLM 调用 → 输出解析 → 后处理 → 返回。任何一个环节出问题，最终表现都是「回答不对」。你不知道是检索没找到对的内容，还是 prompt 写偏了，还是模型本身抽风。就像快递丢了，你不知道是仓库没发货、路上丢了、还是快递员放错地方——没有追踪，你只能瞎猜。

**成本关联**：每次调用都烧 token，都花钱。你不知道谁在烧、烧了多少、烧得值不值，月底账单来了只能傻眼。传统接口调用成本几乎为零，大模型是按字收费，可观测性直接关系到钱袋子。

这三件事叠加，让 AI 应用的可观测性变成了「必须有」而不是「最好有」。

---

## 二、三大支柱：日志、追踪、指标

可观测性有三个支柱，缺一不可。

**日志**：记录「发生了什么」。某次请求的 prompt 是什么、模型返回了什么、花了多少 token、耗时多少。就像飞机的黑匣子，出事了能回放。

**追踪**：记录「请求经过了哪些环节、每个环节花了多久」。把一次请求的完整调用链串起来，从 API 到 LLM 到返回，一目了然。就像快递的物流信息，你能看到包裹在哪个节点、停了多久。

**指标**：记录「整体趋势」。延迟的 P95、错误率、token 消耗、成本曲线。日志和追踪解决「单次请求怎么了」，指标解决「系统整体健康吗」。就像体检报告，单次看病是日志，长期血压曲线是指标。

光有日志，找不到全链路；光有追踪，不知道整体趋势；光有指标，知道出事了但不知道出在哪。三个要一起上。

---

## 三、日志设计

AI 应用的日志不能瞎记，要**结构化**，而且要记对东西。

**结构化日志**：别用 `print("调用了模型")` 这种人类可读但机器难解析的格式。用 JSON，每条日志一个对象，包含 `timestamp`、`trace_id`、`event`、`data`。这样你才能按 trace_id 聚合、按 event 筛选、按字段做分析。就像病历——手写潦草没人看得懂，结构化电子病历才能检索和统计。

**必须记录的内容**：

- **prompt**：用户输入 + 系统提示 + 检索到的上下文，完整记下来。用户投诉时，你得能还原「模型当时看到了什么」。注意脱敏，别把用户隐私写进日志。
- **completion**：模型返回的完整内容。至少存一份，或者存 hash + 前 200 字预览，方便排查「答非所问」类问题。
- **token 数**：input_tokens、output_tokens，分开记。算成本、做优化都靠这个。
- **延迟**：从发请求到收到最后一个 token 的耗时，首 token 延迟（TTFT）如果能量也记上。延迟异常往往是上游限流或网络问题的信号。
- **模型**：调用了哪个模型、什么版本。模型升级后效果变差，你得能对上号。

**trace_id**：每条日志都要带上。一个用户请求可能触发多次 LLM 调用、多次检索、多次工具调用，用 trace_id 把它们串成一条链。没有 trace_id，日志就是一堆散沙。

---

## 四、链路追踪

链路追踪解决的是「一次请求到底经过了什么」。

典型的 AI 应用调用链：**API 入口** → **输入校验/守门人** → **RAG 检索**（Embedding + 向量搜索）→ **Prompt 组装** → **LLM 调用** → **输出解析** → **后处理/守门人** → **返回**。

每个环节都是一个 **Span**（跨度）。根 Span 是整个请求，子 Span 是各个步骤。每个 Span 记录：开始时间、结束时间、父 Span ID、以及该步骤的元数据（比如检索到了几条文档、调用了哪个模型）。

有了完整的 Trace，用户说「刚才那个回答不对」时，你输入 trace_id，就能看到：检索阶段返回了 3 条文档，相关性最高 0.6（偏低）；LLM 调用用了 2.3 秒；输出被守门人拦截了一次，因为检测到敏感词。问题可能出在检索质量，而不是模型——这种判断，没有链路追踪你根本做不出来。

类比一下：传统分布式追踪是看「请求从网关到 A 服务到 B 服务」的路径；AI 应用的追踪是看「请求从入口到检索到 LLM 到输出」的路径。本质一样，只是环节换成了 AI 特有的组件。

## 五、关键指标

指标用来监控系统健康，这几类必须有。

**延迟**：P50、P95、P99，以及首 token 延迟（TTFT）。用户感知的「卡不卡」主要看 TTFT；整体响应时间看 P95。延迟突然升高，可能是上游限流、网络抖动、或者检索/模型变慢。

**token 消耗**：按模型、按用户、按接口维度统计。日消耗、周消耗、趋势图。超预算前能预警，异常用户能发现。

**错误率**：4xx、5xx、超时、内容违规被拒，都算错误。错误率 spike 通常是上游故障或配置问题，要能快速发现。

**用户满意度**：如果有点赞/点踩、评分、反馈，可以做成指标。满意度下降可能是模型效果变差，也可能是 prompt 被改坏了。

**成本**：按模型、按用户的费用曲线。设置预算阈值，超 80% 预警，超 100% 自动熔断或降级。大模型是按量付费，成本指标和业务指标一样重要。

## 六、工具推荐

不想从零造轮子，可以看看这些现成方案。

**LangSmith**：LangChain 官方出品，和 LangChain 生态深度集成。Trace、日志、评估、数据集管理都有。适合用 LangChain 的团队，开箱即用。缺点是绑定 LangChain，不用 LangChain 的话接入成本高。

**Langfuse**：开源，支持多种框架（LangChain、LlamaIndex、直接调 OpenAI 也行）。Trace、指标、成本追踪、prompt 版本管理都有，界面清晰。可以自托管，数据在自己手里。适合想掌控数据、又不想从零写的团队。

**Phoenix**：Arize 开源，偏重评估和调试。Trace 有，但强项是模型评估、幻觉检测、RAG 质量分析。适合做模型效果优化、RAG 调优的场景。

**自建方案**：用 OpenTelemetry 打 Span，用 Prometheus 拉指标，用 Loki 或 ELK 存日志，自己拼一套。灵活度高，但开发和维护成本大。适合有专门基础设施团队、对定制化要求高的公司。

选型建议：**快速上线用 Langfuse 或 LangSmith**（看你用不用 LangChain）；**重评估和 RAG 调优用 Phoenix**；**要完全自控、数据不出内网用自建**。

## 七、实战：用 Langfuse 搭建可观测性

下面用 Langfuse 举个接入例子，几行代码就能把一次 LLM 调用 trace 起来。

先装包、配环境变量：

```bash
pip install langfuse
```

```bash
export LANGFUSE_SECRET_KEY="sk-lf-..."
export LANGFUSE_PUBLIC_KEY="pk-lf-..."
export LANGFUSE_HOST="https://cloud.langfuse.com"  # 自托管则改成你的地址
```

在代码里用 context manager 包住你的逻辑：

```python
from langfuse import get_client

langfuse = get_client()

# 整个请求是一个 trace，内部步骤是 span
with langfuse.start_as_current_observation(as_type="span", name="handle_request") as span:
    span.update(input=user_query)

    # RAG 检索
    with langfuse.start_as_current_observation(as_type="span", name="rag_retrieval") as rag_span:
        docs = retriever.search(user_query)
        rag_span.update(output={"doc_count": len(docs)})

    # LLM 调用
    with langfuse.start_as_current_observation(as_type="generation", name="llm_call", model="gpt-4") as gen:
        response = llm.invoke(messages)
        gen.update(input=messages, output=response.content)

# 短生命周期应用结束时 flush，确保数据发出去
langfuse.flush()
```

跑起来后，打开 Langfuse 控制台，能看到完整的 trace 树：每个 span 的耗时、输入输出、token 数。用户投诉时，根据 trace_id 或时间范围搜一下，点进去就能看到那次请求的完整链路。

Langfuse 也支持 LangChain、LlamaIndex 的自动埋点，装个 callback 就行，不用手写这么多 span。上面这段是展示「最小可用的手动埋点」长什么样。

## 八、告警策略

什么情况需要告警？别贪多，否则容易麻木，真正出事了反而忽略。

**服务不可用**：错误率超过阈值（比如 5%）、连续 N 次失败、健康检查失败。这类必须第一时间触达，电话或短信。

**延迟突增**：P95 延迟比基线高 50% 以上，或者超过业务 SLA（比如 10 秒）。可能是上游限流、网络问题、或者某次部署引入了性能退化。

**成本超预算**：日/周/月 token 消耗或费用超过设定阈值。大模型按量付费，一个异常循环或恶意用户就能让账单爆炸，要有预算告警。

**异常流量**：某个用户或接口的 QPS 突然暴增。可能是爬虫、可能是 bug 导致的重复调用，需要人工介入判断。

告警要分级：P0 直接打电话，P1 发钉钉/飞书，P2 发邮件。避免所有告警一个级别，最后大家都不看了。

---

## 结尾

AI 应用的可观测性，本质是给「黑盒」装上窗户。用户说回答错了，你能从日志里看到当时的 prompt 和 completion；能从 trace 里看到是检索、模型还是后处理出了问题；能从指标里看到延迟、成本、错误率的趋势。

不装这套东西，出了问题只能靠猜、靠用户复现、靠群里问「谁改了什么」。装上之后，从「用户说不对」到「定位到根因」可能只要十几分钟。

如果你已经在用大模型做业务，不妨回头看看：下一次用户说「回答错了」的时候，你知道从哪查吗？
