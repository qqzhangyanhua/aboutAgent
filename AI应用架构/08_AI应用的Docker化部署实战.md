# AI 应用的 Docker 化部署实战

「在我电脑上能跑啊。」——这句话听过吧？开发拍胸脯保证，运维一部署就炸。Python 版本对不上，CUDA 驱动冲突，依赖库少装了一个，模型路径写死了……AI 应用的环境复杂度，比传统 Web 高一个数量级。Docker 不是银弹，但能把「在我电脑上能跑」变成「在哪儿都能跑」。这篇文章聊聊 AI 应用怎么容器化、怎么编排、生产环境有哪些坑要躲。

---

## 一、为什么 AI 应用特别需要容器化

传统 Web 应用：装个 Node 或 Python，拉依赖，跑起来。AI 应用呢？Python 版本要 3.10，PyTorch 要 2.1，CUDA 要 11.8，transformers 要 4.36，少一个版本对不上，直接报错。这叫**依赖地狱**——不是夸张，是真地狱。

更狠的是 GPU。你本地用的是 RTX 4090，驱动 535；服务器是 A100，驱动 525。同一份代码，本地推理嗖嗖的，服务器上「CUDA version mismatch」糊你一脸。GPU 驱动、CUDA 版本、cuDNN、PyTorch 的 CUDA 变体，这四个东西必须严丝合缝对上，差一点就崩。容器化把这一整套环境打成包，开发和生产用同一个镜像，**环境一致性**从玄学变成可复现。

还有模型文件。7B 模型十几个 G，13B 更大，70B 直接上百 G。怎么传？U 盘拷？网盘拖？服务器上手动下载？容器化之后，要么打进镜像（一次构建到处跑），要么挂载共享存储（多实例共用），要么启动时远程拉取——至少有个明确的策略，而不是「运维你看着办」。

总结：AI 应用容器化的核心诉求就三个——**依赖隔离**、**环境一致**、**模型分发可控**。不搞容器也能跑，但每换一台机器就踩一遍坑，谁受得了。

多说一句：虚拟机和容器不一样。虚拟机是整台机器虚拟化，笨重；容器是进程级隔离，轻量，启动秒级。AI 推理服务要快速扩缩容，容器更适合。虚拟机适合那种「一机一用、长期跑」的场景。

## 二、Docker 基础：集装箱运输的比喻

Docker 的本质，用一句话说：**把应用和它的运行环境一起打包，像集装箱一样运到哪儿都能用**。

以前运货：散装，每到一个港口要重新装卸、检查、配齐。现在：货装进集装箱，箱子规格统一，吊起来就能走，到哪儿开箱即用。Docker 镜像就是那个集装箱——里面装着你的代码、Python 解释器、系统库、依赖包，甚至模型文件。构建一次，推送到镜像仓库，任何装了 Docker 的机器都能拉下来跑，不用再「在我这儿装一遍试试」。

几个关键概念：

- **镜像**：只读的模板，构建一次，到处复用。
- **容器**：镜像跑起来的实例，一个镜像可以起多个容器，就像同一个集装箱可以复制很多份。
- **Dockerfile**：构建镜像的配方——先装什么、再装什么、最后把代码拷进去，一步步写清楚。
- **Volume**：挂载的持久化存储，容器删了数据还在，适合放模型、日志、数据库文件。

理解这个比喻，后面写 Dockerfile、配 Compose 就不会晕。你就是在设计：这个集装箱里装啥、怎么装、运到目的地后和哪些别的集装箱配合干活。

## 三、AI 应用的 Dockerfile 怎么写

AI 应用的 Dockerfile 和普通 Python 应用不太一样，有几个地方要特别注意。

**基础镜像**：别用 `python:latest`，版本飘忽不定。用 `python:3.10-slim` 或 `python:3.11-slim`，锁死版本。要跑 GPU 的话，用 NVIDIA 官方镜像，比如 `nvidia/cuda:11.8-cudnn8-runtime-ubuntu22.04`，再在上面装 Python。或者用 `nvidia/cuda:11.8-cudnn8-devel` 如果你要编译扩展——但 runtime 通常够用，镜像更小。顺带说一句，slim 和 alpine 的区别：slim 基于 Debian，兼容性好；alpine 更小但有些 C 扩展编译会出问题，AI 依赖多，一般不折腾 alpine。

**依赖安装**：先拷 `requirements.txt`，再 `pip install`，利用 Docker 的层缓存。改代码不会触发重新装依赖，构建快。注意：如果有私有源或需要 `--no-cache-dir`，显式写上。PyTorch、transformers 这类大包，可以考虑用 `--no-cache-dir` 减小镜像体积，但构建会慢一点。顺序也有讲究：把变化少的放前面，比如先装系统依赖再装 Python 包，这样改了一行代码不会从第一层开始重建。

**模型文件**：这是最容易踩坑的。如果模型不大（几百 M），可以 `COPY` 进镜像，一次构建到处跑，部署最简单。如果模型几十 G，打进镜像会让镜像巨大，拉取慢、占空间，不推荐。更好的做法是**挂载 Volume** 或**启动时下载**，后面单独讲。还有个细节：模型路径别写死，用环境变量 `MODEL_PATH`，这样同一镜像可以挂不同目录、拉不同模型，灵活。

**工作目录和用户**：别用 root 跑应用，`RUN useradd` 建个非 root 用户，`USER` 切过去。安全规范，生产环境必备。工作目录 `WORKDIR` 设好，后面的 `COPY`、`CMD` 都基于这个路径，别搞出一堆 `/app`、`/code` 混用的混乱局面。

**启动命令**：`CMD` 或 `ENTRYPOINT` 写清楚，比如 `uvicorn app:app --host 0.0.0.0 --port 8000`。注意 `0.0.0.0`，否则容器内只监听 localhost，外面访问不到。

常见坑：

- 忘了装 `curl` 或 `wget`，健康检查失败。
- 时区不对，日志时间戳乱，排查问题的时候对不上。
- `pip install` 没加 `--no-cache-dir`，镜像多出几百 M。
- 多阶段构建时，运行阶段忘了拷模型或依赖，起不来。

写完之后 `docker build` 跑一遍，`docker run` 起个容器测一下，能正常推理再往下走。本地验证通过，再考虑推镜像仓库、上生产。

## 四、GPU 容器：NVIDIA Container Toolkit

CPU 推理慢，生产环境多半要上 GPU。Docker 默认看不到宿主机的 GPU，需要 **NVIDIA Container Toolkit**（以前叫 nvidia-docker）来打通。

安装很简单：加 NVIDIA 的 apt 源，装 `nvidia-container-toolkit`，然后 `systemctl restart docker`。装完之后，用 `--gpus all` 或 `--gpus '"device=0"'` 启动容器，容器内就能用 `nvidia-smi` 看到 GPU 了。原理是：Toolkit 在容器启动时把宿主机的 GPU 设备和驱动库挂进去，容器里的应用就能直接调 CUDA。没装 Toolkit 的话，容器里 `nvidia-smi` 会报错，PyTorch 也会退回到 CPU，推理慢到你怀疑人生。

版本对应要小心：宿主机的 NVIDIA 驱动版本要大于等于容器内 CUDA 要求的版本。比如容器用的是 CUDA 11.8，宿主机驱动至少得支持 11.8。驱动太老，容器起不来；驱动太新一般没问题，向下兼容。查版本对应关系，去 NVIDIA 官网看 CUDA Compatibility 文档，别猜。

多卡场景：`--gpus '"device=0,1"'` 指定用哪几张卡，或者 `--gpus all` 全用。K8s 里用 `nvidia.com/gpu: 1` 这样的 resource limit 来申请 GPU。Docker Compose 里写 `deploy.resources.reservations.devices`，后面编排那节会提到。

验证方式：`docker run --gpus all nvidia/cuda:11.8-base-ubuntu22.04 nvidia-smi`，能跑出 GPU 信息就说明 Toolkit 配好了。这一步别省，很多人以为装好了，结果一跑推理就卡在 CPU 上。

## 五、Docker Compose 编排多服务

AI 应用很少单兵作战，通常是模型服务 + API 网关 + 向量数据库 + Redis 缓存，一堆服务要一起跑。Docker Compose 就是为这种场景生的——一个 `docker-compose.yml` 把多个容器定义好，`docker compose up` 一键全起。

典型结构：一个 `llm-service` 跑 vLLM 或 TGI，对外提供推理 API；一个 `api-gateway` 做鉴权、限流、路由；一个 `qdrant` 或 `milvus` 做向量检索；一个 `redis` 做缓存。各服务用 `networks` 连到同一个网络，用服务名互相访问，比如 `http://llm-service:8000`。Compose 会自动做服务发现，不用配 IP，服务名就是主机名。

依赖关系用 `depends_on` 写清楚：gateway 依赖 llm-service 和 redis，llm-service 依赖 redis（如果做 KV cache）。注意 `depends_on` 只保证启动顺序，不保证服务就绪——容器起来了不代表应用已经监听端口。所以要在应用里做重试，或者用 `healthcheck` 配合 `condition: service_healthy`，等模型加载完、数据库连上再算就绪。否则 gateway 一启动就调 llm-service，人家还在加载模型，502 一堆。

GPU 给谁用：只有模型服务需要，在 `llm-service` 的 `deploy.resources.reservations` 里加 `devices: - driver: nvidia ...`。其他服务纯 CPU 就行。

环境变量用 `.env` 文件或 `environment` 配，敏感信息别写死在 yml 里，用 `env_file` 或 secrets。Volume 挂载模型目录、日志目录，数据持久化。

端口暴露：只把 gateway 的端口暴露给宿主机，其他服务内部互通就行。模型服务、数据库、Redis 都不需要直接对外，减少攻击面。`ports: "8000:8000"` 只给 gateway 配，llm-service 用 `expose: 8000` 就够了，不映射到宿主机。

## 六、模型文件的三种处理策略

模型动不动几十 G，怎么放、怎么用，有三种主流做法。

**打进镜像**：`COPY models/ /app/models`。优点：部署最简单，拉镜像就能跑，环境完全自包含，适合离线部署、边缘节点。缺点：镜像巨大，构建慢、推送慢、拉取也慢；模型一更新就要重新构建镜像，CI/CD 流水线压力大。适合：模型小（<2G）、不常更新、对部署简洁度要求高的场景。小团队、模型就一个、懒得搞复杂流程，这种最省心。

**挂载 Volume**：宿主机或 NFS 上放模型，容器启动时 `-v /data/models:/app/models` 挂进去。优点：模型和镜像分离，更新模型不用重建镜像，`docker pull` 拉的是小镜像；多容器可以挂同一个目录，共享模型，省磁盘。缺点：部署前要确保模型文件已经在目标机器上，需要额外的分发流程——rsync、对象存储同步、专门的模型管理服务，总得有一套。适合：模型大、多实例共享、有专门存储系统的生产环境。

**远程加载**：容器启动时从 Hugging Face、S3、内网文件服务下载模型到本地目录。优点：镜像最小，同一镜像可以拉不同模型，灵活。缺点：首次启动慢，依赖网络，要处理下载失败、断点续传。适合：模型经常换、多租户不同模型、边缘节点按需拉取的场景。

选哪种？模型小且固定就打进镜像；模型大且共享就挂 Volume；模型多变就远程加载。也可以组合：基础镜像不含模型，启动脚本根据环境变量决定是挂载还是下载，一份镜像多种用法。

还有个细节：远程加载时，模型目录可以挂 Volume，这样容器重启不用重新下载。第一次启动拉模型到 Volume，后续启动直接复用，省时间。

## 七、生产部署注意事项

容器能跑起来只是第一步，生产环境还有一堆细节要处理。

**健康检查**：`HEALTHCHECK` 或 Compose 的 `healthcheck` 必须配。模型服务可以 `curl` 一个轻量接口，比如 `/health`；数据库、Redis 用各自的 ping 命令。不配健康检查，负载均衡不知道实例挂了，流量还会打过去，雪崩。健康检查别写太重——别每次检查都跑一遍推理，一个简单的 HTTP 200 就行，否则检查本身就成了瓶颈。

**资源限制**：`memory`、`cpus` 要设上限，防止单个容器吃光宿主机。GPU 用 `deploy.resources.limits` 限制卡数。不限制，一个服务 OOM 可能拖垮整台机器。模型服务尤其吃内存，7B 模型加载就要十几 G，`memory` 设太小会 OOM kill，设太大又浪费，按模型规格算一算再配。

**日志**： stdout/stderr 打到控制台，Docker 会收集；但最好统一格式（JSON），方便后续用 Loki、ELK 解析。日志别写死到容器内路径，一重启就没了，挂 Volume 或直接推日志服务。

**镜像瘦身**：多阶段构建，构建阶段装编译工具，运行阶段只拷二进制和依赖，镜像能小一半。`pip install` 加 `--no-cache-dir`，删掉 `apt-get` 的缓存，`rm -rf /var/lib/apt/lists/*`。镜像小，拉取快，节点扩容时省时间。AI 镜像本来就大，能瘦一点是一点，10G 和 15G 拉取时间差不少。

**安全**：非 root 用户跑、镜像定期扫漏洞、敏感信息用 secrets 别写死在配置里。这些是老生常谈，但 AI 应用经常要读 API Key、连数据库，漏一个就够喝一壶。

**镜像仓库**：生产环境别用 Docker Hub 的 public 镜像当基础镜像，版本可能被篡改。有条件的用私有 Harbor，或至少 pin 到具体 digest，别用 `latest`。AI 模型镜像更大，推拉都慢，选离部署节点近的仓库，能省不少时间。

## 八、小结与下一篇预告

「在我电脑上能跑」的终极解决方案，就是把「你的电脑」打包带走。Docker 让 AI 应用的环境从玄学变成可复现，依赖隔离、GPU 支持、多服务编排，一套流程跑通，开发和生产少扯皮。

从架构系列的前几篇一路看下来，你已经有了后端分层、流式输出、网关、缓存、多租户、可观测、成本控制的底子。容器化是最后一块拼图——把这一切塞进集装箱，运到服务器上，稳稳跑起来。下一阶段就是规模化和成本优化，那又是另一套学问了。
