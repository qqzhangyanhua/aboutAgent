# 流式输出的工程实现：SSE / WebSocket / 分块传输

你有没有试过问 ChatGPT 一个复杂问题，然后盯着屏幕干等？前几秒还好，过了十几二十秒，你开始怀疑：是不是卡了？是不是网络断了？要不要刷新？

其实模型在后台吭哧吭哧算着呢，只是你啥也看不到。就像你跟人聊天，对方沉默三十秒，你肯定以为他掉线了——哪怕他只是在认真思考。

**流式输出**干的事，就是让 AI「边想边说」。模型每算出一个 token 就推给你，你立刻能看到，体验从「等一坨结果」变成「看它慢慢写」。这篇文章聊聊这件事在工程上怎么实现。

---

## 一、为什么 AI 应用需要流式输出

两个核心原因：**长耗时**和**用户感知**。

大模型生成一段回复，少则几秒，多则几十秒。如果等全部生成完再一次性返回，用户会经历漫长的「白屏期」。心理学上有个概念叫**可感知延迟**——人对超过 1 秒的等待就开始焦虑，超过 10 秒就容易放弃。流式输出把「等 30 秒看结果」拆成「每秒看到几个字」，用户知道系统在工作，耐心会好很多。

打个比方：餐厅上菜。一种是等所有菜做好一起端上来，你饿着肚子干等二十分钟；另一种是凉菜先上、热菜陆续来，你边吃边等。后者体验好得多，流式输出就是后者。

---

## 二、三种技术方案对比

实现「边算边推」的技术路线主要有三条：**SSE**、**WebSocket**、**HTTP Chunked Transfer**。

| 方案 | 方向 | 协议 | 适用场景 |
|------|------|------|----------|
| SSE | 服务端→客户端 | HTTP | AI 回复、通知推送 |
| WebSocket | 双向 | 独立协议 | 聊天室、实时协作 |
| HTTP Chunked | 服务端→客户端 | HTTP | 通用流式响应 |

**SSE**（Server-Sent Events）是「单向广播」：服务端持续往客户端推数据，客户端只负责接收。基于 HTTP，实现简单，浏览器原生支持 `EventSource`。AI 场景里，模型输出是单向的，SSE 天然契合。

**WebSocket** 是「双向通道」：建立连接后，双方随时收发。适合需要频繁交互的场景，比如多轮对话里用户随时打断、或者需要实时传心跳。但实现比 SSE 复杂，对「只推不拉」的 AI 回复来说有点杀鸡用牛刀。

**HTTP Chunked Transfer** 是 HTTP 1.1 自带的机制：响应体可以分块传输，不必等全部生成完再发。任何支持流式响应的 HTTP 客户端都能用，不依赖特殊 API。缺点是格式需要自己约定，没有 SSE 那种现成的 `data:` 规范。

## 三、SSE 详解

SSE 是目前 AI 应用里用得最多的方案，原因很简单：**够用、好写、兼容性好**。

### 协议格式

SSE 的响应长这样：

```
Content-Type: text/event-stream
Cache-Control: no-cache
Connection: keep-alive

data: {"token": "你"}

data: {"token": "好"}

data: {"token": "，"}

data: {"token": "世界"}

data: [DONE]
```

每条消息以 `data:` 开头，后面跟内容，用两个换行分隔。`[DONE]` 是常见的结束标记，不是标准规定，只是约定俗成。

### 后端实现（Python/FastAPI）

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import json

app = FastAPI()

async def generate_tokens(prompt: str):
    """模拟大模型逐 token 生成"""
    response = "你好，我是AI助手。"
    for char in response:
        yield json.dumps({"token": char}) + "\n"

@app.post("/chat/stream")
async def chat_stream(prompt: str):
    async def event_generator():
        async for chunk in generate_tokens(prompt):
            yield f"data: {chunk}\n\n"
        yield "data: [DONE]\n\n"

    return StreamingResponse(
        event_generator(),
        media_type="text/event-stream",
        headers={
            "Cache-Control": "no-cache",
            "Connection": "keep-alive",
            "X-Accel-Buffering": "no",  # 禁用 Nginx 缓冲
        },
    )
```

关键点：用 `StreamingResponse` 配合 `async generator`，每 `yield` 一块就立刻发出去。`X-Accel-Buffering: no` 是为了避免 Nginx 等反向代理把流缓冲掉。

### 前端消费（原生 JS）

```javascript
const eventSource = new EventSource("/chat/stream?prompt=你好");

eventSource.onmessage = (event) => {
  const data = event.data;
  if (data === "[DONE]") {
    eventSource.close();
    return;
  }
  const parsed = JSON.parse(data);
  document.getElementById("output").innerText += parsed.token;
};
```

`EventSource` 是浏览器原生 API，自动处理重连、解析 `data:` 行。但注意：**EventSource 只支持 GET**，如果要用 POST 传 prompt，得用 `fetch` 手动处理流。

## 四、WebSocket 详解

WebSocket 适合「双向、长连接、高频交互」的场景。

比如：用户一边看 AI 写代码，一边随时点「停」；或者多人在线协作，一个人改 prompt，其他人实时看到输出变化。这类需求里，客户端需要主动发消息，SSE 的单向特性就不够了。

WebSocket 建立连接后，服务端和客户端都可以随时 `send`。格式通常用 JSON，自己定义 `type` 区分消息类型：

```python
# 服务端伪代码
await websocket.send_json({"type": "token", "content": "你"})
await websocket.send_json({"type": "token", "content": "好"})
await websocket.send_json({"type": "done"})
```

```javascript
// 客户端
ws.onmessage = (event) => {
  const msg = JSON.parse(event.data);
  if (msg.type === "token") outputEl.textContent += msg.content;
  if (msg.type === "done") ws.close();
};
```

代价是：要自己维护连接状态、心跳、重连逻辑，部署时还要考虑 WebSocket 的代理配置（很多负载均衡默认不转发 WebSocket）。

## 五、实战：用 FastAPI + SSE 实现流式 AI 回复

下面是一套完整的、可运行的实现，对接 OpenAI 兼容的 API（如 OpenAI、国产大模型等）。

```python
from fastapi import FastAPI, HTTPException
from fastapi.responses import StreamingResponse
import httpx
import json

app = FastAPI()

async def stream_from_llm(prompt: str):
    """调用大模型 API，流式返回 token"""
    async with httpx.AsyncClient() as client:
        async with client.stream(
            "POST",
            "https://api.openai.com/v1/chat/completions",
            json={
                "model": "gpt-4",
                "messages": [{"role": "user", "content": prompt}],
                "stream": True,
            },
            headers={"Authorization": "Bearer YOUR_API_KEY"},
        ) as response:
            if response.status_code != 200:
                raise HTTPException(status_code=502, detail="LLM API error")
            async for line in response.aiter_lines():
                if line.startswith("data: "):
                    data = line[6:]
                    if data.strip() == "[DONE]":
                        break
                    try:
                        chunk = json.loads(data)
                        delta = chunk.get("choices", [{}])[0].get("delta", {})
                        content = delta.get("content", "")
                        if content:
                            yield json.dumps({"token": content})
                    except json.JSONDecodeError:
                        pass

@app.post("/v1/chat/stream")
async def chat_stream(request: dict):
    prompt = request.get("prompt", "")
    if not prompt:
        raise HTTPException(status_code=400, detail="prompt required")

    async def event_generator():
        try:
            async for chunk in stream_from_llm(prompt):
                yield f"data: {chunk}\n\n"
            yield "data: [DONE]\n\n"
        except Exception as e:
            yield f"data: {{\"error\": \"{str(e)}\"}}\n\n"

    return StreamingResponse(
        event_generator(),
        media_type="text/event-stream",
        headers={
            "Cache-Control": "no-cache",
            "Connection": "keep-alive",
            "X-Accel-Buffering": "no",
        },
    )
```

要点：`stream=True` 让 API 返回流式响应；`response.aiter_lines()` 逐行读取；解析每行的 `delta.content` 推给前端。

## 六、前端如何消费流式数据

因为要 POST 传 prompt，不能用 `EventSource`，改用 `fetch` 的流式读取。

### React 示例

```jsx
function ChatStream({ prompt }) {
  const [text, setText] = useState("");
  const [loading, setLoading] = useState(false);

  const stream = async () => {
    setLoading(true);
    setText("");
    const res = await fetch("/v1/chat/stream", {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify({ prompt }),
    });
    const reader = res.body.getReader();
    const decoder = new TextDecoder();
    while (true) {
      const { done, value } = await reader.read();
      if (done) break;
      const chunk = decoder.decode(value, { stream: true });
      const lines = chunk.split("\n").filter((l) => l.startsWith("data: "));
      for (const line of lines) {
        const data = line.slice(6);
        if (data === "[DONE]") return;
        try {
          const parsed = JSON.parse(data);
          if (parsed.token) setText((prev) => prev + parsed.token);
        } catch (_) {}
      }
    }
    setLoading(false);
  };

  return (
    <div>
      <button onClick={stream} disabled={loading}>发送</button>
      <pre>{text}</pre>
    </div>
  );
}
```

### Vue 示例

```vue
<template>
  <div>
    <button @click="stream" :disabled="loading">发送</button>
    <pre>{{ text }}</pre>
  </div>
</template>

<script setup>
import { ref } from "vue";

const text = ref("");
const loading = ref(false);

async function stream() {
  loading.value = true;
  text.value = "";
  const res = await fetch("/v1/chat/stream", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ prompt: "你好" }),
  });
  const reader = res.body.getReader();
  const decoder = new TextDecoder();
  while (true) {
    const { done, value } = await reader.read();
    if (done) break;
    const chunk = decoder.decode(value, { stream: true });
    chunk.split("\n")
      .filter((l) => l.startsWith("data: "))
      .forEach((line) => {
        const data = line.slice(6);
        if (data === "[DONE]") return;
        try {
          const parsed = JSON.parse(data);
          if (parsed.token) text.value += parsed.token;
        } catch (_) {}
      });
  }
  loading.value = false;
}
</script>
```

核心逻辑一样：`res.body.getReader()` 拿到流，`TextDecoder` 解码，按行解析 `data:`，拼到 UI 上。

## 七、流式输出的工程挑战

流式不是银弹，上线后会遇到一堆细节问题。

**错误处理**：模型 API 可能中途超时、限流、返回 5xx。流已经开了，不能像普通请求那样直接返回错误码。常见做法是在流里塞一条 `{"error": "xxx"}` 消息，前端解析后提示用户，并允许重试。

**断线重连**：网络抖动、代理超时都会导致连接中断。SSE 的 `EventSource` 有自动重连，但用 `fetch` 就得自己实现：记录已接收的 token 数，重连时带 `last_event_id` 或 `offset`，让服务端从断点续传。很多场景下「从头再来」也能接受，看业务要求。

**Token 统计**：计费、限流都要算 token 数。流式场景下，要么服务端在推每个 token 时累加，最后在 `[DONE]` 前发一条 `{"usage": {...}}`；要么等流结束后，由后端异步统计。别指望前端自己数，用户可能关页面。

**日志与追踪**：流式请求持续时间长，日志不能只打「请求开始」和「请求结束」。需要在关键节点打日志：首 token 延迟、每秒 token 数、是否异常终止，方便排查「为什么这次特别慢」。

## 八、选型建议

- **纯 AI 回复、单向推送**：用 **SSE**。简单、稳定、浏览器支持好，FastAPI / Flask / Express 都能轻松实现。
- **需要用户随时打断、或双向实时交互**：用 **WebSocket**。愿意多写点代码、处理连接管理的话，体验更灵活。
- **已有 HTTP 客户端、不想引入新协议**：用 **HTTP Chunked**，自己定义 `\n` 分隔的 JSON 行即可，兼容性最好，但前端要自己解析。

大多数 AI 应用，SSE 就够了。先把「边想边说」跑起来，再根据需求决定要不要上 WebSocket。

---

流式输出本质上是在解决「等待焦虑」——把漫长的黑盒过程，变成可见的、持续的反馈。技术选型没有绝对的对错，关键是理解每种方案的适用场景，然后选一个够用、好维护的。下次你看到 ChatGPT 一个字一个字往外蹦，可以想想：背后可能就是一条 SSE 连接，服务端在那边 `yield`，你在这边 `append`，简单，但有效。
