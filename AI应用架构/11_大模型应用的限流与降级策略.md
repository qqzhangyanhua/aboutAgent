# 大模型应用的限流与降级策略

双十一零点，你们的 AI 客服准时上线。前半小时一切正常，用户问啥答啥，响应时间稳稳在一秒内。到了凌晨一点，促销活动加码，流量突然翻了三倍，你眼睁睁看着监控面板上的平均响应时间从 1 秒爬到 5 秒、10 秒，最后直接飙到 30 秒——用户开始骂娘，客服电话被打爆，老板在群里问「AI 怎么挂了」。其实没挂，只是被挤爆了。大模型推理慢、成本高、上游 API 还有速率限制，流量一上来，不提前做好限流和降级，就是这种下场。这篇把限流算法、多维度限流、降级策略、熔断机制讲清楚，让你下次大促不至于手忙脚乱。

顺便说一句：灰度发布帮你「慢慢放量」，限流降级帮你「扛住洪峰」。这两个是配套的，一个管「怎么上」，一个管「上了之后别崩」。

---

## 一、为什么大模型应用特别需要限流

传统接口，一个请求几十毫秒就返回了，数据库查一下、缓存读一下，撑得住高并发。大模型不一样——单次推理动不动几秒十几秒，GPU 就那么几块，排队的人一多，谁都别想快。而且贵：GPT-4 一个请求几毛钱，流量一上来，账单能把你吓一跳。更狠的是，上游 API 厂商自己就有限制：OpenAI 每分钟多少请求、每用户每天多少 token，超了直接 429 把你拒之门外。你不自己限流，等上游帮你限，那就是集体超时、集体失败，用户体验烂到家。

打个比方：传统接口像自助餐厅，大家排队取餐，速度快，翻台率高；大模型像米其林三星，一桌菜要做半小时，厨房就那么大，来一百桌客人同时点单，厨师直接崩溃。限流就是「门口发号」，控制同时进店的人数，保证已经在吃的客人能正常吃完，而不是所有人都挤在门口干等、最后谁也吃不上。

成本这块再啰嗦一句：大模型按 token 计费，一个长对话可能吃掉几万 token，顶几十个短问答。你不限流，来个恶意用户或者爬虫疯狂刷长文本，一晚上能把一个月的预算烧光。限流不只是保稳定，也是保钱包。

---

## 二、限流算法：令牌桶、漏桶和滑动窗口

限流的核心是「控制单位时间内通过的请求数」，具体怎么控，有三种常见算法，各有各的脾气。

**令牌桶**：想象一个桶，每隔固定时间往里面丢一个令牌，桶有容量上限。请求来了，先拿一个令牌，拿得到就放行，拿不到就拒绝或排队。桶里平时可以攒令牌，流量低的时候攒着，流量突然上来可以「 burst 」一下，短时间多放几个请求。适合「平时平稳、偶尔有尖峰」的场景，比如用户突然连续发好几条消息，你允许他 burst 几下，但长期平均速率还是控住的。实现简单，Redis 就能做，很多网关默认就用这个。

**漏桶**：桶底有个洞，水（请求）从上面进、从下面漏，漏的速率是固定的。不管上面倒多猛，下面漏的速度不变，多出来的水要么溢出（拒绝），要么在桶里排队。和令牌桶的区别：漏桶是「匀速输出」，不允许 burst；令牌桶允许短时突发。漏桶更适合「下游消化能力固定」的场景，比如你调用的推理服务 QPS 上限就是 10，那你必须严格按 10 个/秒往外放，多了也没用，只会把队列撑爆。

**滑动窗口**：把时间切成小格子，比如每分钟 60 个格子，每个格子记一下这一秒有多少请求。当前时间点的「窗口」是过去 60 秒，窗口内总请求数不能超阈值。比固定窗口更平滑——固定窗口是「每分钟重置」，会导致窗口边界处流量突增（比如 59 秒和 1 秒各来一波，两波都能过）；滑动窗口没有这种边界效应，限流更精确。代价是计算和存储稍复杂，要维护一个滑动的时间序列。

选型上：简单场景用令牌桶，要严格匀速用漏桶，对边界敏感、要求精确的用滑动窗口。大多数大模型应用的限流，令牌桶就够用了。

实现时有个坑要注意：分布式环境下，限流状态存在哪？单机内存里的话，多实例各自为政，总 QPS 会变成 N 倍；存 Redis 的话，每次请求都要网络往返，延迟会上去。折中方案是「本地桶 + 定期同步」：每个实例维护一个本地桶，定期从 Redis 拉取全局配额做校正，既能限住总量，又不会每次请求都打 Redis。具体实现看你的延迟要求和一致性要求，没有银弹。

## 三、多维度限流：按用户、按接口、按模型、按 Token

光「每秒 N 个请求」不够，大模型场景要分维度限，不然会出现「一个土豪用户把额度全占了，其他用户饿死」的情况。

**按用户限流**：每个用户独立一个桶，免费用户每分钟 5 次，付费用户 50 次，VIP 不限。这样既防止单用户滥用，又保证多用户之间的公平。实现时用 user_id 做 key，每个 key 一个限流计数器，Redis 的 INCR + EXPIRE 就能搞定，或者用现成的限流中间件。

**按接口限流**：不同接口消耗不一样。简单问答可能 1 秒一个，长文档分析可能要 30 秒，你不能把它们混在一起算。给「重接口」单独设更低的 QPS，比如文档分析每分钟 2 次，简单问答每分钟 20 次。这样重活不会把轻活挤死。

**按模型限流**：GPT-4 和 GPT-3.5 的配额是分开的，你调用的时候也要分开限。大模型贵、慢，限得严一点；小模型便宜、快，可以放宽松。否则大家都抢 GPT-4，GPT-3.5 闲着也是闲着。

**按 Token 消耗限流**：上游 API 经常按 token 计费、按 token 限流，你光数请求数不够，还得数 token。比如每个用户每天 10 万 token，超了就不给调大模型，只能走小模型或缓存。这样既控成本，又符合上游的约束。实现时要注意：token 数是请求返回后才知道的，你可以在请求前根据历史平均或用户等级预估一个上限，超了先拒绝；或者先放行，返回后扣减额度，下次请求时检查——后者更精确但实现复杂，前者简单但可能误伤。看业务需求选。

实际落地时，这几个维度往往叠加：先按用户限总 QPS，再按接口限单接口 QPS，再按模型限单模型 QPS，最后按 token 限日消耗。一层层筛，哪一层超了就在哪一层拒绝或降级。

限流后的处理方式也有讲究：直接拒绝是最简单，但用户体验差；排队等待可以保留请求，但队列太长会超时，而且占着连接不释放；返回降级结果（比如走缓存、走小模型）是更友好的做法，用户至少能拿到东西。所以限流和降级经常绑在一起：限流触发时，不一定是 429，可以自动走降级链路，用户无感知。

## 四、降级策略：五星级到快餐到方便面

限流是「拦」，降级是「换」。流量太大、服务扛不住的时候，不能直接报错，得想办法让用户还能用上，哪怕体验差一点。降级就是「降级体验、保住可用性」。

可以想象成吃饭：正常情况下去五星级酒店，服务好、环境好；人太多排不上队了，降一级去快餐店，快、便宜、能吃饱；快餐店也爆满了，再降一级，回家泡方便面，好歹不饿肚子。大模型应用的降级链路类似：**大模型 → 小模型 → 缓存 → 规则兜底**。

第一级：大模型（GPT-4、Claude 等）正常服务，效果最好。第二级：大模型扛不住了，自动切到小模型（GPT-3.5、小参数开源模型），效果略差但快、便宜，大部分简单问题能答。第三级：小模型也扛不住，或者命中缓存了，直接返回缓存里的历史回答，零延迟、零成本。第四级：缓存也没有，上规则兜底——关键词匹配、模板回复、或者直接告诉用户「当前服务繁忙，请稍后再试」，至少给个明确反馈，而不是白屏或 500 错误。

规则兜底不是敷衍，可以做得挺聪明。比如客服场景，常见问题「怎么退款」「订单在哪查」，完全可以做成关键词 + 模板，命中率不低；再比如「当前排队人数较多，预计等待 3 分钟，您也可以先看看帮助中心」，给用户选择，比干等着强。兜底层设计好了，降级体验不会太差，用户能接受。

每一级都要提前配置好触发条件：大模型延迟超过多少秒、错误率超过多少、队列满到什么程度，就自动触发下一级降级。降级是自动的，不是人工切，否则真出事了等你反应过来，用户已经跑光了。

降级链路的顺序可以按业务调。有的团队把「缓存」放在「小模型」前面，因为缓存零成本、零延迟，命中率高的话能省一大笔；有的团队把「小模型」放前面，因为缓存容易过期、相似问题匹配不准，小模型至少能生成新内容。没有标准答案，看你的缓存命中率、用户对「新鲜度」的要求、以及成本结构。多跑几轮压测，看哪种组合在你场景下更划算。

## 五、模型路由降级：GPT-4 扛不住时自动切 GPT-3.5

模型路由降级是降级策略里最常见的一种：同一个请求，优先走大模型，大模型不行了自动走小模型。

实现上，在路由层加一个「健康检查」和「降级开关」。比如调用 GPT-4，如果连续 3 次超时、或者 5 分钟内错误率超过 20%，就自动把流量切到 GPT-3.5。GPT-3.5 便宜、快、稳定，大部分场景效果够用，用户感知可能是「回答稍微简单了点」，但总比一直转圈强。等 GPT-4 恢复了，再慢慢把流量切回去，可以设一个「试探」机制：先放 5% 流量回 GPT-4，没问题再逐步放量，避免刚恢复又被一波打挂。

健康检查可以主动做，也可以被动做。主动：定时发探测请求，看上游响不响应；被动：正常请求的成败、延迟作为信号。被动更真实，因为探测请求和真实请求可能走不同路径；主动能提前发现问题，不用等用户撞上。两者结合最好，被动为主、主动兜底。

还有个细节：不同接口的降级策略可以不一样。简单问答可以大胆降，GPT-3.5 完全够用；代码生成、长文档分析这类复杂任务，可能宁愿排队等 GPT-4，也不要降级到小模型糊弄，否则用户会骂「什么破 AI」。按业务重要性分级，核心场景保质量，边缘场景保可用。

模型路由降级还可以和「负载均衡」结合。比如你有多个上游 API 供应商（OpenAI、Azure、自建），某个挂了，自动把流量切到其他家，用户无感知。这叫「多活」，比单点依赖稳得多，当然成本也高——要维护多套对接、多套监控。小团队可以先单点 + 熔断 + 降级，等规模上去了再考虑多活。

## 六、优雅兜底：给用户有意义的反馈

降级不是为了省事，是为了让用户知道「发生了什么、该怎么办」。最烂的做法是直接返回 500 或者超时白屏，用户一脸懵：是我网坏了？还是你们挂了？还是我操作错了？

优雅兜底的意思是：**无论走到哪一级降级，都要给用户一个明确、有用的反馈**。比如「当前访问量较大，已为您切换至快速模式，回答可能略简略」——用户知道是降级了，预期被管理好了，不会觉得是 bug。再比如「该问题需要深度分析，预计等待 1～2 分钟，您也可以选择简化回答」——给用户选择权，等得起就等，等不起就降级。缓存命中时可以说「根据历史相似问题，为您推荐以下回答」——坦诚是缓存，用户反而觉得你们聪明。

错误的时候更要人话化。「服务暂时不可用，请 30 秒后重试」比「Internal Server Error」强一万倍。「您今日的免费额度已用完，升级会员可继续使用」比「429 Too Many Requests」强一万倍。技术术语留在日志里，给用户看的要像人说的话。

兜底文案最好提前写好，存在配置里，按场景切换。限流触发时显示什么、降级触发时显示什么、熔断触发时显示什么，都要有预案。别等到真出事了现场编，容易说错话、或者暴露内部信息。有条件的话，让产品或运营过一遍文案，技术同学写的往往太「工程师思维」，用户看不懂。

## 七、熔断机制：上游 API 挂了怎么办

限流和降级解决的是「流量太大」和「主动降级体验」的问题，熔断解决的是「上游挂了」的问题。你调 OpenAI 的 API，人家机房炸了、或者账号被封了，你这边会疯狂重试、疯狂超时，把你自己也拖死。熔断就是：**检测到上游持续失败，就暂时不再请求，等它恢复再说**。

经典熔断有三个状态：关闭、打开、半开。关闭时正常请求；失败率或超时率超过阈值，变成打开，直接拒绝所有请求，不再调用上游，快速失败；过一段时间（比如 30 秒），变成半开，放少量请求去试探上游好了没，成功了就恢复关闭，失败了就继续打开。这样上游挂的时候，你不会跟着一起挂，而是快速失败、返回降级结果，等上游好了再自动恢复。

大模型场景里，熔断要配合降级用：熔断打开时，不是直接报错，而是走降级链路——切小模型、走缓存、或者规则兜底。用户感知是「服务变慢了/变简单了」，而不是「服务挂了」。等上游恢复，熔断关闭，再慢慢把流量切回大模型。

熔断的阈值要调好。太敏感：上游偶尔抖一下，你就熔断，频繁开关，用户体验割裂；太迟钝：上游已经挂了，你还傻乎乎重试，拖死自己。一般用「连续失败次数」或「时间窗口内失败率」做判断，比如 10 秒内失败 5 次就熔断，或者失败率超过 50% 持续 30 秒就熔断。具体数值要结合你的上游稳定性、超时设置、业务容忍度来调，没有万能配置。上线前用混沌工程模拟一下上游挂掉，看看熔断能不能及时触发、恢复能不能自动完成，心里有数再放量。

## 八、监控与告警：限流降级不能黑盒运行

限流、降级、熔断都配好了，还得能看见。谁被限了、谁走了降级、熔断什么时候开的，都要有监控和日志。否则出了问题你都不知道是限流触发了还是真的挂了，排查起来抓瞎。

关键指标：限流拒绝率、降级触发次数、熔断开关次数、各层级（大模型/小模型/缓存/规则）的流量占比。这些做成大盘，实时看；异常时告警，比如限流拒绝率突然飙高、降级占比超过 50%，说明上游或者流量出问题了，要介入。日志里记录每个请求走了哪条路、为什么（超时/错误率/队列满），方便事后分析。限流降级不是「配完就忘」，要当成日常运维的一部分，定期看数据、调阈值。

## 九、小结与下一篇预告

大模型应用天生「重」：推理慢、成本高、上游有限制。流量一大，不提前做好限流和降级，就是集体超时、集体崩溃。限流用令牌桶、漏桶或滑动窗口，按用户、接口、模型、Token 多维度控；降级从大模型一路降到小模型、缓存、规则兜底，像从五星级到快餐到方便面，保不住体验就保可用性；熔断在上游挂掉时快速失败，配合降级给用户有意义的反馈，而不是白屏和 500。把这些都配好，大促来了至少能扛住，不会手忙脚乱。

灰度发布、限流降级都搞定了，你的 AI 应用已经能在流量和故障面前站稳脚跟。但还有一个问题绕不开：用户数据、对话记录、模型输出，这些怎么存、怎么传、怎么合规？下一篇聊 **数据安全与合规**——大模型应用在隐私和监管面前该怎么设计，才能既好用又不踩雷。
