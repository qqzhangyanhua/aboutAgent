# 多租户 AI 平台架构设计

你的 AI 产品要卖给 100 家公司。A 公司上传了内部财报，B 公司灌进了客户名单，C 公司塞了一堆竞品分析。有一天 A 公司的员工问 AI：「我们 Q3 的营收是多少？」——系统要是把 B 公司的数据也掺进来，或者让 B 公司的人能搜到 A 公司的文档，这单生意就黄了。

每家的数据不能互相看到，这就是**多租户**。听起来简单，做起来一堆坑。这篇文章聊聊 SaaS 化 AI 平台的多租户架构该怎么设计。

---

## 一、什么是多租户

多租户不是什么新概念，传统 SaaS 早就玩透了。用**公寓楼**来类比最直观。

一栋楼，大家共享基础设施：同一套水电管网、同一部电梯、同一个物业。但每家关上门，就是自己的地盘。你不会进邻居家上厕所，邻居也不会来你家蹭网。楼是共享的，空间是隔离的。

AI 平台的多租户也一样。**共享**的是：模型推理服务、向量数据库集群、API 网关、算力资源。**隔离**的是：每家企业的知识库、对话历史、训练数据、业务配置。A 公司调用的模型和 B 公司用的是同一套 GPU，但 A 公司喂给模型的数据，B 公司永远看不到。

多租户的本质，就是在**共享**和**隔离**之间划一条清晰的线。线划对了，成本摊薄、运维省心；线划错了，数据泄露、合规暴雷。

---

## 二、AI 平台多租户的特殊挑战

传统 SaaS 多租户，主要操心的是数据库行级隔离、用户权限、资源配额。AI 平台在此基础上，多了几层麻烦。

**数据隔离更复杂**。不只是「这条记录属于哪个租户」，而是：向量库里的 Embedding、RAG 检索时的上下文、对话历史里的敏感信息、微调用的训练集——每一层都要隔离。漏一层，就可能串数据。比如向量库没有按租户分 namespace，A 公司上传的文档，可能被 B 公司的 RAG 检索到。

**模型和配置要隔离**。不同租户可能用不同的模型：有的要 GPT-4，有的用国产模型；有的要 32K 上下文，有的 4K 就够了。Prompt 模板、系统指令、温度参数，都可能按租户定制。你不能把 A 公司的 Prompt 套到 B 公司的请求上，也不能让一个租户的异常配置影响其他租户。

**成本分摊要算清楚**。大模型按 token 计费，一个租户疯狂调用，成本可能炸掉。你需要在调用链的每一环打上**租户 ID**，按租户统计 token 消耗、API 调用次数、存储用量，才能做账单、做配额、做限流。

**自定义需求多**。有的租户要接自己的知识库，有的要接私有模型，有的要自定义工作流。多租户架构不能只支持「大家用同一套配置」，要能扩展**租户级配置**，而不影响其他租户。

---

## 三、三种隔离模式

多租户的隔离程度，可以从「完全隔离」到「完全共享」拉一条光谱。常见有三种模式。

**完全隔离**：每个租户一套独立环境——独立的数据库、独立的向量库、独立的推理服务。就像每家独栋别墅，水电自己管。数据绝对安全，但成本高、运维重，适合对合规要求极高的金融、医疗客户。租户一多，机器数量线性增长，撑不住。

**共享基础设施、隔离数据**：这是最主流的做法。模型服务、向量库、网关都是共享的，但数据层按租户隔离——每条记录带 `tenant_id`，查询时强制过滤；向量库按租户分 namespace 或分 collection；对话历史按租户分表或分库。就像公寓楼，共享电梯和管网，各家关上门各自为政。成本可控，隔离靠逻辑实现，需要严格把控每一层。

**完全共享**：所有租户共享同一套数据存储，只靠应用层的权限控制区分。成本最低，但风险最大——一个 SQL 漏写 `tenant_id`，就可能全量泄露。适合对隔离要求不高的场景，比如内部工具、Demo 环境。生产级 SaaS 一般不推荐。

实际选型时，大多数 AI 平台会选择**第二种**：共享算力和模型，隔离数据和配置。在成本和安全性之间找到平衡点。

---

## 四、数据隔离设计

数据隔离是多租户的核心。AI 平台涉及的数据类型多，每一类都要想清楚怎么隔离。

**向量库隔离**。RAG 场景下，向量库存的是文档的 Embedding。如果所有租户的向量混在一起，检索时可能串数据。做法有两种：一是**按租户分 collection**（或 namespace），每个租户一个独立的 collection，查询时只查该租户的；二是**单 collection 加 tenant_id 过滤**，每条向量记录带 `tenant_id`，检索时在 filter 里强制加上 `tenant_id = 当前租户`。前者隔离更彻底，后者存储更紧凑，但必须保证 filter 不会被绕过。

**知识库隔离**。知识库不只是向量，还有原始文档、解析后的 chunk、元数据。文档存储（如 S3、OSS）要按租户分 bucket 或分前缀；数据库里知识库的元信息要带 `tenant_id`；文档解析、chunk 生成、Embedding 写入的流水线，全程携带租户上下文，不能丢。

**对话历史隔离**。用户的聊天记录可能包含敏感信息。存储时，对话表必须有 `tenant_id` 和 `user_id`，查询时双重过滤。如果做对话的向量化存储（用于记忆、检索），同样要按租户隔离，避免 A 公司的对话被 B 公司的 RAG 搜到。

**原则**：所有持久化数据，要么在 schema 里显式带 `tenant_id`，要么在物理上分库分表。中间件、ORM、查询构建器要支持**租户上下文自动注入**，避免开发手写漏了过滤条件。

## 五、模型和配置隔离

不同租户可能用不同的模型、不同的 Prompt、不同的参数。这一层也要隔离。

**模型隔离**。租户 A 买的是 GPT-4 套餐，租户 B 买的是国产模型套餐。网关或路由层要根据 `tenant_id` 查租户配置，决定把请求转发到哪个模型。可以维护一张「租户 → 模型」映射表，或者租户配置里有 `default_model` 字段。调用时，从请求上下文取出 `tenant_id`，查表，再路由。

**Prompt 和系统指令隔离**。有的租户要自定义系统 Prompt，比如「你是一个专业的法律顾问」；有的要加业务规则，比如「回答时必须引用知识库」。这些配置存在租户配置表里，调用模型前按租户加载，注入到请求中。注意不要串：A 的 Prompt 不能出现在 B 的请求里。

**参数隔离**。temperature、max_tokens、top_p 等，也可以按租户配置。有的租户要创造性高一点，temperature 调高；有的要稳定输出，temperature 调低。这些参数和模型、Prompt 一起，组成租户的「推理配置」，统一管理。

**实现方式**：在网关或编排层，请求进来先解析 `tenant_id`（从 API Key、JWT、Header 等获取），再加载该租户的配置，组装成完整的模型请求。配置可以缓存，避免每次查库，但要支持租户配置变更后的热更新。

## 六、成本分摊和计量

多租户平台要算清楚每个租户用了多少资源，才能做计费、做配额、做成本分析。

**Token 计量**。大模型按 token 计费，每次调用要记录：租户 ID、模型、输入 token 数、输出 token 数、时间戳。可以在网关层统一拦截，解析响应里的 `usage` 字段，写入计量表。按租户聚合，就能得到每个租户的日/月消耗。

**API 调用次数**。除了 token，调用次数本身也可以作为计费维度。每次请求计数，按租户汇总。

**存储计量**。知识库文档量、向量数量、对话历史条数，按租户统计。可以用定时任务扫描，或者写入时累加计数器。

**配额和限流**。根据计量结果，对超配额租户做限流或拒绝。比如某租户本月 token 额度 100 万，用了 95 万时告警，用了 100 万后拒绝新请求。限流逻辑要在网关层实现，按 `tenant_id` 维度的令牌桶或滑动窗口。

**账单**。计量数据汇总后，可以按租户生成账单，对接财务系统。也可以做成本分析，看哪些租户消耗大、哪些模型贵，优化定价和资源分配。

## 七、实际架构方案

下面是一个典型的多租户 AI 平台架构，按层次拆解。

**接入层**：用户请求通过 API 网关进入。网关负责认证、解析租户身份（从 API Key 或 JWT 中提取 `tenant_id`）、限流、路由。每个请求打上 `tenant_id`，贯穿后续所有环节。

**编排层**：负责组装 RAG、调用模型、管理对话。从租户配置中心加载该租户的模型、Prompt、知识库绑定关系；从向量库检索时带上 `tenant_id` 过滤；调用模型时注入租户的 Prompt 和参数；写入对话历史时带上 `tenant_id`。

**模型层**：大模型网关，支持多模型、多 Key。编排层传入目标模型，网关负责转发、负载均衡、降级。计量中间件在网关处拦截请求和响应，按 `tenant_id` 记录 token 消耗。

**数据层**：关系型数据库存租户元数据、用户、知识库配置、对话历史，所有表带 `tenant_id`，查询强制过滤。向量库按租户分 collection 或单 collection 加 filter。对象存储按租户分前缀存文档。

**配置中心**：租户的模型配置、Prompt 模板、额度、功能开关，集中管理。支持按租户热更新，编排层按需拉取或订阅变更。

架构图可以这样画：最上层是「租户 A / 租户 B / 租户 C」的请求入口，汇聚到 API 网关；网关下游是编排服务，编排服务连接「租户配置中心」「向量库（按租户隔离）」「模型网关」「关系库（按 tenant_id 过滤）」「对象存储（按租户分桶）」；模型网关连接多个模型供应商；计量数据写入「计量库」，供账单和配额使用。

## 八、安全注意事项

多租户架构的安全，核心是**防止租户间数据泄露**和**防止租户越权**。

**租户 ID 不可伪造**。`tenant_id` 必须从可信来源获取——API Key 绑定租户、JWT 由平台签发且包含租户声明。不能依赖客户端传的 Header，否则恶意用户伪造 `tenant_id` 就能访问其他租户数据。

**所有查询必须带租户过滤**。ORM、查询构建器要支持「租户上下文」——设置当前租户后，所有查询自动加上 `tenant_id` 条件。避免开发手写 SQL 时漏掉。对关键查询做代码审查和自动化检查。

**日志脱敏**。日志里可能包含用户输入、模型输出，这些可能涉及租户敏感信息。生产环境日志要脱敏，或者按租户隔离存储，避免日志泄露导致跨租户信息暴露。

**密钥隔离**。如果租户可以配置自己的模型 API Key（用于私有模型），这些 Key 要加密存储，按租户隔离，使用时不能串到其他租户的请求里。

**审计**。关键操作——数据导出、配置变更、跨租户管理操作——要留审计日志，便于事后追溯和合规检查。

---

## 结尾

多租户 AI 平台，本质是在「共享」和「隔离」之间做精细的切割。共享算力、共享模型、共享运维，把成本摊薄；隔离数据、隔离配置、隔离计量，把安全和合规守住。

从「单租户 Demo」到「多租户 SaaS」，架构要提前想清楚。数据层、配置层、计量层，每一层都要带上租户的烙印。否则等客户多了再 retrofitting，改造成本会高很多。

如果你正在做或打算做 AI 产品的商业化，多租户这一关迟早要过。先把线划清楚，后面的路会好走很多。
