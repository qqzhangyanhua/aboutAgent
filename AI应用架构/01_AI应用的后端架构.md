# AI 应用的后端架构：从单体到微服务

你的 AI Demo 跑起来了。调通了 API，对话能来回，效果还不错。然后呢？

从能跑到能扛住 1000 个用户，中间差的不止几行代码——差的是一个**架构**。就像你刚学会开车，在空地上转几圈没问题，但真要上高速、跑长途、载一车人，你得先搞清楚车是怎么设计的，哪里是瓶颈，哪里能扩容。AI 应用的后端也一样：Demo 阶段怎么简单怎么来，真要上线，就得认真想想架构这件事。

---

## 一、AI 应用后端架构的特殊挑战

传统 Web 应用的后端，请求进来、查库、算一算、返回，几十毫秒到几百毫秒搞定。AI 应用不一样，它有三座大山：**长耗时**、**高成本**、**不确定性**。

先说长耗时。一次模型调用，少则几秒，多则几十秒。用户发一句「帮我写篇 3000 字的文章」，你总不能让人家盯着空白页等半分钟。这跟点外卖不一样——点外卖你下单后可以干别的，但聊天界面卡在那儿，用户会以为坏了。所以 AI 后端天然要考虑**异步**、**流式输出**、**超时处理**，这些在传统 CRUD 里很少见。

再说高成本。每次调用都要烧 token，GPT-4 级别的模型，一千 token 几毛钱，用户聊嗨了，一天几百轮，你的账单能吓死人。传统应用加个用户，边际成本几乎为零；AI 应用加个用户，可能就是在给自己加债。所以你得有**用量统计**、**预算控制**、**模型路由**（简单任务用小模型），否则月底对账时欲哭无泪。

最后说不确定性。模型 API 可能抖、可能挂、可能限流，返回的内容可能超长、可能违规、可能跑偏。传统接口要么成功要么失败，状态清晰；AI 的返回是「大概对了」还是「完全跑题」，你得自己判断。所以需要**重试机制**、**降级方案**、**输出校验**，把不确定性兜住。

这三座大山，决定了 AI 应用的后端不能照搬传统 Web 的架构思路，得专门设计。

---

## 二、阶段一：单体架构

**单体架构**就是所有逻辑塞在一个服务里：接收请求、调模型、存数据、返回结果，一条龙搞定。像一个人开一家小店，进货、收银、打扫全自己来。

这种架构适合 **MVP** 和早期验证。你刚有个想法，想快速跑通，没必要一上来就搞微服务。一个 Python 脚本、一个 FastAPI 服务、直接调 OpenAI，能跑就行。开发快、部署简单、调试方便，出了问题一眼就能找到。

但单体的瓶颈也很明显。模型调用是 CPU/IO 密集的，一个请求卡住，后面的都得等。用户一多，请求排队，响应时间直线上升。而且所有逻辑耦合在一起，改一处可能牵一发动全身，扩展性差。就像小店生意好了，一个人忙不过来，但你没法只多雇一个收银员——因为收银、进货、打扫都绑在一个人身上。

所以单体架构的适用场景很清晰：**用户量小、请求频率低、还在验证阶段**。一旦过了这个阶段，就得考虑拆。

---

## 三、阶段二：分层架构

**分层架构**把逻辑按职责拆开：**API 层**负责接收请求、认证、限流；**业务逻辑层**负责编排、流程控制；**AI 推理层**专门调模型；**数据层**负责存储和检索。各层各干各的，通过接口通信。

这有点像餐厅的后厨分工。前台接待、点单、结账是一拨人，切菜、炒菜、摆盘是另一拨人，采购、仓储又是另一拨。每拨人专注自己的事，效率高，也方便扩展——生意好了，多招几个厨师就行，不用把前台也拉去炒菜。

分层的好处是**职责清晰**。API 层只管「谁在请求、请求合不合法」；业务逻辑层只管「这个请求该怎么处理、该调哪些服务」；推理层只管「把 prompt 扔给模型、把结果拿回来」。出了问题，你能快速定位是认证挂了、逻辑错了，还是模型超时了。

另一个好处是**可替换**。推理层可以换成不同的模型服务，业务逻辑层不用改；数据层可以从 SQLite 换成 PostgreSQL，其他层也不用动。这种解耦让迭代和试错成本大大降低。

分层架构适合**用户量上来、功能开始复杂**的阶段。你已经不是「能跑就行」，而是「要稳、要快、要能改」。

---

## 四、阶段三：微服务架构

**微服务架构**把系统拆成多个独立服务，每个服务可以单独部署、单独扩缩容。AI 应用里，最典型的是把**推理服务**独立出来——专门负责调模型的那些逻辑，单独跑在一组机器上，根据负载动态扩缩。

为什么推理要独立？因为它是**最吃资源**的那一块。模型调用耗时长、占内存、烧钱，如果和业务逻辑混在一起，你扩业务就得连推理一起扩，成本高、效率低。把推理拆出去，业务服务可以轻量级多开几个，推理服务根据队列深度单独扩，各扩各的，互不干扰。

这就像物流公司。仓储、运输、配送、客服，各是各的团队。双十一订单爆了，多调几辆货车、多招几个配送员就行，不用把客服也拉去搬货。每个环节按自己的节奏扩容，整体才能扛住。

微服务还有一层好处：**技术选型自由**。推理服务可以用 Python 写（调模型方便），业务服务可以用 Go 写（高并发省资源），网关可以用 Nginx 或 Kong。不同服务用最合适的语言和框架，不必强求统一。

但微服务也有代价。服务多了，部署、监控、排查问题都变复杂。网络调用代替了函数调用，延迟和失败模式都多了。所以微服务适合**规模已经上来、团队能驾驭**的阶段，不适合一上来就搞。

## 五、关键组件设计

不管选哪种架构，有几个**关键组件**是绕不开的。

**请求队列**。当请求量超过推理能力时，不能直接拒绝，而是先入队，后台慢慢处理。就像银行取号，人多时排队等，总比「满了请明天再来」好。队列还能做优先级：VIP 用户、付费任务可以插队，免费用户排队等。

**异步处理**。长耗时任务不能同步阻塞。用户发起一个「生成 5000 字报告」的请求，API 立即返回一个任务 ID，后台 Worker 慢慢跑，跑完了写结果，用户通过轮询或 WebSocket 拿结果。这样 API 不会被长任务拖死，用户也能知道「在做了」而不是「卡住了」。

**回调机制**。异步任务完成后，怎么通知用户？可以轮询（用户隔几秒问一次「好了没」），可以 WebSocket 推送，也可以回调 URL（任务完成时往用户提供的地址 POST 一下）。选哪种取决于你的场景：实时聊天用 WebSocket，批量任务用回调更省资源。

**缓存层**。同样的 prompt，同样的模型，结果可以缓存。用户问「什么是机器学习」，第一次调模型，第二次直接读缓存，省时省钱。缓存要设计好失效策略：模型升级了、prompt 模板改了，旧缓存得清掉。

这些组件不是「有了就行」，而是要根据业务调。请求量小，队列可以简单点；用户能接受轮询，就不一定上 WebSocket。关键是**知道有这些选项**，需要时能拿出来用。

## 六、架构选择决策树

怎么选架构？可以按这个思路走。

**用户量**：日活几十、几百，单体够用；日活几千、上万，分层起步；日活十万级、推理 QPS 高，考虑微服务。

**团队规模**：一两个人，别搞微服务，维护成本吃不消；三五人以上，分层可以；十人以上、有专人负责基础设施，微服务才玩得转。

**业务阶段**：MVP、验证想法，单体最快；产品定型、开始规模化，分层；已经规模化、要精细控制成本和性能，微服务。

**推理特性**：如果推理是「调一次 API 等结果」的同步模式，分层就能应付；如果是「任务丢队列、多 Worker 并发处理」的异步模式，微服务的优势才明显。

一句话：**别提前过度设计，但也别等到爆了再救火**。根据当前阶段选合适的，留好扩展空间，等瓶颈真的来了再升级。

## 七、一个实际的 AI 应用架构图

下面是一个典型的、已经上规模的 AI 应用架构，你可以对照着理解各层在干什么。

```
                    用户端（Web / App / API）
                              │
                              ▼
                    ┌─────────────────────┐
                    │      网关层          │
                    │  认证 / 限流 / 路由   │
                    └──────────┬──────────┘
                               │
              ┌────────────────┼────────────────┐
              │                │                 │
              ▼                ▼                 ▼
    ┌─────────────┐  ┌─────────────┐  ┌─────────────┐
    │  会话管理    │  │  任务队列    │  │  用量统计    │
    │ Session Mgr │  │ Task Queue  │  │ Usage Track │
    └──────┬──────┘  └──────┬──────┘  └──────┬──────┘
           │                │                 │
           └────────────────┼─────────────────┘
                            │
                            ▼
                    ┌─────────────────────┐
                    │      编排层          │
                    │  业务逻辑 / 流程控制   │
                    └──────────┬──────────┘
                               │
              ┌────────────────┼────────────────┐
              │                │                 │
              ▼                ▼                 ▼
    ┌─────────────┐  ┌─────────────┐  ┌─────────────┐
    │  推理服务    │  │  检索服务    │  │  工具执行    │
    │ LLM Infer   │  │  RAG/向量   │  │ Skill Run   │
    └──────┬──────┘  └──────┬──────┘  └──────┬──────┘
           │                │                 │
           └────────────────┼─────────────────┘
                            │
                            ▼
                    ┌─────────────────────┐
                    │      数据层          │
                    │  DB / Redis / 向量库  │
                    └─────────────────────┘
```

**解读**：

- **网关层**：第一道门。谁在请求、请求频率合不合法、往哪路由，都在这里解决。认证失败、限流触发，直接挡在外面，不浪费后续资源。

- **会话管理 / 任务队列 / 用量统计**：三个横切能力。会话管理负责「这个用户之前的对话在哪」；任务队列负责「请求太多时先排队」；用量统计负责「这个用户今天烧了多少 token」。它们为编排层提供支撑，但不参与具体业务逻辑。

- **编排层**：大脑。收到请求后，决定要调推理、要检索、还是要跑工具，按流程串联起来。编排层是无状态的，可以水平扩展，状态都存在下面的数据层。

- **推理 / 检索 / 工具执行**：三个能力服务。推理专门调模型，可以独立扩缩；检索负责 RAG、向量搜索；工具执行负责跑代码、调 API 等。它们各自独立，编排层按需调用。

- **数据层**：持久化。对话历史、用户数据、向量索引，都存在这里。推理服务不存状态，重启、扩容都不丢数据。

这张图不是让你照抄，而是帮你建立**分层思维**。你的业务可能不需要检索、不需要工具，但「网关 - 编排 - 能力服务 - 数据」这个骨架，大部分 AI 应用都适用。

---

## 结尾

AI 应用的后端架构，本质是在**简单**和**可扩展**之间找平衡。MVP 阶段单体够用，别一上来就微服务；规模上来了，分层、拆服务、加队列加缓存，一步步来。关键是知道每个阶段该有什么、不该有什么，别提前过度设计，也别等到爆了才后悔。

从能跑到能扛住 1000 个用户，差的不是魔法，而是一张清晰的架构图，和按阶段迭代的耐心。如果你正在做 AI 应用，不妨对照一下自己现在在哪个阶段，下一步该往哪走。有问题可以留言聊聊，一起把架构这件事捋清楚。
