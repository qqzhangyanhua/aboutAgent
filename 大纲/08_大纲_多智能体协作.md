## 章节大纲

### 开头引入

- 切入点：前面我们搭的智能体都是"单打独斗"——一个 AI 又查资料、又推理、又写代码。就像一个人同时当项目经理、开发、测试，能干但效率不高
- 类比：现实中复杂项目都是团队协作的。AI 也一样，让不同的 AI 扮演不同角色，术业有专攻
- 核心问题：怎么让多个智能体互相配合，比一个智能体做得更好？

### 一、为什么需要多智能体？

**要点：**
- 单智能体的天花板：上下文窗口有限、角色切换容易混乱、复杂任务容易跑偏
- 多智能体的好处：
  - 专业化：每个 Agent 专注一件事，prompt 更聚焦
  - 互相校验：一个写代码一个 Review，质量更高
  - 并行处理：不相关的子任务可以同时进行
- 三种经典模式预告：主从模式、辩论模式、流水线模式

### 二、模式一：主从模式（Orchestrator-Worker）

**要点：**
- 一个"老板"Agent 拆解任务 + 分配子任务，多个"员工"Agent 各自执行
- 适合场景：复杂任务分解（比如"帮我写一篇调研报告"→ 拆成搜索、总结、排版）
- 老板负责统筹全局，员工不需要互相通信

**核心代码片段：**

```python
class OrchestratorAgent:
    def __init__(self):
        self.workers = {
            "researcher": Agent(system_prompt="你是一个信息调研专家，负责搜索和收集资料。"),
            "writer": Agent(system_prompt="你是一个技术写手，负责把资料整理成文章。"),
            "reviewer": Agent(system_prompt="你是一个严格的审稿人，负责检查文章质量。"),
        }

    def run(self, task):
        # 第一步：老板拆解任务
        plan = self.plan(task)
        # plan = [{"worker": "researcher", "subtask": "搜索 RAG 最新论文"},
        #         {"worker": "writer", "subtask": "根据资料写文章"},
        #         {"worker": "reviewer", "subtask": "审阅并给出修改意见"}]

        results = {}
        for step in plan:
            worker = self.workers[step["worker"]]
            context = self._build_context(step, results)
            results[step["worker"]] = worker.run(context)

        return results["reviewer"]  # 最终输出经过审阅的结果

    def plan(self, task):
        response = ask_llm(f"""你是一个项目经理。请把以下任务拆解成子任务，指定执行者。
可用的执行者：researcher（调研）、writer（写作）、reviewer（审阅）
任务：{task}
输出 JSON 格式的执行计划。""")
        return json.loads(response)
```

### 三、模式二：辩论模式（Debate）

**要点：**
- 两个或多个 Agent 对同一问题给出不同角度的答案，互相质疑，最后由裁判 Agent 综合
- 适合场景：需要多角度分析的决策（比如"这个技术方案行不行"）
- 核心机制：多轮辩论 + 最终裁决

**核心代码片段：**

```python
class DebateSystem:
    def __init__(self):
        self.agents = [
            Agent(system_prompt="你是乐观派技术顾问，倾向于看到方案的优势和可能性。"),
            Agent(system_prompt="你是悲观派技术顾问，专注于发现风险和潜在问题。"),
        ]
        self.judge = Agent(system_prompt="""你是技术评审裁判。
阅读双方的辩论内容，给出客观、平衡的最终结论。""")

    def debate(self, question, rounds=3):
        history = []

        for round_num in range(rounds):
            for i, agent in enumerate(self.agents):
                context = f"议题：{question}\n\n之前的讨论：\n"
                context += "\n".join(history[-4:])  # 最近的发言
                context += f"\n\n请发表你的第 {round_num+1} 轮观点："

                response = agent.run(context)
                label = "正方" if i == 0 else "反方"
                history.append(f"【{label}第{round_num+1}轮】{response}")

        # 裁判做最终裁决
        verdict = self.judge.run(
            f"议题：{question}\n\n完整辩论记录：\n" + "\n\n".join(history)
            + "\n\n请给出你的最终评判和建议：")
        return verdict
```

### 四、模式三：流水线模式（Pipeline）

**要点：**
- 多个 Agent 串行处理，每个 Agent 的输出是下一个的输入
- 类比工厂流水线：原材料 → 加工 → 质检 → 包装
- 适合场景：有明确先后顺序的多步骤任务（比如"翻译 → 润色 → 校对"）

**核心代码片段：**

```python
class Pipeline:
    def __init__(self):
        self.stages = [
            {"name": "drafter",
             "agent": Agent(system_prompt="你是初稿写手。根据主题写一篇技术博客初稿，重点是内容完整性。")},
            {"name": "polisher",
             "agent": Agent(system_prompt="你是文字润色专家。在不改变内容的前提下，让文章更生动、口语化，去掉 AI 味。")},
            {"name": "fact_checker",
             "agent": Agent(system_prompt="你是事实核查员。检查文章中的技术描述是否准确，标出可疑的地方。")},
            {"name": "formatter",
             "agent": Agent(system_prompt="你是排版专家。把文章格式化为微信公众号适合的 Markdown 格式，加上合适的小标题和分段。")},
        ]

    def run(self, initial_input):
        current_output = initial_input
        for stage in self.stages:
            print(f"📎 阶段：{stage['name']}")
            current_output = stage["agent"].run(
                f"请处理以下内容：\n\n{current_output}")
            print(f"   ✅ 完成，输出 {len(current_output)} 字")
        return current_output
```

### 五、三种模式对比

```
| 维度         | 主从模式              | 辩论模式              | 流水线模式            |
|-------------|----------------------|----------------------|----------------------|
| 通信方式     | 星型（老板→员工）      | 全连接（互相对话）      | 线性（前→后）         |
| 适合任务     | 可拆解的复杂任务       | 需要多角度的决策/分析   | 有明确步骤的顺序任务   |
| Agent 数量   | 1 + N               | 2-3 + 裁判            | N 个串行             |
| 并行能力     | 高（员工可并行）       | 低（需要轮流发言）      | 无（严格顺序）        |
| 成本         | 中等                 | 高（多轮对话）          | 中等                 |
| 典型应用     | 调研报告、项目执行     | 技术方案评审、风险评估   | 内容生产、数据处理     |
```

### 六、实战踩坑与优化建议

**要点：**
- Agent 之间传递信息时要控制长度，不然下游 Agent 上下文爆炸
- 每个 Agent 的 system prompt 越聚焦越好，不要让它"啥都干"
- 调试技巧：给每个 Agent 的输出加前缀标记，方便看是谁说的
- 成本控制：不是所有任务都需要多 Agent，简单任务单 Agent 就够了，别为了花哨而花哨
