## 章节大纲

### 开头引入

- 切入点：前面所有文章我们一直用 ChromaDB，因为它装起来最简单。但你的项目要上生产了，100 万条向量、需要高并发、还得支持持久化——ChromaDB 还够用吗？
- 核心问题：向量数据库这么多，到底怎么选？别跟我说"看需求"，给我实测数据
- 本文思路：同一个测试集，四个数据库，从安装到查询全跑一遍

### 一、先搞清楚向量数据库在干什么

**要点：**
- 向量数据库的核心功能：存向量 + 按相似度搜索
- 和传统数据库的区别：MySQL 找"等于"，向量库找"像"
- 关键指标：
  - **查询延迟**：一次检索多快
  - **召回率**：能不能找到最相关的
  - **可扩展性**：数据量从 1 万到 1 亿怎么办
  - **部署复杂度**：能不能 pip install 就跑，还是得搭集群

### 二、四位选手介绍

**ChromaDB：**
- 定位：开发者友好的嵌入式向量库
- 语言：Python
- 特点：pip install 即用、API 简单、适合原型和小项目
- 局限：单机、大数据量下性能衰减

**FAISS（Facebook AI Similarity Search）：**
- 定位：Meta 开源的高性能向量搜索库
- 语言：C++ 底层，Python 接口
- 特点：速度极快、支持 GPU 加速、多种索引类型
- 局限：只是一个库不是数据库（无持久化、无 CRUD、无 API 服务）

**Qdrant：**
- 定位：生产级向量数据库
- 语言：Rust
- 特点：性能好、有完整 REST API、支持过滤、支持云服务
- 局限：生态相对较新

**Milvus：**
- 定位：分布式向量数据库（企业级）
- 语言：Go + C++
- 特点：支持十亿级向量、分布式架构、Zilliz 云服务
- 局限：部署重（需要 etcd + MinIO + 多组件）

### 四、性能实测

**要点：**
- 测试环境说明：MacBook M2 / 16GB，单机环境
- 测试数据：10K / 100K / 1M 条 384 维向量
- 测试维度：
  - 插入速度（条/秒）
  - 查询延迟（毫秒/次）
  - 内存占用
  - 磁盘占用

**核心代码片段（性能测试框架）：**

```python
import time
import numpy as np

def benchmark_insert(db_client, vectors, batch_size=1000):
    start = time.time()
    for i in range(0, len(vectors), batch_size):
        batch = vectors[i:i+batch_size]
        db_client.insert(batch)  # 各库的 insert 方法
    elapsed = time.time() - start
    return len(vectors) / elapsed  # 条/秒

def benchmark_query(db_client, queries, k=10):
    latencies = []
    for q in queries:
        start = time.time()
        db_client.search(q, k=k)
        latencies.append((time.time() - start) * 1000)
    return {
        "avg_ms": np.mean(latencies),
        "p95_ms": np.percentile(latencies, 95),
        "p99_ms": np.percentile(latencies, 99)
    }
```

### 五、特色功能对比

**要点：**
- 元数据过滤：哪些库支持 where 条件（ChromaDB ✅, Qdrant ✅, Milvus ✅, FAISS ❌）
- 持久化：ChromaDB 文件持久化、FAISS 需手动 save/load、Qdrant 原生持久化、Milvus 需外部存储
- 多租户：Qdrant 有 namespace、Milvus 有 partition
- 全文搜索：Qdrant 开始支持、Milvus 支持混合搜索
- 云服务：ChromaDB 刚推出、Qdrant Cloud 成熟、Zilliz Cloud（Milvus 托管）
- FAISS 特色：GPU 加速、多种索引算法（IVF、HNSW、PQ 量化）

### 六、选型决策树

```
你的数据量？
├── < 10 万条
│   ├── 需要快速原型 → ChromaDB
│   └── 需要极致速度 → FAISS
├── 10 万 - 1000 万条
│   ├── 单机部署 → Qdrant
│   └── 需要云服务 → Qdrant Cloud
└── > 1000 万条
    ├── 有运维能力 → Milvus 自建集群
    └── 不想运维 → Zilliz Cloud
```

### 七、迁移建议

**要点：**
- 从 ChromaDB 迁移到 Qdrant 的代码改动量
- 抽象层的思路：写一个统一的 VectorStore 接口，底层可替换
- 实际建议：先用 ChromaDB 开发，验证可行后再切换

**核心代码片段：**

```python
from abc import ABC, abstractmethod

class VectorStore(ABC):
    @abstractmethod
    def add(self, ids, vectors, documents, metadatas=None): ...
    @abstractmethod
    def search(self, query_vector, top_k=5): ...
    @abstractmethod
    def delete(self, ids): ...

class ChromaStore(VectorStore):
    def search(self, query_vector, top_k=5):
        return self.collection.query(query_embeddings=[query_vector], n_results=top_k)

class QdrantStore(VectorStore):
    def search(self, query_vector, top_k=5):
        return self.client.search(self.collection_name, query_vector=query_vector, limit=top_k)
```

### 写在最后

- 衔接语：向量数据库是 RAG 的存储引擎，选对了事半功倍。但光检索还不够，智能体需要更强的规划能力——下一篇我们来聊 Planning 机制，让 AI 先想清楚再动手
- 关键观点：别在选型上纠结太久。绝大多数项目的数据量，ChromaDB 就够了。等你真的遇到性能瓶颈再迁移，比提前"架构优化"划算得多
