# AI 做设计：Figma AI + Galileo + v0.dev 实测

老李是个后端，写接口一把好手，但让他做个 landing page，他对着空白画布发呆半小时，最后憋出一版：标题居中、按钮巨大、配色像十年前政府网站。产品经理看了一眼，委婉地说「要不咱找个设计师？」老李心想：设计师排期排到下周，老板明天就要看 demo。他硬着头皮继续改，改到凌晨两点，越改越丑，最后索性摆烂——反正功能能跑就行。**程序员做设计，就像让厨子去修水管**：不是干不了，是专业不对口，干得憋屈。

这种故事不少见。你身边肯定有程序员做的内部系统——功能齐全、逻辑清晰，但界面一眼就能看出「这不是设计师干的」。按钮大小不一、颜色撞得眼花、留白要么挤成一片要么空得慌。用户嘴上不说，心里已经在打分了。

好消息是，AI 开始掺和设计这摊事了。从「文字描述生成 UI」到「设计稿直接变代码」，工具越来越多，门槛越来越低。今天实测三款：**Figma AI**（设计领域老大哥加 AI）、**Galileo AI**（从文字生成 UI）、**v0.dev**（Vercel 出品，从描述到前端代码）。看看它们能不能把老李从「配色灾难」里捞出来。实测基于 2025 年初的版本，后续功能可能有更新。

---

## 一、设计的痛点：不是每个团队都有设计师

先说清楚为什么程序员做设计这么难。

设计这事儿，跟写代码是两套脑子。写代码讲究逻辑、结构、可维护性；做设计讲究视觉、节奏、留白、层次。程序员脑子里装的是「这个 div 要不要加个 flex」，设计师脑子里装的是「用户视线会先落在哪儿」。你让一个习惯用 `margin: 0 auto` 的人去考虑「视觉焦点」和「呼吸感」，相当于让一个只会做番茄炒蛋的人去摆盘——菜能熟，但摆出来像食堂大锅饭。

更现实的问题是：**小团队养不起设计师**。一个全职 UI/UX 设计师，月薪动辄两三万，很多创业公司、小项目组根本请不起。活儿就落在产品经理或程序员头上。产品经理好歹懂点用户心理，程序员呢？能跑就行，好看是 bonus。结果就是：功能齐全、界面感人，用户打开第一眼就想关掉。

还有个尴尬：就算请了设计师，排期也排不过来。一个需求从「要个界面」到「设计稿出来」可能得等一周，开发等着设计稿才能开工，设计等着需求文档才能动笔，一环扣一环，谁都急。**AI 做设计**，某种程度上是在填这个「等待」的坑——能不能先出个草稿、先跑个 demo，别让进度卡在设计这一环。

所以「AI 做设计」这事儿，本质是**给没有设计能力的人借一双设计师的手**。能不能借到、借得好不好，得实测了才知道。

## 二、三款 AI 设计工具简介

先把三个家伙拎出来认个脸。三个工具各有侧重，别搞混了。

**Figma AI**：Figma 是设计圈的老大哥，Sketch、Adobe XD 的对手，很多团队的设计稿都在 Figma 里。Figma AI 是它内置的 AI 能力，包括 AI 填充、布局建议、自动生成原型等。定位是**辅助设计师**，不是替代设计师——你已经有设计稿了，AI 帮你加速、帮你补细节。

**Galileo AI**：一个独立的 AI 设计工具，主打「从文字到 UI」。你输入一段描述，比如「做一个 SaaS 产品的登录页，简洁现代，蓝色主色」，它直接给你生成一整套 UI 设计稿。定位是**从零到一**——你没有设计稿，只有想法，Galileo 帮你把想法变成画面。相当于「你口述需求，它给你画出来」，省掉从空白画布开始的恐惧。

**v0.dev**：Vercel 出的，跟 Galileo 有点像，也是「描述 → 产出」，但产出物不一样。Galileo 出的是设计稿（图），v0 出的是**代码**——可运行的 React 组件，带 Tailwind CSS，复制粘贴就能用。定位是**设计到代码一步到位**——你描述需求，它直接给你能跑的前端。

三个工具，三条路线：Figma AI 是「设计稿加速器」，Galileo 是「想法变设计稿」，v0 是「想法变代码」。下面逐个实测。

顺便提一嘴：这三款都不是「一键出成品」的魔法——AI 能大幅降低门槛，但不能完全替代人的判断。生成的结果你得看、得改、得取舍。抱着「丢进去一句话就完事」的预期，多半会失望；抱着「有个靠谱的初稿，我在此基础上打磨」的心态，体验会好很多。

## 三、Figma AI 实测：老大哥的 AI 能干啥

Figma AI 不是单独的产品，而是嵌在 Figma 里的能力。你得先有 Figma 账号，打开设计文件，才能用到。没玩过 Figma 的，先去官网注册个免费账号，新建一个文件，在右侧面板或右键菜单里能找到 AI 相关的入口。

**AI 填充**：选中一个矩形或文本框，让 AI 帮你填内容。比如你画了个「用户头像」的占位框，点 AI 填充，它会生成一张随机头像图；你画了个「产品描述」的文本框，它会生成一段假文案。实测下来，对占位内容、假数据很实用——以前得去 unsplash 找图、去 lorem ipsum 复制文字，现在点一下就完事。但生成的内容风格比较随机，想要特定风格得多试几次。

**布局建议**：你选中几个元素，让 AI 给布局建议。比如几个卡片横着排还是竖着排、间距多大、要不要加网格。AI 会给出几个方案，你选一个应用。对「不知道怎么摆」的人有帮助，但建议偏常规——不会给你特别大胆的布局，更多是「安全牌」。想玩花的，得自己来。

**自动原型**：根据画好的页面，AI 可以帮你加交互、连成原型。比如「点击登录按钮跳转到首页」这种简单流程，它能自动连。复杂一点的用户路径，还是得自己拖。

**智能重命名**：批量给图层、组件起名，AI 能根据内容推断——比如一个放用户头像的框，它会建议命名为 `avatar` 或 `user-avatar`。对规范命名、方便后续开发挺有用。

**总体感受**：Figma AI 像**设计稿里的 Copilot**——你干活，它打下手。适合已经会用 Figma、有基本设计能力的人，用来提速。如果你连 Figma 都没摸过、画个矩形都费劲，Figma AI 帮不了太多——它不会从零教你做设计。

还有个细节：Figma AI 对中文支持一般，你让它生成假文案，出来的多半是英文。要是做面向国内用户的产品，得自己替换。不过占位图、布局建议这些不涉及语言，影响不大。

## 四、Galileo AI 实测：从描述词到完整 UI

Galileo 的玩法很简单：打开网站，输入一段描述，等几秒，出一套 UI。不需要会设计软件，会打字就行，门槛极低。

实测输入：「一个 B 端 SaaS 的仪表盘首页，包含侧边栏导航、顶部用户信息、主区域有数据卡片和图表，深色主题，科技感。」

生成结果：一套完整的界面布局——侧边栏、顶栏、卡片、图表的占位，配色偏深蓝灰，整体风格统一。细节上，图标、文字、间距都像模像样，不像随手画的。生成速度大概十秒左右，比从零画快一个数量级。

但有几个问题：一是**内容不可控**——卡片里显示什么数据、图表是什么类型，描述里没写清楚的话，AI 会自己发挥；二是**修改成本**——想改某个卡片的样式，得在生成结果上手动调，没有「改描述重新生成」那么方便；三是**输出格式**——生成的是设计稿/图片，不是代码，要变成网页还得再走一步。

**适用场景**：快速出个 UI 方案、给老板或客户看个大概、头脑风暴时把想法可视化。适合「我有个想法，想先看看长什么样」的阶段。如果你要的是最终可用的设计稿，可能还得在 Galileo 的基础上再精修。

多说一嘴：Galileo 的描述词越具体，生成结果越可控。你只说「做个登录页」，它可能给你十种风格随机挑一个；你说「极简风格、白色背景、蓝色主按钮、左边大图右边表单」，它就能往这个方向靠。**描述是杠杆**，多写几个关键词，能省不少试错。

## 五、v0.dev 实测：从一句话到可运行的 React 组件

v0 的卖点是**直接出代码**。你描述需求，它给你 React + Tailwind 的组件，复制到项目里就能跑。官网 v0.dev，用 Vercel 账号登录即可，免费版每天有额度限制，但对个人够用。

实测输入：「一个简洁的登录页，左边是品牌 logo 和 slogan，右边是邮箱密码输入框和登录按钮，整体白色背景，蓝色主色。」

生成结果：一个完整的 React 组件，结构清晰，Tailwind 类名规整，布局是左右分栏，左边放 logo 和文案，右边是表单。代码可以直接复制，在 Next.js 或普通 React 项目里用。迭代也方便——你可以在对话框里说「把按钮改成圆角」「加一个忘记密码的链接」，它会基于当前版本修改，再输出新代码。

**优势**：设计到代码一步到位，省掉「设计稿 → 切图 → 写代码」的中间环节。对程序员特别友好——你本来就要写代码，v0 帮你把「画界面」这步跳过去了。而且输出的是标准 React，不是黑盒，你可以随便改。

**局限**：复杂交互、动画、响应式细节，v0 不一定能一次搞定，可能得你手动补。另外它偏向「组件级」产出，一屏一屏生成，整站级的页面结构还得你自己搭。大项目还是得靠人搭架子，v0 负责填组件。

实测还有个发现：v0 对 shadcn/ui 这类组件库很熟，你描述「用 shadcn 的 Button 和 Input」，它能准确引用。如果你项目里已经用了 shadcn，生成的代码直接能融入，不用大改。这点对追求统一设计系统的团队挺友好。

还有个技巧：描述时带上「响应式」「移动端适配」，v0 会在代码里加 breakpoint、做 flex 或 grid 的响应式布局。不说的话，它可能默认只出桌面版，手机上看就崩了。

## 六、三者对比：一张表说清楚

| 维度 | Figma AI | Galileo AI | v0.dev |
|------|----------|------------|--------|
| **定位** | 设计稿内的 AI 助手 | 从文字生成 UI 设计稿 | 从描述生成前端代码 |
| **使用门槛** | 需要会 Figma | 会打字就行 | 会描述需求、懂一点前端更好 |
| **输出物** | 设计稿内的元素/布局 | 完整 UI 设计稿（图） | 可运行的 React 组件 |
| **价格** | Figma 订阅内（有免费版） | 有免费额度，高级功能付费 | 免费版 + 付费 Pro |
| **适用场景** | 已有设计稿，加速制作 | 从零出方案、快速可视化想法 | 要代码、要能跑、程序员主导 |

简单说：**有设计师、用 Figma** → 上 Figma AI；**没设计师、先出图** → 用 Galileo；**没设计师、直接要代码** → 用 v0。

价格方面，三款都有免费档，够个人和小团队试水。真要高频用、要更多额度，再考虑付费。建议先白嫖一段时间，确认工作流跑得通再掏钱。

还有个选择逻辑：你要是**视觉优先**——先得让老板、客户点头，再考虑实现——Galileo 或 Figma 更适合，出的图好看、好展示。你要是**效率优先**——赶紧跑起来、能 demo 就行——v0 更直接，省掉「设计稿转代码」这道工序。

最后提醒：工具在迭代，功能会变。写这篇文章时的情况，过半年可能就不一样了。建议自己去官网试一轮，以实际体验为准。

## 七、从设计到代码的工作流：怎么串起来

这三个工具可以组合用，形成一条「想法 → 设计 → 代码」的流水线。

**方案一：Galileo + 手写代码**。用 Galileo 快速出个 UI 方案，看整体布局和风格合不合意。满意了，照着设计稿自己写代码（或让 AI 根据设计稿描述生成代码）。适合对视觉有要求、但团队没有设计师的情况。

**方案二：v0 直接出代码**。需求想清楚了，直接上 v0，描述 → 代码 → 复制到项目 → 微调。适合程序员主导、对「设计稿」这个中间产物不感冒的情况。你要的是能跑的页面，v0 一步到位。

**方案三：Figma + v0**。设计师在 Figma 里出稿，用 Figma AI 加速。定稿后，把设计稿的描述（或截图）给 v0，让 v0 生成对应代码。这样设计稿和代码能对上，设计师和程序员都省事。

**方案四：Galileo + Figma + v0**。用 Galileo 快速出初版方案，导入 Figma 精修（或直接在 Galileo 里调），定稿后用 v0 把关键页面生成代码。适合从零启动、想快速迭代的项目。

**方案五：纯 v0 迭代**。适合独立开发者或小团队：需求想清楚，直接 v0 出第一版，跑起来看效果，不满意就对话里改——「把卡片间距调大」「加个加载状态」——几轮下来，页面就成型了。全程不碰 Figma，不碰设计稿，代码即设计。

选哪种，看你手头有什么、要什么。有 Figma 文件就优先用 Figma AI；从零开始就 Galileo 或 v0 二选一；要代码就直接 v0。

核心思路：**别指望一个工具包打天下**。每个工具擅长的不一样，按你的阶段和需求挑着用，能串就串，串不起来就单点突破。

再提醒一句：不管用哪个工具，**人的审美判断不能省**。AI 生成的东西，风格容易趋同——你看多了会发现，不同人用同一工具，出来的界面都有点「那个味儿」。真要做出辨识度，还是得人在布局、配色、细节上把关。AI 是加速器，不是自动驾驶，方向盘还得你握着。

## 八、结尾：老李的 landing page 有救了

回到开头的老李。他要是早点知道这些工具，就不用对着空白画布发呆了。用 Galileo 先出个大概，或者直接上 v0 描述需求拿代码，十分钟能跑起来一个能看的页面。再在 Figma 里微调下配色、间距，或者让 v0 迭代几轮，丑是不至于了，至少能拿得出手。

老板明天要 demo？今晚用 v0 出个登录页、用 Galileo 出个仪表盘草图，拼一拼，能交差。设计师排期紧？先用 AI 顶一版，等设计师有空再精修。**救急不救穷**，但救急也够用了。

AI 做设计，不是要取代设计师，而是**给没有设计能力的人一条活路**。程序员、产品经理、小团队，都能借上力。工具会越来越成熟，门槛会越来越低，以后「不会设计」可能不再是不能做界面的借口了。

当然，真要做出有辨识度的产品，还是得靠人——AI 能帮你快速出方案，但审美、品牌感、细节打磨，这些得靠积累。工具是拐杖，不是腿。先用起来，别让「不会设计」卡住进度；用着用着，慢慢也能培养出一点感觉。
