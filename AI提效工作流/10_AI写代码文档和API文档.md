# AI 写代码文档 + API 文档自动生成

小张入职第一天，领到一台电脑、一个 Git 仓库地址、一句「先熟悉下代码」。他 clone 下来，打开 README，上面写着「本项目用于业务系统开发」，没了。点进 src 目录，几百个文件，函数名有的叫 `handleData`，有的叫 `processStuff`，还有 `doSomething`——鬼知道它们到底干什么的。问老同事，老同事说「你看下 API 文档」，小张找到文档链接，打开一看，最后更新时间是两年前。接口早就改了三轮，文档还停留在上古版本。他叹了口气：**这代码，跟没文档差不多**。

程序员最讨厌写什么？文档。但文档偏偏是最重要的——新人入职、接口交接、半年后自己回来看代码，全靠它。现在有个好消息：**用 AI 写文档，这套脏活累活能省掉一大半**。从函数注释到 README，从 API 文档到自动保鲜，今天聊聊怎么玩。

---

## 一、程序员为什么讨厌写文档

先说说为什么文档这么重要，却又这么难产。

写文档像**给房子画户型图**：房子是你盖的，你心里门儿清，但画成图给别人看，就得把每个房间、每扇门、每条动线都标清楚。麻烦不说，房子还在不断装修——今天加个隔断，明天拆堵墙，户型图就得跟着改。改着改着就忘了，最后户型图和实际房子对不上，新人拿着图找厕所，结果走进了杂物间。

**写的时候烦**。代码写完了，脑子已经转不动了，还要切到「解释模式」：这个函数干嘛的、参数什么意思、返回值长什么样、有没有坑……写一遍等于把逻辑再过一遍，累。而且很多「显而易见」的东西，写出来觉得废话，不写又怕别人看不懂，纠结。

**改的时候更烦**。代码迭代快，文档迭代慢。改了一行逻辑，文档要不要改？改。改完发现还有三处引用同一段逻辑，一起改。改着改着发现文档和代码又对不上了，索性摆烂：反正也没人看。

**结果**：文档要么没有，要么过时，要么写得像天书。新人入职靠口口相传，接口交接靠「你调一下试试」，半年后自己回来看代码，像在看别人写的。这种恶性循环，很多团队都经历过。

## 二、AI 写文档的三个层次

AI 写文档不是一刀切，而是分层次的——就像做饭，有凉拌、小炒、炖汤，难度和火候都不一样。

**第一层：函数注释 / Docstring**。给单个函数、类、方法写说明，解释参数、返回值、可能的异常。粒度最小，AI 最擅长——它能看到代码，能理解逻辑，生成一段人话描述不难。Cursor、Copilot 都能在写代码时自动补注释，你写个函数名，它帮你把 Docstring 填好。

**第二层：README / 架构文档**。给整个项目写概述：这是什么、怎么跑、目录结构、核心模块、怎么贡献。需要「全局视角」，得把整个项目喂给 AI，让它从上帝视角写一份总览。适合项目初始化、交接文档、开源项目介绍。

**第三层：API 文档**。给接口写说明：路径、方法、参数、请求示例、响应格式、错误码。传统做法是手写 Swagger/OpenAPI，或者用框架的注解——但很多老项目没这套，路由散落在代码里，文档靠人工维护，迟早过时。AI 可以分析路由代码，自动生成 OpenAPI Spec，再导出成 Swagger UI 或静态文档。

三层递进：函数级 → 项目级 → 接口级。越往上，需要的上下文越多，AI 的发挥空间越大，但你也得越会「喂料」。

## 三、用 AI 生成 Docstring 实测

先说最细的一层：函数注释。

**Cursor**：选中一个函数，按 Cmd+K 或右键「Edit with AI」，输入「给这个函数写 Docstring」，它会根据函数名、参数、返回值生成一段说明。实测下来，对简单函数效果不错——比如一个 `formatDate(timestamp)`，它能准确写出「将时间戳格式化为可读日期字符串」。但对复杂逻辑、业务含义强的函数，容易写成「正确的废话」：参数列一遍，逻辑复述一遍，没有解释「为什么这么设计」「有什么坑」。

**Copilot**：在函数定义下方直接回车，它会自动补全 Docstring。风格偏标准——Google 风格、NumPy 风格都能模仿，参数和返回值格式规整。但同样的问题：对业务逻辑的理解有限，生成的注释偏「语法层面」的描述，缺乏「语义层面」的洞察。

**局限**：AI 不知道你这个函数在业务里扮演什么角色、和哪些模块耦合、历史上出过什么 bug。这些「上下文」得你自己补。所以 AI 生成的 Docstring 适合当**初稿**——先让它填满，你再删掉啰嗦的、补上关键的，比从零写快很多。

## 四、用 AI 生成 README 和项目文档

项目级的文档，需要「整盘棋」的视角。

**做法**：把项目结构、关键文件、甚至部分核心代码一起发给 AI，让它写一份 README 或架构文档。比如在 Cursor 里，用 `@codebase` 或选中整个项目目录，然后问：「根据这个项目的结构和代码，写一份 README，包含项目简介、技术栈、目录结构、如何运行、如何贡献。」

**效果**：AI 能根据目录名、文件名、import 关系推断出项目的大致结构。比如看到 `src/api/`、`src/services/`、`src/models/`，它会写出「本项目采用分层架构，api 层负责路由，services 层负责业务逻辑……」这种概述。对技术栈的识别也不错——看到 `package.json` 里的依赖、`requirements.txt` 里的库，能准确列出。

**坑**：AI 容易「脑补」。如果项目结构混乱、命名随意，它可能推断错误。比如一个叫 `utils` 的文件夹其实塞满了业务逻辑，AI 会按字面意思写成「工具函数」。所以生成后要**人工校对**，尤其是架构图、模块职责这类关键信息，别让 AI 自由发挥。

**适用场景**：新项目初始化、老项目补文档、开源项目写介绍。你相当于有了一个「文档初稿生成器」，省掉从零搭框架的功夫。

## 五、API 文档自动生成：从代码到 OpenAPI Spec

API 文档是最容易过时的——接口改一次，文档就得跟一次，人工维护成本高得离谱。

**思路**：让 AI 分析路由代码，自动生成 OpenAPI Spec（也就是 Swagger 的规范格式），再导出成可读的 API 文档。

**怎么做**：把路由定义、控制器、请求/响应模型一起给 AI。比如一个 FastAPI 或 Express 项目，路由和参数定义都在代码里，AI 可以解析出：`GET /api/users/:id` 返回用户信息，需要 `id` 参数，响应是 `{ id, name, email }`。你让 AI「根据这些路由代码生成 OpenAPI 3.0 的 YAML」，它就能输出一份规范的 Spec 文件。再用 Swagger UI、Redoc 或静态站点生成器渲染，一份可交互的 API 文档就出来了。

**对老项目**：如果代码里没有 OpenAPI 注解，就得靠 AI「读代码推断」。把路由文件、handler 函数、请求体解析逻辑发给 AI，让它逆向工程出一份 Spec。准确度取决于代码的规整程度——如果路由清晰、参数类型明确，AI 能推个八九不离十；如果到处是 `any`、动态参数、隐式依赖，就容易漏或错。

**进阶**：把这一步接到 CI 里。每次代码提交，自动跑一遍「从代码生成 OpenAPI」，和上次的版本 diff，如果有变化就提醒：接口改了，文档记得更新。这样文档和代码至少能「同源」，不会各走各的。

## 六、文档保鲜策略：怎么让文档跟着代码一起更新

文档最大的敌人不是「写不出来」，是「写出来就死了」——代码在演进，文档在原地踏步。

**CI 集成**：在 CI 流水线里加一步「文档校验」。比如用 `swagger-cli validate` 检查 OpenAPI 文件是否合法，用自定义脚本检查 README 里的「如何运行」是否和实际命令一致。文档和代码一起提交，CI 一起跑，至少能保证「能通过校验的文档」和「能跑通的代码」是配套的。

**Pre-commit Hook**：在 git commit 前触发检查。比如「如果修改了 `src/api/` 下的文件，必须同时修改 `openapi.yaml`」。强制养成「改代码必改文档」的习惯。一开始会烦，习惯之后能避免很多「文档滞后」的坑。

**单源真相**：尽量让文档从代码「生成」而不是「手写」。Docstring 在代码里，README 的「如何运行」从 `package.json` 的 scripts 里读，API 文档从路由代码生成。这样代码是唯一的真相源，文档只是它的衍生品，改代码 = 改文档，不会脱节。

**现实**：很多团队做不到这么严格，至少可以定个规矩：**大版本发布前，文档必须过一遍**。不追求实时同步，但别让文档落后超过一个版本。

## 七、AI 写文档的常见问题

AI 写文档省时间，但有几个坑得心里有数。

**过于啰嗦**。AI 喜欢把一句话拆成三句，把「返回用户列表」写成「本接口用于获取系统中所有用户的列表信息，以数组形式返回」。读起来像产品经理写的 PRD，不像程序员写的文档。解决：生成后自己删，或者提示词里加「简洁、直接、不要废话」。

**术语堆砌**。AI 可能用一堆「高内聚低耦合」「面向接口编程」之类的词，看起来专业，实际没信息量。好的文档应该让人「看完能干活」，不是「看完觉得作者很厉害」。解决：提示词里强调「用大白话，让新人能看懂」。

**和代码不一致**。AI 生成文档时，如果代码已经改了但 AI 看到的是旧版本，或者 AI 推断错了，文档就会和实际代码对不上。这是最危险的——错误文档比没文档更害人。解决：生成后**必须人工核对**，尤其是参数、返回值、错误码这类硬信息，一条条对。

**没有立场**。AI 写的文档往往四平八稳，该提醒的坑、该强调的注意事项，它可能漏掉。比如「这个接口在高并发下可能超时，调用方需要重试」——这种「踩坑经验」AI 写不出来，得你自己补。

总结：**AI 适合当写文档的助手，不适合当唯一作者**。用它提速，用你把关。

## 八、结尾

回到开头的小张。如果他的团队用上 AI 写文档：函数有 Docstring，项目有 README，API 有从代码自动生成的 Swagger，文档更新接在 CI 里——他入职第一天至少能「按图索骥」，不用到处问人。文档不会完美，但不会是一片荒漠。

写文档这事儿，以前是「不想写、写了也不想改」；现在有了 AI，至少「写」和「改」的成本都下来了。剩下的，就是养成习惯：代码和文档一起维护，别让文档变成摆设。
